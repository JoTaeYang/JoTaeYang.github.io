Lesson 10 · Phase 3 — 서빙 스택

vLLM 엔진 파라미터

지금까지 배운 개념 하나하나가 vllm serve의 플래그 하나하나다. 새 지식이 아니라 번역이다.

이번 레슨의 단 하나의 승리 실무에서 마주칠 실제 vllm serve 명령 한 줄을 읽고, 각 플래그가 메모리와 처리량에 무슨 짓을 하는지 — 그리고 어떤 레슨의 개념인지 — 즉시 말할 수 있다.

1실전 명령 한 줄

배포 현장에서 이런 명령을 만나게 된다:

vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --gpu-memory-utilization 0.92 \
  --max-model-len 8192 \
  --max-num-seqs 128 \
  --max-num-batched-tokens 2048 \
  --dtype bfloat16 \
  --quantization fp8 \
  --kv-cache-dtype fp8 \
  --tensor-parallel-size 2 \
  --speculative-config '{"method":"mtp","num_speculative_tokens":2}'

겁먹을 것 없다. 이 플래그들은 전부 우리가 이미 유도해 본 물리량 — KV 캐시 예산(L2·L4), 반복(iteration)당 토큰 예산(L5), 파라미터당 바이트(L6·L7), 읽기 1회당 토큰 수(L9) — 를 서버에게 알려주는 숫자일 뿐이다.

2그룹 ① 메모리 예산 — "KV 캐시에 얼마를 줄 것인가"

vLLM은 시작할 때 GPU 메모리를 미리 나눈다: 가중치 + 액티베이션을 빼고, 남는 전부를 KV 캐시 블록으로 예약한다(L2의 두 번째 병목, L4의 배치 상한이 바로 이 예산이다).

플래그기본값역할
--gpu-memory-utilization0.92 이 vLLM 인스턴스가 쓸 GPU 메모리 비율(0~1). 나머지 8%는 드라이버/다른 프로세스 몫. 올리면 KV 예산↑, 너무 올리면 시작 시 OOM.
--max-model-len모델 config 시퀀스(프롬프트+생성) 최대 길이. 시퀀스 1개가 차지할 수 있는 최대 KV를 결정 — 줄이면 같은 예산에 더 많은 동시 사용자가 들어간다.
--kv-cache-dtypeauto KV 캐시만 따로 양자화(fp8 등). L6의 "파라미터당 바이트" 논리를 KV 바이트에 적용 — 캐시 용량 2배, 디코드 때 읽을 바이트도 절반.
--block-size16 KV 캐시를 나누는 페이지 크기(토큰 단위). vLLM의 간판 PagedAttention — OS 가상 메모리의 페이지와 같은 발상. (심화는 다음 레슨감)
A100 80 GB × utilization 0.9273.6 GB 예산
− Llama-3.1-8B 가중치 (FP16, L7: 2 bytes/param)− 16 GB
− 액티베이션/워크스페이스 (대략)− ~3 GB
KV 캐시 예산≈ 55 GB
KV/토큰 (32층 × 8 KV헤드 × 128차원 × 2 bytes × K,V)0.125 MB
캐시 가능한 총 토큰 → 8192 길이 기준 동시 시퀀스~44만 tok → ~54개

vLLM은 시작 로그에 이 계산 결과를 # GPU blocks로 찍어준다. 위 손계산과 로그를 대조하는 것이 Phase 3 첫 실습이 된다.

3그룹 ② 스케줄러 — "한 반복에 무엇을 태울 것인가"

L5에서 그린 반복별 타임라인을 기억하자. 스케줄러는 매 반복마다 "누구를 태울까"를 정하는데, 그 정원이 딱 두 개의 플래그다:

플래그역할우리 언어로
--max-num-seqs 한 반복에 함께 도는 시퀀스 수 상한 L4의 배치 크기 상한. 보통 KV 예산이 먼저 바닥나서, 이 값보다 적게 돈다.
--max-num-batched-tokens 한 반복에 처리할 토큰 수 상한 L5의 chunked prefill 청크 예산. 긴 프롬프트도 이 크기로 잘려 디코드 사이에 끼워진다. 작게 = 디코드 지연(ITL)↓, 크게 = prefill 처리량↑.

chunked prefill(V1 엔진 기본 활성)은 이 두 숫자로 완전히 기술된다: 매 반복, 디코드 중인 시퀀스들(각 1토큰) + 남는 토큰 예산만큼의 prefill 청크. L5에서 유저 A·B로 손으로 굴린 그 스케줄이, 이 두 플래그의 값 그대로다.

4그룹 ③ 바이트 줄이기, 읽기당 토큰 늘리기

Phase 2의 두 손잡이가 각각 플래그 하나씩이다:

플래그레슨역할
--dtypeL7 가중치/연산의 기본 정밀도(auto·bfloat16·float16). 양자화가 아니라 "원본 포맷" 선택.
--quantization (-q)L6 양자화 방식(fp8·awq·gptq 등). 파라미터당 바이트↓ → 읽을 바이트↓ + 같은 GPU에 큰 모델. 이미 양자화된 체크포인트는 자동 감지되는 경우가 많다.
--speculative-configL9 JSON 하나로 투기적 디코딩 설정. method: mtp(자기-투기, GLM/DeepSeek), eagle/eagle3, draft_model(별도 초안 모델), ngram(프롬프트에서 초안 검색 — 모델조차 불필요). num_speculative_tokens = L9의 K.
L9의 경고가 여기서 실전이 된다 --quantization과 --speculative-config를 함께 켤 때, 양자화로 초안 품질이 떨어지면 acceptance length가 줄어 합친 효과가 각각의 곱보다 작거나, 심하면 역효과일 수 있다. 반드시 둘 다 켠 상태로 측정한다.

5그룹 ④ 모델이 GPU 하나에 안 들어갈 때

플래그역할
--tensor-parallel-size (TP) 각 레이어의 행렬을 N개 GPU에 세로로 쪼갠다 — 모든 GPU가 매 토큰 함께 일한다. 가중치도 KV도 1/N씩. 한 노드 안 GPU끼리(NVLink) 쓰는 기본 수단.
--pipeline-parallel-size (PP) 레이어를 가로로 쪼개 GPU마다 다른 층을 담당 — 노드를 넘어갈 때 주로 사용. TP×PP = 총 GPU 수.

왜 TP가 통신을 많이 하고도 이득인지(All-Reduce, NVLink 대역폭)는 Phase 4의 좋은 입구다 — 지금은 "세로 vs 가로"만 가져가면 된다.

6서버 손잡이 vs 요청 손잡이

헷갈리기 쉬운 마지막 구분: 지금까지 본 것은 전부 엔진(서버) 파라미터 — 시작할 때 한 번 정하고 모든 사용자가 공유한다. 반면 temperature, top_p, max_tokens 같은 SamplingParams는 요청마다 API 호출에 실려 온다. 성능 튜닝은 전자, 출력 스타일은 후자다. (단, 요청의 max_tokens·프롬프트 길이 분포는 KV 예산을 통해 서버 성능에 되먹임된다 — 남 얘기가 아니다.)
