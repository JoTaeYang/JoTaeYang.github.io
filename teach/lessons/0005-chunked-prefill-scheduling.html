Lesson 05 · Phase 1 — The Workload

Prefill 스케줄링과 Chunked Prefill

한 사용자의 긴 프롬프트가 다른 사용자의 답변을 멈추게 하는 문제(prefill stalling)와, 그걸 OS 스케줄링처럼 푸는 방법.

이번 레슨의 단 하나의 승리 두 사용자의 요청을 iteration 단위로 따라가며, prefill stalling이 왜 생기고 chunked prefill이 그걸 어떻게 없애는지 이해한다. 그리고 이 전부가 OS 시분할/SMT와 같은 구조임을 본다.

배경: 프리필과 디코드는 성격이 정반대다

Lesson 2에서 봤듯, 추론은 두 단계다. 그런데 둘의 하드웨어 프로필이 정반대다:

성격노는 자원
프리필 (프롬프트 통째로)compute-bound메모리 대역폭이 남음
디코드 (토큰 1개씩)memory-bound계산 유닛이 남음

iteration은 배치의 모든 사용자가 함께 한 스텝 전진하는 단위다. 한 스텝이 끝나야 다음 토큰들이 시작된다 — 이 사실이 문제의 씨앗이다.

문제를 보자 — 유저 A, B 두 명

규칙: 프리필 예산 = 512토큰/스텝, 디코드 = 사용자당 1토큰/스텝. A 프롬프트 1,500토큰(t=0 도착), B 프롬프트 1,000토큰(A가 답하는 중 도착).

❌ Chunked prefill 없이 (통짜 프리필)

iter하는 일AB
0A 프리필 1,500 통째로 (무거움)프리필—
1A 디코드 → A토큰1🟢 술술—
2A 디코드 → A토큰2🟢 술술← B 도착
3B 프리필 1,000 통째로 (매우 무거움)🔴 멈춤!프리필
4A 디코드 + B 디코드🟢 재개🟢

iter 3에서 B의 프리필 1,000토큰이 그 스텝의 계산을 통째로 잡아먹어 스텝이 길어지고, A는 그동안 다음 토큰을 못 받는다. A 화면에서 글자가 술술 나오다 뚝 끊긴다. 이것이 prefill stalling.

✅ Chunked prefill 적용 (예산 512/스텝)

iter하는 일 (프리필 조각 + 디코드 섞기)AB
1A프리필[512]프리필 1/3—
2A프리필[512]프리필 2/3—
3A프리필[476] → A 프리필 완료첫 토큰 임박—
4A디코드 → A토큰1🟢—
5A디코드 → A토큰2🟢← B 도착
6B프리필[512] + A디코드 → A토큰3🟢 안 멈춤!프리필 1/2
7B프리필[488] + A디코드 → A토큰4 → B 완료🟢프리필 완료
8A디코드 + B디코드 → A토큰5, B토큰1🟢🟢

B의 프리필이 512토큰씩 조각나 들어오니 한 스텝의 계산량에 상한이 걸린다. 그래서 A는 매 스텝 토큰을 하나씩 계속 받는다 — 멈춤이 없다.

왜 A가 안 멈추나 — 자원 섞기 (진짜 이유)

iter 6을 자원 관점에서 뜯으면, 두 작업이 서로 다른 자원을 쓴다:

iter 6: B프리필 512토큰 ┃ A디코드 1토큰 ─────────────────┃───────────────── 계산 유닛 : 거의 다 씀 ███████ ┃ ░ 조금 메모리대역: 조금 ░ ┃ ███████ 거의 다 씀

프리필은 계산 유닛을 쓰지만 대역폭을 남기고, 디코드는 대역폭을 쓰지만 계산 유닛을 남긴다. 둘을 한 스텝에 섞으면 서로의 놀던 자원을 메꿔 GPU가 양쪽으로 꽉 찬다. B를 프리필하는 “김에” A 디코드가 거의 공짜로 얹혀 간다. (Lesson 1의 “배치 1은 계산 유닛이 논다”를 정반대에서 푸는 것.)

핵심: 쪼개는 대상은 '메모리'가 아니라 '계산 시간' A가 멈춘 이유는 메모리가 부족해서가 아니라, 그 한 스텝이 너무 길어서였다. chunked prefill은 한 스텝에 넣는 프리필 계산량(스텝 길이)에 상한을 둔다. (B의 KV 캐시 메모리를 조각내 할당하는 건 별개 축 — 그건 PagedAttention의 일이다.)

이건 정확히 OS 스케줄링이다

서빙 엔진OS 세계
GPU 하나CPU 코어 하나
iteration 한 스텝타임 슬라이스 (quantum)
요청 A, B프로세스
chunk 예산 (512)퀀텀 길이
KV 캐시프로세스별 상태(context)
  • 연속 배칭 = 라운드 로빈: 요청이 끝나면 슬롯에서 빠지고 대기열의 새 요청이 들어온다.
  • prefill stalling = head-of-line blocking: 한 작업이 코어를 안 놓아 뒤가 밀린다.
  • 통짜 프리필 = yield 안 하는 협조적(cooperative) 태스크 → chunked prefill = 선점형(preemptive) 타임슬라이싱.
더 정확한 비유: SMT(하이퍼스레딩) 프리필(compute-bound)과 디코드(memory-bound)를 한 스텝에 섞어 서로의 놀던 실행 유닛을 채우는 건, 성격 다른 두 스레드를 한 코어에 얹는 SMT와 같다. 현대 LLM 서빙 엔진 = 라운드 로빈(공정성) + SMT(자원 충전)를 iteration 단위로 합친 스케줄러. 단, 진짜 선점은 커널 도중이 아니라 iteration 경계에서만 일어난다(그래서 chunk 크기가 퀀텀).

트레이드오프 (튜닝 knob)

  • 새 요청(B)의 TTFT는 살짝 늘어남 — 프리필을 여러 스텝에 나눴으니.
  • 대신 진행 중인 요청(A)의 지연은 안정 + 전체 처리량 향상.
  • chunk 크기가 knob: 작게 → 디코드 더 안정, TTFT↑ / 크게 → 시작 빠름, 끊길 위험↑.
Chunked prefill이 A의 디코드를 멈추지 않게 하는 원리는?
스스로 확인 (retrieval practice) 종이를 덮고: ⑴ prefill stalling은 어떤 자원의 문제인가(메모리? 계산 시간?)? ⑵ 프리필과 디코드를 섞으면 이득인 이유는? ⑶ chunk 크기를 줄이면 무엇이 좋아지고 무엇이 나빠지나?  + 복습: 연속 배칭이 정적 배칭 대비 없애는 낭비는?(L4)
