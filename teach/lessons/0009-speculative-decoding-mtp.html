Lesson 09 · Phase 2 — The Knobs · (Sionic 간판)

MTP / Speculative Decoding

가중치를 한 번 읽고 토큰을 여러 개 확정한다. Sionic이 ~100 → ~600 tok/s를 만든 알고리즘적 돌파구 — 그리고 그 큰 숫자를 우리 손으로 계산한다.

이번 레슨의 단 하나의 승리 "가중치 1회 읽기당 토큰 여러 개"라는 아이디어를 이해하고, acceptance length 하나로 Sionic 글의 6.65 → 1.9 → 0.67 GB/토큰을 직접 계산한다. 그리고 이게 양자화와 달리 품질을 훼손하지 않는(lossless) 이유를 안다.

1다시, 낭비의 근원 (L1)

Lesson 1: 디코드는 토큰 하나를 만들려고 가중치 전체를 읽는다. 읽는 데 대부분의 시간이 가고(메모리 바운드), 계산 유닛은 논다. 배칭(L4)은 그 읽기를 여러 사용자가 나눠 쓰게 했다. 이번엔 다른 각도다:

질문 이미 가중치를 다 읽어왔는데, 같은 한 번의 읽기로 한 사용자의 토큰을 여러 개 만들 순 없을까?

2Draft-then-Verify: 추측하고 검증한다

Speculative decoding의 핵심은 두 역할로 나누는 것이다:

  • Draft(초안): 작고 빠른 예측기가 다음 K개 토큰을 후딱 추측한다 (싸다).
  • Verify(검증): 큰 모델이 그 K개를 단 한 번의 forward pass로 한꺼번에 검증하고, 자기 예측과 맞는 것만 받아들인다.
초안: 큰모델이 "The cat sat on the ___" 다음을 draft가 추측 → [mat] [and] [slept] (3개 한 번에 제안) 검증: 큰모델이 이 3개를 한 번의 pass로 병렬 확인 [mat] ✓ [and] ✓ [slept] ✗ → 앞 2개 확정 + 큰모델의 진짜 다음 토큰 1개 → 가중치 1회 읽기로 토큰 3개 전진 (2 확정 + 1 보정)
왜 검증이 싼가 — L5의 그 원리 제안된 K개를 검증하는 건 모든 위치를 한 번에 계산하는 병렬 pass다 — 프리필과 똑같이 compute-bound. 즉 가중치는 한 번만 읽고(비싼 것 1회), 계산(싼 것)만 K배. Lesson 1에서 계산은 메모리보다 940배 쌌던 걸 기억하라 — 그래서 K개 검증이 거의 공짜다.

3품질을 훼손하지 않는다 (lossless) — 양자화와의 결정적 차이

검증 규칙(각 초안 토큰을 min(1, p/q) 확률로 수락)은 수학적으로 최종 출력이 "큰 모델 혼자 디코딩했을 때와 완전히 동일한" 분포가 되도록 보장한다[1].

핵심 대비 양자화(L6)는 빠른 대신 품질을 조금 내준다(lossy). Speculative/MTP는 빠르면서 품질이 그대로다(lossless). 틀린 초안은 검증에서 버려지므로, 초안이 나빠도 느려질 뿐 틀리지 않는다.

4MTP: 초안 모델을 따로 두지 않는다

Multi-Token Prediction은 별도의 draft 모델 없이, 큰 모델 자신에게 미래 토큰을 예측하는 추가 헤드 몇 개를 달아 그 출력을 초안으로 쓴다(self-speculation)[2]. GLM·DeepSeek·Gemma 등이 이 방식. 그래서 GLM-5.2 글이 "MTP"라 부른다.

5초안은 어디서, 어떻게 나오나 (Q&A에서)

5a. "작고 빠른 예측기"는 LLM 안에 있나, 밖에 있나 → 방식에 따라 둘 다

draft는 어디에추가 비용
고전 spec. decoding[1]밖 — 별도의 작은 모델 (예: 70B 옆에 1B)작은 모델의 가중치 + KV 캐시 메모리 (L2/L4의 배치 상한을 갉아먹음); 토크나이저·분포도 맞아야 함
MTP (GLM·DeepSeek)안 — 몸통(80층)에 붙은 추가 예측 헤드거의 없음 — 얇은 헤드 몇 개; 몸통의 문맥 이해를 공유해 초안 품질도 좋음
토큰 → 몸통 80층 (L7) → 최종 벡터 ─┬→ 기존 헤드: t+1 예측 ← 원래 있던 것 ├→ MTP 헤드 1: t+2 예측 ← 추가 └→ MTP 헤드 2: t+3 예측 ← 추가

즉 MTP의 예측기는 별도의 뇌가 아니라, 같은 뇌에 달린 추가 출력구다 — 그래서 self-speculation.

5b. t+2를 찍으려면 t+1을 모르는데? → 두 가지 설계

  • 병렬형 (Medusa, 원조 MTP 논문[2]): 각 헤드가 같은 최종 벡터에서 t+1·t+2·t+3를 독립적으로 찍는다. 가능한 이유 — 최종 벡터는 "다음 토큰 하나"가 아니라 문맥의 궤적 전체를 담고 있다(L7). 단, 멀수록 정확도↓.
  • 연쇄형 (DeepSeek-V3·GLM 계열): 초안 생성도 작게 자기회귀를 돈다 — 단 진짜 t+1 대신 "자기가 방금 찍은 t+1"을 조건으로, 스텝당 비용은 몸통 80층이 아니라 얇은 MTP 모듈 1개. 그래서 초안이 싸다.
연쇄형: 몸통 80층 → 최종 벡터 → t+1 초안 확정 [최종 벡터 + t+1 초안의 임베딩] → MTP 모듈(얇은 1층) → t+2 초안 [그 출력 + t+2 초안의 임베딩] → 같은 MTP 모듈 → t+3 초안

5c. 둘 다 성립하는 이유 — 틀려도 검증이 잘라낸다

초안: [mat] [and] [slept] 검증: [mat] ✓ [and] ✗ → 첫 불일치에서 그 뒤는 볼 것도 없이 폐기 확정: [mat] + 큰 모델의 진짜 다음 토큰
이 설계의 정수 초안은 맞을 필요가 없고, 자주 맞기만 하면 된다. t+1 추측이 틀리면 그 위에 쌓은 t+2는 어차피 같이 버려진다 — 맹찍기의 대가는 틀림이 아니라 느려짐(acceptance length↓)뿐이다 (§3의 lossless 안전망). 그리고 검증에는 이 문제가 없다: 초안 토큰들이 이미 주어져 있으니 프리필처럼 모든 위치를 병렬 계산 — "생성은 순차, 검증은 병렬"인 이유.

6Sionic의 숫자를 직접 계산하자

딱 하나의 지표가 이득을 정한다 — acceptance length: 큰 모델 pass 1회당 평균 몇 토큰이 확정되나.

공식 토큰당 가중치 바이트 = (pass 1회에 읽는 가중치) ÷ (acceptance length)

GLM-5.2는 MoE라 pass 1회에 활성 전문가 6.65 GB를 읽는다(글의 값). 그럼:

MTP 없음 (acceptance = 1)6.65 ÷ 1 = 6.65 GB/토큰
MTP (acceptance ≈ 3.5)6.65 ÷ 3.5 ≈ 1.9 GB/토큰
MTP (acceptance ≈ 10)6.65 ÷ 10 ≈ 0.67 GB/토큰

글에 나온 6.65 → 1.9 → 0.67 GB/토큰이 정확히 이것이다. 토큰당 읽는 바이트가 줄면 L1 공식대로 토큰당 시간이 줄어 → 글의 ~100 → ~600 tok/s가 나온다. 당신은 이제 그 간판 숫자를 스스로 유도할 수 있다.

7공짜는 아니다 — 양자화와의 얽힘 (L6 회수)

이득은 전적으로 acceptance length에 달렸고, 그건 초안이 얼마나 자주 맞느냐다. 여기서 L6의 떡밥이 회수된다:

  • 양자화 정확도 저하 → 초안 품질↓ → acceptance length↓ → 오히려 처리량↓. Sionic이 경고한 바로 그 얽힘[3].
  • 워크로드가 좌우: 코드·반복 텍스트는 예측 쉬워 acceptance↑, 무작위·창의적 텍스트는↓.
  • temperature·수락 임계값 튜닝으로 품질을 지키며 acceptance를 최대화 — 글의 "MTP 파라미터 튜닝" 층.
Sionic 3층 최적화와의 연결 이제 글의 구조가 보인다: ⓐ MTP 파라미터 튜닝(이 레슨) → ⓑ 런타임(SGLang 패치) → ⓒ 커널. 그리고 이 모두가 양자화·acceptance가 수치적으로 얽혀 조율돼야 한다는 것. Phase 3·4에서 ⓑⓒ로 내려간다.
Speculative decoding / MTP가 속도를 얻는 핵심 원리는?
스스로 확인 (retrieval practice) 종이를 덮고: ⑴ 검증 pass가 싼 이유는(어떤 바운드)? ⑵ acceptance length가 5면 6.65GB/pass 모델의 토큰당 바이트는? ⑶ 양자화가 MTP 처리량을 낮출 수 있는 경로는? ⑷ 스펙decoding이 lossless인 이유는? ⑸ t+2 초안은 t+1을 모르는데 어떻게 가능한가 (두 설계 + 안전망)?
