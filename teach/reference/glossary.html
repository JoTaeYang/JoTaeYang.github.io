Glossary

Shared nomenclature for the course. Every lesson uses these exact terms. Grows as we go — a term appears here the first lesson that needs it.

추론 Inference P0
Running a trained model to produce output. For an LLM: turning a prompt into generated tokens. Distinct from training. Serving = doing inference for many users, reliably and fast.
토큰 Token P0
The unit an LLM reads and generates — a word-piece (~0.75 English words). A model generates one token at a time, feeding each back in to produce the next.
자기회귀 디코딩 Autoregressive decoding P0
The one-token-at-a-time generation loop. Each new token requires a full pass through the model. This sequential, one-at-a-time nature is the root cause of inference being memory-bound.
파라미터 / 가중치 Parameter / Weight P0
One of the model's learned numbers — a coefficient in a multiply-add. "70B model" = 70 billion of them. Set during training, fixed during inference. Collectively they are the model's knowledge.
파라미터당 바이트 Bytes per parameter P0
How many bytes one parameter occupies — a choice of number format, not the value itself (FP16=2B, FP8=1B, FP4=0.5B). model bytes = param count × bytes/param; quantization shrinks the second factor only. Like int vs short vs char in C.
레이어 / 블록 Layer / Transformer block P1
One repeating processing stage that refines the token vectors and returns them in the same shape, so stages compose (f₈₀(…f₁(x))). Each layer = an attention sub-layer + an FFN sub-layer, with its own parameters. Stacking many (~80) gives depth: shallow layers catch surface/grammar, deep layers catch meaning/intent.
FFN Feed-Forward Network P1
The per-token sub-layer inside a transformer block: after attention lets tokens exchange info, the FFN lets each token transform that info on its own (two matrix multiplies + nonlinearity). Holds a large share of a model's parameters.
임베딩 Embedding P1
The lookup table mapping each token to a learned vector ("meaning coordinates"). Similar meanings land at nearby vectors, which is why attention's Q·K dot product measures relatedness. The table is itself parameters.
분산 표현 Distributed representation P1
Meaning lives not in a single vector axis but across combinations of axes (directions in the space). Individual coordinates are usually not human-interpretable; concepts like "gender" exist as learned directions (hence king−man+woman≈queen).
경사하강법 Gradient descent P1
The training procedure: repeatedly nudge every parameter a tiny bit in the direction that reduces next-token prediction error. Over trillions of tokens the vector space self-organizes. (Training only — inference just runs the fixed result.)
메모리 대역폭 Memory bandwidth P0
How fast the GPU can move data between its HBM memory and its compute units, in bytes/second (e.g. B300 ≈ 8 TB/s). Often the true speed limit of decode, not raw compute.
HBM High-Bandwidth Memory P0
The large, fast DRAM stacked on a GPU package where model weights and the KV cache live. Big (tens–hundreds of GB) but slow relative to on-chip memory; every weight byte must travel from here.
FLOP / FLOPS Floating-point op(s) / per second P0
FLOP = one floating-point arithmetic operation. FLOPS = operations per second (a rate). A GPU's headline "15 PFLOPS" is its compute ceiling.
연산 강도 Arithmetic intensity P0
FLOPs performed per byte moved from memory (FLOP/byte). Low intensity → memory-bound; high intensity → compute-bound. Batch-1 LLM decode sits near 1 FLOP/byte — deeply memory-bound.
루프라인 모델 Roofline model P0
A plot of achievable performance vs. arithmetic intensity. The "roof" has a slanted (bandwidth-limited) part and a flat (compute-limited) part. Where your workload lands tells you which resource to optimize.
메모리 바운드 / 컴퓨트 바운드 Memory-bound / Compute-bound P0
A workload is memory-bound if it finishes faster only by moving bytes faster, and compute-bound if only by doing math faster. LLM decode is the former.
어텐션 Attention P1
The mechanism by which each token pulls information from other tokens, weighted by relevance. Every token becomes three vectors: Query ("what I'm looking for"), Key ("what I advertise"), Value ("the content I hand over"). A token's Q is dot-producted with every past token's K → softmax → weights → weighted sum of the Vs. Because a token attends to all past tokens, past K/V must be kept — the root of the KV cache and of memory-bound decode.
셀프 어텐션 Self-attention P1
Attention where tokens attend to other tokens within the same sequence (rather than to a separate input). The core operation of the transformer.
헤드 Attention head P1
One parallel "view" of attention. A model runs many heads at once, each learning a different kind of relation (grammar, topic, coreference…). Each head normally has its own Q/K/V.
MHA / GQA / MQA Multi-Head / Grouped-Query / Multi-Query Attention P1
A spectrum trading KV-cache size for quality. MHA: every Q head has its own K/V (biggest cache). GQA: groups of Q heads share one K/V set (today's default — ~¼ cache, near-MHA quality). MQA: all Q heads share one K/V (smallest cache, some quality loss). KV-cache size scales with the number of K/V heads, not Q heads.
위치 인코딩 / RoPE Positional encoding / Rotary Position Embedding P1
Bare attention is permutation-invariant — Q/K/V carry no order, so "동물 길" and "길 동물" look identical. Order must be injected. RoPE (today's standard) rotates Q and K by an angle proportional to position, so the Q·K dot product encodes relative distance. Its behavior beyond the trained length is a main cause of long-context breakdown (extended via position interpolation / YaRN).
롱컨텍스트 Long context P1
Same attention mechanism at any distance, but quality drops far back: softmax dilution over many candidates, lost-in-the-middle (models attend to start/end better than middle), and degradation past the trained length. It is both a memory problem (KV cache ∝ context) and a quality problem.
헤드 특화 Head specialization (interpretability) P1
Empirically, specific attention heads take on identifiable roles — induction heads ([A][B]…[A]→[B] copying), previous/duplicate-token heads, name-mover heads (do coreference-like routing, IOI circuit), syntactic/positional heads. Found via attention-pattern analysis and causal activation patching. Not all heads are cleanly interpretable (many are polysemantic).
트랜스포머 Transformer P1
The neural-network architecture built on self-attention (Vaswani et al., 2017, "Attention Is All You Need"). The backbone of every modern LLM. Replaced sequential RNNs — parallelizable, good at long-range relations.
KV 캐시 KV cache P1
A store of the Key and Value vectors of all past tokens. Since a past token's K/V never change, caching them avoids recomputing the whole context each step — turning a quadratic cost into linear. Grows with sequence length × batch size and becomes the second big memory consumer after weights; it caps how large a batch you can serve.
프리필 vs 디코드 Prefill vs Decode P1
Prefill: process the whole prompt at once (parallel, compute-bound, fast) and fill the KV cache. Sets TTFT (time to first token). Decode: generate tokens one by one (sequential, memory-bound, the dominant cost), reading past K/V from the cache and appending the new token's. Sets TPOT (time per output token).
배칭 Batching P1
Serving multiple requests together so a single weight-read produces tokens for many users at once — the main lever for throughput. Amortizes the expensive weight-read across N users (per-token memory cost ÷ N). Bounded by KV-cache memory (batch × context).
처리량 vs 지연 Throughput vs Latency P1
Throughput: total tokens/sec across all users — sets cost-per-user; batching raises it. Latency: one user's speed — TTFT (first token) and TPOT (per token). Batching leaves per-user latency ~flat while lifting throughput, until compute/memory limits bite.
정적 vs 연속 배칭 Static vs Continuous batching P1
Static: fix the batch, wait for all to finish — finished users' slots sit idle (wasted GPU). Continuous (iteration-level): as a request finishes, a queued one takes its slot, keeping the batch full. ~2–4× throughput, better tail latency. The default in vLLM/SGLang. Caveat: prefill stalling.
PagedAttention PagedAttention P1
vLLM's technique to store the KV cache in fixed-size pages (like OS virtual memory) instead of one big contiguous block, eliminating fragmentation waste and letting more requests fit — i.e. bigger batches. A memory-axis optimization (distinct from chunked prefill's compute/time-axis one).
Prefill stalling Prefill stalling P1
When a new request's (heavy, compute-bound) prefill runs in one iteration, that iteration becomes long and all in-flight decode users freeze until it ends — a latency spike. It is a compute-time problem (the step is too long), not a memory-capacity one. The serving analog of head-of-line blocking.
Chunked prefill Chunked prefill P1
Split a prompt's prefill into fixed-size chunks (a per-iteration compute budget) and interleave a chunk with ongoing decodes each step. Caps step length → no stalling; and mixing compute-bound prefill with memory-bound decode fills both resources (SMT-like). Knob: chunk size trades new-request TTFT vs decode stability. Origin: SARATHI (2023); default in vLLM/SGLang.
양자화 Quantization P2
Storing weights/activations with fewer bits per number (FP16 → FP8 → FP4/INT4) by rounding a continuous range onto a few representable levels via a scale factor. Reduces bytes per parameter (not the parameter count). Double win: less memory traffic and faster low-precision tensor-core math. Cost: accuracy loss, worst on math/code/reasoning at 4-bit.
부동소수점 포맷 FP16 / FP8 (E4M3, E5M2) / FP4 P2
Number formats split into sign · exponent · mantissa. FP16/BF16 = 2 bytes (baseline). FP8 = 1 byte, two flavors: E4M3 (more mantissa → precision, ±448) and E5M2 (more exponent → range). FP4 = 0.5 byte (~16 levels). Exponent trades range, mantissa trades precision.
텐서 코어 Tensor Core P2
Specialized GPU units that do matrix multiply-accumulate very fast, and faster still at lower precision — this is why quantization also speeds up compute. Blackwell's headline "15 PFLOPS" is an FP4 tensor-core number. (Blackwell's 5th-gen tensor-core instruction = tcgen05, covered in Phase 4.)
weight-only vs weight+activation Quantization scope P2
Weight-only: quantize only stored weights (helps memory), compute in higher precision — safer. Weight+activation: quantize the running activations too — unlocks low-precision tensor-core math but is harder (activations have outliers). GPTQ/AWQ are popular 4-bit weight-only methods.
MoE Mixture of Experts P2
A model split into many "expert" sub-networks; each token activates only a few (e.g. top-8). GLM-5.2 is 753B params but only a fraction runs per token. Detailed in Phase 2.
스펙큘러티브 디코딩 Speculative decoding P2
Draft-then-verify: a cheap predictor drafts the next K tokens, then the big model verifies all K in one parallel forward pass (compute-bound, like prefill) — so one expensive weight-read advances multiple tokens. The min(1, p/q) acceptance rule makes the output distribution identical to the big model decoding alone (lossless — unlike quantization). A bad draft only makes it slower, never wrong. (Leviathan et al., 2022.)
MTP Multi-Token Prediction P2
Speculative decoding without a separate draft model: the big model carries extra heads that predict future tokens, and their output is the draft (self-speculation). Used by GLM/DeepSeek-style models — this is what the Sionic post tunes. Same lossless verify rule; the gain still lives or dies by acceptance length.
Acceptance length Acceptance length P2
Average number of tokens confirmed per big-model pass. The single number that sets the speculative/MTP gain: bytes per token = bytes per pass ÷ acceptance length — exactly the Sionic post's 6.65 → 1.9 → 0.67 GB/token. Depends on draft quality (quantization accuracy loss lowers it — the quantization↔MTP coupling) and on workload predictability (code ↑, creative text ↓).
tcgen05 · TMEM · Split-K · FP8/FP4 P4
Blackwell tensor-core kernel machinery from the Sionic post — defined when we reach the CUDA phase.
