LLM 추론 최적화 — Zero부터 CUDA 커널까지

미션: 실무 LLM 추론 서빙/최적화. 촉매: Sionic의 GLM-5.2 x10 글을 처음부터 끝까지 스스로 읽어내기.

코스를 관통하는 한 문장 토큰 하나의 비용은 곱셈이 아니라 가중치를 메모리에서 끌어오는 시간이다 — 배칭·양자화·MTP·커널, 전부 이 읽기 비용을 줄이거나 나눠 쓰는 방법이다.

Phase 0 — 멘탈 모델 ✓ 완료

모든 최적화를 읽어낼 단 하나의 공식.

  1. L1왜 LLM 추론은 '메모리' 문제인가— 메모리 바운드, 940배, 상한 ~114 tok/s

Phase 1 — 워크로드 ✓ 완료

디코드 루프가 실제로 뭘 하는지: 서빙 처리량을 스스로 추론하는 틀.

  1. L2디코드 루프와 KV 캐시— prefill vs decode, 두 번째 메모리 병목
  2. L3어텐션이란 무엇인가— Q·K·V, 모든 최적화 문제의 뿌리
  3. L4배칭: 처리량 vs 메모리— 읽기 1회를 N명이 분담, KV 캐시가 상한
  4. L5Prefill 스케줄링과 Chunked Prefill— stalling, 유저 A·B 타임라인, OS/SMT 비유

Foundations — 개념의 바닥 (곁가지 심화) ✓ 완료

질문에서 태어난 레슨들 — 블랙박스를 남기지 않는다.

  1. L7모델은 결국 숫자다: 파라미터·레이어·임베딩— 파라미터당 바이트, 벡터, "그것"→"동물"
  2. L8순서 · 롱컨텍스트 · 헤드 특화— RoPE, lost-in-the-middle, induction head

Phase 2 — 조절 손잡이 ◉ 진행 중

읽는 바이트 자체를 줄이는 레버들 — Sionic 글의 심장.

  1. L6양자화 (FP16 → FP8 → FP4)— 파라미터당 바이트↓, double win, 정확도 대가
  2. L9MTP / Speculative Decoding— 읽기 1회로 토큰 여러 개, 6.65→0.67 GB/토큰
  3. L11MoE (예정)— 753B 중 일부만 활성, 6.65 GB/pass의 출처

Phase 3 — 서빙 스택 ◉ 진행 중 · 실습형

vLLM/SGLang을 실제 GPU에서: tok/s 측정, FP8 vs FP16, KV OOM 재현, CUDA graph.

  1. L10vLLM 엔진 파라미터— 배운 개념이 플래그가 된다: KV 예산, 스케줄러, -q, speculative-config

Phase 4 — GPU 구조와 CUDA 커널 예정

직접 커널을 쓴다 — tcgen05, 커널 퓨전. 이후 캡스톤: Sionic 글 완독.
