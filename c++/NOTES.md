# Teaching Notes

- 한국어로 가르친다. 기술 용어는 영어 병기 (예: 링버퍼(ring buffer)).
- 사용자는 깊이 우선을 선호 — 레슨을 서두르지 말고 한 주제를 바닥까지.
- C++ 기본기가 불안하다고 자가 평가 → 초반 레슨은 C++/메모리 기초를 이 저장소의 실제 코드와 연결해서 진행.
- 기본 CS 지식(OS, 자료구조, 네트워크 이론)을 명시적으로 요청함 — 코드만 다루지 말 것.
- 이 저장소 자체(NetLib, FighterServer)가 살아있는 교재. 추상적 예제보다 실제 코드 인용을 우선.
- **레슨은 5개 배치로 제작** (2026-07-06 요청): 로드맵이 보이도록 한 번에 5개씩 만들되, 사용자는 순서대로 하나씩 실습 통과 후 진행 권장.
- **파생 문서 규칙**: 레슨을 보다 생긴 질문은 `NNNN-K-슬러그.html`(예: 0001-1-...)로 만들어 부모 레슨과 양방향 링크. 질문이 나올 때 즉석 제작이 원칙.
- **모든 레슨에 "초보자 질문 세트" 섹션 포함** (2026-07-06 요청): 완전 초보자가 그 레슨을 읽고 던질 법한 질문 6~8개를 접이식(details)으로 — 답을 보기 전에 스스로 생각하게 유도. 질문은 레슨 본문이 당연하게 넘어간 전제(타입 별칭, 용어, "왜?")를 겨냥할 것.
- **시스템 설계 레슨은 "실패 시나리오 우선"으로** (2026-07-12 발견, [[0004]]): 사용자는 C++ 문법에선 흔들려도 "왜 이렇게 안 하면 깨지나" 직관이 강함. D(IOCP)·E(락프리) 레슨은 "무엇을 하는가"보다 "이렇게 안 하면 어떤 크래시가 나는가"로 열 것. 레슨 서술을 일부러 반박 가능한 형태로 내놓으면 관여도가 올라감(사용자가 사전할당 이유를 스스로 정정한 사례).
- **면접은 목표 아님 — 2026-07-12 재확인.** 사용자가 "면접에서 먹힐까" 질문했으나, 확인 결과 **미션 불변**(엔지니어링 실력 자체가 목적). MISSION.md 수정 안 함. 코딩테스트·CS 단답 대비 트랙은 추가하지 않는다.
- **손코딩 실습은 기둥 F에 몰아서** (2026-07-12 사용자 결정). 에이전트는 레슨마다 손코딩 과제를 붙이자고 제안했으나 사용자가 원안 유지 선택. → **귀결**: 재인(객관식 통과) vs 생성(백지 구현) 격차를 막을 장치가 간격 복습뿐이 됨. 따라서 **퀴즈 통과만으로 넘기지 말고, 며칠 뒤 "보기 없이 말로 설명" 자유회상 재검을 더 자주 건다**(오늘 중간 패딩 망각을 이 방식으로 잡아냄).
