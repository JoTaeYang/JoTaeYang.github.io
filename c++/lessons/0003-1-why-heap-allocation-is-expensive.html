"비싼 연산"의 정체 — new 한 번에 실제로 벌어지는 일

레슨 3은 new를 "OS 할당자를 거치는 비싼 연산"이라고 한 단어로 뭉갰다. 그 안에는 최소 다섯 단계, 최악의 경우 커널 왕복 두 번이 들어있다. 하나도 빼지 않고 펼친다.

비유부터 — 공책 vs 창고 사물함

스택 = 공책에 이어 쓰기힙 = 창고에서 사물함 빌리기
연필이 지금 몇 줄에 있는지만 알면 된다. 3줄 필요하면 연필을 3줄 내린다. 끝. 다 쓰면 3줄 올린다. 끝.관리인(할당자)이 있다. ①줄 서기 ②장부 뒤지기 ③큰 것 쪼개기 ④꼬리표 붙이기 ⑤줄에서 나가기. 창고가 꽉 차면 건물주(커널)에게 신청서를 낸다.

진짜 이름으로 — new의 호출 사슬

new stSESSION
  → operator new(크기)     // 컴파일러가 부르는 함수
  → malloc(크기)            // C 런타임의 할당자
  → HeapAlloc(...)          // Windows 힙 = "관리인"

관리인 안에서 (빠른 경로)

  1. 락을 잡는다. 여러 스레드가 동시에 new할 수 있으니 장부가 꼬이지 않게 한 번에 하나만. → 워커 4개가 동시에 부르면 3개는 기다린다.
  2. 프리 리스트를 뒤진다. "빈 덩어리" 연결 목록을 따라가며 충분히 큰 것을 찾는다. 메모리 여기저기를 점프하며 읽으므로 캐시 미스가 잦다(레슨 7 주제).
  3. 쪼갠다(split). 100칸에서 48칸을 떼주고 남은 52칸을 다시 목록에 넣는다.
  4. 헤더를 쓴다. 돌려주는 주소 바로 앞에 크기·플래그를 기록한다. ← 레슨 3 퀴즈 2(delete vs delete[])가 UB인 이유가 정확히 이것: 헤더 위치가 어긋나면 관리인 장부가 깨진다.
  5. 락을 푼다.

창고가 꽉 찼을 때 (느린 경로)

6. VirtualAlloc 으로 커널에게 메모리를 더 요청 — 시스템 콜. 여기가 진짜 비싼 지점이다.

시스템 콜이 왜 비싼가
CPU는 유저 모드(내 프로그램)와 커널 모드(운영체제) 두 권한 수준으로 돈다. 시스템 콜은 그 문을 넘는 일이다: 하던 것 전부 저장 → 권한 수준 변경 → 커널로 점프 → 요청 정당성 검사 → 페이지 테이블(주소 장부) 수정 → 권한 복구 → 저장한 것 복원 → 복귀.
내 방 문을 여는 것 vs 공항 보안검색대를 통과하는 것의 차이.

7. 그런데 아직 끝이 아니다. 커널은 "방 배정했다"고 종이에만 써준 상태(주소 공간 예약/커밋). 실제 물리 메모리는 없다. 그 자리에 처음 쓰는 순간 → 페이지 폴트 → 또 커널 진입 → 빈 물리 페이지 확보 → 0으로 전부 지움(앞 프로세스 데이터 유출 방지 — 보안) → 페이지 테이블에 연결 → 복귀.

한 문장
new 한 번은 최악의 경우 락 경쟁 + 장부 검색 + 커널 왕복 두 번이다.

그동안 스택은

sub rsp, 48        ; 이게 전부다

기계어 명령 딱 하나 — 스택 포인터에서 48을 뺀다. 크기는 컴파일 타임에 이미 계산돼 있다. 락도, 장부도, 헤더도, 시스템 콜도 없다. 해제는 함수 끝에서 rsp를 되돌리는 게 전부 — 공짜. 게다가 스택 꼭대기는 프로그램에서 가장 자주 건드리는 메모리라 거의 항상 캐시에 뜨겁다.

비용 자릿수 (환경마다 다름 — 자릿수만 볼 것)

하는 일대략
스택 할당 (sub rsp)1 사이클 미만 — 사실상 0
힙 할당, 빠른 경로수십 나노초
힙 할당, 락 경쟁수백 ns ~ 수 μs
힙 할당, VirtualAlloc까지수 μs
첫 접근 시 페이지 폴트수 μs

스택 대비 100배~10000배. 이것이 "비싼 연산"의 실체다.

그래서 이 서버의 설계

초당 수만 패킷. 패킷마다 new면 위 다섯 단계를 초당 수만 번 반복하고, 워커 스레드들이 ①번 락에서 서로를 막는다 — 코어를 8개 줘도 할당 구간에선 한 줄로 서니 병렬성이 죽는다.

그래서 CMessageBuffer는 new가 아니라 풀에서 Alloc()으로 빌려온다(CServer.cpp:375). 풀은 시작 시 큰 덩어리를 한 번 받아두고 빌려주기/돌려받기만 한다 — 장부 검색도, 쪼개기도, 시스템 콜도 없이.

이것이 세션 사전할당의 이유 중 "비싸다" 쪽 절반이다. 나머지 절반(use-after-free 회피)은 정확성 문제로, 레슨 3 FAQ와 이후 IOCP 레슨에서 다룬다.
