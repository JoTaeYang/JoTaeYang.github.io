Lesson 0006-1 · 파생 심화 · 레슨 6에서 파생 (지적: "가상 메모리 내용에 뭔가 빠진 것 같다")

주소 변환의 실제 기계

레슨 6은 무엇을 말했다 — 주소는 거짓말이고, 페이지 단위이며, 커밋은 미뤄진다. 하지만 어떻게를 건너뛰었다. "MMU가 페이지 테이블을 보고 번역한다"는 한 줄 뒤에 실제로 무슨 기계가 도는지, 그리고 그 기계가 어떻게 nullptr 덫·코드 보호·TLB를 하나의 메커니즘으로 설명하는지를 펼친다.

1. 가상 주소는 두 조각이다

번역의 첫 걸음은 자르기다. 페이지가 4KB = 212바이트이므로, 주소의 하위 12비트는 "페이지 안에서 몇 번째 바이트인가"를 그대로 표현할 수 있다.

가상 주소 0x7FF6A0001234 (48비트 사용)

  [ 상위 36비트 = 가상 페이지 번호 ] [ 하위 12비트 = 오프셋 ]
         0x7FF6A0001                        0x234
핵심 — 오프셋은 번역되지 않는다
번역이 바꾸는 것은 페이지 번호뿐이다. 오프셋 12비트는 손대지 않고 그대로 통과한다. 가상 페이지 0x7FF6A0001이 물리 프레임 0x00A73으로 번역되면, 최종 물리 주소는 0x00A73234 — 뒤 세 자리는 그대로다.
그래서 "한 페이지 안에서의 상대 위치"는 가상이든 물리든 항상 같다. 16진수로 읽을 때 마지막 3자리가 오프셋이라고 외워두면 평생 쓴다.

2. 페이지 테이블 엔트리 — 주소 + 권한

페이지 테이블의 한 칸(PTE, 8바이트)은 단순히 "물리 프레임 번호"만 담지 않는다. 권한 비트가 같이 들어있다:

비트뜻0이면 / 위반하면
Present이 페이지가 지금 물리 메모리에 매핑돼 있나0 → 페이지 폴트
R/W쓰기 허용인가0인데 쓰면 → 폴트(액세스 위반)
U/S유저 모드가 접근 가능한가0인데 유저가 접근 → 폴트
NX실행 금지 (bit 63)1인데 실행하면 → 폴트 (DEP)
레슨 6의 두 사실이 여기서 하나로 합쳐진다
  • nullptr 역참조가 크래시인 이유 = 0번지 근처 페이지의 Present가 0(아예 매핑 안 함). 접근 → 폴트 → 커널이 "이 주소는 유효하지 않음" 판정 → 0xC0000005.
  • 코드 영역이 읽기+실행 전용인 이유 = 그 페이지들의 R/W가 0. 실수로 함수 본문을 덮어쓰려 하면 하드웨어가 막는다.
  • 스택·힙에서 코드 실행이 막히는 이유(DEP) = 그 페이지들의 NX가 1. 버퍼 오버플로로 셸코드를 심어도 실행이 안 된다.
세 가지가 전부 같은 기계다: PTE의 비트를 CPU가 매 접근마다 검사한다. 소프트웨어 검사가 아니라 하드웨어가 공짜로 해준다.

3. 왜 페이지 테이블은 4단계인가 — 계산해보면 안다

"페이지 번호 → 프레임 번호" 표를 평평하게 하나로 만든다고 해보자:

가상 페이지 번호 = 36비트 → 표의 칸 수 = 2^36 개
한 칸(PTE) = 8바이트
표 크기 = 2^36 × 8 = 2^39 = 512 GB     ← 프로세스 하나당!

불가능하다. 128TB 주소 공간 중 실제로 쓰는 건 몇 MB인데, 표가 512GB일 수는 없다.

해법은 트리다. 페이지 번호 36비트를 9비트씩 네 조각으로 쪼개, 4단계로 내려간다:

[9비트][9비트][9비트][9비트][12비트 오프셋]
  PML4  PDPT   PD     PT      offset
   ↓     ↓      ↓      ↓
  512칸 512칸  512칸  512칸    ← 각 단계는 512개 엔트리
                              512 × 8바이트 = 4096바이트 = 정확히 한 페이지!

아름다운 점 둘:

  • 각 테이블이 정확히 한 페이지에 딱 맞는다(512 × 8 = 4KB). 우연이 아니라 그렇게 설계한 것.
  • 안 쓰는 가지는 아예 만들지 않는다. 128TB 중 몇 MB만 쓰는 프로세스는 테이블이 실제로 몇 KB면 된다. 512GB → 수 KB.

9×4 + 12 = 48비트. 이것이 x64가 실제로 48비트 주소만 쓰는 이유다(총 256TB = 유저 128TB + 커널 128TB). 최신 CPU는 5단계(57비트)까지 확장 가능하다. OSTEP 20장

4. 대가와 해결책 — TLB

4단계 트리에는 대가가 있다. 주소 하나를 번역하려면 메모리를 4번 읽어야 한다(각 단계 테이블을). 그리고 그 4번을 모든 메모리 접근마다 한다면?

players[3172].chHP를 한 번 읽는 데 메모리 접근 5번(번역 4 + 실제 1). 5배 느려진다. 이대로면 가상 메모리는 못 쓸 물건이다.

그래서 CPU 안에 TLB(Translation Lookaside Buffer)가 있다 — "최근에 번역한 페이지 번호 → 프레임 번호" 결과를 담는 작고 아주 빠른 캐시(보통 수십~수천 개 엔트리).

  • TLB 히트: 번역이 사실상 공짜(1사이클 수준). 대부분의 접근이 여기 해당.
  • TLB 미스: 그때만 4단계를 실제로 걷는다(page table walk).
이것이 레슨 7(캐시)과 이어지는 지점
TLB는 페이지 단위의 지역성에 의존한다. 한 페이지(4KB) 안을 계속 훑으면 번역 결과 하나로 4096바이트를 다 커버한다 → TLB 히트.
반대로 여기저기 흩어진 주소를 점프하면 매번 다른 페이지 → TLB 미스 폭발.
파생 3-1에서 "프리 리스트를 따라가는 건 캐시 미스가 잦다"고 한 것이 정확히 이 이야기다. 링버퍼가 연속된 메모리인 것, 섹터가 list를 버리고 vector를 택한 것(레슨 7) — 전부 TLB와 캐시를 위한 선택이다.

5. 전체 그림 — 한 번의 메모리 접근에서 벌어지는 일

players[3172].chHP 를 읽는다
   ↓
① 가상 주소 계산 (base + 48 × 3172 + 오프셋)   ← 레슨 2에서 배운 그것
   ↓
② 주소를 쪼갠다: 상위 36비트=페이지번호, 하위 12비트=오프셋
   ↓
③ TLB에 페이지번호가 있나?
     있음 → 프레임번호 즉시 획득 (거의 공짜)
     없음 → 4단계 페이지 테이블 walk (메모리 4번 읽기) → TLB에 저장
   ↓
④ PTE의 권한 비트 검사 (Present? R/W? U/S? NX?)
     위반 → 폴트 → 크래시(0xC0000005) 또는 커널이 처리
   ↓
⑤ 물리 주소 = 프레임번호 + (통과한) 오프셋
   ↓
⑥ 그 물리 주소를 CPU 캐시에서 찾는다 (여기서부터 레슨 7)

6. 드릴 (답은 접기)

D1. 가상 주소 0x00007FF6A0009ABC 에서 오프셋은? 가상 페이지 번호는?
D2. 같은 페이지 안의 두 변수를 번갈아 읽는다. TLB 미스는 몇 번?
D3. 페이지 테이블을 평평하게(1단계) 만들면 왜 실용적이지 않은가 — 숫자로.
답 확인 (먼저 스스로)

D1: 오프셋 = 하위 12비트 = 16진수 마지막 3자리 = 0xABC. 가상 페이지 번호 = 나머지 = 0x7FF6A0009.
D2: 첫 접근에서 1번(콜드 미스), 그 뒤로는 0번. 같은 페이지면 번역 결과가 같으므로 TLB 엔트리 하나를 계속 재사용한다. — 이것이 "연속 메모리를 순차 접근하라"의 근거다.
D3: 페이지 번호 36비트 → 236칸 × 8바이트 = 512GB, 그것도 프로세스마다. 실제 사용량은 수 MB인데 표가 512GB인 건 말이 안 된다. 4단계 트리는 안 쓰는 가지를 생략해 수 KB로 줄인다.
