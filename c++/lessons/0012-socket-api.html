Lesson 0012 · 네트워크 · Winsock

소켓 API: 에코 서버 손코딩

모든 TCP 서버는 같은 사다리로 시작한다: WSAStartup → socket → bind → listen → accept → recv/send. 오늘의 목표: 그 사다리로 동작하는 서버를 처음부터 직접 만들어 실제 접속을 받아 본다. 미션의 첫 이정표다.

왜 이걸 배우나
CServer::Start(11~83행)와 AcceptThread(99행~)는 정확히 이 사다리다. IOCP라는 화려한 부분을 걷어내면 뼈대는 오늘 만드는 에코 서버와 같다. 뼈대를 자기 손으로 세워 본 사람만이 나중에 "IOCP가 무엇을 대체했는지"를 안다.

1. 지식 — 사다리의 각 단

왼쪽이 우리가 오늘 쓸 호출, 오른쪽이 실제 CServer의 같은 줄이다:

호출하는 일CServer.cpp
WSAStartup(MAKEWORD(2,2), &wsa)Winsock 라이브러리 초기화 (Windows 전용 의식)17행
socket(AF_INET, SOCK_STREAM, 0)소켓 생성: IPv4(AF_INET) + TCP(SOCK_STREAM)22행
bind(sock, &addr, size)소켓에 (IP, 포트) 부착 — "20000번 문은 내 것"33행
listen(sock, SOMAXCONN)접속 대기 시작. 백로그 = 미처리 연결 대기열 길이40행
accept(sock, ...)완성된 연결 하나를 꺼낸다 → 새 소켓 반환109행
recv / send그 연결로 바이트 읽기/쓰기(IOCP 버전은 WSARecv/WSASend)

두 가지 통찰이 사다리보다 중요하다:

  • 소켓은 두 종류다. listen 소켓은 연결을 낳기만 하는 소켓(데이터 안 흐름), accept가 돌려준 소켓이 실제 대화 소켓이다. CServer가 m_listen_sock과 세션별 sock을 구분해 들고 있는 이유.
  • 기본 동작은 블로킹이다. accept는 접속이 올 때까지, recv는 데이터가 올 때까지 그 스레드를 재운다. CServer가 accept 전용 스레드를 따로 두는(66행) 직접적 이유다 — 메인 흐름이 accept에 갇히면 안 되니까.

2. 스킬 — 45줄 에코 서버

scratch\echo_server.cpp. TODO 두 곳은 표를 보고 직접 채워라:

#include <cstdio>
#include <winsock2.h>
#include <ws2tcpip.h>
#pragma comment(lib, "ws2_32.lib")

int main() {
    WSADATA wsa;
    WSAStartup(MAKEWORD(2, 2), &wsa);

    SOCKET listenSock = socket(AF_INET, SOCK_STREAM, 0);

    SOCKADDR_IN addr = {};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = htonl(INADDR_ANY);    // 0.0.0.0 (레슨 11)
    addr.sin_port = htons(20001);                // 왜 htons인지 말할 수 있어야 한다

    // TODO(1): bind와 listen을 호출하라. 각각 SOCKET_ERROR 검사 포함.
    //          (힌트: bind는 (SOCKADDR*)&addr 캐스팅 필요 — CServer.cpp:33)

    printf("에코 서버 대기 중 (포트 20001)...\n");

    while (true) {
        SOCKADDR_IN clientAddr; int len = sizeof(clientAddr);
        SOCKET client = accept(listenSock, (SOCKADDR*)&clientAddr, &len);
        if (client == INVALID_SOCKET) break;
        printf("접속!\n");

        char buf[512];
        while (true) {
            int n = recv(client, buf, sizeof(buf), 0);
            if (n <= 0) break;      // 0=상대가 정상 종료(FIN), 음수=오류

            // TODO(2): 받은 n바이트를 그대로 돌려보내라(send).
            //          send의 반환값(실제 보낸 바이트 수)도 확인할 것.
        }
        printf("종료 (recv 반환 관찰: 정상 종료면 0)\n");
        closesocket(client);
    }
    closesocket(listenSock);
    WSACleanup();
}
cl /EHsc /nologo echo_server.cpp && echo_server.exe

테스트 — 두 번째 터미널에서 PowerShell 한 줄 클라이언트:

$c = New-Object Net.Sockets.TcpClient("127.0.0.1", 20001); $s = $c.GetStream()
$msg = [Text.Encoding]::ASCII.GetBytes("hello server"); $s.Write($msg, 0, $msg.Length)
$buf = New-Object byte[] 512; $n = $s.Read($buf, 0, 512)
[Text.Encoding]::ASCII.GetString($buf, 0, $n)      # → "hello server"가 돌아오면 성공!
$c.Close()

성공 기준: 보낸 문자열이 그대로 돌아오고, 클라이언트를 닫으면 서버 콘솔에 "종료"가 찍힌다. 이 순간을 기억하라 — 여러분이 처음부터 손으로 만든 첫 네트워크 서버다.

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

SOCKET이라는 타입의 정체는 뭔가요?

커널이 관리하는 연결 객체를 가리키는 번호표(핸들)다. 실제 연결 상태(버퍼, 시퀀스 번호, 4튜플)는 전부 커널 안에 있고, 우리는 번호표를 제시하며 "이 연결로 보내/받아 줘"라고 부탁할 뿐이다. 파일 핸들과 같은 사상이라 Windows에서는 소켓을 CreateIoCompletionPort에 파일 핸들처럼 넘길 수 있다 — IOCP 아크의 문을 여는 열쇠가 되는 사실이다.

bind가 SOCKET_ERROR로 실패하는 흔한 이유는요?

1위는 "이미 그 포트를 누가 쓰고 있음"(WSAEADDRINUSE, 10048) — 서버를 두 번 띄웠거나 방금 죽인 서버의 연결이 TIME_WAIT(레슨 15)에 남은 경우다. netstat -ano | findstr :20001로 범인을 찾아라. CServer.cpp:36이 실패 시 WSAGetLastError()를 찍는 것도 이 진단을 위해서다. 에러 코드를 읽는 습관이 네트워크 디버깅의 절반이다.

recv가 0을 반환하는 것과 음수를 반환하는 것의 차이는요?

0 = 상대가 예의 바르게 떠남(FIN을 보내고 정상 종료 — "더 보낼 것 없음"의 신호). 음수(SOCKET_ERROR) = 사고(연결 리셋, 강제 종료 등 — WSAGetLastError로 사유 확인). 두 경우 모두 이 연결은 끝이므로 정리 경로로 보내야 한다. 이 구분은 IOCP에서도 그대로 이어진다: 완료 크기 0 = 정상 종료 통지.

send가 내가 준 것보다 적게 보낼 수도 있다고요?

그렇다. send는 "커널 송신 버퍼에 복사한 바이트 수"를 반환하는데, 버퍼가 거의 차 있으면 일부만 받아 줄 수 있다. 100바이트를 보내려 했는데 60만 반환됐다면 나머지 40은 다시 보내야 한다 — Beej's Guide의 유명한 "handling partial send()s" 문제다. 실전 서버가 세션별 SendQ를 두고 송신을 상태 기계로 관리하는 근본 이유 중 하나.

listen의 SOMAXCONN(백로그)은 정확히 뭘 제한하나요?

핸드셰이크가 끝났지만 아직 accept로 꺼내가지 않은 연결들의 대기열 길이다. 우리가 accept를 게을리하면 대기열이 차고, 그 뒤에 오는 SYN은 거절당한다. 접속 폭주(오픈 직후!)에 대비해 accept를 빨리 도는 전용 스레드를 두는 이유 — AcceptThread가 accept→세션 생성 외에 아무 일도 안 하는 것이 바로 그 설계다.

이 에코 서버에 클라이언트 두 명이 붙으면 어떻게 되나요?

안 된다 — 첫 클라이언트의 recv 루프에 갇혀 있는 동안 두 번째는 accept조차 되지 않는다(백로그에서 대기만). 직접 확인해 보라: PowerShell 클라이언트를 두 창에서 띄워 보면 두 번째가 응답을 못 받는다. 이 답답함을 몸으로 느끼는 것이 오늘의 숨은 목표다 — 레슨 14가 정확히 이 문제에서 시작한다.
