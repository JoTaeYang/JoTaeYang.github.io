상속과 virtual: 엔진이 게임을 호출하는 방법

NetLib은 FighterServer의 존재를 모른다. 그런데도 패킷이 오면 정확히 CFighterServer::OnRecv가 실행된다. 오늘의 목표: 이 마법의 정체(vtable)를 해부하고, 이 코드베이스 전체의 뼈대 설계를 읽는다.

왜 이걸 배우나
이 저장소의 가장 중요한 설계 결정 하나를 꼽으라면 class CFighterServer : public CServer다. 네트워크 엔진(재사용 가능)과 게임 로직(교체 가능)을 가르는 칼이 상속+virtual이고, 손코딩으로 재작성할 때 가장 먼저 그릴 선이다. 레슨 1의 "객체=바이트"가 여기서 vptr로 다시 등장한다.

1. 지식 — 세 단계로 이해하기

① 상속: 멤버를 물려받고, 뒤에 이어 붙는다

class CFighterServer : public CServer   // CFighterServer.h:3

메모리 관점(레슨 1의 언어로): CFighterServer 객체는 CServer 부분이 앞에, 자기 멤버(players, sectors…)가 뒤에 붙은 하나의 바이트 덩어리다. 그래서 CServer* 포인터로 CFighterServer 객체를 가리켜도 앞부분(CServer 부분)은 완벽히 유효하다 — 이것이 "자식은 부모인 척할 수 있다"의 물리적 실체다.

② virtual: 어떤 함수를 부를지 실행 시점에 정한다

CServer.h:44-48 — 엔진이 선언만 하고 구현하지 않는 다섯 함수:

virtual bool OnConnectionRequest(const WCHAR* cwIP, unsigned short sPort) = 0;
virtual void OnClientJoin(unsigned __int64 SessionID) = 0;
virtual void OnClientLeave(unsigned __int64 SessionID) = 0;
virtual void OnGameUpdate() = 0;
virtual void OnRecv(unsigned __int64 SessionID, CMessageBuffer* pBuffer) = 0;

virtual이 없는 함수 호출은 컴파일 시점에 대상이 박힌다(정적 바인딩). virtual이 있으면: 객체 맨 앞에 숨은 포인터(vptr, 8바이트)가 들어가고, 이것이 그 클래스의 함수 주소표(vtable)를 가리킨다. 호출은 "vptr을 따라가 표에서 N번째 함수를 꺼내 부른다"로 번역된다. CFighterServer 객체의 vptr은 CFighterServer의 표를 가리키므로, 엔진이 this->OnRecv(...)를 불러도 게임의 OnRecv가 실행된다. learncpp: The virtual table

③ = 0 (순수 가상): "구현은 네가 해라"라는 강제 계약

= 0은 두 가지를 뜻한다: CServer 자체로는 객체를 만들 수 없고(추상 클래스), 상속받는 쪽은 다섯 함수를 구현하지 않으면 컴파일이 안 된다. 즉 CServer는 "네트워크는 내가 다 해줄 테니, 이 다섯 순간에 뭘 할지만 네가 정해라"라는 인터페이스 계약서다.

이 구조의 이름: 템플릿 메서드 / 헐리우드 원칙

"우리를 부르지 마라, 우리가 부르겠다(Don't call us, we'll call you)." 제어의 흐름이 엔진에 있고, 게임 코드는 콜백으로 불려 나온다. 이 서버의 실제 호출 흐름:

[엔진 GameThread]  5ms마다: JoinProcess → PacketProcess → OnGameUpdate() → ...
                                │                              │
                                ▼ (vtable 경유)                 ▼ (vtable 경유)
[게임]            OnClientJoin: 캐릭터 생성·주변 통보     OnGameUpdate: 이동·프레임 처리

덕분에 NetLib은 FighterServer 없이도 컴파일되는 독립 라이브러리다(CMakeLists.txt에서 정적 lib로 분리된 이유). 채팅 서버를 만들고 싶으면 CServer를 상속해 다섯 함수만 새로 쓰면 된다.

주의: virtual의 비용
vptr 8바이트 + 호출마다 표 조회 한 번 + 인라인 최적화 방해. 그래서 이 서버는 경계에만 virtual을 쓴다 — 초당 수만 번 불리는 내부 함수(RecvMsgDiv, SendPost)는 전부 non-virtual이다. "다형성은 아키텍처 경계에, 핫패스에는 정적 호출" — 손코딩할 때 그대로 가져갈 원칙이다.

2. 스킬 — vtable을 눈으로 확인하기

scratch\virtual_check.cpp — 실행 전에 세 출력(①②③)을 예측하라:

#include <cstdio>

class Engine {
public:
    void Run() { OnTick(); }              // 엔진의 흐름: OnTick이 뭔지 모른 채 부른다
    virtual void OnTick() = 0;            // 계약
    virtual ~Engine() {}
};

class Game : public Engine {
public:
    void OnTick() override { printf("① Game::OnTick 실행!\n"); }
};

class Plain  { int x; };
class WithVT { int x; virtual void f() {} };

int main() {
    Game g;
    Engine* e = &g;      // 부모 포인터로 자식을 가리킨다
    e->Run();            // ① 어느 OnTick이 불릴까?

    printf("② sizeof(Plain)  = %zu\n", sizeof(Plain));    // int 하나면?
    printf("③ sizeof(WithVT) = %zu\n", sizeof(WithVT));   // virtual을 붙이면?
}
cl /EHsc /nologo virtual_check.cpp && virtual_check.exe

예측 포인트: ③이 ②보다 8+α 커지는 이유를 레슨 1의 정렬 규칙까지 동원해 설명해 보라 (int 4 + vptr 8 → 정렬 때문에 16). 추가 실험: OnTick의 virtual을 지우면 어떤 컴파일 에러/동작 변화가 생기는지 관찰하라.

3. 확인 퀴즈

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

protected는 뭔가요? private랑 뭐가 달라요?

private는 "나만", protected는 "나와 내 자식들만"이다. CServer.h가 정확한 용례다: SendPacket·Disconnect는 protected — 게임(자식)은 써야 하니까. 세션 배열·IOCP 핸들은 private — 자식이라도 엔진 내부를 직접 만지면 불변식이 깨지니까. 접근 지정자는 메모리에 흔적이 없다(레슨 1 질문 세트에서 확인한 그대로).

override 키워드는 꼭 써야 하나요? 이 코드는 안 쓰던데요.

문법상 선택이지만 실무에서는 필수 습관이다. override를 붙이면 "부모에 같은 시그니처의 virtual이 없을 때" 컴파일 에러가 난다 — 오타(PakcetProc_Attack1 같은!)나 인자 타입 불일치로 오버라이드인 줄 알았는데 새 함수를 만든 사고를 잡아 준다. 이 코드베이스는 C++03 스타일이라 안 썼을 뿐, 여러분이 손코딩할 때는 붙여라.

추상 클래스는 객체를 못 만든다면서 CServer의 생성은 어떻게 되는 건가요?

CServer s;는 컴파일 에러지만, CFighterServer f;를 만들면 그 안의 CServer 부분은 함께 태어난다. 생성 순서는 부모 먼저 → 자식 나중, 소멸은 역순. 즉 추상 클래스는 "단독으로는 못 태어나고, 항상 자식의 앞부분으로만 존재하는" 클래스다.

virtual ~Engine() {} 소멸자에는 왜 virtual을 붙였나요?

Engine* e = new Game; delete e; 상황 때문이다. 소멸자가 non-virtual이면 Engine의 소멸자만 불리고 Game 부분(예: Game이 소유한 힙 자원)은 정리되지 않는다 — UB이자 누수. 규칙: 부모 포인터로 delete될 가능성이 있는 클래스는 소멸자를 virtual로. 레슨 3(수명)과 이번 레슨이 만나는 지점이다.

vtable 조회가 느리다면 얼마나 느린 건가요?

메모리 읽기 한 번 + 간접 호출 한 번 — 나노초 단위다. 문제는 절대 비용이 아니라 빈도와 최적화 방해다. 5ms마다 한 번 불리는 OnGameUpdate에선 완전히 무시 가능하지만, 패킷당 수십 번 불리는 내부 루프라면 인라인이 막히는 것까지 겹쳐 체감된다. 그래서 이 서버의 원칙이 "경계에만 virtual"이다. 측정 없이 겁먹지도, 무신경하게 남발하지도 말 것.

다중 상속도 되나요? 왜 이 코드는 하나만 상속하죠?

C++은 다중 상속을 허용하지만(메모리 레이아웃이 복잡해진다 — 부모 두 벌이 이어 붙고 vptr도 여럿), 실무에서는 혼란 대비 이득이 작아 "구현 상속은 한 개, 필요하면 인터페이스(순수 가상만 있는 클래스)만 추가"가 통례다. 이 서버는 역할이 "CServer의 게임 구현" 하나뿐이므로 단일 상속이 자연스럽다.

상속 말고 콜백을 넘기는 방법(함수 포인터, std::function)도 있지 않나요?

있다. Go 포트는 실제로 클로저/인터페이스를, C# 포트는 델리게이트 스타일을 쓴다. C++에서도 std::function 5개를 등록받는 설계가 가능하다. 상속 방식의 장점은 다섯 콜백+상태(players 등)가 한 클래스로 묶이고, 순수 가상 덕에 하나라도 빠뜨리면 컴파일 에러라는 것. 어느 쪽이든 "엔진은 계약만 알고 구현은 모른다"는 본질은 같다 — 재작성 때 직접 골라 보게 될 것이다.
