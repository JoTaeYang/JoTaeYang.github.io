캐스팅 4형제: 타입을 바꿔 말하기

(SOCKADDR*)&serveraddr, (CServer*)lpParam, (long*)&recvTPS — 이 서버는 캐스팅으로 가득하다. 오늘의 목표: C 스타일 캐스팅 (T) 뒤에 숨은 네 가지 서로 다른 연산을 구별하고, 각각이 언제 안전하고 언제 위험한지 안다.

왜 이걸 배우나
저수준 코드는 타입 경계를 끊임없이 넘는다. 소켓 주소 구조체를 API에 넘길 때(SOCKADDR*), void* 컨텍스트에서 this를 되찾을 때(레슨 20), 원자 연산에 변수를 넘길 때(long*). 이 캐스팅들이 무엇을 하는지 모르면 "왜 여기선 되고 저기선 크래시하는지"를 영원히 운으로 넘긴다. 재작성 시 반드시 손대는 부분이다.

1. 지식 — (T)x 하나에 숨은 네 연산

C++은 캐스팅을 의도별로 네 개의 명명된 연산자로 나눴다. C 스타일 (T)x는 이 넷을 순서대로 시도하는 뭉뚱그린 도구다 — 편하지만 의도가 안 보이고, 실수로 위험한 것이 선택돼도 경고가 없다.

① static_cast — "관련된 타입 사이의 정직한 변환"

double d = 3.9;
int i = static_cast<int>(d);        // 3 (값을 실제로 변환)
int idx = static_cast<int>(sectors[y][x].size());   // CSector.cpp의 실제 패턴

숫자 타입 간 변환, 관련된 포인터(상속 관계의 부모↔자식) 변환 등 컴파일러가 타당함을 확인할 수 있는 변환이다. size_t→int처럼 값을 진짜로 바꾸거나(비트 재해석이 아님), 논리적으로 연결된 타입을 오간다. CSector가 size()(size_t)를 iSectorIdx(int)에 넣을 때 이 변환이 일어난다(레슨 16).

② reinterpret_cast — "이 비트를 저 타입으로 우겨 읽기"

bind(m_listen_sock, (SOCKADDR*)&serveraddr, sizeof(serveraddr));   // CServer.cpp:33
CServer* pLanServer = (CServer*)lpParam;                          // CServer.cpp:101

가장 위험하고, 저수준에서 가장 흔하다. 비트를 그대로 두고 타입 이름표만 바꾼다. 두 사례:

  • SOCKADDR 캐스팅: 소켓 API는 SOCKADDR*(일반형)를 받지만 우리는 SOCKADDR_IN*(IPv4 전용)을 갖고 있다. 두 구조체는 첫 부분 레이아웃이 호환되게 설계돼서(주소 패밀리가 맨 앞), 비트를 그대로 SOCKADDR*로 봐도 API가 올바르게 읽는다. "레이아웃 호환을 아는 사람만" 안전한 캐스팅이다.
  • void*→CServer* 복원: 레슨 20의 this 되찾기. this를 void*로 넣었다가 같은 타입으로 되돌리는 왕복이라 안전하다(넣은 것과 꺼내는 타입이 일치).

위험한 이유: 컴파일러가 맞는지 검사하지 않는다. (CCharacter*)lpParam처럼 엉뚱한 타입으로 복원해도 컴파일은 통과하고, 실행 시 남의 메모리를 CCharacter로 오독한다(레슨 3의 UB). "네가 안다고 했으니 믿는다"가 reinterpret_cast의 계약이다.

③ const_cast — "const 딱지를 떼거나 붙이기"

const char* p = ...;
char* q = const_cast<char*>(p);     // const 제거 (레슨 4의 const 계약을 벗김)

오직 const/volatile 한정자만 바꾼다. const 인터페이스를 우회해야 하는 드문 경우(주로 옛 C API와의 접점)에 쓴다. 원본이 진짜 const면 쓰기는 UB다 — 딱지만 떼는 것이지 원본의 불변성이 사라지는 게 아니다. 이 서버에서는 거의 안 쓰지만, 알아둬야 캐스팅 4형제가 완성된다.

④ dynamic_cast — "상속 계층에서 안전하게 내려가기"

CServer* base = ...;
CFighterServer* derived = dynamic_cast<CFighterServer*>(base);   // 실패 시 nullptr

유일하게 실행 시점에 타입을 검사하는 캐스팅(레슨 5의 vtable/RTTI 이용). base가 진짜 CFighterServer면 포인터를, 아니면 nullptr를 준다. 안전하지만 런타임 비용이 있고, 이 서버는 "base 포인터로 다형성을 쓰되 다운캐스트는 안 하는" 설계라 실제로는 안 쓴다. 그래도 "안전한 다운캐스트가 필요하면 이것"이라는 카드는 알아야 한다.

(CServer*)lpParam은 컴파일러가 "static_cast로 되나? 안 되면 reinterpret_cast? const_cast?"를 조용히 시도해 가장 먼저 되는 것을 고른다. 의도가 "관련 타입 변환(static)"이었는데 실수로 "비트 재해석(reinterpret)"이 선택돼도 경고가 없다. 그래서 현대 C++은 명명된 캐스팅을 권한다 — 의도가 코드에 드러나고, 위험한 것(reinterpret)이 눈에 띄며, grep으로 찾을 수 있다. 이 코드베이스는 전통적 C 스타일을 쓰지만, 당신이 재작성할 땐 명명된 캐스팅으로.

2. 스킬 — 네 캐스팅의 차이를 눈으로

scratch\cast_check.cpp:

#include <cstdio>

int main() {
    // ① static_cast: 값을 진짜로 변환
    double d = 3.9;
    printf("static:  (int)3.9 = %d  (값이 3으로 바뀜)\n", static_cast<int>(d));

    // ② reinterpret_cast: 비트는 그대로, 해석만 바꿈
    float f = 3.9f;
    unsigned int bits = *reinterpret_cast<unsigned int*>(&f);   // float의 비트를 int로 훔쳐봄
    printf("reinterpret: 3.9f의 실제 비트 = 0x%08X  (값 변환 아님!)\n", bits);

    // this 왕복 시뮬레이션 (레슨 20의 스레드 패턴)
    int original = 42;
    void* asVoid = &original;                       // 타입 지운 주소로 넣기
    int* back = reinterpret_cast<int*>(asVoid);      // 같은 타입으로 되찾기
    printf("왕복 복원: %d  (넣은 타입과 꺼낸 타입이 같아 안전)\n", *back);

    // ③ 위험 시연: 엉뚱한 타입으로 되찾으면? (개념만 — 주석 유지 권장)
    // double* wrong = reinterpret_cast(asVoid);
    // printf("%f\n", *wrong);   // int 4바이트를 double 8바이트로 오독 → 쓰레기/크래시

    // static_cast는 이런 무관한 변환을 아예 거부한다:
    // int* p = static_cast(asVoid);   // 컴파일 에러! (void*→int*는 명시적 reinterpret 필요)
    printf("\nstatic_cast로 void*→int*를 시도하면 컴파일 에러가 난다(주석 풀어 확인).\n");
}
cl /EHsc /nologo cast_check.cpp && cast_check.exe

확인: ① static은 값(3.9→3)을, reinterpret는 비트(3.9f의 IEEE754 패턴)를 본다는 차이. ② this 왕복이 같은 타입이라 안전한 것. 실험: 마지막 주석(static_cast<int*>(asVoid))을 풀어 컴파일 에러를 보라 — static_cast가 "위험한 변환은 거부"하는 안전장치임을 확인하는 것이다. 반면 reinterpret_cast는 거부하지 않는다.

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

(SOCKADDR*)&serveraddr가 왜 안전한가요? 다른 구조체인데요.

두 구조체가 레이아웃 호환이 되도록 설계됐기 때문이다. SOCKADDR(일반)과 SOCKADDR_IN(IPv4)은 맨 앞에 같은 위치·크기의 "주소 패밀리" 필드를 두도록 표준이 정했다. 소켓 API는 그 첫 필드로 실제 타입을 판별한 뒤 알맞게 읽는다. 즉 "겉은 일반형으로 받고, 속은 패밀리 필드로 구분"하는 C 시대의 다형성 기법이다. 레이아웃 호환이 보장된 경우에만 이 reinterpret가 안전하다 — 아무 구조체나 이렇게 캐스팅하면 UB다.

InterlockedIncrement((long*)&recvTPS) 처럼 (long*)로 캐스팅하는 건 뭔가요?

원자 연산 API가 long*를 요구하는데 변수 타입이 살짝 달라(예: unsigned int) 비트가 같은 타입으로 맞춰주는 reinterpret다. long과 unsigned int가 x64 Windows에서 둘 다 4바이트라 비트 해석이 일치해 동작한다. 다만 이건 "크기·표현이 같음을 아는" 저수준 트릭이라, 엄밀히는 이식성 논란이 있다. 기둥 E(원자 연산)에서 이 캐스팅이 왜 필요하고 어디까지 안전한지 정면으로 다룬다 — 지금은 "API 시그니처에 타입을 맞추는 reinterpret"로만 이해하면 된다.

static_cast와 C 스타일 (int)는 결과가 같은데 왜 굳이 긴 걸 쓰나요?

숫자 변환에선 결과가 같다. 차이는 포인터·타입 오용을 막는 데 있다. (T*)는 무관한 포인터 변환(위험한 reinterpret)도 조용히 해버리지만, static_cast<T*>는 관련 없는 포인터 변환을 컴파일 에러로 거부한다(실습에서 확인). 즉 static_cast는 "안전한 변환만 허용하는 좁은 문"이라, 실수로 위험한 캐스팅을 하면 컴파일러가 막아준다. 안전한 자리에도 습관적으로 명명 캐스팅을 쓰면, 위험한 자리에서 reinterpret_cast가 눈에 확 띈다.

dynamic_cast는 안전하다면서 왜 이 서버는 안 쓰나요?

필요가 없게 설계됐기 때문이다. 이 서버는 CServer(base)의 virtual 콜백으로 게임 코드를 부를 뿐(레슨 5), "base 포인터를 CFighterServer로 되돌려서 게임 전용 기능을 호출"하는 다운캐스트를 하지 않는다. 다형성을 올바로 쓰면 다운캐스트가 거의 불필요해진다 — dynamic_cast가 잦다면 설계에 냄새가 난다는 신호이기도 하다. 게다가 런타임 검사 비용과 RTTI 활성화가 필요해 고성능 코드는 꺼리는 편이다.

왜 이렇게 캐스팅이 많죠? 애초에 안 넘나들면 안 되나요?

저수준 서버의 숙명이다. C API(소켓, 스레드, 원자 연산)와 C++ 객체 세계가 만나는 경계에서 캐스팅이 발생한다 — void* 콜백(레슨 20), 일반형 구조체(SOCKADDR), API 시그니처 맞추기(long*). 상위 응용 코드(게임 로직)에는 캐스팅이 거의 없는 걸 보라. 캐스팅의 밀도는 "이 코드가 얼마나 저수준인가"의 지표다. 그래서 캐스팅을 경계에 격리하고 그 안쪽은 타입 안전하게 유지하는 게 좋은 설계다.

reinterpret_cast로 만든 포인터로 값을 읽는 게 항상 되나요?

아니다. 두 함정이 있다. ① 정렬(레슨 1): 4바이트 정렬 안 된 주소를 int*로 읽으면 일부 상황에서 문제. ② strict aliasing 규칙: 컴파일러는 "서로 다른 타입의 포인터는 같은 메모리를 안 가리킨다"고 가정해 최적화하는데, reinterpret로 이를 어기면 최적화가 코드를 깨뜨릴 수 있다(실습의 float 비트 훔쳐보기는 사실 엄밀히는 이 규칙 위반 — 실무에선 memcpy가 정석). 저수준 트릭은 "동작하는 것처럼 보여도 표준상 회색지대"인 경우가 있어, 경계를 아는 게 중요하다. 기둥 E에서 memory model과 함께 재론한다.
