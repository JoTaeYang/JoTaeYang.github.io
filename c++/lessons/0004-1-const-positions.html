Lesson 0004-1 · 파생 심화 · 레슨 4에서 파생 (질문: "const 위치에 따라 의미가 달라지는 문법은 알 필요가 없나?")

const의 위치가 바꾸는 것

알아야 한다 — 그것도 필수다. NetLib 시그니처의 절반이 const WCHAR*다. 이 문법을 못 읽으면 "누가 무엇을 못 바꾸는지"라는 계약을 못 읽는다. 규칙은 딱 하나다.

단 하나의 규칙
포인터 선언에서 *를 찾아라. const가 *의 왼쪽에 있으면 → 가리키는 대상이 const(*p를 못 바꿈). const가 *의 오른쪽에 있으면 → 포인터 자신이 const(p를 다른 곳으로 못 옮김).

세 가지 경우 — 같은 부품, 다른 계약

const WCHAR* p;    // ① *의 왼쪽 const → 대상이 const
WCHAR* const p;    // ② *의 오른쪽 const → 포인터가 const
const WCHAR* const p; // ③ 양쪽 → 둘 다 const
바꿀 수 있나: p[0] = 'x'
(대상 수정)		바꿀 수 있나: p = other
(포인터 재지정)
① const WCHAR* p❌ 불가 (대상 const)✅ 가능
② WCHAR* const p✅ 가능❌ 불가 (포인터 const)
③ const WCHAR* const p❌ 불가❌ 불가

const WCHAR*와 WCHAR const*는 완전히 같다. 컴파일러는 const를 바로 왼쪽 것에 붙이는데, 맨 앞에 오면 예외적으로 오른쪽(WCHAR)에 붙인다. 그래서 둘 다 "대상이 const". 헷갈리면 머릿속에서 const를 * 바로 왼쪽으로 옮겨 읽어라 — 의미가 안 변한다.

왜 NetLib은 ①(const WCHAR*)만 쓰는가

bool CServer::Start(const WCHAR* cwIP, ...);   // CServer.cpp:11
virtual bool OnConnectionRequest(const WCHAR* cwIP, ...) = 0;  // CServer.h:44

const WCHAR* cwIP는 호출자에게 주는 서면 약속이다: "네 문자열의 주소만 받아서(8바이트, 복사 안 함) 읽기만 하고 절대 안 고친다." 두 가지 이득:

  • 안전: 함수가 실수로 호출자의 문자열을 훼손하는 걸 컴파일러가 막는다. 계약 위반 시 빌드가 멈춘다 — 런타임 버그가 아니라.
  • 유연: 문자열 리터럴 L"127.0.0.1"은 그 자체가 읽기 전용이다. 파라미터가 const여야 리터럴을 넘길 수 있다. WCHAR*(non-const)였다면 리터럴 전달이 거부된다.

포인터 자신(cwIP)이 const일 필요는 없다 — 함수 안에서 cwIP++로 훑든 말든 호출자와 무관하니까. 그래서 ②·③이 아니라 ①이다. 계약은 "대상 불변"이면 충분하다.

리턴에서의 const — GetSector의 두 겹

const std::vector<CCharacter*>& GetSector(int iX, int iY)  // CSector.h:24

여기 const가 두 가지를 동시에 말한다:

  • &(참조 리턴): 벡터를 복사하지 않는다. 섹터에 100명이 있어도 800바이트 복사가 아니라 내부 벡터를 그대로 빌려준다.
  • const: 빌려준 그 벡터에 원소를 추가·삭제 못 한다. 호출자가 서버 내부 상태를 건드리는 걸 막는 울타리.
함정 — const는 한 겹만 막는다
const vector<CCharacter*>&에서 const가 지키는 건 벡터다. 그 안의 CCharacter*가 가리키는 캐릭터는 여전히 수정 가능하다 — GetSector(...)[0]->SetHP(...)는 컴파일된다. const는 "벡터에 손대지 마"이지 "캐릭터에 손대지 마"가 아니다. (레슨 4의 for (CCharacter* pOther : GetSector(...)) 순회가 이래서 성립한다.)

곁가지 — 앞으로 만날 두 형태 (지금은 눈만)

  • 값 파라미터의 const: void f(const int x) — 어차피 복사본이라 호출자에겐 무의미. 그래서 시그니처엔 거의 안 쓴다. const가 의미를 갖는 건 포인터·참조 파라미터일 때다 (레슨 4의 주제와 정확히 연결).
  • 멤버 함수 뒤 const: int GetSize() const — "이 메서드는 객체 자신을 안 바꾼다"는 약속. 이 저장소엔 아직 없지만 표준 라이브러리 도처에 있다. 레슨 16(vector 내부)에서 다시 만난다.

드릴 — 각 시그니처가 "무엇을 못 하게" 막는가 (답은 접기)

void a(const BYTE* p);      // CSysLog: 무엇이 불변?
void b(IN_ADDR* pAddr);     // CServer::DomainToIP의 두 번째 인자: 왜 const가 없나?
CCharacter* const c = &players[0];  // c에 무엇을 못 하나?
답 확인 (먼저 스스로 답할 것)

a: * 왼쪽 const → 가리키는 바이트들이 불변. 함수는 버퍼를 읽어 로그로 찍기만 하고 원본을 안 고친다. (포인터 p는 함수 안에서 옮겨도 됨.)
b: DomainToIP(const WCHAR* cwIP, IN_ADDR* pAddr)에서 pAddr는 일부러 non-const — 이 함수의 목적이 결과(찾은 IP)를 그 주소에 써주는 것이기 때문. const였다면 결과를 못 쓴다. const의 유무가 곧 "입력이냐 출력이냐"를 말한다.
c: * 오른쪽 const → 포인터 c를 다른 데로 못 옮긴다(c = &players[1] 불가). 하지만 c->SetHP(0)은 가능 — 대상은 const가 아니니까.
