static의 네 가지 얼굴

static은 C++에서 가장 문맥에 따라 뜻이 갈리는 키워드다. 오늘의 목표: 이 서버에 실제로 쓰인 네 가지 static을 구별하고, "static 스레드 함수에 왜 this를 인자로 넘기는가"라는 코드베이스의 관례를 완전히 설명한다.

왜 이걸 배우나
CServer.cpp:72에 이런 주석이 있다: "this 넘겨주는 이유: static 함수는 클래스 멤버 변수에 접근할 수 없으므로 넘겨준다." 이 한 줄을 진짜로 이해하려면 static의 얼굴 넷 중 둘(멤버 함수, 멤버 변수)을 알아야 한다. CMessageBuffer::pool이 왜 모든 버퍼가 공유하는 단 하나의 풀인지도 여기서 풀린다. 배치 4를 닫는 통합 레슨이다.

1. 지식 — 위치가 뜻을 바꾼다

static은 어디에 붙느냐에 따라 완전히 다른 네 가지를 뜻한다. 헷갈리는 이유이자, 하나씩 떼면 쉬운 이유다.

얼굴 ① 정적 멤버 변수 — 클래스당 하나, 모두가 공유

보통 멤버 변수는 객체마다 따로다(레슨 1: 각 CCharacter가 자기 shX를 가짐). static 멤버는 반대다 — 클래스 전체에 딱 하나, 모든 객체가 공유한다. CMessageBuffer.h:159:

static CMemoryPool<CMessageBuffer>* pool;   // 모든 CMessageBuffer가 공유하는 단 하나의 풀

버퍼를 100만 개 만들어도 풀은 하나다. 당연하다 — 풀의 존재 이유가 "모두가 같은 창고에서 빌리고 반납"이니까(레슨 3). 객체마다 풀이 따로면 공유의 의미가 없다. GetAllocCount()가 전체 발급 수를 세는 것도 pool이 하나라서 가능하다.

주의: static 멤버는 선언과 별도로 .cpp에 정의가 한 번 있어야 한다(CMemoryPool<...>* CMessageBuffer::pool = ...;). "어느 객체에도 안 속하니 저장 공간을 따로 잡아줘야 한다"는 뜻 — 레슨 22(링크)에서 왜인지 완전히 다룬다.

얼굴 ② 정적 멤버 함수 — this가 없다

보통 멤버 함수는 숨은 this를 받는다(레슨 1 질문 세트: obj.f() → f(&obj)). static 멤버 함수는 this가 없다 — 특정 객체에 속하지 않고 클래스에 속한다. 그래서 멤버 변수에 접근할 수 없다(어느 객체의 변수인지 모르니까).

CServer.h:99-102의 스레드 함수들이 전부 static인 이유가 이것이다:

static unsigned int WINAPI AcceptThread(LPVOID lpParam);
static unsigned int WINAPI WorkerThread(LPVOID lpParam);

왜 반드시 static이어야 하나? _beginthreadex(Windows API)는 스레드 시작 함수로 정해진 모양의 함수 포인터(unsigned(__stdcall*)(void*))를 요구한다. 그런데 일반 멤버 함수는 숨은 this 때문에 그 모양과 맞지 않는다 — OS는 this를 넣어줄 방법을 모른다. static 멤버 함수는 this가 없어 모양이 딱 맞는다.

①과 ②의 결합 — 코드베이스의 그 관례

이제 그 주석이 완전히 읽힌다. 스레드 함수는 static이어야 해서 this가 없다. 하지만 스레드 안에서 세션 배열·IOCP 핸들 같은 멤버 변수는 써야 한다. 해법: this를 인자로 직접 넘긴다.

// 스레드 생성 시 — 마지막 직전 인자로 this를 밀어넣는다 (CServer.cpp:66)
hAcceptThread = (HANDLE)_beginthreadex(NULL, 0, AcceptThread, (LPVOID)this, 0, NULL);
                                                              └── this를 LPVOID로 ──┘

// static 함수 안에서 — 받은 인자를 CServer*로 복원해 멤버에 접근 (CServer.cpp:101)
unsigned int WINAPI CServer::AcceptThread(LPVOID lpParam) {
    CServer* pLanServer = (CServer*)lpParam;   // 잃어버린 this를 손으로 되찾는다
    ... pLanServer->m_listen_sock ...          // 이제 멤버 접근 가능
}

OS API에 콜백을 넘길 때의 고전적 관용구다: "static 함수 + void*로 컨텍스트(this) 전달". C 스타일 콜백과 C++ 객체를 잇는 다리다. Go/C# 포트는 클로저·델리게이트로 이 다리를 자동으로 놓지만, C++에서는 손으로 놓는다.

얼굴 ③ 정적 지역 변수 — 함수 안에서 살아남는다

함수 지역 변수는 함수가 끝나면 사라진다(레슨 3: 스택). 앞에 static을 붙이면 스택이 아니라 전역 영역에 살고, 함수 호출 사이에 값이 유지되며, 최초 한 번만 초기화된다.

int nextId() {
    static int counter = 0;   // 최초 호출 때 딱 한 번 0으로 초기화
    return ++counter;         // 호출할 때마다 1, 2, 3... (값이 살아남는다)
}

이 서버가 직접 쓰진 않지만, 알아야 하는 이유: C++11부터 정적 지역 변수의 초기화는 스레드 안전이 보장된다 — 멀티스레드에서 "한 번만 초기화"가 지켜진다(싱글턴 관용구의 기반). 기둥 E에서 초기화 경쟁을 이야기할 때 되돌아올 개념이다.

얼굴 ④ 파일 스코프 static — 이 파일 안에서만 (내부 링키지)

함수나 전역 변수 앞의 static은 "이 이름을 이 .cpp 파일 밖에서 못 보게 숨긴다"는 뜻이다(내부 링키지). 다른 파일의 같은 이름과 충돌하지 않는다. 이 개념은 코드베이스 지도가 언급한 실제 사건과 닿아 있다: Profile.cpp와 ProfileTLS.cpp가 같은 이름의 심볼을 정의해 중복 심볼 링크 에러가 났고, 그래서 하나(Profile.cpp)를 빌드에서 뺐다. 만약 그 함수들이 파일 스코프 static이었다면 애초에 충돌하지 않았을 것이다 — static의 부재가 낳은 문제다. 링크의 세계는 레슨 22에서 정면으로 다룬다.

얼굴위치뜻이 서버의 예
① 멤버 변수클래스 안클래스당 하나, 공유CMessageBuffer::pool
② 멤버 함수클래스 안this 없음AcceptThread 등 스레드 함수
③ 지역 변수함수 안호출 간 값 유지, 1회 초기화(직접 사용 없음 — 개념)
④ 파일 스코프함수/전역 앞파일 밖에서 안 보임(Profile 중복 심볼의 반례)

2. 스킬 — 네 얼굴을 한 프로그램에서 확인

scratch\static_check.cpp:

#include <cstdio>

struct Counter {
    static int total;        // 얼굴 ①: 모든 Counter가 공유
    int id;
    Counter() { id = ++total; }             // 태어날 때마다 공유 total 증가

    static void Report() {   // 얼굴 ②: this 없음 → id(멤버)엔 접근 불가!
        printf("지금까지 생성된 총 개수: %d\n", total);   // static 멤버는 OK
        // printf("%d", id);  // ← 주석 풀면 컴파일 에러: this가 없어 id에 접근 불가
    }
};
int Counter::total = 0;      // 얼굴 ①의 정의 (파일에 딱 한 번)

int tick() {
    static int n = 0;        // 얼굴 ③: 호출 간 유지
    return ++n;
}

int main() {
    Counter a, b, c;
    Counter::Report();                       // 3 — 객체 없이 클래스 이름으로 호출
    printf("a.id=%d b.id=%d c.id=%d\n", a.id, b.id, c.id);   // 1 2 3 (각자 다름)

    printf("tick: %d %d %d\n", tick(), tick(), tick());       // 1 2 3 (n이 살아남음)
    // TODO: Report()가 id를 못 찍는 이유를 한 문장으로 주석에 적어보라.
}
cl /EHsc /nologo static_check.cpp && static_check.exe

확인: ① Report()를 객체 없이 Counter::로 부를 수 있는가(멤버 함수인데!). ② total은 하나인데 id는 객체마다 다른가. ③ tick이 1,2,3으로 값을 유지하는가. 실험: Report 안의 id 주석을 풀고, 나는 에러 메시지가 "this가 없다"는 취지인지 확인하라 — 그 에러가 "왜 스레드 함수에 this를 인자로 넘겨야 하는가"의 직접 증거다.

왜 하나의 키워드에 뜻을 네 개나 몰아넣었나요? 헷갈리게.

역사적 부채다. C에서 물려받은 static(얼굴 ③④)에 C++이 클래스 개념(얼굴 ①②)을 얹으며 같은 키워드를 재활용했다. 관통하는 느슨한 공통점은 "정적 저장 기간" 또는 "제한된 범위"지만, 실무에서는 위치로 구별하는 게 빠르다: 클래스 안 변수/함수냐, 함수 안 변수냐, 파일 스코프냐. 이 셋만 물으면 즉시 갈린다.

static 멤버 함수는 그럼 일반 함수랑 뭐가 다른가요?

거의 같다 — this가 없다는 점에서. 차이는 이름 공간과 접근 권한이다: Counter::Report처럼 클래스 이름에 소속되고, 클래스의 private static 멤버(total)에 접근할 수 있다. "특정 객체와 무관하지만 이 클래스에 논리적으로 속하는 기능"에 쓴다. 스레드 함수를 전역 함수가 아니라 static 멤버로 둔 것도 "CServer의 일부"라는 소속을 표현하면서 private 멤버 접근 권한을 얻기 위해서다.

this를 LPVOID로 캐스팅해 넘기는데, 이게 안전한가요?

이 패턴에서는 안전하다. this(CServer*)를 void*로 넣었다가 같은 CServer*로 되돌리는 왕복이라, 원래 타입으로 정확히 복원된다(레슨 2: 포인터는 주소일 뿐, void*는 "타입 지운 주소"). 위험은 엉뚱한 타입으로 복원할 때뿐인데, 여기선 넣는 쪽과 받는 쪽이 같은 코드라 타입이 보장된다. 레슨 11(캐스팅 4형제, 배치 5 예정)에서 이 왕복의 안전 조건을 형식적으로 다룬다.

정적 지역 변수(얼굴 ③)와 전역 변수는 뭐가 다르죠? 둘 다 안 사라지는데.

수명은 같지만(프로그램 끝까지) 가시성이 다르다. 전역 변수는 아무 데서나 접근 가능하지만, 정적 지역 변수는 그 함수 안에서만 보인다 — "값은 유지하되 남이 못 건드리게 캡슐화"다. tick()의 counter를 전역으로 빼면 아무나 조작할 수 있어 버그 표면이 넓어진다. "필요한 범위로 좁혀라"는 원칙(레슨 5의 private, 레슨 6의 스레드 로컬)의 또 다른 형태다.

멤버 함수 대신 람다(closure)를 스레드에 넘기면 this 문제가 없어지지 않나요?

현대 C++(std::thread + 람다)에서는 그렇다 — 람다가 this를 캡처해 자동으로 다리를 놓아준다. 하지만 이 코드는 Windows의 _beginthreadex라는 C API를 직접 쓰고, 그건 순수 함수 포인터만 받으므로 람다 캡처를 못 받는다(캡처 없는 람다만 함수 포인터로 변환 가능). 그래서 "static 함수 + void* 컨텍스트"라는 C식 다리가 필요하다. 저수준 OS API와 붙는 코드에서 자주 만나는 제약이다 — IOCP(기둥 D)에서도 같은 패턴을 다시 본다.

배치 4를 다 봤는데, 이제 뭘 할 수 있게 된 건가요?

이 서버의 부품 클래스들이 문법적으로 읽히는 상태가 됐다: vector 기반 컨테이너(16), 자원 소유 클래스와 복사 정책(17), 직렬화 연산자(18), 템플릿 컨테이너(19), static 멤버·스레드 함수 관례(20). 즉 CMessageBuffer·CRingBuffer·CLockFree* 의 선언과 구조가 눈에 들어온다. 아직 못 하는 것: 그 안의 CAS·메모리 오더링(기둥 E), IOCP 이벤트 흐름(기둥 D). 다음 배치는 캐스팅·컴파일/링크 모델로 저수준 준비를 마저 한 뒤 CS 심화로 넘어간다.
