CPU 캐시와 지역성: 섹터가 list를 버린 이유

RAM 접근은 CPU에게 수백 사이클짜리 여행이다. 오늘의 목표: 캐시라인 개념으로 이 서버가 섹터 컨테이너를 std::list에서 std::vector로 바꾼 결정을 스스로 정당화하고, 그 차이를 직접 측정한다.

왜 이걸 배우나
같은 O(N)인데 10배 느린 코드가 있다 — Big-O가 안 알려주는 부분이 캐시다. 이 코드베이스에는 그 증거가 박제되어 있다: 코드베이스 지도의 "섹터 list→vector 마이그레이션", 그리고 CMessageBuffer 주석의 "참조 카운트를 별도 힙에서 멤버로 옮겼다(캐시 지역성)". 손코딩 재작성 때 여러분이 똑같이 내려야 할 종류의 결정이다.

1. 지식 — 계층, 라인, 지역성

메모리는 계층이다

계층대략적 지연비유 (L1=1초라면)
L1 캐시 (~32KB)~4 사이클1초 — 책상 위
L2 캐시 (~256KB+)~12 사이클3초 — 책상 서랍
L3 캐시 (수 MB)~40 사이클10초 — 책장
RAM~200+ 사이클1분 — 도서관 왕복

CPU는 RAM에서 직접 일하지 않는다. 필요한 데이터를 캐시로 가져와서 일한다. CS:APP 6장이 이 표의 원전이다.

캐시라인: 운반 단위는 64바이트

캐시는 데이터를 1바이트씩이 아니라 64바이트 캐시라인 단위로 가져온다. players[i].shX 한 필드(2바이트)를 읽는 순간, 그 주변 64바이트 — 사실상 CCharacter 하나(48바이트, 레슨 1)의 대부분 — 가 통째로 L1에 올라온다. 직후 players[i].shY를 읽으면? 공짜다.

지역성: 캐시가 보상하는 두 가지 행동

  • 공간 지역성: 방금 읽은 곳 근처를 읽는다 → 같은 캐시라인, 공짜. 배열 순회가 빠른 이유.
  • 시간 지역성: 방금 읽은 곳을 또 읽는다 → 아직 캐시에 있음, 공짜. 루프 변수가 공짜인 이유.

여기에 보너스: 메모리를 순서대로 읽으면 하드웨어 프리페처가 패턴을 감지하고 다음 라인을 미리 가져다 놓는다. 배열 순회는 사실상 RAM 지연을 숨긴 채 달린다.

사례 해부: 섹터의 list → vector

CSector.cpp의 현재 모습:

std::vector<CCharacter*>& list = sectors[curshY][curshX];
CCharacter* pMoved = list.back();          // swap-remove: 마지막을 내 자리로
list[pCharacter->iSectorIdx] = pMoved;     // O(1), 그리고 전부 연속 메모리
list.pop_back();
std::list (연결 리스트)std::vector (동적 배열)
메모리 배치노드가 힙에 뿔뿔이 → 순회마다 캐시 미스 행진포인터들이 연속 → 라인 하나에 8개씩
순회(브로드캐스트마다!)노드당 포인터 추적 = 도서관 왕복 N번프리페처가 미리 날라줌
중간 삭제O(1) — 그러나 그 노드를 찾는 게 이미 비쌈원래 O(N)… swap-remove로 O(1)

핵심 통찰: list의 유일한 장점(중간 삭제 O(1))을 순서를 포기하는 대가로(swap-remove + iSectorIdx로 내 위치 기억) vector에서도 얻어냈다. 섹터 안 캐릭터의 순서는 아무 의미가 없으니 공짜 거래다. 이것이 자료구조 선택의 실전 문법이다: "필요 없는 성질(순서)을 팔아 필요한 성질(지역성)을 산다."

2. 스킬 — 캐시 미스를 측정하기

scratch\cache_check.cpp — 같은 덧셈 6400만 번, 순서만 다르다:

#include <cstdio>
#include <windows.h>

const int N = 8000;                 // 8000×8000 int = 256MB … 캐시보다 훨씬 크게
static int g_m[N][N];               // 전역(커밋은 첫 접근 때 — 레슨 6!)

int main() {
    long long sum = 0; DWORD t;

    t = GetTickCount();
    for (int r = 0; r < N; r++) for (int c = 0; c < N; c++) sum += g_m[r][c];
    printf("행 우선(연속 접근):   %lu ms\n", GetTickCount() - t);

    t = GetTickCount();
    for (int c = 0; c < N; c++) for (int r = 0; r < N; r++) sum += g_m[r][c];
    printf("열 우선(32KB 점프):  %lu ms\n", GetTickCount() - t);

    return (int)(sum & 1);          // 최적화로 루프가 날아가는 것 방지
}
cl /EHsc /O2 /nologo cache_check.cpp && cache_check.exe

행 우선은 캐시라인의 16개 int를 다 쓰고 넘어가지만, 열 우선은 라인마다 int 하나만 쓰고 버린다. 몇 배 차이가 나는지 기록해 두라 — 같은 연산 횟수, 같은 Big-O다.

캐시는 내가 코드로 제어할 수 있나요?

직접 명령할 수는 없다(하드웨어가 자동 관리). 우리가 할 수 있는 것은 캐시가 좋아하는 접근 패턴을 만드는 것뿐이다: 연속 배치(vector), 순차 순회, 자주 같이 쓰는 데이터를 한 구조체에(CMessageBuffer가 refcount를 멤버로 들인 이유), 안 쓰는 데이터를 핫 루프에서 치우기. "제어 불가, 그러나 유혹 가능"이라 기억하라.

vector의 중간 삭제가 O(N)이라는 게 정확히 왜죠?

중간 원소를 빼면 순서를 유지하기 위해 뒤의 원소를 전부 한 칸씩 앞으로 복사해야 하기 때문이다. swap-remove는 그 전제를 뒤집는다: 순서 유지를 포기하면 "마지막 원소를 빈자리에 꽂고 크기만 줄이는" 두 번의 대입으로 끝난다. 단, 옮겨진 원소의 위치가 바뀌므로 iSectorIdx처럼 각자 자기 위치를 기억시켜야 한다 — CSector.cpp:25가 정확히 그 갱신이다.

레슨 1의 "멤버 순서만 바꾸면 40바이트" 떡밥이 캐시랑 무슨 상관인가요?

이제 회수할 수 있다. 48바이트 CCharacter는 64바이트 라인에 1개 + 다음 것이 걸쳐 들어간다. 40바이트로 줄이면 캐릭터가 라인 경계에 걸치는 빈도가 줄고, 배열 전체 순회 시 읽는 라인 수도 준다. 더 나아가 진짜 고수의 수는 핫 필드 분리다: 매 프레임 만지는 shX/shY만 별도 배열로 빼면(SoA 구조) 이동 처리 루프의 캐시 효율이 극대화된다. MMO 서버들이 실제로 쓰는 기법이다.

포인터의 배열(vector<CCharacter*>)이면 결국 따라간 곳은 흩어져 있잖아요?

날카로운 지적이다. 포인터 목록은 연속이지만 역참조 대상은 점프다. 다만 이 서버의 대상은 players라는 하나의 연속 배열 안에 있으므로 완전 무작위 힙 노드보다 훨씬 낫다(같은 페이지·인접 라인에 몰려 있을 확률이 높다). 완벽하진 않지만 list(노드 자체도, 연결도 흩어짐)와는 급이 다르다. "한 단계 간접 참조까지 계산에 넣는" 이 사고방식이 서버 성능 감각의 핵심이다.

멀티스레드에서 캐시는 어떻게 되나요? 코어마다 따로 있잖아요.

L1/L2는 코어마다 따로 있고, 하드웨어가 코어 간 일관성을 자동으로 맞춘다(캐시 일관성 프로토콜). 문제는 그 비용이다: 두 코어가 같은 캐시라인을 서로 쓰면 라인이 코어 사이를 핑퐁친다 — 서로 다른 변수여도 같은 64바이트에 있으면 당한다(거짓 공유, false sharing). 이 서버의 ProfileTLS가 스레드별 배열을 따로 두는 이유가 정확히 이것이다. 멀티스레드 아크에서 정면으로 다룬다.

실무에서 캐시 미스를 어떻게 확인하죠? 감으로 하나요?

측정한다. 오늘 실습처럼 시간 비교가 1차 도구이고, 본격적으로는 VTune·perf 같은 프로파일러가 캐시 미스 카운터(하드웨어가 세 준다)를 보여준다. 이 코드베이스의 ProfileTLS도 "어디가 느린가"를 찾는 1차 도구다. 원칙: 추측 금지, 측정 우선 — Bench/RESULTS.md가 그 문화의 증거물이다.
