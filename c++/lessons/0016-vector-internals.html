std::vector 내부: capacity, 재할당, 무효화

vector는 "마법의 늘어나는 배열"이 아니다. 포인터 하나, 크기 둘로 된 아주 단순한 구조체다. 오늘의 목표: 그 세 필드를 직접 그리고, 이 서버의 섹터 코드에 숨은 무효화 함정을 찾아낸다.

왜 이걸 배우나
CSector는 33×43개 섹터마다 std::vector<CCharacter*>를 두고, 매 프레임 브로드캐스트로 순회한다(레슨 7의 캐시 이야기). vector가 언제 통째로 이사하는지(재할당), 그때 붙잡고 있던 포인터·인덱스가 어떻게 썩는지(무효화)를 모르면 "가끔 크래시하는" 지옥의 버그를 심게 된다. swap-remove가 왜 그렇게 짜였는지도 여기서 완성된다.

1. 지식 — vector의 정체는 세 필드다

포인터 + size + capacity

vector는 개념적으로 이것뿐이다:

template<typename T>
class vector {
    T*     _data;      // 힙에 할당된 배열의 시작 (레슨 3: new[])
    size_t _size;      // 지금 들어있는 원소 수
    size_t _capacity;  // 재할당 없이 담을 수 있는 최대치
};

size ≤ capacity가 핵심 불변식이다. size()는 "몇 개 있나", capacity()는 "몇 개까지 견디나". 둘은 다르다 — 이 구분을 놓치면 vector를 영원히 오해한다.

재할당(reallocation): 꽉 차면 통째로 이사한다

push_back인데 size == capacity라면? vector는:

  1. 더 큰 배열을 힙에 새로 할당한다 (보통 capacity를 1.5~2배로 — MSVC는 1.5배)
  2. 기존 원소를 전부 새 배열로 복사(또는 이동)한다
  3. 옛 배열을 delete[]한다
  4. _data가 새 주소를 가리키게 바꾼다

2배씩 늘리는 이유(분할 상환): N번 push_back의 총 복사 비용이 O(N)에 수렴한다 — 매번 1개씩 늘리면 O(N²)가 된다. 이것이 "vector의 push_back은 평균 O(1)"의 진짜 의미다. cppreference: vector

재할당 = 대이동 = 무효화
3단계에서 옛 배열이 사라진다. 그러면 재할당 전에 얻어둔 모든 것이 썩는다: 원소를 가리키던 포인터·참조·iterator 전부가 댕글링(레슨 3)이 된다. 이것이 vector의 제1함정, "iterator 무효화"다.

이 서버는 이 함정을 어떻게 피했나 — 두 겹의 방어

방어 1: 원소가 포인터다. CSector의 vector는 CCharacter가 아니라 CCharacter*를 담는다(레슨 2·7). 재할당이 일어나도 이사하는 건 8바이트 주소들이고, 캐릭터 실물은 players 배열에 그대로 있다. 즉 "섹터 vector가 재할당돼도 캐릭터 객체 자체의 주소는 안 변한다." players 배열로 캐릭터를 가리키는 다른 코드들이 안전한 이유다.

방어 2: iterator 대신 인덱스를 저장한다. 그런데 문제가 하나 더 있다. 재할당되면 vector 안에서의 위치를 가리키던 iterator는 썩는다. 그래서 CCharacter는 자기 위치를 iterator가 아니라 정수 인덱스 iSectorIdx로 기억한다:

// SectorAddCharacter — 넣을 때 내 인덱스를 스스로 기록
pCharacter->iSectorIdx = (int)sectors[shY][shX].size();   // 새 원소는 맨 끝
sectors[shY][shX].push_back(pCharacter);

인덱스 3은 재할당 후에도 "새 배열의 3번째"로 유효하다 — 주소가 아니라 순번이니까. iterator 무효화를 인덱스로 우회한 것이다.

swap-remove 재방문 — 이제 완전히 읽힌다

void CSector::SectorRemoveCharacter(CCharacter* pCharacter) {
    std::vector<CCharacter*>& list = sectors[curshY][curshX];
    CCharacter* pMoved = list.back();          // 마지막 원소
    list[pCharacter->iSectorIdx] = pMoved;      // 내 자리에 마지막을 덮어씀
    pMoved->iSectorIdx = pCharacter->iSectorIdx; // 옮겨간 놈에게 새 인덱스 통보!
    list.pop_back();                            // 크기 1 감소 (capacity는 그대로)
}

레슨 7에서는 "순서를 포기해 O(1) 삭제를 샀다"고 했다. 이제 세 번째 줄의 진짜 의미가 보인다: 마지막 원소를 옮겼으면 그 원소의 iSectorIdx도 갱신해야 한다. 안 하면 pMoved는 자기가 아직 맨 끝에 있다고 착각하고, 다음 삭제 때 엉뚱한 자리를 건드린다. 인덱스로 위치를 관리하는 설계의 대가이자 규율이다. 그리고 pop_back은 size만 줄이고 capacity는 안 건드린다 — 그래서 반복 입퇴장에도 재할당이 거의 안 일어난다(캐시 친화, 레슨 7).

2. 스킬 — 재할당과 무효화를 눈으로 목격하기

scratch\vector_check.cpp — 각 push_back의 capacity 변화를 예측하고 실행하라:

#include <cstdio>
#include <vector>

int main() {
    std::vector<int> v;
    printf("초기: size=%zu cap=%zu\n", v.size(), v.capacity());

    int prevCap = -1;
    for (int i = 0; i < 10; i++) {
        v.push_back(i);
        if ((int)v.capacity() != prevCap) {           // capacity가 바뀐 순간 = 재할당
            printf("push %2d → size=%2zu cap=%2zu  ← 재할당!\n", i, v.size(), v.capacity());
            prevCap = (int)v.capacity();
        }
    }

    // 무효화 실험 — 이게 핵심이다:
    std::vector<int> w;
    w.push_back(100);
    int* p = &w[0];                    // 0번 원소의 주소를 붙잡는다
    printf("\n재할당 전: p=%p, *p=%d\n", (void*)p, *p);
    for (int i = 0; i < 50; i++) w.push_back(i);   // 강제로 여러 번 재할당
    printf("재할당 후: p=%p, w[0]=%d\n", (void*)p, w[0]);
    printf("같은 주소인가: %d  (다르면 p는 이제 댕글링!)\n", p == &w[0]);
    // *p 를 읽으면? → 해제된 메모리 접근. 주석 풀고 관찰(값이 깨지거나 크래시)
    // printf("*p = %d\n", *p);
}
cl /EHsc /nologo vector_check.cpp && vector_check.exe

확인: ① capacity가 0→1→2→3→4... 가 아니라 계단식(1→2→3→4→6→9...MSVC 1.5배)으로 뛰는가. ② 재할당 후 p == &w[0]이 거짓인가 — 붙잡아둔 포인터가 썩은 증거다. 이 순간이 "왜 CSector가 포인터+인덱스를 쓰는지"의 실물 증명이다.

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

size와 capacity가 다르면, 남는 capacity 자리에는 뭐가 들어있나요?

아무것도 아니다 — 할당은 됐지만 아직 생성되지 않은 원시 메모리다(레슨 6의 커밋된 페이지지만 쓰레기값). v[size()]처럼 그 자리를 인덱스로 접근하면 UB다. vector가 size를 따로 들고 있는 이유가 이것 — "어디까지가 진짜 원소인가"의 경계다. 레슨 9의 CMessageBuffer가 BufferSize(공간)와 WritePos(데이터 끝)를 구분한 것과 똑같은 구조다.

reserve()는 뭔가요? 언제 쓰죠?

capacity를 미리 키워두는 함수다. 최종 크기를 안다면 v.reserve(9500)로 재할당을 한 번에 끝내 — 중간의 여러 번 대이동과 무효화를 전부 없앤다. 서버 초기화처럼 "곧 N개가 들어올 것"이 확실할 때 성능+안전 둘 다 얻는다. 이 서버가 세션 배열을 아예 new[]로 고정 할당한 것(레슨 3)도 같은 정신의 극단판이다.

pop_back으로 지워도 capacity가 안 줄면 메모리 낭비 아닌가요?

맞지만 의도된 트레이드오프다. 섹터는 캐릭터가 끊임없이 드나드는 곳이라, 줄일 때마다 배열을 다시 잡으면 곧바로 또 늘려야 해서 재할당이 폭증한다(진동). capacity를 유지하면 "한 번 커진 섹터는 그 크기를 재사용"하므로 안정적이다. 정말 줄이고 싶으면 shrink_to_fit()이 있지만, 게임서버 핫패스에서는 거의 안 쓴다 — 메모리 몇 KB보다 재할당 없는 예측 가능성이 값지다.

list[iSectorIdx] = pMoved 처럼 []로 쓰는 건 재할당을 안 일으키나요?

안 일으킨다. []는 이미 있는 원소를 덮어쓸 뿐 size를 늘리지 않는다 — 재할당은 오직 size가 capacity를 넘으려 할 때(push_back, insert 등)만 발생한다. 그래서 swap-remove의 덮어쓰기+pop_back 조합은 재할당·무효화를 전혀 유발하지 않는 안전한 O(1) 연산이다. 설계자가 이걸 알고 짠 것이다.

[] 접근은 범위 검사를 하나요? at()과 뭐가 다르죠?

[]는 검사 안 한다(C 배열처럼 빠르되 범위 밖이면 UB). at()은 검사하고 범위를 벗어나면 예외를 던진다. 게임서버 핫패스는 성능과 "인덱스는 내가 통제한다"는 확신 때문에 거의 []를 쓴다 — CSector도 그렇다. 대신 그 확신이 틀리면(iSectorIdx 갱신을 빠뜨리면) 조용한 오염이 된다. 속도의 대가는 규율이다.

vector 대신 처음부터 고정 배열을 쓰면 재할당 걱정이 없지 않나요?

맞고, 그게 세션 배열의 선택이다(최대치가 고정이므로). 하지만 섹터당 인원은 0명일 수도 수백 명일 수도 있어 최대치로 고정하면(예: 섹터마다 9500칸) 33×43×9500 = 엄청난 낭비다. vector는 "대부분 비어있고 가끔 붐비는" 섹터에 딱 맞는다 — 실제 인원만큼만 메모리를 쓰되 필요하면 늘어난다. 자료구조 선택은 데이터의 분포를 아는 데서 나온다는 걸 보여주는 좋은 대비다.
