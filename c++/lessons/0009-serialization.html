직렬화: 객체를 전선에 태우는 법

네트워크는 바이트 스트림만 나른다 — 객체도, 타입도, 의미도 없다. 오늘의 목표: CMessageBuffer의 << 연산자가 하는 일을 미니 직렬화 버퍼로 직접 재현한다. 레슨 1(객체=바이트)이 드디어 수확기를 맞는다.

왜 이걸 배우나
CMessageBuffer(423줄)는 이 서버의 모든 패킷이 태어나고 읽히는 곳이다. 손코딩 재작성 목록에서 링버퍼와 함께 가장 먼저 만들게 될 부품이며, 경계 검사(CheckWrite/CheckRead)는 조작 패킷을 막는 보안 방어선이기도 하다.

1. 지식 — 직렬화 버퍼의 3요소

① 버퍼 + 두 개의 커서

CMessageBuffer.h:141-152의 상태는 놀랍게 단순하다:

int m_iReadPos;              // 다음에 읽을 위치
int m_iWritePos;             // 다음에 쓸 위치
unsigned char* m_chBuffer;   // 그냥 바이트 배열

쓰면 WritePos가 전진, 읽으면 ReadPos가 전진. ReadPos == WritePos가 되면 다 읽은 것. 큐를 배열 위에 눕힌 모양새다.

② 넣기/빼기: memcpy + 커서 전진

buffer << shX가 내부에서 하는 일의 전부:

// 개념 코드 — CMessageBuffer::operator<<(short)의 뼈대
if (CheckWrite(sizeof(short))) {                     // ← 먼저 경계 검사!
    memcpy(m_chBuffer + m_iWritePos, &shValue, sizeof(short));
    m_iWritePos += sizeof(short);
}
return *this;                    // 자신을 반환 → buffer << a << b << c 체이닝

객체의 2바이트를 버퍼의 현재 위치에 그대로 복사한다. 레슨 1에서 배웠듯 short는 바이트 2개일 뿐이니까. >>는 정확히 반대다(버퍼→변수, ReadPos 전진). 반환 타입이 CMessageBuffer&(레슨 4의 참조!)라서 buf << byType << shX << shY처럼 이어 쓸 수 있다.

③ 경계 검사: 신뢰하지 않는 입력에 대한 방어선

CMessageBuffer.h:119-138 실제 코드:

bool CheckRead(int iSize)
{
    if (m_iReadPos + iSize > m_iWritePos)   // 데이터 끝을 넘는 읽기?
    {
        m_bError = true;                     // 에러 표시만 하고
        return false;                        // 조용히 거부
    }
    return true;
}

시나리오: 악의적 클라이언트가 "이동 패킷"이라며 2바이트만 보냈다. 핸들러는 buf >> dir >> shX >> shY로 5바이트를 읽으려 한다. CheckRead가 없다면? 버퍼 뒤의 남의 메모리를 읽는다(정보 유출/오염). 있다면? m_bError가 켜지고, 파싱 후 IsError() 확인 → 즉시 연결 종료. "패킷은 유죄 추정"이 서버 보안의 제1원칙이다.

덤: 헤더 프레이밍과 pack(1)

페이로드 앞에는 CServer.h:3-17의 4바이트 헤더가 붙는다 — 그리고 그 구조체는 #pragma pack(push,1)로 감싸져 있다. 이제 이유를 완전하게 말할 수 있다: 헤더는 전선을 타는 바이트 배치 그 자체이므로, 컴파일러 패딩(레슨 1)이 끼어들면 상대방과 약속한 4바이트가 깨진다. 메모리용 구조체는 패딩 허용, 전송용 구조체는 pack(1) — 레슨 1 질문 세트의 복선 회수다.

2. 스킬 — 미니 직렬화 버퍼를 직접 만들기

이번엔 베끼지 말고 만든다. scratch\serial_check.cpp의 빈칸을 채워라:

#include <cstdio>
#include <cstring>   // memcpy

class MiniBuffer {
    unsigned char buf[64];
    int readPos = 0, writePos = 0;
    bool error = false;
public:
    MiniBuffer& operator<<(short v) {
        // TODO(1): 경계 검사 후 memcpy + writePos 전진 (위의 개념 코드 참고)
        return *this;
    }
    MiniBuffer& operator>>(short& v) {
        // TODO(2): CheckRead에 해당하는 검사 후 memcpy + readPos 전진
        return *this;
    }
    bool IsError() const { return error; }
};

int main() {
    MiniBuffer b;
    short x = 777, y = -25, outX = 0, outY = 0;

    b << x << y;
    b >> outX >> outY;
    printf("복원: %d %d (기대: 777 -25), error=%d\n", outX, outY, b.IsError());

    short extra;
    b >> extra;                        // 있는 것보다 더 읽기 — 공격 시뮬레이션
    printf("과잉 읽기 후 error=%d (기대: 1)\n", b.IsError());
}

두 출력이 기대대로 나오면 통과. 여러분은 방금 CMessageBuffer의 심장을 손코딩했다. 막히면 에이전트에게 — 답을 보여주기 전에 힌트부터 줄 것이다.

3. 확인 퀴즈

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

memcpy가 정확히 뭐 하는 함수죠?

memcpy(목적지, 출발지, 바이트수) — 타입도 의미도 묻지 않고 바이트를 그 수만큼 복사한다. 직렬화가 memcpy의 고향이다: "short의 의미"는 버리고 "short가 놓인 2바이트"만 나르는 것이 정확히 우리가 원하는 일이니까. 단, 겹치는 영역 복사는 UB(그때는 memmove), 그리고 목적지 공간 확보는 전적으로 호출자 책임 — 그래서 CheckWrite가 먼저다.

왜 구조체를 통째로 memcpy해서 보내지 않고 필드를 하나씩 << 하나요?

통째 복사는 ① 패딩까지 나가고(쓰레기 바이트 = 낭비 + 정보 유출), ② 구조체 레이아웃이 컴파일러/버전마다 달라질 수 있으며, ③ "이 패킷에 어떤 필드가 어떤 순서로 들어가는가"가 코드에 안 보인다. 필드 단위 <<는 패킷 명세가 코드 그 자체가 된다 — MakePacket_MoveStart를 읽으면 와이어 포맷이 그대로 보인다.

엔디언(endian)이라는 게 있다던데 여기선 왜 신경 안 쓰나요?

멀티바이트 값의 바이트 순서 문제다. x86/x64는 리틀 엔디언(낮은 바이트 먼저): short 777(0x0309)은 메모리에 09 03으로 눕는다. 인터넷 표준 프로토콜은 빅 엔디언을 쓰지만, 이 서버는 클라이언트와 서버가 모두 x86 계열이라는 전제로 리틀 엔디언 그대로 보낸다 — 게임업계의 흔한 실용적 선택이다. 다른 아키텍처와 통신해야 하는 순간 htons/ntohs 같은 변환이 필요해진다. 네트워크 아크에서 재론한다.

float도 그냥 memcpy로 보내도 되나요? 소수점인데요.

된다 — float도 그냥 4바이트다(IEEE 754라는 표준 배치). CMessageBuffer.h에 operator<<(float)가 태연히 있는 이유다. 양끝이 같은 표준을 쓰는 한(사실상 모든 현대 CPU) 바이트 복사로 완벽히 복원된다. 주의할 것은 오히려 bool이나 열거형처럼 크기가 처리계마다 다를 수 있는 타입 — 그래서 프로토콜에는 BYTE/short/int처럼 크기가 못 박힌 타입만 쓴다.

에러가 나면 예외를 던지지 왜 플래그(m_bError)만 세우나요?

설계 선택이다. 이 버퍼는 초당 수십만 번 불리는 핫패스에 있고, 예외는 (드물게 발생해도) 코드 경로를 복잡하게 하며 게임서버 업계는 전통적으로 예외를 꺼린다. 플래그 방식의 계약은 "연산은 조용히 무시되고, 파싱이 끝난 뒤 IsError() 한 번으로 판정한다" — 검사 지점이 한 곳으로 모여 오히려 빠뜨리기 어렵다. 대신 반드시 검사해야 한다는 규율이 필요하다.

AddRef/DecRef는 뭔가요? 왜 버퍼에 참조 카운트가 있죠?

브로드캐스트 때문이다. 이동 패킷 하나를 주변 30명에게 보낼 때 버퍼를 30벌 복사하는 대신, 한 버퍼를 30개 세션의 SendQ가 공유하고 카운트를 30으로 올린다. 각 전송이 끝날 때마다 DecRef, 0이 되는 순간 풀로 반납. "여럿이 드는 객체는 마지막 사람이 문을 잠근다" — 레슨 3에서 예고한 공유 수명 문제의 해법이며, 멀티스레드 아크에서 이 카운트가 왜 volatile LONG+Interlocked여야 하는지 다룬다.
