컴파일과 링크: .cpp에서 .exe까지

NetLib.lib는 어떻게 만들어지고, FighterServer.exe는 그걸 어떻게 흡수하는가? 오늘의 목표: 소스가 실행 파일이 되는 두 단계(컴파일→링크)를 분리해서 이해하고, "중복 심볼"·"템플릿은 헤더에" 같은 실무 수수께끼를 푼다.

왜 이걸 배우나
빌드 에러는 초보자를 가장 자주 좌절시키는 벽이고, 그 대부분은 컴파일과 링크를 한 덩어리로 오해해서 생긴다. 이 서버는 NetLib(정적 라이브러리)과 FighterServer(실행 파일)로 분리돼 있고(코드베이스 지도), 그 경계에서 링크가 일어난다. 재작성할 때 프로젝트 구조를 스스로 세우려면 이 모델이 필수다. 레슨 19(템플릿)·20(static ④)에서 남긴 실도 여기서 회수된다.

1. 지식 — 두 단계는 완전히 다른 일이다

단계 1: 컴파일 — .cpp 하나를 .obj 하나로 (독립적으로)

컴파일러는 .cpp 파일 하나씩을 따로 기계어(.obj)로 번역한다. 이 단위를 번역 단위(translation unit)라 한다. 결정적 사실: 각 .cpp는 다른 .cpp의 존재를 모른다. CServer.cpp를 컴파일할 때 컴파일러는 CFighterServer.cpp를 보지 않는다.

그럼 CServer.cpp가 CMessageBuffer::Alloc()을 부르는데 그 구현이 다른 파일(CMessageBuffer.cpp)에 있으면? 컴파일러는 헤더(.h)에서 본 선언만으로 "이런 함수가 어딘가 있다고 치고" 호출 코드를 만든다 — 실제 주소는 비워둔 채. 이 "나중에 채울 빈칸"이 링크의 일거리다.

헤더의 역할: 약속을 공유하는 방법

#include "CMessageBuffer.h"는 사실상 그 헤더 내용을 복사해 붙여넣는 것이다. 여러 .cpp가 같은 헤더를 include하면 모두 같은 선언(약속)을 공유한다 — "CMessageBuffer에는 Alloc이라는 함수가 있다"를. 구현은 각자 몰라도, 약속만 같으면 컴파일이 된다. 그래서 규칙: 헤더엔 선언, .cpp엔 정의.

#pragma once(이 서버의 모든 헤더 첫 줄)의 정체도 여기서 나온다: 한 .cpp가 A.h와 B.h를 include하고 둘 다 C.h를 include하면 C.h가 두 번 붙어 "중복 정의" 에러가 난다. #pragma once는 "이 헤더는 번역 단위당 한 번만 붙여라"라는 지시다.

단계 2: 링크 — .obj들의 빈칸을 서로 채운다

링커는 모든 .obj를 모아 빈칸(미해결 심볼)을 실제 주소로 연결한다. CServer.obj의 "CMessageBuffer::Alloc 주소 = ?"를 CMessageBuffer.obj가 제공하는 실제 주소로 메운다. 모든 빈칸이 채워지면 .exe 완성. 하나라도 못 채우면 그 악명 높은 "unresolved external symbol" 링크 에러다 — 컴파일은 다 됐는데(선언은 있었으니) 정의를 못 찾은 것.

정적 라이브러리: .obj들을 묶은 꾸러미

NetLib.lib는 NetLib의 모든 .obj(CServer, CMessageBuffer, CRingBuffer, CLockFree*, ...)를 하나로 묶은 아카이브다. FighterServer를 링크할 때 이 .lib를 던져주면, 링커가 필요한 .obj를 꺼내 FighterServer의 빈칸을 채운다. "네트워크 엔진을 한 덩어리로 재사용"의 물리적 실체다 — 레슨 5에서 "NetLib은 독립 라이브러리"라 한 것이 이 빌드 구조로 실현된 것이다.

추가로 Common.h의 #pragma comment(lib, "ws2_32.lib")는 "이 외부 라이브러리(Winsock)도 링크 단계에 포함하라"는 지시다. 소켓 함수들의 실제 구현이 그 안에 있다.

이제 두 수수께끼가 풀린다

수수께끼 1 (레슨 19): 템플릿은 왜 헤더에? 링커는 이미 만들어진 .obj의 빈칸만 채운다 — 코드를 생성하진 못한다. 그런데 CLockFreeStack<int>는 사용하는 .cpp에서 그 순간 생성돼야 한다(레슨 19의 인스턴스화). 구현이 .cpp에 숨어 있으면 그 .cpp만 컴파일될 뿐, 다른 곳에서 <int>로 찍어낼 코드가 없다 → 링크 에러. 그래서 템플릿 구현은 모두가 볼 수 있게 헤더에 둔다.

수수께끼 2 (레슨 20의 ④): Profile 중복 심볼. Profile.cpp와 ProfileTLS.cpp가 같은 이름의 함수를 각자 정의했다. 둘 다 링크되면 링커는 "같은 심볼이 두 곳에 — 어느 주소를 쓰지?"라며 중복 정의 에러를 낸다. 해법이 하나를 빌드에서 제외한 것(코드베이스 지도). 만약 그 함수들이 파일 스코프 static(내부 링키지, 레슨 20)이었다면 서로 안 보여 충돌하지 않았을 것이다 — static ④의 존재 이유가 링크 레벨에서 증명된다.

한 장의 그림
CServer.cpp ─┐ (컴파일)
CMessageBuffer.cpp ─┼→ 각각 .obj (빈칸 있음) ─┐
CRingBuffer.cpp ─┘                          │ (링크: 빈칸 채우기)
                                            ├→ NetLib.lib (묶음)
FighterServer.cpp ──→ .obj ─────────────────┴+ ws2_32.lib → FighterServer.exe
컴파일은 "번역", 링크는 "연결". 컴파일 에러는 문법/선언 문제, 링크 에러는 정의를 못 찾은 문제. 에러 메시지에 "unresolved external"이 보이면 링크 단계다.

2. 스킬 — 컴파일과 링크를 손으로 분리하기

지금까지는 cl file.cpp로 한 방에 했지만, 오늘은 두 단계를 쪼개서 눈으로 본다. 파일 두 개를 만들라.

scratch\math.cpp:

int add(int a, int b) { return a + b; }   // 정의(구현)

scratch\main2.cpp:

#include <cstdio>
int add(int a, int b);                     // 선언만! (헤더 역할)
int main() { printf("%d\n", add(3, 4)); }  // 빈칸을 두고 호출
# 1) 각각 컴파일만 (/c = 링크 안 함) → .obj 두 개 생성
cl /c /nologo math.cpp
cl /c /nologo main2.cpp        # add의 주소는 빈칸으로 남는다 — 그래도 컴파일 성공!

# 2) 링크만 → 빈칸을 채워 exe 생성
cl /nologo main2.obj math.obj  # → main2.exe (7 출력)

# 3) 링크 에러를 일부러 내보기: math.obj 없이 링크
cl /nologo main2.obj           # LNK2019 unresolved external symbol "add" — 이게 링크 에러!

확인: ① main2.cpp가 add의 구현 없이도 컴파일된다(선언만으로 충분). ② math.obj를 빼면 컴파일이 아니라 링크에서 실패하고, 메시지가 "unresolved external symbol add"인 것. 실험: main2.cpp의 선언을 int add(int);(인자 하나)로 바꾸면 어느 단계에서 실패하는가? (답: 링크 — 선언과 정의의 시그니처가 달라 심볼 이름이 안 맞는다. C++ 이름 맹글링의 결과다.)

컴파일 에러와 링크 에러를 어떻게 구별하나요?

에러 코드와 메시지로 구별된다. MSVC 기준 컴파일 에러는 C####(예: C2065 "선언되지 않은 식별자") — 문법·타입·선언 문제라 파일명과 줄 번호가 찍힌다. 링크 에러는 LNK####(예: LNK2019 "unresolved external", LNK2005 "already defined") — 줄 번호가 없고 심볼 이름이 나온다. "줄 번호 없이 심볼 이름만 나오면 링크 단계"가 빠른 판별법이다.

헤더에 함수 구현을 그냥 다 넣으면 편하지 않나요?

두 가지 문제가 생긴다. ① 중복 정의(LNK2005): 여러 .cpp가 그 헤더를 include하면 각 .obj가 같은 함수 정의를 갖게 되어 링크 시 충돌한다(#pragma once도 이건 못 막는다 — 파일 간 문제라서). ② 재컴파일 폭증: 헤더를 고치면 그걸 include한 모든 .cpp를 다시 컴파일해야 한다. 그래서 "선언은 헤더, 정의는 .cpp"가 원칙이다. 예외가 둘 있다: inline 함수와 템플릿(수수께끼 1) — 이들은 헤더에 정의를 둬도 되게 특별 규칙(ODR 예외)이 있다.

정적 라이브러리(.lib)와 동적 라이브러리(.dll)는 뭐가 다르죠?

정적(.lib): 링크 시점에 필요한 코드가 .exe 안으로 복사되어 흡수된다. NetLib.lib가 이 방식 — FighterServer.exe 하나만 배포하면 된다. 동적(.dll): 코드가 .exe 밖에 남고 실행 시점에 로드된다(ws2_32.dll이 이 방식 — Windows에 이미 있으니 안 들고 다녀도 됨). 정적은 배포가 단순하고 호출이 약간 빠르지만 exe가 커지고, 동적은 여러 프로그램이 코드를 공유하고 개별 업데이트가 가능하다. 게임서버는 자기 코드를 대개 정적으로, OS 라이브러리는 동적으로 쓴다.

이름 맹글링(name mangling)이 뭔가요?

C++은 오버로딩(레슨 18)을 지원해서 add(int,int)와 add(double,double)이 공존한다. 링커는 이름만으로 구별해야 하므로, 컴파일러가 함수 이름에 인자 타입 정보를 인코딩한다(예: ?add@@YAHHH@Z). 그래서 실습에서 선언을 add(int)로 바꾸면 맹글된 이름이 달라져 "add(int,int)의 정의"를 못 찾는다. 참고로 extern "C"는 이 맹글링을 끄는데, C 라이브러리와 링크할 때 쓴다 — 저수준에서 가끔 만난다.

CMake는 이 과정에서 뭘 하나요?

CMake는 컴파일러가 아니라 빌드 지휘자다. "어떤 .cpp들을 컴파일해 NetLib.lib로 묶고, FighterServer는 그 lib와 ws2_32를 링크하라"는 계획을 CMakeLists.txt에 적으면, CMake가 그걸 실제 빌드 시스템(MSVC 프로젝트, Makefile 등)으로 변환한다. 즉 CMake는 "누가 무엇에 의존하는가"의 그래프를 관리한다. 이 서버의 루트 CMakeLists가 NetLib→FighterServer 의존 순서를 정하는 것이 그 일이다. 재작성 시 이 파일부터 세우게 된다.

#include를 "복사 붙여넣기"라고 했는데, 그럼 컴파일이 느려지지 않나요?

느려진다 — 큰 헤더(특히 windows.h)를 많은 .cpp가 include하면 같은 내용을 반복 파싱한다. 그래서 이 서버는 Common.h에 공통 include를 모으고, 실무에서는 미리 컴파일된 헤더(PCH)로 이 비용을 줄인다(자주 안 바뀌는 헤더를 한 번만 파싱해 캐시). 대형 C++ 프로젝트의 빌드 시간 최적화가 하나의 기술 분야인 이유다. C++20의 모듈이 이 문제의 근본 해결책으로 도입됐지만 아직 과도기다.
