Lesson 0014 · 네트워크 · I/O 모델

블로킹의 한계: 9500명을 어떻게 받을 것인가

레슨 12의 에코 서버는 두 번째 손님을 받지 못했다. 오늘의 목표: 그 한계를 실험으로 확인하고, I/O 모델의 진화(스레드-퍼-클라이언트 → select → 비동기 완료 통지)를 따라가 왜 IOCP인가에 스스로 도달한다.

왜 이걸 배우나
IOCP를 "고성능이라니까" 쓰는 사람과, 다른 모델의 어디가 부러지는지 보고 쓰는 사람의 격차는 장애 상황에서 드러난다. 이 레슨은 코드보다 사고 과정이다 — 면접이든 설계 리뷰든 "왜 IOCP였나"에 대한 여러분의 답이 여기서 만들어진다. 배치 4(IOCP 아크)의 관문.

1. 지식 — 세 번의 진화

0단계의 문제: recv는 스레드를 인질로 잡는다

블로킹 recv는 "데이터가 올 때까지" 스레드를 재운다. 클라이언트 A의 recv에 갇힌 스레드는 B의 데이터가 도착해도 못 움직인다. 스레드 하나로는 연결 하나 — 레슨 12에서 직접 목격한 그 벽이다.

진화 ①: 클라이언트마다 스레드를 준다 (thread-per-client)

accept할 때마다 _beginthreadex로 전담 스레드를 만들면? 소규모(수십 명)에서는 실제로 잘 동작하고 코드도 직관적이다. 그러나 9500명이면:

  • 스택 9500개: 1MB씩 예약(레슨 6) → 주소 공간 ~9.3GB가 스택으로 증발.
  • 컨텍스트 스위칭 폭풍: 코어는 8개인데 실행 대기 스레드가 수천 개 — OS는 수 마이크로초짜리 문맥 교환을 초당 수십만 번 하며 캐시(레슨 7)를 계속 오염시킨다. CPU가 일이 아니라 교대에 소모된다.
  • 스레드 대부분은 어차피 recv에서 자고 있다 — 9500개 중 실제 일하는 것은 한 줌. 자원 낭비의 정의다.

진화 ②: 한 스레드가 여러 소켓을 감시한다 (select — readiness 모델)

"어느 소켓이 읽을 준비가 됐는지" 목록으로 물어보는 API. 스레드 하나로 여러 연결을 처리할 수 있게 됐다. 그러나 Windows의 select는 한 번에 64개 제한이 있고, 무엇보다 매 호출마다 전체 소켓 목록을 커널에 넘기고 전체를 훑는 O(N)이다. 9500개면 "누가 준비됐나"를 묻는 비용 자체가 서버를 잡아먹는다.

진화 ③: 방향을 뒤집는다 — 완료 통지 (IOCP — completion 모델)

readiness 모델은 "읽을 수 있게 되면 알려줘, 읽기는 내가 할게"다. IOCP는 한발 더 간다: "읽기를 미리 걸어둘 테니(WSARecv), 다 읽어서 내 버퍼에 넣어놓고 완료됐다고 알려줘."

[readiness]  나: 준비됐어?→아니→준비됐어?→응!→그럼 recv (복사는 내 몫, 물어보는 비용 O(N))
[completion] 나: 이 버퍼에 읽어놔 (WSARecv 등록)
             …다른 일 하다가…
             커널: 다 됐다. 3172번 세션, 87바이트. (완료 큐에서 O(1)로 꺼냄)

완료 통지들은 하나의 큐(완료 포트)에 모이고, 소수의 워커 스레드(이 서버는 2개 — main.cpp:17의 세 번째 인자)가 GetQueuedCompletionStatus로 꺼내 처리한다. 스레드 수가 연결 수가 아니라 코어 수에 비례한다 — 9500연결이 스레드 2개로 돌아가는 마법의 정체는 "기다림을 전부 커널에 위임했다"는 것뿐이다. 레슨 10의 복선도 여기서 회수된다: 커널이 "내 버퍼에 직접" 쓰려면 미리 내어줄 연속 공간이 필요하다 — 그것이 링버퍼의 Direct 계열 + 2세그먼트 WSABUF였다.

2. 스킬 — 한계를 실험으로 확인하기

레슨 12의 에코 서버를 그대로 사용한다. 실험 A — 블로킹의 벽:

# 터미널 1: echo_server.exe 실행
# 터미널 2: 클라이언트 접속 후 send 없이 대기 (연결만 물고 있는다)
$c1 = New-Object Net.Sockets.TcpClient("127.0.0.1", 20001)
# 터미널 3: 두 번째 클라이언트로 데이터 전송 시도
$c2 = New-Object Net.Sockets.TcpClient("127.0.0.1", 20001); $s2 = $c2.GetStream()
$m = [Text.Encoding]::ASCII.GetBytes("hi"); $s2.Write($m, 0, 2)
$buf = New-Object byte[] 16; $s2.ReadTimeout = 3000
try { $n = $s2.Read($buf, 0, 16); "응답: $([Text.Encoding]::ASCII.GetString($buf,0,$n))" }
catch { "3초간 응답 없음 — c1의 recv에 인질로 잡혀 있다!" }

실험 B — 진화 ①을 직접 구현: 에코 서버의 accept 루프에서 클라이언트 처리부를 _beginthreadex로 넘겨 thread-per-client 에코 서버로 개조하라 (클라이언트 처리 함수를 static으로 빼고 SOCKET을 인자로). 개조 후 실험 A를 다시 — 이번엔 c2가 즉시 응답받아야 한다. 그리고 작업 관리자에서 스레드 수를 관찰하라. 이 개조가 이번 배치의 핵심 손코딩 과제다. 완성하면 여러분은 "동시성 서버"를 처음 만든 것이다.

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

컨텍스트 스위칭이 정확히 뭐고 왜 비싼가요?

OS가 코어에서 스레드 A를 내리고 B를 올리는 작업이다: A의 레지스터 전부 저장 → B의 것 복원 → 커널 모드 왕복(레슨 6). 직접 비용은 마이크로초 수준이지만 진짜 비용은 간접이다 — B가 올라오면 캐시와 TLB에는 A의 데이터가 가득해서(레슨 7) 한동안 미스 행진을 한다. "스레드는 공짜 동시성"이라는 착각을 깨는 것이 이 레슨의 절반이다.

스레드를 코어 수만큼만 만들고 블로킹 recv를 쓰면 안 되나요?

8코어에 스레드 8개면 동시에 8명의 recv에만 갇힐 수 있다 — 나머지 9492명의 데이터는 그 8명 중 누군가 말할 때까지 방치된다. 블로킹 모델의 본질적 결함은 "스레드가 특정 소켓에 묶인다"는 것이다. IOCP의 워커는 어느 소켓이든 완료된 것부터 집는다 — 스레드와 소켓의 결합을 끊은 것이 혁신의 핵심이다.

워커가 2개인 근거는요? 코어가 8개면 8개가 낫지 않나요?

이 서버의 워커는 "완료 꺼내기 → RecvMsgDiv → 큐 넣기"만 한다(무거운 게임 로직은 GameThread 몫). 일이 가벼우니 2개로 족하고, 더 늘리면 completeRecvQ 등 공유 자원 경합만 는다. 통례는 "코어 수 ~ 코어×2에서 시작해 측정으로 조정" — MSDN의 IOCP 문서도 같은 조언을 한다. 정답 숫자가 아니라 측정 습관이 답이다.

select의 64개 제한은 왜 있는 건가요? 리눅스도 그런가요?

Windows의 FD_SETSIZE 기본값이 64일 뿐 늘릴 수는 있다 — 그러나 "매 호출 전체 목록 복사+스캔 O(N)"이라는 구조적 비용은 그대로다. 리눅스도 같은 문제를 겪고 epoll(readiness지만 O(1) 통지)로 진화했다. Windows의 답이 IOCP다. 참고로 이 저장소의 GoFighter가 리눅스에서 돌면 Go 런타임이 내부적으로 epoll을 쓴다 — 같은 문제, 언어별 다른 포장이다.

비동기면 WSARecv를 건 순간과 완료 사이에 내 버퍼는 어떻게 되는 건가요?

핵심을 짚었다. 그 구간 동안 버퍼(링버퍼의 빈 구간)의 소유권은 커널에 넘어가 있다 — 우리가 건드리면 안 되고, 세션이 그 사이 죽어도 버퍼를 회수하면 안 된다(커널이 아직 쓸 수 있으므로). "진행 중인 IO가 있는 한 세션을 릴리즈하지 않는다"는 규칙이 필요해지고, 그것이 바로 lIOCount 참조 카운트의 존재 이유다. IOCP 아크(배치 4)의 최대 난제를 지금 예고한 것이다.

C10K 문제라는 말을 들어봤는데 이건가요?

정확히 이것이다. 1999년 Dan Kegel이 "커넥션 1만 개(C10K)를 한 머신에서"를 화두로 던졌고, thread-per-client의 붕괴와 이벤트 기반 모델(epoll/kqueue/IOCP)로의 전환을 정리했다. 이 서버의 9500세션은 의도적으로 그 규모다. 이후 C10M(천만) 논의로 이어졌지만, 게임서버 실무 감각은 C10K 세대의 해법 위에 서 있다.
