Lesson 0008 · CS 기초 · 비트 연산

비트와 바이트: 세션 Key 해부

이 서버의 세션 ID는 숫자 하나에 두 개의 값을 숨겨 놓았다: 상위 17비트는 배열 인덱스, 하위 47비트는 고유 번호. 오늘의 목표: 시프트와 마스크로 이 Key를 직접 만들고 쪼갤 수 있게 된다.

왜 이걸 배우나
CServer.h:24-26의 GET_SESSIONINDEX(KEY, INDEX) INDEX = ((KEY) >> 47)은 이 서버에서 가장 자주 실행되는 코드 축에 든다. 프로토콜의 0x89, 최대 페이로드 4096, 그리고 락프리 자료구조의 "포인터+카운터를 한 덩어리로" 기법까지 — 전부 비트 감각 위에 서 있다. 네트워크·락프리 아크 진입 전 마지막 기초 체력이다.

1. 지식 — 16진수, 시프트, 마스크

16진수: 비트를 읽기 위한 안경

16진수 한 자리 = 정확히 4비트. 그래서 0x89를 보면 즉시 1000 1001이 보인다(8=1000, 9=1001). 10진수 137은 그게 안 보인다. 프로토콜 코드·주소·플래그를 전부 16진수로 쓰는 이유는 멋이 아니라 비트가 보이는 표기법이기 때문이다.

시프트: 자리 옮기기 = 곱셈/나눗셈, 그리고 "꺼내기"

x << n   // 왼쪽으로 n비트: x × 2ⁿ. 아래는 0으로 채움
x >> n   // 오른쪽으로 n비트: x ÷ 2ⁿ. 아래 n비트는 버려짐!

"버려진다"가 핵심이다. Key >> 47은 하위 47비트(고유 번호)를 버리고 상위 17비트(인덱스)만 남긴다 — 두 값이 포개진 숫자에서 위층만 꺼내는 연산이다.

마스크: AND로 "원하는 비트만 남기기"

x & 0xFF                  // 하위 8비트만 남김 (나머지는 0과 AND → 0)
Key & 0x7FFFFFFFFFFF      // 하위 47비트(고유 번호)만 남김

0x7FFFFFFFFFFF = 하위 47비트가 전부 1인 수. 일반형으로 외워라: 하위 n비트 마스크 = (1ULL << n) - 1. (1을 n칸 올리면 1000…0, 1을 빼면 0111…1.)

조립: OR로 두 값을 한 숫자에 포개기

세션 생성 시 서버가 하는 일의 본질:

unsigned __int64 Key = ((unsigned __int64)index << 47) | uniqueNo;
//   [ 17비트 index ][        47비트 uniqueNo        ]  ← 64비트 하나에 두 값

왜 이런 짓을 하나? ① 인덱스: players[Key >> 47]로 O(1) 접근(레슨 2). ② 고유 번호: 같은 인덱스 자리가 재사용됐을 때, 죽은 세션의 옛 Key로 산 세션을 건드리는 사고를 uniqueNo 불일치로 검출한다 — 세션 수준의 ABA 문제 방어이고, 락프리 풀의 unique 태그와 같은 사상이다. 이 한 줄에 이 서버의 안전 철학이 압축돼 있다.

참고로 17비트인 이유: 최대 세션 9500 < 2¹⁷ = 131,072. 47비트 고유 번호는 초당 1만 접속이 와도 44만 년을 버틴다.

2. 스킬 — Key를 만들고 쪼개기

scratch\bits_check.cpp — 실행 전에 세 출력을 손으로 계산하라:

#include <cstdio>

typedef unsigned long long U64;

U64 MakeKey(U64 index, U64 uniqueNo) {
    return (index << 47) | (uniqueNo & ((1ULL << 47) - 1));
}

int main() {
    U64 key = MakeKey(3172, 900000001);

    U64 index  = key >> 47;                    // 서버의 GET_SESSIONINDEX와 동일
    U64 unique = key & ((1ULL << 47) - 1);

    printf("key    = 0x%016llX\n", key);
    printf("index  = %llu (기대: 3172)\n", index);
    printf("unique = %llu (기대: 900000001)\n", unique);

    // 세션 재사용 검출 시뮬레이션:
    U64 staleKey = MakeKey(3172, 900000001);   // 죽은 세션의 옛 Key
    U64 newKey   = MakeKey(3172, 900000777);   // 같은 자리의 새 세션
    printf("같은 인덱스인가:   %d\n", (staleKey >> 47) == (newKey >> 47));
    printf("같은 세션인가:     %d\n", staleKey == newKey);   // ← 이게 방어선
}
cl /EHsc /nologo bits_check.cpp && bits_check.exe

보너스 도전: MakeKey에서 | 를 +로 바꿔도 되는가? (답: uniqueNo가 47비트를 안 넘는 한 결과는 같다. 그럼에도 OR을 쓰는 이유는 "겹치지 않는 비트의 조립"이라는 의도가 드러나기 때문.)

3. 확인 퀴즈

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

0x89 = 1000 1001로 바꾸는 걸 매번 어떻게 하나요?

16진수 한 자리씩 4비트로 치환하면 끝이다: 8→1000, 9→1001. 16개만 외우면 되는데 사실 절반은 자명하다(0~7은 3비트 이진수 그대로, 8~F는 거기에 최상위 1을 켠 것). A=1010, C=1100, F=1111만 익숙해지면 나머지는 따라온다. 프로그래머 계산기(Windows 계산기의 프로그래머 모드)로 검산하는 습관도 좋다.

&와 &&는 뭐가 다르죠? |와 ||도요?

&/|는 비트별 연산(64개 비트 각각에 AND/OR), &&/||는 논리 연산(전체를 참/거짓 하나로 취급)이다. 5 & 2 = 0 (101과 010은 겹치는 비트 없음)이지만 5 && 2 = 1 (둘 다 0이 아니므로 참). 조건문에 &를 잘못 쓰면 종종 통과되는 것처럼 보여서 더 위험한 버그가 된다.

부호 있는 정수를 >>하면 어떻게 되나요?

함정 지대다. 음수(int)를 오른쪽 시프트하면 대부분의 컴파일러가 부호 비트를 복사해 채우지만(산술 시프트), 이식성 있는 보장이 아니었다(C++20에서야 확정). 그래서 비트 조작은 unsigned로가 철칙이다 — 서버의 Key가 unsigned __int64인 이유이고, 실습 코드가 1ULL(unsigned long long 리터럴)을 고집한 이유다. 1 << 47이라 쓰면 32비트 int 1을 47칸 밀어 UB가 난다!

비트 플래그(여러 상태를 한 변수에)는 어떻게 쓰는 건가요?

같은 원리의 응용이다: 켜기 flags |= FLAG_A, 끄기 flags &= ~FLAG_A(~는 비트 반전), 검사 if (flags & FLAG_A). 각 플래그를 1<<0, 1<<1, 1<<2…로 정의해 서로 다른 비트를 쓰게 한다. Windows API가 WSA_FLAG_OVERLAPPED 같은 플래그를 OR로 조합받는 방식 그대로다.

4096이나 8192 같은 숫자가 계속 나오는데 우연인가요?

전부 2의 거듭제곱이다: 4096=2¹², 8192=2¹³. 이유는 ① 페이지 크기(4KB)·캐시라인(64B)과 궁합이 맞고, ② x % 8192를 컴파일러가 x & 8191(한 사이클)로 바꿀 수 있다 — 링버퍼(레슨 10)의 회전 연산이 정확히 이 최적화를 노린다. "크기는 2의 거듭제곱으로"는 시스템 프로그래밍의 기본 예법이다.

17비트에 세션 수 9500이면 낭비 아닌가요? 14비트(16384)면 충분한데.

맞는 계산이고, 그게 트레이드오프 사고의 시작이다. 인덱스 비트를 줄이면 고유 번호 비트가 늘어나 재사용 검출이 더 튼튼해진다. 반대로 인덱스 여유를 두면 최대 세션을 코드 수정 없이 13만까지 키울 수 있다. 이 서버는 후자를 골랐다. 정답이 있는 문제가 아니라 어느 쪽 여유가 더 값진가의 문제 — 재작성 때 여러분이 직접 정하게 된다.
