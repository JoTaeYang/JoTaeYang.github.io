연산자 오버로딩: << 는 어떻게 함수가 되는가

buf << shX << shY는 문법 마법이 아니라 함수 호출 두 번이다. 오늘의 목표: <<가 어떤 함수로 번역되는지 정확히 알고, 레슨 9에서 만든 미니 버퍼의 연산자를 설계 관점에서 다시 짠다.

왜 이걸 배우나
CMessageBuffer는 <</>>를 타입별로 20개 넘게 오버로딩해서, 모든 MakePacket_* 함수가 "패킷 명세처럼 읽히게" 만든다(레슨 9). 재작성 목록에서 CMessageBuffer는 초반 부품이고, 그 인터페이스의 심장이 연산자다. 왜 void가 아니라 자기 참조를 반환하는지, 왜 매개변수가 char와 char&로 갈리는지 — 오늘 전부 설명된다.

1. 지식 — 연산자는 특별한 이름의 함수다

<< 의 번역

C++ 컴파일러는 a << b를 보면 a.operator<<(b)라는 함수 호출로 바꾼다. 즉 "왼쪽 피연산자의 멤버 함수를, 오른쪽 피연산자를 인자로 부른다". CMessageBuffer의 선언(CMessageBuffer.h:76):

CMessageBuffer& operator<<(short shValue);

이 한 줄을 셋으로 뜯어보자 — 각 부분에 설계 결정이 하나씩 박혀 있다:

부분의미왜 이렇게?
operator<<연산자 <<의 함수 이름buf << x 문법을 쓰기 위한 약속된 이름
(short shValue)오른쪽 피연산자를 값으로 받음short는 작으니 복사가 싸다(레슨 4)
CMessageBuffer& 반환자기 자신의 참조를 돌려줌체이닝(<< a << b)을 위해 (레슨 17의 대입 연산자와 같은 패턴)

체이닝이 성립하는 원리 — 왼쪽부터 접힌다

buf << shX << shY는 <<가 좌결합이라 이렇게 묶인다:

(buf << shX) << shY
= (buf.operator<<(shX)).operator<<(shY)
   └── buf& 반환 ──┘
= buf.operator<<(shX) 가 buf 자신을 돌려주니, 그 buf에 다시 << shY

만약 반환이 void였다면? (buf << shX)의 결과가 없어서 ... << shY를 붙일 대상이 사라진다 → 컴파일 에러. "자기 참조 반환"이 체이닝의 엔진이다. 참조(&)로 돌려주는 이유도 중요하다 — 값으로 돌려주면 매번 CMessageBuffer 복사본이 생기는데, 레슨 17에서 배웠듯 이 클래스는 복사되면 안 된다. 참조 반환은 "같은 원본을 계속 넘긴다"는 뜻이다.

넣기(<<)와 빼기(>>)의 매개변수가 다른 이유

CMessageBuffer& operator<<(short shValue);    // 넣기: 값을 받는다 (읽기만)
CMessageBuffer& operator>>(short& shValue);   // 빼기: 참조를 받는다 (채워줘야)

<<는 버퍼에 값을 복사해 넣으므로 원본을 안 건드린다 → 값 전달로 충분. >>는 버퍼에서 꺼낸 값을 호출자의 변수에 채워줘야 하므로 그 변수의 참조가 필요하다(레슨 4의 출력 인자). 그래서 buf >> dir >> shX가 dir·shX를 실제로 바꾼다. 매개변수 하나의 & 유무가 "읽기냐 쓰기냐"를 가른다.

오버로딩 = 같은 이름, 다른 매개변수

CMessageBuffer에는 operator<<(char), operator<<(short), operator<<(int), operator<<(float)... 가 전부 있다(레슨 9). 이름은 같지만 매개변수 타입이 달라서, 컴파일러가 buf << shX의 shX 타입(short)을 보고 맞는 오버로드를 컴파일 시점에 고른다. 이것이 "오버로드 해석(overload resolution)"이다 — 실행 비용 0. 덕분에 buf << 어떤타입이든 그 타입 크기만큼 정확히 직렬화된다.

2. 스킬 — 미니 버퍼에 연산자를 제대로 얹기

레슨 9의 MiniBuffer를 이번엔 연산자 설계에 집중해 완성한다. scratch\operator_check.cpp:

#include <cstdio>
#include <cstring>

class MiniBuffer {
    unsigned char buf[64];
    int readPos = 0, writePos = 0;
public:
    // TODO(1): operator<<(short) — 반환 타입을 무엇으로? 왜?
    MiniBuffer& operator<<(short v) {
        memcpy(buf + writePos, &v, sizeof(v)); writePos += sizeof(v);
        return _____;              // ← 체이닝이 되려면 무엇을 반환해야 하나?
    }
    // TODO(2): operator<<(char) 오버로드를 추가하라 (1바이트).
    //          같은 이름, 다른 매개변수 — 컴파일러가 골라준다.

    // 빼기: 왜 매개변수가 short& 인가?
    MiniBuffer& operator>>(short& v) {
        memcpy(&v, buf + readPos, sizeof(v)); readPos += sizeof(v);
        return *this;
    }
};

int main() {
    MiniBuffer b;
    short x = 777, y = -25;
    char  type = 5;

    b << type << x << y;        // 체이닝 + 오버로드 선택이 동시에 일어난다

    char t; short ox, oy;
    // TODO(3): operator>>(char&)도 추가한 뒤, b >> t >> ox >> oy; 로 복원
    printf("복원: type=%d x=%d y=%d\n", t, ox, oy);   // 5 777 -25 기대
}
cl /EHsc /nologo operator_check.cpp && operator_check.exe

통과 기준: 5 777 -25. 실험: TODO(1)의 반환을 시험 삼아 void로 바꿔 보라 — b << type << x 줄에서 나는 컴파일 에러 메시지를 직접 읽어라. 그 에러가 "체이닝은 반환값을 먹고 산다"의 증거다.

3. 확인 퀴즈

4. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

<<는 원래 비트 시프트(레슨 8) 아니었나요? 어떻게 둘 다 되죠?

피연산자 타입으로 구별된다. int << int는 내장 비트 시프트, CMessageBuffer << short는 우리가 정의한 함수. 컴파일러는 왼쪽 피연산자가 클래스면 operator<< 멤버를 찾고, 내장 타입이면 시프트로 해석한다. 같은 기호의 이중생활은 이 코스에 자주 나온다(레슨 2의 *, 레슨 4의 &). "기호가 아니라 문맥이 의미를 정한다"가 C++ 읽기의 핵심 습관이다.

std::cout << "hello" 도 같은 원리인가요?

정확히 같다. cout은 std::ostream 객체이고 <<는 그 스트림에 대한 연산자 오버로드다. 반환도 ostream&(자기 참조)라서 cout << a << b << c 체이닝이 된다 — CMessageBuffer의 설계는 사실 표준 스트림 관용구를 게임 패킷에 그대로 빌려온 것이다. 그래서 C++ 개발자에게 buf << x가 즉시 "직렬화구나"로 읽힌다. 좋은 인터페이스는 익숙한 관용구를 재사용한다.

연산자를 멤버 함수 말고 밖에서(전역 함수로) 정의할 수도 있나요?

있다. operator<<(MiniBuffer& b, short v)처럼 자유 함수로도 가능하고, 왼쪽 피연산자가 내 클래스가 아닐 때(예: cout << myObject — 왼쪽이 ostream이라 내가 멤버를 못 넣음)는 반드시 자유 함수여야 한다. CMessageBuffer는 왼쪽이 자기 자신이라 멤버 함수가 자연스럽다. 규칙: "왼쪽 피연산자가 내 타입이면 멤버로, 아니면 자유 함수로."

아무 연산자나 다 오버로딩할 수 있나요? +, ==, [] 같은 것도?

대부분 가능하다(+ - * / == < [] () -> 등). 단 . :: ?: sizeof 등 몇 개는 금지다. 그러나 할 수 있다와 해야 한다는 다르다 — 남용하면 코드가 암호가 된다. 좋은 기준: "연산자의 통념적 의미와 맞을 때만." <<=스트림에 넣기는 관용에 맞아 훌륭하지만, 예컨대 벡터 덧셈이 아닌데 +를 엉뚱하게 쓰면 독이다. Core Guidelines도 "놀라지 않게(least surprise)"를 강조한다.

오버로드가 20개면 함수가 20개인데, 호출할 때 20개를 다 뒤지나요?

실행 시점엔 안 뒤진다 — 컴파일 시점에 피연산자 타입으로 딱 하나를 골라 그 호출로 못 박는다(정적 바인딩, 레슨 5의 non-virtual과 같은 방식). 그래서 오버로드가 아무리 많아도 실행 비용은 직접 함수 호출 하나와 같다. virtual 함수의 vtable 조회(레슨 5)와 대비되는 점 — 오버로딩은 컴파일 타임, 다형성은 런타임이다.

float도 << 하나로 되던데, short로 넘기면 float 오버로드가 안 불릴 위험은요?

좋은 우려다. C++은 정확히 맞는 오버로드가 없으면 암시적 변환으로 근사한 것을 고를 수 있어, 의도치 않은 오버로드가 선택되는 함정이 있다(예: 딱 맞는 게 없으면 short→int 승격 후 int 버전 호출). CMessageBuffer가 char/short/int/float/int64/double을 빠짐없이 오버로딩한 이유가 이것 — 모든 프로토콜 타입에 정확한 오버로드를 둬서 "엉뚱한 크기로 직렬화되는" 사고를 원천 차단한 것이다. 직렬화에서 크기 오판은 곧 프로토콜 붕괴(레슨 13)다.
