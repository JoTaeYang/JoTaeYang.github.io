포인터: 주소를 값으로 다루기

레슨 1에서 객체는 바이트 덩어리이고 어딘가의 주소에 놓인다는 걸 배웠다. 포인터는 그 주소를 담는 8바이트 변수일 뿐이다. 오늘의 목표: players[index]가 실제로 어떤 주소 계산인지 손으로 재현한다.

왜 이걸 배우나
이 서버는 포인터로 돌아간다. 세션 배열, CCharacter* players, 섹터가 들고 있는 CCharacter* 목록, CMessageBuffer* 전달 — 전부 "객체를 복사하지 않고 주소만 돌린다"는 하나의 아이디어다. 이게 흔들리면 뒤의 모든 레슨이 흔들린다.

1. 지식 — 네 가지 기호

기호읽는 법뜻
CCharacter* p"CCharacter를 가리키는 포인터 p"p는 CCharacter가 놓인 주소를 담는 8바이트 변수
&obj"obj의 주소"객체가 놓인 위치를 얻는다
*p"p가 가리키는 곳"주소를 따라가서 객체 자체에 접근 (역참조)
p->shX"(*p).shX의 축약"따라간 곳의 멤버 접근

실제 코드에서 확인하자. CFighterServer.cpp의 SendAround:

for (CCharacter* pOther : sectors->GetSector(iSectorX, iSectorY))
{
    if (pOther->SessionKey == players[index].SessionKey)  // 포인터는 ->, 객체는 .

섹터는 캐릭터의 복사본이 아니라 주소들을 보관한다. 캐릭터 하나는 48바이트지만 주소는 8바이트이고, 무엇보다 — 주소를 따라가면 모두가 같은 원본을 본다. HP가 깎이면 섹터를 통해 보든 players 배열로 보든 같은 값이다.

배열 인덱싱은 포인터 연산이다

CFighterServer.cpp:29에서 players = new CCharacter[uiMaxSession]; — players는 첫 캐릭터의 주소다. 그러면:

players[index]  ==  *(players + index)
// 주소 계산으로는:  players가 담은 주소 + index × sizeof(CCharacter)
//                =  players + index × 48

포인터에 1을 더하면 1바이트가 아니라 대상 타입 크기만큼 이동한다. 컴파일러가 레슨 1에서 배운 sizeof를 알고 있기 때문에 가능한 일이다. learncpp: Pointer arithmetic

"없음"을 표현하기: nullptr

생성자는 players = nullptr;로 시작한다. nullptr는 "아직 아무것도 가리키지 않음"이라는 약속된 주소값(0)이다. nullptr를 역참조하면 크래시한다 — 그래서 이 서버의 FindSession은 "살아있는 세션이면 포인터, 아니면 NULL"을 반환하고 호출자가 반드시 검사한다.

2. 손으로 계산해 보기

players가 주소 0x1000을 담고 있다고 하자 (sizeof(CCharacter)=48, 레슨 1).

  • &players[0] = 0x1000
  • &players[1] = 0x1000 + 48 = 0x1030
  • &players[2] = 0x1060
  • &players[1].shX = 0x1030 + 14(shX의 오프셋) = 0x103E

배열 인덱싱이 곱셈 한 번 + 덧셈 한 번이라는 것 — 이것이 배열이 O(1) 접근인 이유다.

3. 스킬 — 직접 검증하기

scratch\pointer_check.cpp로 저장하고 실행하라:

#include <cstdio>
#include <windows.h>
#include "../FighterServer/CCharacter.h"

int main() {
    CCharacter* players = new CCharacter[3];

    printf("players[0]: %p\n", (void*)&players[0]);
    printf("players[1]: %p\n", (void*)&players[1]);   // 정확히 48(0x30) 차이나는가?
    printf("*(players+1)과 같은가: %d\n", &players[1] == (players + 1));

    CCharacter* p = &players[1];
    p->shX = 777;
    printf("players[1].shX = %d\n", players[1].shX);   // 777 — 같은 원본!

    delete[] players;
}
cl /EHsc /nologo pointer_check.cpp && pointer_check.exe

두 주소의 차가 0x30(=48)인지, p->shX에 쓴 값이 players[1]로 보이는지 확인하라.

5. 초보자 질문 세트

클릭하기 전에 먼저 스스로 답해 보라.

포인터랑 배열은 같은 건가요? players는 배열인가요 포인터인가요?

다르다. players는 포인터(주소를 담는 8바이트 변수)이고, new CCharacter[9500]이 만든 힙 위의 456,000바이트 덩어리가 배열이다. 포인터는 그 배열의 첫 원소를 가리킬 뿐이다. 혼동이 생기는 이유는 배열 이름이 대부분의 문맥에서 첫 원소의 주소로 자동 변환(decay)되기 때문 — 그래서 인덱싱 문법이 똑같아 보인다.

*가 선언에서도 나오고 역참조에서도 나오는데 헷갈려요.

같은 기호가 두 가지 역할을 한다. 선언에서 CCharacter* p의 *는 "p는 포인터 타입"이라는 표시이고, 식(expression)에서 *p의 *는 "따라가라"는 연산이다. 읽을 때 "선언이면 타입의 일부, 식이면 역참조"로 구분하면 된다. &도 마찬가지로 이중 역할이 있다(식에서는 주소 얻기, 선언에서는 참조 — 레슨 4에서).

nullptr를 역참조하면 정확히 무슨 일이 일어나요?

OS가 프로세스를 죽인다(액세스 위반, 0xC0000005). 주소 0 근처 페이지는 OS가 일부러 접근 금지로 막아 두었기 때문이다 — 실수를 조용히 지나치지 않고 즉시 크래시시켜 주는 안전장치다. 이 서버의 CCrashDump가 하는 일이 바로 이런 크래시 순간의 메모리 전체를 덤프 파일로 남겨 사후 분석을 가능하게 하는 것이다.

왜 섹터는 CCharacter를 통째로 저장하지 않고 포인터를 저장하나요?

두 가지 이유. ① 단일 원본: 캐릭터의 진짜 데이터는 players 배열 한 곳에만 있어야 한다. 섹터에 복사본을 두면 HP를 깎을 때 두 군데를 갱신해야 하고, 하나를 빠뜨리는 순간 버그다. ② 이동 비용: 캐릭터가 섹터를 옮길 때 8바이트 주소만 옮기면 되지, 48바이트를 복사할 필요가 없다. "데이터는 한 곳에, 나머지는 참조"는 서버 설계의 기본 원칙이다.

%p로 찍힌 주소가 실행할 때마다 달라져요. 왜죠?

ASLR(주소 공간 배치 무작위화) 때문이다. OS가 보안을 위해 프로그램·힙·스택의 시작 위치를 매번 바꾼다. 그리고 애초에 이 주소는 물리 메모리 주소가 아니라 가상 주소다 — 프로세스마다 자기만의 주소 공간을 갖고, OS가 뒤에서 물리 메모리로 변환해 준다. 가상 메모리는 CS 기초 레슨에서 제대로 다룬다.

포인터의 포인터(CCharacter**)도 있나요? 어디에 쓰죠?

있다. 포인터도 8바이트짜리 변수이므로 그것의 주소를 담는 변수를 또 만들 수 있다. 쓰임새: "포인터 자체를 바꿔치기해야 할 때". 예로 이 서버의 락프리 스택은 top 포인터를 CAS로 교체하는데, 그런 코드는 "포인터를 담은 자리"를 다뤄야 하므로 이중 포인터 개념이 전제된다. 지금은 "포인터도 그냥 변수라서 주소를 가질 수 있다"까지만 소화하면 충분하다.

delete[] players를 안 하면 어떻게 되나요?

프로그램이 끝날 때까지 그 456KB는 반환되지 않는다 — 메모리 누수(leak). 서버는 몇 주씩 켜 두는 프로그램이라 누수가 쌓이면 결국 메모리가 바닥난다. 언제 new/delete를 해야 하고 어떻게 자동화하는지(RAII)가 바로 다음 레슨의 주제다.
