스택, 힙, 객체의 수명

모든 객체는 태어나고 죽는다. 문제는 누가, 언제 죽이느냐다. 오늘의 목표: 이 서버의 players가 왜 힙에, around가 왜 스택에 사는지 설명할 수 있게 된다.

왜 이걸 배우나
서버가 죽는 버그의 왕은 수명 버그다: 이미 해제된 메모리 사용(use-after-free), 이중 해제, 누수. 이 서버의 가장 정교한 장치 — lIOCount 참조카운트, CMessageBuffer의 AddRef/DecRef, 세션 릴리즈를 GameThread로 몰아넣는 설계 — 는 전부 "멀티스레드에서 객체를 언제 죽여도 안전한가"라는 수명 문제의 답이다. 오늘은 그 1인분 버전(싱글스레드 수명)을 완전히 잡는다.

1. 지식 — 메모리의 두 구역

스택(stack): 함수와 함께 자동으로 태어나고 죽는다

함수가 호출되면 그 함수의 지역 변수들을 담을 공간(스택 프레임)이 쌓이고, 함수가 return하면 통째로 사라진다. SendAround를 보라:

void CFighterServer::SendAround(DWORD index, CMessageBuffer* buffer, bool bSendMe)
{
    st_SECTOR_AROUND around;   // 스택에 생성 — new 없음
    int iSectorX;
    int iSectorY;
    ...
}   // ← 여기서 around는 자동으로 소멸. 코드 한 줄 안 썼는데.

빠르고(포인터 하나 옮기는 비용), 해제를 잊을 수 없다. 대신 크기가 컴파일 타임에 정해져야 하고, 함수가 끝나면 무조건 죽으며, 스택 전체 크기가 작다(Windows 기본 1MB).

힙(heap): 내가 태어나게 하고, 내가 죽여야 한다

CFighterServer.cpp:29:

players = new CCharacter[uiMaxSession];   // 힙에 9500개 × 48바이트 = 약 456KB

new는 힙에서 공간을 받아 생성자를 부르고 주소를 돌려준다. 이 객체는 함수가 끝나도 delete할 때까지 살아 있다. 크기를 실행 시점에 정할 수 있고(uiMaxSession은 런타임 인자!), 함수 수명과 무관하게 오래 살 수 있다. 대가: 느리고(할당자 호출), 죽이는 책임이 나에게 있다.

왜 players는 힙이어야만 하는가 — 3연타

  1. 크기를 런타임에 안다: uiMaxSession은 main이 넘겨주는 값. 스택 배열은 컴파일 타임 크기만 가능.
  2. 너무 크다: 456KB는 1MB 스택에 올리기 부담스럽다.
  3. 수명이 함수보다 길다: Start()가 return해도 서버가 도는 내내 살아야 한다.

죽이는 책임: 짝을 맞춰라

소멸자가 그 책임을 진다:

CFighterServer::~CFighterServer()
{
    delete[] players;   // new[]로 만들었으니 delete[]
    delete sectors;     // new로 만들었으니 delete
}

규칙: new ↔ delete, new[] ↔ delete[]. 짝이 어긋나면 미정의 동작(UB)이다. 그리고 이 코드가 보여주는 것이 바로 RAII의 원형이다: "자원의 해제를 소멸자에 묶어라. 그러면 객체가 죽을 때 자원도 반드시 따라 죽는다." C++ Core Guidelines: RAII

3대 수명 버그
누수(leak): delete를 잊음 — 장수 프로세스인 서버의 암살자. 댕글링(dangling): delete된 주소를 계속 들고 있다가 역참조 — 크래시 또는 더 나쁘게는 조용한 오염. 이중 해제(double free): 같은 주소를 두 번 delete — 힙 구조 파괴. 셋 다 "주소를 든 자와 죽이는 자가 여럿"일 때 터진다. 이 서버가 CMessageBuffer에 참조카운트를 단 이유다.

2. 스킬 — 수명을 눈으로 보기

scratch\lifetime_check.cpp:

#include <cstdio>

struct Probe {
    int id;
    Probe(int i) : id(i)  { printf("  [%d] 생성 (주소 %p)\n", id, (void*)this); }
    ~Probe()              { printf("  [%d] 소멸\n", id); }
};

Probe* leak_demo() {
    Probe stackObj(1);                 // 스택
    Probe* heapObj = new Probe(2);     // 힙
    printf("함수 끝나기 직전\n");
    return heapObj;
}   // 여기서 [1]만 자동 소멸하는 것을 관찰하라

int main() {
    Probe* p = leak_demo();
    printf("함수는 끝났지만 [2]는 살아있다: id=%d\n", p->id);
    delete p;                          // 이 줄을 지우고 다시 실행 → [2] 소멸이 안 찍힌다 = 누수
}
cl /EHsc /nologo lifetime_check.cpp && lifetime_check.exe

실습 순서: ① 그대로 실행하고 출력 순서를 예측→확인. ② delete p;를 지우고 [2]의 소멸이 사라지는 것 확인. ③ delete p;를 두 번 써서 이중 해제가 어떻게 되는지 관찰(크래시하거나, 더 무섭게도 아무 일 없어 보일 수 있다 — 그것이 UB다).

스택과 힙은 물리적으로 다른 메모리인가요?

아니다. 같은 RAM이다. 다른 것은 관리 방식이다. 스택은 "쌓고-걷어내기"만 하는 단순한 영역(그래서 빠름), 힙은 아무 때나 아무 크기나 빌리고 반납하는 자유 저장소(그래서 관리 비용이 있음). 프로세스의 가상 주소 공간 안에서 서로 다른 구역을 차지할 뿐이다.

생성자/소멸자가 정확히 뭐예요? 안 만들면요?

생성자는 객체가 태어나는 순간, 소멸자는 죽는 순간 자동 호출되는 함수다. 안 만들면 컴파일러가 "아무것도 안 하는" 기본판을 만들어 준다 — 그래서 레슨 1의 CCharacter는 소멸자가 없어도 된다(들고 있는 자원이 없으니까). 반면 CFighterServer는 힙 메모리를 소유하므로 소멸자에서 반납해야 한다. 원칙: 자원을 소유한 클래스만 소멸자가 필요하다.

delete를 하면 포인터 변수는 어떻게 되나요?

그대로다! delete는 가리키던 객체를 죽일 뿐, 포인터 변수에는 죽은 주소가 그대로 남는다(댕글링 포인터). 그래서 방어적으로 delete p; p = nullptr;를 쓰는 관례가 있다 — nullptr 역참조는 즉시 크래시라서 조용한 오염보다 백배 낫다. CFighterServer 생성자가 players = nullptr로 시작하는 것도 같은 정신이다.

new를 했는데 메모리가 부족하면요?

기본 동작은 std::bad_alloc 예외를 던진다(포인터가 nullptr로 오는 게 아니다 — C의 malloc과 다른 점). 64비트 서버에서는 가상 주소 공간이 커서 흔치 않지만, 물리 메모리+페이지파일이 바닥나면 일어난다.

이 서버가 시작 시점에 세션 배열(new stSESSION[uiMaxSession])을 한 번에 전부 할당하는 진짜 이유는 따로 있다 — 할당 실패 방지는 부차적 효과일 뿐이다. 핵심은 운영 중엔 세션을 new/delete하지 않는다는 것:

  • 정확성(진짜 킬러): IOCP에선 종료 시점에도 비동기 IO가 커널에 걸려 있을 수 있다. 그 OVERLAPPED가 세션 객체 안에 사는데, 세션을 delete하면 뒤늦은 완료 통지가 해제된 메모리를 건드린다(use-after-free → 크래시). 그래서 메모리는 절대 반납하지 않고, lIOCount(진행 중 IO 수)가 0이 될 때까지 릴리즈를 미룬 뒤 슬롯 인덱스만 락프리 스택으로 재사용한다.
  • 성능: 접속/종료마다 new/delete를 하면 여러 워커 스레드가 힙 전역 락에서 직렬화되고 단편화된다. 미리 잡아두고 인덱스로 빌려주기/돌려받기 하면 O(1)이다.

두 이유 모두 뒤(멀티스레드·락프리 레슨)에서 해부한다. 앵커: lIOCount가 0일 때 릴리즈하는 로직은 CServer.cpp의 IO 완료 처리부에 있다.

왜 new/delete를 매번 하지 않고 메모리 풀(CMemoryPool)을 쓰나요?

속도와 예측 가능성. new/delete는 OS 할당자를 거치는 비싼 연산이고, 초당 수만 패킷이 오가는 서버에서 패킷마다 new를 하면 그 비용이 지배적이 된다. 풀은 시작할 때 큰 덩어리를 받아 놓고 "빌려주기/돌려받기"만 하므로 몇 나노초면 된다. CMessageBuffer가 매번 new되지 않고 풀에서 Alloc되는 이유다. 풀 자체는 락프리 레슨에서 해부한다.

"OS 할당자를 거치는 비싼 연산"이 구체적으로 무슨 절차인지(락 → 프리 리스트 검색 → 쪼개기 → 헤더 쓰기, 그리고 최악의 경우 커널 왕복 두 번)는 파생 문서 3-1: "비싼 연산"의 정체에서 생략 없이 펼친다.

스택 1MB를 넘게 쓰면 어떻게 되나요?

스택 오버플로 — 즉시 크래시(0xC00000FD). 흔한 원인 둘: 함수 안에 거대한 지역 배열 선언(char buf[2000000]), 그리고 끝없는 재귀. "큰 것, 크기를 모르는 것, 오래 살 것은 힙으로"가 규칙인 이유다.

요즘은 스마트 포인터(unique_ptr) 쓰라던데, 이 코드는 왜 생 포인터죠?

좋은 지적이고, 현대 C++라면 std::unique_ptr<CCharacter[]> players가 정석이다(소멸자 코드 자체가 필요 없어진다). 이 코드베이스는 게임서버 업계의 전통적 스타일(C++14, 수동 관리)로 쓰였다 — 수동 수명 관리를 먼저 이해해야 스마트 포인터가 "무엇을 자동화해 주는지" 보이기 때문에, 이 코스에서는 원리를 먼저 배우고 스마트 포인터를 그 위에 얹는다.
