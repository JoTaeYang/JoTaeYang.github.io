객체를 바이트로 보기

클래스는 마법이 아니다. 그냥 연속된 바이트 덩어리다. 오늘의 목표: CCharacter의 크기를 계산으로 예측하고, 컴파일러로 검증한다.

왜 이걸 먼저 배우나
이 서버의 거의 모든 핵심 기법이 "객체 = 바이트"라는 사실 위에 서 있다. 패킷 직렬화(CMessageBuffer)는 객체를 바이트로 복사하는 것이고, 메모리 풀의 128비트 CAS는 16바이트 정렬을 요구하며, 섹터를 list에서 vector로 바꾼 이유는 바이트가 메모리에 어떻게 배치되는지가 성능을 좌우하기 때문이다. 이 레슨이 그 전부의 주춧돌이다.

1. 지식 — 세 가지 규칙

규칙 1: 멤버는 선언 순서대로 배치된다

객체는 멤버 변수를 선언한 순서대로 메모리에 늘어놓은 것이다. 우리 교재인 CCharacter.h를 보자 (실제 코드 그대로):

class CCharacter {
    __int64 SessionKey;        // 8바이트
    DWORD dwAction;            // 4바이트 (unsigned long)
    BYTE byDirection;          // 1바이트
    BYTE byMoveDirection;      // 1바이트
    short shX;                 // 2바이트
    short shY;                 // 2바이트
    st_SECTOR_POS CurSectorPos;// int 2개 = 8바이트
    st_SECTOR_POS OldSectorPos;// 8바이트
    int iSectorIdx;            // 4바이트
    char chHP;                 // 1바이트
};                             // 합계: 39바이트... 정말?

단순 합은 39바이트다. 하지만 sizeof(CCharacter)는 39가 아니다. 규칙 2 때문이다.

규칙 2: 정렬(alignment) — 타입은 자기 크기의 배수 주소에만 놓인다

CPU는 4바이트 int를 4의 배수 주소에서, 8바이트 __int64를 8의 배수 주소에서 읽을 때 가장 빠르다(일부 아키텍처에선 그렇지 않으면 크래시). 그래서 컴파일러는 멤버 사이에 패딩(padding)이라는 빈 바이트를 끼워 넣어 각 멤버를 제 정렬에 맞춘다. cppreference: Alignment

규칙 3: 전체 크기는 가장 큰 정렬의 배수로 올림된다

배열 CCharacter arr[2]에서 arr[1]의 멤버들도 정렬이 맞아야 하므로, 구조체 전체 크기는 멤버 중 가장 큰 정렬 요구(여기선 __int64의 8)의 배수로 올림된다. MS Learn: Alignment

2. 손으로 계산해 보기

오프셋(객체 시작 기준 몇 번째 바이트인지)을 하나씩 따라가자:

dwAction4 / 48–118은 4의 배수 ✓
byDirection1 / 112
byMoveDirection1 / 113

shY2 / 216–17
패딩218–19다음 멤버가 4정렬 필요

따라서 sizeof(CCharacter) == 48. 39바이트 데이터에 9바이트가 패딩이다.

떡밥
멤버 순서만 바꾸면 이 클래스는 40바이트가 된다 — 8바이트(17%) 절약. 왜 그런지, 언제 그럴 가치가 있는지는 캐시 레슨에서 다룬다.

3. 스킬 — 직접 검증하기

예측을 컴파일러에게 채점받자. 아래 파일을 e:\2026_por\Study\scratch\sizeof_check.cpp로 저장하고 실행해 보라:

#include <cstdio>
#include <cstddef>   // offsetof
#include <windows.h>
#include "../FighterServer/CCharacter.h"

int main() {
    printf("sizeof(CCharacter) = %zu\n", sizeof(CCharacter));
    printf("offsetof shY         = %zu\n", offsetof(CCharacter, shY));
    printf("offsetof CurSectorPos= %zu\n", offsetof(CCharacter, CurSectorPos));
    printf("offsetof chHP        = %zu\n", offsetof(CCharacter, chHP));
}

Visual Studio 개발자 명령 프롬프트(또는 cl이 잡히는 셸)에서:

cl /EHsc /nologo sizeof_check.cpp && sizeof_check.exe

출력이 위 표와 일치하는지 확인하라. 불일치하면 그게 오늘의 가장 값진 배움이다 — 표의 어느 줄이 틀렸는지 추적해 보고, 막히면 에이전트(나)에게 물어보라.

보너스: 비주얼 확인
Visual Studio 디버거로 CCharacter 객체를 만들고 조사식 창에 &obj,48을 넣거나, 메모리 창(디버그 → 창 → 메모리)에 &obj를 넣으면 방금 계산한 레이아웃을 눈으로 볼 수 있다. MSVC 전용 트릭: 컴파일 옵션 /d1reportSingleClassLayoutCCharacter를 주면 컴파일러가 레이아웃을 직접 출력해 준다.

4. 확인 퀴즈

5. 초보자 질문 세트

이 레슨을 읽은 초보자가 물어볼 법한 질문들이다. 클릭하기 전에 먼저 스스로 답해 보라 — 답이 안 나오면 그게 지금 메워야 할 구멍이다.

DWORD, BYTE, __int64… 이게 다 뭔가요? int랑 뭐가 달라요?

전부 Windows 헤더(windows.h)가 만들어 둔 별칭(typedef)이다. BYTE = unsigned char(1바이트), WORD = unsigned short(2바이트), DWORD = unsigned long(4바이트), __int64 = MSVC의 8바이트 정수(표준 C++의 long long과 같음). Windows API가 이 이름들을 쓰기 때문에 이 코드베이스도 따라 쓴다. 새 이름이 나올 때마다 "몇 바이트짜리 무슨 부호의 정수인가"로 번역하면 된다.

패딩 바이트 안에는 무슨 값이 들어 있나요?

아무 의미 없는 쓰레기값이다. 컴파일러는 패딩을 초기화할 의무가 없다. 이게 실전에서 중요한 이유: 객체를 통째로 memcpy해서 네트워크로 보내면 쓰레기 패딩까지 같이 나간다(정보 유출 + 낭비). 이 서버의 Protocol.h가 #pragma pack(push, 1)으로 패킷 구조체의 패딩을 제거하는 이유가 바로 이것이다 — 직렬화 레슨에서 자세히 다룬다.

패딩이 낭비라면 그냥 항상 없애면 안 되나요?

#pragma pack(1)로 없앨 수 있지만 공짜가 아니다. 정렬이 깨진 멤버를 읽을 때 CPU가 추가 작업을 하므로 느려질 수 있고, 일부 상황(예: 정렬을 요구하는 원자 연산, SIMD)에서는 아예 오동작한다. 그래서 원칙은: 메모리 안에서 쓰는 객체는 패딩을 허용하고, 네트워크/파일로 나가는 바이트 배치만 pack으로 통제한다. 이 코드베이스도 정확히 그렇게 한다.

클래스에 함수(InitPlayer 등)도 있는데, 함수는 48바이트 어디에 있나요?

객체 안에 없다. 멤버 함수의 기계어 코드는 프로그램 전체에 한 벌만 있고(코드 영역), 객체 100만 개를 만들어도 복사되지 않는다. obj.InitPlayer(...)는 사실 "InitPlayer 함수를 호출하되 this라는 숨은 인자로 &obj를 넘겨라"로 번역될 뿐이다. 그래서 sizeof에는 데이터 멤버만 반영된다. (예외: virtual 함수가 있으면 vptr이라는 포인터 8바이트가 객체에 추가된다 — 상속 레슨에서 다룬다. CServer의 OnRecv 등이 실제 사례다.)

st_SECTOR_POS는 구조체인데, 구조체 안의 구조체는 어떻게 배치되나요?

그 자리에 통째로 펼쳐진다. CurSectorPos는 오프셋 20에서 시작하는 8바이트(int iX, int iY)일 뿐, 별도의 장소를 가리키는 참조 같은 게 아니다. 중첩 구조체의 정렬 요구는 "그 구조체 멤버 중 가장 큰 정렬"(여기선 int의 4)을 물려받는다. 이 성질 덕분에 C++ 객체는 자바 객체와 달리 한 덩어리 연속 메모리다 — 캐시 레슨에서 이게 왜 성능의 핵심인지 본다.

sizeof는 프로그램이 실행될 때 계산하나요?

아니다. 컴파일 시점에 이미 상수 48로 박힌다. 타입의 레이아웃은 컴파일러가 코드를 번역하는 순간 완전히 확정되기 때문이다. 그래서 static_assert(sizeof(CCharacter) == 48, "layout changed!")처럼 컴파일 자체를 실패시키는 안전장치로도 쓸 수 있다 — 프로토콜 구조체 크기를 지키는 실무 기법이다.

포인터가 8바이트라는데, 32비트에선 4바이트라던데요?

맞다. 포인터 크기 = 그 플랫폼의 주소 폭이다. x86(32비트)은 4바이트, x64는 8바이트. 이 프로젝트는 CMakeLists.txt에서 x64를 강제하는데, 이유 중 하나가 InterlockedCompareExchange128(128비트 CAS)이 x64 전용 명령이기 때문이다. 즉 "포인터 8바이트 + 카운터 8바이트 = 16바이트를 한 번에 원자적으로 교체"하는 락프리 기법이 x64 주소 폭을 전제로 설계돼 있다.

public/private을 바꾸면 메모리 배치도 바뀌나요?

배치는 바뀌지 않는다(같은 접근 지정자 구간 안에서는 선언 순서 보장). 접근 지정자는 컴파일러가 누구의 접근을 허용할지만 검사하는 컴파일 타임 규칙이고, 실행 시점 메모리에는 흔적이 없다. private 멤버도 메모리 창으로 보면 그냥 바이트다 — "정보 은닉"은 보안 장치가 아니라 설계 규율이라는 뜻.
