# 레슨 3 — 사용자가 세션 사전할당의 "진짜 이유"를 스스로 교정

2026-07-12, 레슨 3(스택·힙·수명)의 FAQ가 "세션 배열을 한 번에 할당하는 이유 중 하나 = 운영 중 할당 실패로 죽는 일 방지"라고 적은 것을 **사용자가 반박**: "그게 아니라 멀티스레드 환경에서 할당/해제가 위험해서, 해제가 필요 없게 미리 만들고 재사용하는 것"이라고 정정.

코드로 검증 → **사용자가 옳음.** 근거:
- [[codebase-map]] / CServer.cpp:51 `new stSESSION[uiMaxSession]` — 부팅 시 1회 배열 할당.
- CServer.cpp:54 `sessionIndexStack`(CLockFreeStack) — 슬롯을 인덱스로 claim/return, 런타임 new/delete 없음.
- stSESSION.h:57 `lIOCount`, CServer.cpp:446 `InterlockedDecrement(&lIOCount)==0` 에서만 릴리즈.

진짜 이유 2가지로 정리(레슨 본문도 이대로 수정함):
- **A. 정확성(킬러)**: OVERLAPPED가 세션 객체 안에 살아, 종료 후 뒤늦은 IO 완료가 해제 메모리를 건드리는 use-after-free를 막으려면 메모리를 반납하면 안 됨. lIOCount=0까지 릴리즈 지연 + 인덱스 재사용.
- **B. 성능**: 접속/종료마다 new/delete → 멀티스레드 힙 전역 락 직렬화·단편화.

원래 레슨의 "할당 실패 방지"는 부차적 효과였을 뿐. FAQ 문장 교체 완료([[0003-stack-heap-lifetime]] line 98 부근).

## Implications

- **ZPD 재조정 신호**: 사용자는 C++ 문법(패딩 산수 등)에서는 아직 흔들리지만, **시스템 설계 의도(왜 이 구조인가)** 직관은 레슨 서술을 능가하는 수준. 앞으로 D(IOCP)·E(락프리) 레슨은 "무엇을 하는가"보다 **"왜 이렇게 안 하면 깨지는가"(실패 시나리오 우선)** 로 짜는 게 더 잘 맞는다.
- lIOCount 릴리즈 메커니즘·use-after-free는 이미 사용자 머릿속에 씨앗이 있음 → 멀티스레드 레슨에서 이 앵커(CServer.cpp:446, lSendFlag CAS→2 @719)로 곧장 연결.
- 레슨을 "정답 전달"이 아니라 **반박 대상**으로 내놓으면 사용자가 더 깊이 관여함 — 향후 레슨에 의도적으로 "이 설명의 약점은?" 트리거를 넣는 것을 검토.
