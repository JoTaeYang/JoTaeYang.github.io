# 레슨 1 통과 — 단, 패딩 규칙 구분과 포인터 크기에 혼동

2026-07-06, 레슨 1의 sizeof 실습을 직접 컴파일해 검증 성공(48바이트 예측 일치). 레이아웃 계산 절차 자체는 손에 잡혔다는 증거. 그러나 퀴즈 1·3을 틀렸다:

- **퀴즈 1 (중간 패딩의 이유)**: 멤버 사이 패딩(다음 멤버의 정렬 요구)과 꼬리 패딩(구조체 전체 크기 = 최대 정렬의 배수)이라는 **두 규칙을 구분하지 못함**. "8의 배수" 규칙을 중간 패딩에도 적용하려 한 것으로 추정.
- **퀴즈 3 (포인터의 크기)**: sizeof(CCharacter*)가 대상 객체 크기(48)와 무관하게 x64에서 8바이트 고정이라는 것이 아직 안 잡힘. **"포인터 = 대상과 분리된 8바이트 주소 변수"라는 심상이 미형성.**

## 갱신 (2026-07-06, 같은 날)

파생 문서 0001-1의 변형 퀴즈 2개(중간 패딩의 근거 규칙, 포인터 크기 불변) **모두 정답** — 두 혼동 모두 즉시 교정됨. 레슨 2 진행 승인. 단, 며칠 뒤 레슨 1 원본 퀴즈로 간격 재검은 여전히 유효.

## 갱신 (2026-07-12, +6일 간격 복습)

레슨 2(포인터) 완료 + 퀴즈 전부 통과. 진입 전 간격 재검(3문항 자유회상) 실시:

- **포인터 크기(구 퀴즈 3 오답 지점)** — 완전 저장됨. "8바이트, 아키텍처가 정함, 대상과 무관" 즉답.
- **포인터 연산 `players[index]`** — 저장됨. `48 × index` 스케일링 정확(시작 주소 base 한 조각만 보강).
- **중간 패딩 규칙** — **6일 뒤 흐려짐**. 꼬리 패딩(규칙 B)은 정확했으나 중간 패딩을 "4바이트씩"으로 오기억. → 실제 CCharacter(shY→CurSectorPos, 18→20 2바이트) 예시로 재교정, 새 드릴 D1/D2 출제. 채점 대기 중.

## Implications

- 간격 복습이 제 역할을 함 — fluency(당일)로는 통과했으나 storage는 중간 패딩에서 부분 실패. 이 지점은 앞으로도 재검 후보(레슨 7 캐시/정렬에서 자연 재등장 예정).
- 드릴 결과: **D2(3멤버, 중간 패딩 위치·양) 정답** → 규칙 A 개념 통과. **D1은 패딩 양 1바이트를 2로 오산** — 개념 아닌 오프셋 산수 슬립. 공식 `패딩 = (정렬 − 오프셋%정렬) % 정렬`로 봉인.
- 레슨 2 통과 + 간격 복습 완료 → **레슨 3(스택·힙·수명) 진입 승인 (2026-07-12)**.
