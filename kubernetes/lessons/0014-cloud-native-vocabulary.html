Kubernetes Mastery · Lesson 0014 어휘 일괄 · 저비중 도메인 3개

클라우드 네이티브 어휘 일괄

비중 낮은 세 도메인(아키텍처 16% · 관측성 8% · 딜리버리 8%)을 한 번에. 손으로 만지기보단 시험에 나올 단어에 이름표를 붙이는 레슨 — 과투자 금지.

이번 레슨의 win: KCNA 에 나오는 "쿠버네티스 바깥의 클라우드 네이티브 세계" 단어들 — CNCF 성숙도, 12-factor, 관측성 3기둥, GitOps, Helm — 을 지금까지 손으로 만진 것과 연결해 암기가 아니라 지도로 기억한다. 이걸로 갭 체크의 마지막 ⬜ 칸들이 닫힌다.

0. 한눈에 — 세 질문

딱 이 세 줄

어디서 왔나 (아키텍처) = 이 소프트웨어들은 누가 만들고 얼마나 검증됐나 → CNCF 와 성숙도 단계. 그리고 앱을 어떻게 지어야 클라우드에서 잘 도나 → 12-factor.

어떻게 보나 (관측성) = 돌아가는 시스템 안을 어떻게 들여다보나 → 3기둥: 메트릭·로그·트레이스.

어떻게 배포하나 (딜리버리) = YAML 을 손으로 안 치고 어떻게 안전하게 굴리나 → GitOps(Git 이 진실) 와 Helm(패키지 매니저).

1. 어디서 왔나 — CNCF 와 성숙도

CNCF(Cloud Native Computing Foundation)는 쿠버네티스를 비롯한 클라우드 네이티브 프로젝트들의 중립적 관리 재단이다. 특정 회사 소유가 아니라는 게 핵심 — 그래서 벤더 종속(lock-in)을 피한다. 프로젝트는 검증된 정도에 따라 세 단계를 밟는다:

단계뜻예
Sandbox초기·실험적. 재단이 "지켜보는 중"신생 프로젝트들
Incubating실사용처가 생기고 성장 중Argo, Cilium (시점따라 변동)
Graduated성숙·안정. 프로덕션에 널리 검증됨Kubernetes, Prometheus, Helm, containerd, etcd
외우는 법 Sandbox → Incubating → Graduated = "모래밭에서 놀다 → 알에서 부화 → 졸업". 왼쪽일수록 실험적, 오른쪽일수록 안정. KCNA 는 이 순서와 "graduated = 가장 성숙"만 물으면 충분하다. 어떤 프로젝트가 어느 단계인지 통째로 외울 필요는 없다 — CNCF Landscape 한 번 훑어보는 걸로 감만.

12-factor app — 클라우드에서 잘 도는 앱을 짓는 12 가지 원칙(2011, Heroku). 전부 외울 필요 없고, 쿠버네티스가 이미 강제하는 것들과 연결하면 저절로 남는다:

12-factor 원칙이미 배운 것과 연결
설정은 환경변수에 (코드에 박지 마라)0005 ConfigMap/Secret 을 env 로 주입한 것
프로세스는 stateless·disposable (언제 죽어도 됨)0002 Pod 는 언제든 교체됨 / 상태는 0011 PV 로 밖에
로그는 이벤트 스트림으로 (파일에 쌓지 말고 stdout 으로)0004·0006 에서 kubectl logs 로 본 것
dev/prod 최대한 동일 (parity)같은 이미지·같은 매니페스트를 환경만 바꿔 배포
마이크로서비스 vs 모놀리스도 이 도메인 단골. 모놀리스 = 한 덩어리 앱, 마이크로서비스 = 작게 쪼개 독립 배포·확장. 쿠버네티스는 후자를 굴리기 좋은 판. OCI(Open Container Initiative)는 "컨테이너 이미지·런타임의 열린 표준" — 0012 에서 "Docker 를 지워도 OCI 이미지는 그대로 돈다"던 그 표준이다.

2. 어떻게 보나 — 관측성 3기둥

돌아가는 시스템 안에서 무슨 일이 일어나는지 아는 능력 = 관측성(observability). 세 종류의 신호로 본다:

기둥답하는 질문대표 도구 (CNCF)
메트릭(Metrics)"지금 얼마나?" — 수치·추세 (CPU, 요청수)Prometheus (+ Grafana 대시보드)
로그(Logs)"무슨 일이 있었나?" — 개별 사건 기록Fluentd, Loki
트레이스(Traces)"이 요청이 어느 서비스들을 거쳤나?" — 경로 추적Jaeger, OpenTelemetry
한 줄 구분 메트릭 = 숫자, 로그 = 문장, 트레이스 = 지도. 마이크로서비스에서 요청 하나가 10개 서비스를 거치면 로그만으론 경로를 못 쫓으니 트레이스가 필요해진 것. Prometheus 는 pull 방식(직접 긁어옴)이라는 점, 그리고 이미 0004·0006 에서 쓴 kubectl logs/describe/events 가 로그 기둥의 손맛이라는 것만 연결하면 KCNA 는 충분.

3. 어떻게 배포하나 — GitOps 와 Helm

GitOps — Git 저장소를 "원하는 상태"의 유일한 진실(source of truth)로 삼는 배포 방식. 사람이 클러스터에 직접 kubectl apply 하지 않고, Git 에 커밋하면 에이전트(ArgoCD/Flux)가 클러스터를 그 상태로 맞춘다.

아하 — 이미 아는 그 루프다 GitOps 의 "실제 상태를 원하는 상태로 계속 맞춘다"는 0007 의 제어 루프와 정확히 같은 구조다. 차이는 원하는 상태가 어디 사느냐뿐 — 컨트롤러는 etcd 를 보고, GitOps 에이전트는 Git 을 본다. 그래서 "클러스터가 Git 과 달라지면 되돌린다(drift 교정)". 롤백 = git revert. 감사 = git log.

Helm — 쿠버네티스의 패키지 매니저(apt/brew 의 쿠버네티스판). 지금까지 손으로 쓴 YAML 뭉치를 차트(chart)라는 템플릿으로 묶어, 값(values.yaml)만 바꿔 설치·업그레이드·롤백한다.

Helm 용어뜻
Chart템플릿화된 매니페스트 묶음 (패키지 자체)
Values차트에 꽂는 설정값 (replicas, image tag…)
Release차트를 클러스터에 설치한 한 번의 인스턴스
KCNA 는 GitOps·Helm 을 개념 수준(무엇을 위한 것인가, 어떤 문제를 푸는가)까지만 묻는다. ArgoCD 설치나 차트 작성 실습은 CKAD/CKA 몫 — 여기선 이름표만 붙이고 넘어간다 (과투자 금지).

4. 관찰 (선택, 1분) — 내 클러스터에도 이미 있다

terminal $ helm version # 설치돼 있으면 graduated 프로젝트 하나를 손에 쥔 것 $ kubectl top nodes # metrics-server 있으면 = 메트릭 기둥의 맛보기 (없으면 "설치 필요" 뜸) $ kubectl logs -n kube-system -l component=kube-apiserver --tail=3 # 로그 기둥

없어도 정상 — 이 레슨은 개념이 목표다. 있으면 "아, 이게 그거였네" 하고 넘어가면 된다.

5. 회상 연습

Q1. CNCF 프로젝트 성숙도 단계를 덜 성숙 → 더 성숙 순서로 옳게 놓은 것은?

Sandbox(실험) → Incubating(성장) → Graduated(성숙·프로덕션 검증). Kubernetes·Prometheus·Helm 은 graduated. 왼쪽일수록 실험적, 오른쪽일수록 안정이다.

Q2. 관측성 3기둥 중 "이 요청 하나가 어떤 서비스들을 순서대로 거쳤는지" 경로를 보여주는 것은?

메트릭=숫자(얼마나), 로그=문장(무슨 일), 트레이스=지도(어느 경로). 마이크로서비스에서 요청이 여러 서비스를 거칠 때 경로를 쫓으려면 트레이스가 필요하다.

Q3. (레슨 0007 복습) GitOps 가 "클러스터를 Git 의 상태로 계속 맞춘다"고 할 때, 이 동작과 구조가 같은 쿠버네티스 개념은?

간격 반복: GitOps 는 0007 의 제어 루프를 클러스터 밖으로 옮긴 것이다. 컨트롤러는 원하는 상태를 etcd 에서 읽고, GitOps 에이전트(ArgoCD/Flux)는 Git 에서 읽는다 — "현재≠원함이면 맞춘다"는 골격은 동일. 그래서 drift 교정·git revert 롤백이 공짜로 따라온다.
판정: Q1·Q2·Q3 를 맞히면 저비중 세 도메인(아키텍처·관측성·딜리버리)이 어휘 수준으로 ✅ — KCNA 범위가 사실상 다 채워진다. 다음은 전 도메인 모의 진단: Fundamentals·오케스트레이션·아키텍처·관측성·딜리버리를 섞은 퀴즈로 마지막 갭을 확인하고 응시 일정을 잡는다.

초보자 질문 (막히면 여기부터)

이 레슨은 단어가 많아 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

CNCF 랑 쿠버네티스는 무슨 관계예요?

CNCF 는 쿠버네티스를 보유·관리하는 재단이다 (구글이 만들어 2015 년 CNCF 에 기증). 쿠버네티스는 CNCF 의 첫 graduated 프로젝트 중 하나. CNCF 는 Prometheus·Helm·etcd 등 수백 개 프로젝트의 벤더 중립적 우산이다.

Sandbox / Incubating / Graduated, 실무에서 어떻게 써먹어요?

기술 선택의 리스크 신호다. graduated 는 프로덕션에 널리 검증됐으니 안심, sandbox 는 유망하지만 갑자기 방향이 바뀌거나 버려질 수 있으니 신중히. "이거 도입해도 되나?"에 대한 CNCF 의 성숙도 도장인 셈.

12-factor 는 쿠버네티스 전용 규칙인가요?

아니다. 2011 년 Heroku 가 정리한 일반적인 클라우드 앱 원칙으로, 쿠버네티스보다 먼저다. 다만 쿠버네티스가 이 원칙들(환경변수 설정, stateless, 로그를 stdout 으로 등)을 자연스럽게 강제해서, 둘이 잘 맞는다. 12 개를 다 외우기보다 "왜 이렇게 지어야 하나"의 감을 잡는 게 KCNA 엔 충분.

메트릭·로그·트레이스, 언제 뭘 봐요?

메트릭: "느려졌다/CPU 튄다" 같은 추세·경보. 로그: 특정 시점에 정확히 뭐가 터졌나(에러 메시지, 스택트레이스). 트레이스: 요청이 여러 서비스를 거쳐 어디서 느려졌나. 보통 메트릭으로 이상을 감지 → 로그로 원인 파악 → 트레이스로 경로 병목 추적, 이 순서로 쓴다.

Prometheus 가 "pull 방식"이라는 게 무슨 뜻이에요?

앱이 Prometheus 로 데이터를 보내는(push) 게 아니라, Prometheus 가 각 앱의 /metrics 엔드포인트를 주기적으로 긁어온다(pull/scrape). 그래서 "이 앱을 감시 대상에 넣어라"만 설정하면 되고, 앱은 Prometheus 주소를 몰라도 된다. KCNA 는 "Prometheus=메트릭, pull 방식" 정도면 충분.

OpenTelemetry 는 뭐예요? Jaeger 랑 뭐가 달라요?

OpenTelemetry(OTel) = 관측성 데이터(트레이스·메트릭·로그)를 만들고 내보내는 벤더 중립 표준·SDK. Jaeger = 그렇게 모인 트레이스를 저장·조회하는 백엔드. OTel 로 계측하고 Jaeger 에서 본다, 는 조합이 흔하다. KCNA 는 "OTel=표준화된 계측"만 알면 된다.

GitOps 랑 그냥 CI/CD 는 뭐가 달라요?

CI/CD 는 보통 파이프라인이 클러스터에 밀어넣는다(push) — kubectl apply. GitOps 는 클러스터 안 에이전트가 Git 을 지켜보다 스스로 끌어와 맞춘다(pull). 차이의 핵심은 "원하는 상태가 Git 에 선언적으로 있고, 실제와 다르면 자동 교정"이라는 점. CI(빌드/테스트)는 그대로 두고 배포(CD) 부분만 GitOps 로 바꾸는 조합이 흔하다.

ArgoCD 랑 Flux, 둘 중 뭘 알아야 해요?

KCNA 는 이름과 역할(둘 다 GitOps 에이전트)만 알면 된다. 실습·선택은 CKAD/CKA·실무 몫이다. 굳이 감을 잡자면 ArgoCD 는 웹 UI 가 강하고, Flux 는 더 가볍고 쿠버네티스 네이티브한 편 — 이 정도면 충분.

Helm chart 가 정확히 뭐예요? 왜 필요해요?

지금까지 손으로 쓴 Deployment·Service·ConfigMap YAML 들을 하나의 패키지로 묶고 값만 바꿔 재사용하게 만든 템플릿이 chart 다. "nginx 를 replicas=3 으로 깔아줘" 를 helm install 한 줄로 — 매번 YAML 을 복붙·수정할 필요가 없다. apt 로 프로그램 깔듯 쿠버네티스 앱을 깐다고 보면 된다.

그럼 kubectl apply 대신 항상 Helm 을 써야 하나요?

아니다. 간단한 몇 개 리소스면 kubectl apply 가 더 명확하다. Helm 은 여러 리소스를 한 단위로 배포·업그레이드·롤백하거나 남이 만든 앱(예: prometheus)을 통째로 설치할 때 빛난다. 규모와 재사용성이 판단 기준 — 학습 단계에선 손으로 YAML 을 쓰는 지금 방식이 오히려 이해에 좋다.
