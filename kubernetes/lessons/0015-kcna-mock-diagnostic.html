KCNA 전 도메인 모의 진단

18문항, 다섯 도메인을 실제 비중대로 섞었다. 정답률보다 어느 도메인에서 흔들리는지가 산출물 — 그 칸만 다시 보고 응시 일정을 잡는다.

이번 레슨의 win: 0001~0014 를 한 판에 모아, 남은 갭이 있는지 확인한다. 16개 이상이면 응시 준비 완료 신호. 찍지 말 것 — 헷갈린 문항은 헷갈린 채로 두는 게 진단에 이롭다. 문항 옆 괄호는 관련 레슨.
배점 비중: Fundamentals 8 · Orchestration 4 · Architecture 3 · Observability 2 · Delivery 1 (실제 KCNA 도메인 비중을 반영). 다 풀고 도메인별로 틀린 개수를 세보면 약한 칸이 드러난다.

1. Kubernetes Fundamentals (~46%)

Q1. (0002) Deployment 를 만들면 Pod 가 생긴다. 그 사이에 자동으로 끼는 리소스는?

Deployment → ReplicaSet → Pod. 롤링 업데이트는 새 ReplicaSet 을 만들어 Pod 를 옮기는 방식이다.

Q2. (0003) Pod 는 죽고 새로 태어나며 IP 가 계속 바뀌는데도 Service(ClusterIP) 주소가 안 바뀌는 이유는?

Service 의 ClusterIP 는 안정적 접점, 실제 대상 Pod 목록은 EndpointSlice 가 관리한다. Pod 가 바뀌면 목록만 갱신된다.

Q3. (0004) liveness probe 와 readiness probe 의 차이로 옳은 것은?

readiness = "지금 트래픽 받을 수 있나"(과녁에서 빼고 넣기), liveness = "살아있나, 아니면 재시작"(RESTARTS 증가).

Q4. (0005) ConfigMap 을 환경변수(env)로 주입한 Pod 에서, ConfigMap 값을 수정하면?

env 주입은 컨테이너 시작 시 한 번 값을 굳힌다. volume 주입은 파일이 갱신되지만, env 는 재생성이 필요하다 — 둘의 결정적 차이.

Q5. (0006) 컨테이너가 Exit Code 137 (OOMKilled) 로 죽었다. 직접 원인은?

limits 는 단속선 — 메모리 limit 초과 시 OOMKill(137). requests 부족은 스케줄 단계의 Pending 문제로, 죽는 게 아니라 못 뜨는 것이다.

Q6. (0009) 안정적 이름(web-0, web-1)과 순서 있는 생성이 필요한 워크로드는?

Deployment 는 "서로 바꿔도 되는 복제본"을 가정한다. 그 가정이 깨질 때(고유 신원) StatefulSet 을 쓴다 — 지우고 살려도 같은 이름으로 부활.

Q7. (0011) PVC 를 만들었더니 Pending 이다. kind 기본 StorageClass 환경에서 이게 정상인 이유는?

동적 프로비저닝 + WaitForFirstConsumer 는 소비자(Pod)가 나타날 때 PV 를 찍어낸다. Pod 를 붙이면 Bound 로 바뀐다 — LoadBalancer 의 영원한 <pending> 과는 다르다.

Q8. (0008) namespace lab 에 만든 Role + RoleBinding 으로 get pods 가 허용됐다. 같은 SA 로 get pods -n default 는?

Role/RoleBinding 은 namespace 경계 안에서만 유효하다. 전 namespace 나 클러스터 자원엔 ClusterRole 이 필요하다.

2. Container Orchestration (~22%)

Q9. (0012) 노드에서 get nodes -o wide 를 보니 CONTAINER-RUNTIME 이 containerd 다. kubelet 이 컨테이너를 실행할 때 거치는 표준 인터페이스는?

CRI = 실행, CNI = 네트워크, CSI = 스토리지. kubelet 은 CRI 를 통해 containerd 같은 런타임에 실행을 위임한다.

Q10. (0013) Pod 에 진짜 네트워크 인터페이스와 IP 를 만들어 주는 것은?

CNI 가 Pod 네트워크(실제 인터페이스+IP)의 도로를 깐다. kube-proxy 는 그 위에 Service 바꿔치기 규칙(이정표)을 세울 뿐이다.

Q11. (0013) 클러스터 안에서 ClusterIP 로 ping 은 안 되는데 curl 은 된다. 이유는?

ClusterIP 는 어떤 인터페이스에도 없다. curl 이 되는 건 목적지가 노드를 떠나기 전에 진짜 Pod IP 로 DNAT 되기 때문 — 그 규칙은 Service 포트에만 있으므로 ICMP 는 허공으로 간다.

Q12. (0007) "현재 상태를 원하는 상태로 수렴시키는" 제어 루프들이 실제로 도는 컴포넌트는?

scheduler 는 "어느 노드에" 한 번 정하고, apiserver 는 관문·저장 중계다. 실제 reconciliation 루프(ReplicaSet, Node, Job 컨트롤러 등)는 controller-manager 안에서 돈다.

3. Cloud Native Architecture (~16%)

Q13. (0014) CNCF 프로젝트 성숙도를 덜 성숙 → 더 성숙 순으로 옳게 놓은 것은?

Sandbox(실험) → Incubating(성장) → Graduated(프로덕션 검증). Kubernetes·Prometheus·Helm 은 graduated.

Q14. (0014/0012) "Docker 를 지워도 컨테이너 이미지는 그대로 돈다"를 보장하는 열린 표준은?

OCI 는 컨테이너 이미지·런타임의 벤더 중립 표준이다. 그래서 런타임을 containerd 로 바꿔도 같은 이미지가 그대로 실행된다.

Q15. (0014/0005) 12-factor 원칙 "설정은 환경에" 는 쿠버네티스의 무엇으로 실현되나?

설정을 코드/이미지에서 분리해 환경별로 주입 — 같은 이미지를 dev/prod 에서 다르게 돌리는 0005 의 핵심이 곧 12-factor 의 config 원칙이다.

4. Cloud Native Observability (~8%)

Q16. (0014) "요청 하나가 어떤 서비스들을 순서대로 거쳤는지" 경로를 보여주는 관측성 기둥은?

메트릭=숫자, 로그=문장, 트레이스=지도. 마이크로서비스에서 요청 경로를 쫓으려면 트레이스(Jaeger/OpenTelemetry)가 필요하다.

Q17. (0014) Prometheus 가 메트릭을 모으는 방식은?

Prometheus 는 pull 방식 — 감시 대상만 등록하면 앱은 Prometheus 주소를 몰라도 된다. KCNA 는 "Prometheus=메트릭, pull" 수준이면 충분.

5. Cloud Native Application Delivery (~8%)

Q18. (0014/0007) GitOps 의 핵심을 가장 잘 설명한 것은?

GitOps 는 0007 의 제어 루프를 클러스터 밖으로 옮긴 것 — 원하는 상태가 etcd 대신 Git 에 있고, drift 가 생기면 되돌린다. ArgoCD/Flux 가 그 에이전트다.
