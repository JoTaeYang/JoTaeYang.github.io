Kubernetes Mastery · Lesson 0018 심화 · Architecture

CNCF 커뮤니티·거버넌스·페르소나

기술이 아니라 사람과 조직을 배우는 레슨 — 누가 프로젝트를 승격시키고, 왜 한 회사가 못 좌우하며, "이 일은 누구 역할인가"를 KCNA 가 어떻게 묻는지.

이번 레슨의 win: KCNA 는 아키텍처 도메인에서 기술만큼 "조직·사람"을 묻는다 — Governing Board 대 TOC의 역할 분리, 프로젝트를 누가 승격시키나, 왜 오픈 거버넌스인가, 그리고 페르소나("이건 개발자 일? 운영 일? 보안 일?"). 이번 레슨으로 지금까지 배운 조각들에 역할 이름표를 붙여, 갭 체크의 마지막 개념 칸을 닫는다.

0. 한눈에 — 세 질문

딱 이 세 줄

누가 재단을 굴리나 = Governing Board(돈·예산·마케팅) 와 TOC(기술 방향·프로젝트 승격 투표) 두 축. 이 분리가 KCNA 단골.

왜 특정 회사가 못 좌우하나 = 오픈 거버넌스 = 벤더 중립. 0012 의 OCI 열린 표준과 같은 뿌리 — 그래서 lock-in 을 피한다.

이 일은 누구 역할인가 = 페르소나: application developer / operations·SRE·platform / security. 지금까지 배운 것에 역할을 매핑한다.

1. CNCF 의 두 축 — Governing Board 대 TOC

CNCF(Cloud Native Computing Foundation)는 Linux Foundation 산하의 재단이다. 쿠버네티스(0014 에서 본 첫 graduated 프로젝트 중 하나)를 비롯한 수백 개 프로젝트의 벤더 중립적 우산. 그런데 재단을 실제로 굴리는 힘은 두 개의 서로 다른 몸통으로 나뉜다 — 이 분리가 KCNA 의 단골 문제다.

기구맡는 일한 줄 기억
Governing Board (GB)비즈니스 — 예산, 마케팅, 회원사 관리, 이벤트(KubeCon)돈과 사업을 결정
TOC (Technical Oversight Committee)기술 방향, 프로젝트 성숙도 승급 투표, 기술 표준·정합성기술과 프로젝트를 결정
아하 — 지갑과 손은 다르다 Governing Board 는 "지갑", TOC 는 "손"이다. GB 는 돈을 걷고 쓰지만 어떤 프로젝트를 받아들일지·승격할지는 절대 못 정한다 — 그건 오직 TOC 의 몫. 반대로 TOC 는 예산이나 마케팅을 정하지 않는다. KCNA 가 노리는 함정이 바로 이것: "프로젝트를 graduated 로 올리는 건 누구?" → 답은 TOC(투표), Governing Board 가 아니다.

2. 성숙도 사다리 — 누가, 어떻게 움직이나

0014 에서 프로젝트 성숙도 사다리를 이름표만 붙였다: Sandbox → Incubating → Graduated. 여기선 그 사다리를 누가 밀어 올리나를 더 깊게 본다. 답은 위에서 봤듯 TOC 의 투표다.

단계뜻 (0014 복습)움직이는 힘
Sandbox초기·실험적. 재단이 "지켜보는 중"TOC 가 받아들임 (진입 장벽 낮음)
Incubating실사용처·기여자 커뮤니티가 생김TOC 투표로 승급
Graduated성숙·안정. 프로덕션에 널리 검증TOC 투표 + 엄격한 기준(보안 감사, 다수 벤더 기여, 거버넌스 문서)
Graduated 승급 기준에는 "한 회사에 종속되지 않을 것"(committer 가 여러 조직에 분산), 공개 거버넌스 문서, 보안 감사, CII/OpenSSF Best Practices 배지 등이 들어간다. 즉 사다리 자체가 벤더 중립성을 강제하는 장치다 — 다음 절의 오픈 거버넌스와 정확히 이어진다.

3. 왜 오픈 거버넌스인가 — 벤더 중립의 뿌리

오픈 거버넌스(open governance) = 프로젝트의 의사결정 권한이 한 회사에 몰려 있지 않고 공개된 규칙과 커뮤니티에 분산된 상태. 핵심 효과는 한 회사가 프로젝트를 좌우하지 못한다는 것 — 그래서 그 회사에 종속되는 lock-in 을 피한다.

아하 — OCI 와 같은 뿌리 이건 새 아이디어가 아니다. 0012·0014 의 OCI(Open Container Initiative) — "컨테이너 이미지·런타임의 열린 표준, 그래서 Docker 를 지워도 OCI 이미지는 그대로 돈다" — 와 정확히 같은 정신이다. OCI 는 포맷을 열어 벤더 중립을 만들고, CNCF 의 오픈 거버넌스는 의사결정을 열어 벤더 중립을 만든다. 둘 다 답하는 질문은 하나다: "이 회사가 망하거나 마음을 바꾸면 나는 갇히나?" — 아니오.
KCNA 함정: "오픈소스"와 "오픈 거버넌스"는 다르다. 코드가 공개(오픈소스)여도 한 회사가 로드맵과 머지 권한을 독점하면 오픈 거버넌스가 아니다. CNCF 가 요구하는 건 코드 공개 + 결정권 분산 둘 다다.

4. 커뮤니티 역할 — 누가 누구인가

오픈 거버넌스는 사람들이 채운다. KCNA 가 이름과 한 줄 역할만 물어보는 커뮤니티 역할들:

역할한 줄
Contributor코드·문서·이슈로 기여하는 누구나 (진입점)
Maintainer프로젝트의 방향·머지 권한을 가진 핵심 관리자
CNCF Ambassador커뮤니티를 전파하는 공인 자원봉사자 (강연·글·멘토링). 기술 결정권이 아니라 홍보·교육 역할
End User Community프로젝트를 운영에 쓰는 조직들의 모임. 벤더가 아니라 사용자 관점의 피드백을 재단에 전달
SIG (Special Interest Group)특정 영역(네트워킹, 스토리지, 보안 등)을 맡는 상설 작업 그룹
Ambassador vs Maintainer 헷갈리지 말 것: maintainer 는 프로젝트 안의 기술 권한, ambassador 는 프로젝트 밖으로의 전파 역할이다. End User Community 가 중요한 이유 — CNCF 는 벤더뿐 아니라 실제 운영 사용자의 목소리를 거버넌스에 넣어 중립성을 지킨다.

5. 페르소나 — "이건 누구 역할?"

KCNA 는 종종 페르소나(persona)를 묻는다: 어떤 작업이 어떤 역할의 관심사냐는 것. 클라우드 네이티브 세계의 세 큰 페르소나와, 지금까지 배운 것을 매핑하면:

페르소나관심사지금까지 배운 것 매핑
Application Developer앱을 짜고 컨테이너로 만들어 배포. "내 코드가 잘 도나"0006 requests(내 앱이 필요한 자원 선언) / 12-factor(0014) 원칙대로 앱 설계
Operations · SRE · Platform클러스터를 안정적으로 운영·확장. "전체가 잘 도나"0006 limits(폭주 방지 상한) / GitOps(0014)로 안전한 딜리버리 / 노드·용량 관리
Security접근 통제·정책·최소 권한. "누가 뭘 할 수 있나"0008 RBAC·ServiceAccount(누가 API 로 무엇을 하나) / Secret 관리
아하 — 경계는 협업 지점이다 페르소나는 담을 치자는 게 아니라 누가 무엇에 책임지나를 나눈 것이다. 0006 의 requests/limits 가 좋은 예: requests 는 "내 앱엔 이만큼 필요"라고 개발자가 선언하고, limits 는 "전체를 지키려면 이 상한"이라고 운영이 건다 — 같은 필드에서 두 페르소나가 협업한다. KCNA 가 "requests 를 정하는 건 누구 관심사?"라고 물으면 이 경계를 떠올리면 된다.

6. 오픈소스 라이선스와 기여

CNCF 프로젝트 대부분은 Apache License 2.0 을 쓴다 — 상업적 사용·수정·배포가 자유롭고 특허 보호 조항이 있어 기업 채택에 안전하다. 이것도 벤더 중립의 일부다. 코드를 기여하려면 보통 서명 절차를 거친다:

용어한 줄
DCO (Developer Certificate of Origin)커밋에 Signed-off-by 를 달아 "이 코드를 낼 권리가 내게 있다"고 서명. 가볍다
CLA (Contributor License Agreement)재단/회사에 기여물의 라이선스 권리를 부여하는 계약 서명. DCO 보다 무겁다
KCNA 는 "Apache 2.0 가 흔하다", "기여하려면 DCO/CLA 서명이 있을 수 있다" 정도의 인지만 물어본다. 라이선스 조항 비교나 SPDX 같은 건 심화 — 이름표만 붙이고 넘어간다.

7. "Cloud Native" 공식 정의 — 배운 조각으로 되짚기

마지막으로, 이 모든 것의 우산인 "Cloud Native" 의 CNCF 공식 정의를 지금까지 배운 조각들로 되짚는다. 정의(요약): 컨테이너·마이크로서비스·불변 인프라·선언적 API 로 확장 가능하고 회복력 있는(resilient) 시스템을 짓는 접근.

정의의 조각이미 손으로 만진 것
컨테이너(Containers)0012 OCI 이미지·런타임 — 표준화된 배포 단위
마이크로서비스(Microservices)0014 작게 쪼개 독립 배포·확장 (vs 모놀리스)
불변 인프라(Immutable Infra)Pod 를 고치지 않고 교체 — 새 이미지로 새 Pod (12-factor disposable, 0014)
선언적 API(Declarative API)"원하는 상태"를 YAML 로 선언 → 컨트롤러 제어 루프가 수렴 (GitOps 도 같은 골격, 0014)
회복력(Resilient)Pod 죽으면 자동 재생성, replicas 로 다중화 — 자가 치유
아하 — 정의는 배운 것의 목차였다 "Cloud Native" 정의는 새 암기 카드가 아니라 지금까지의 레슨 목차다. 컨테이너(0012)로 담고, 마이크로서비스(0014)로 쪼개고, 불변·선언적으로 굴리고(제어 루프), 회복력을 얻는다 — 이 다섯 단어를 각각 내가 만진 것 하나와 연결할 수 있으면, 정의 문제는 이미 풀 수 있는 것이다.

8. 관찰 (선택, 3분) — landscape 를 지도처럼 훑기

이 레슨엔 hands-on 이 없다(순수 개념). 대신 관찰 과제 하나 — landscape.cncf.io 를 한 번 열어 이렇게 훑어본다:

① 프로젝트를 Graduated / Incubating / Sandbox 로 필터해 본다(2절의 사다리 확인). ② 같은 칸(예: Observability)에 여러 벤더의 여러 프로젝트가 있음을 본다 — 이게 벤더 중립의 모습이다. ③ 아무 graduated 프로젝트나 눌러 maintainer 가 여러 회사에 분산돼 있는지 본다(3절의 증거). 외울 필요 없다. "아, 이게 그 지도구나" 하고 감만 잡으면 된다.

9. 회상 연습

Q1. CNCF 에서 어떤 프로젝트를 Graduated 로 승격시킬지 결정(투표)하는 것은?

역할 분리가 핵심: Governing Board 는 "지갑"(돈·마케팅·회원사), TOC 는 "손"(기술 방향·프로젝트 승급 투표). 프로젝트를 사다리 위로 올리는 건 오직 TOC 다. Ambassador 는 전파 역할이라 기술 결정권이 없다.

Q2. (레슨 0012 복습) CNCF 의 오픈 거버넌스가 lock-in 을 피하게 하는 원리와 같은 뿌리를 가진 것은?

둘 다 "이 회사에 갇히지 않는다"를 만든다. OCI 는 포맷을 열어(0012: Docker 를 지워도 OCI 이미지는 돈다) 벤더 중립을 만들고, 오픈 거버넌스는 의사결정을 열어(한 회사가 프로젝트를 좌우 못 함) 벤더 중립을 만든다. RBAC(0008)는 접근 통제, Helm 은 패키징 — 다른 문제다.

Q3. (레슨 0006·0008 복습) KCNA 스타일 페르소나 문제 — "누가 RBAC 규칙과 ServiceAccount 권한을 정의하는가?"에 가장 맞는 페르소나는?

"누가 무엇을 할 수 있나"는 보안 페르소나의 관심사다. 0008 의 RBAC·ServiceAccount 는 최소 권한을 거는 도구이므로 security/platform 역할에 매핑된다. 대조적으로 0006 의 requests 는 개발자, limits 는 운영이 협업하는 지점이었다. Ambassador 는 거버넌스가 아니라 전파 역할이다.
판정: Q1·Q2·Q3 를 맞히면 아키텍처 도메인의 "조직·사람" 축이 개념 수준으로 ✅ — TOC 대 Governing Board, 오픈 거버넌스, 페르소나까지 KCNA 범위가 닫힌다. 남은 건 전 도메인 모의 진단으로 마지막 갭을 확인하고 응시 일정을 잡는 것뿐. 지금까지 배운 기술 조각들에 이제 역할 이름표까지 붙었다.

초보자 질문 (막히면 여기부터)

이 레슨은 조직·사람 이름이 많아 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

CNCF 랑 Linux Foundation 은 무슨 관계예요?

Linux Foundation(LF) 이 상위 우산이고, CNCF 는 그 산하 재단 중 하나다(2015 년 설립). LF 는 리눅스 커널뿐 아니라 여러 산업별 재단을 품는데, 클라우드 네이티브 프로젝트(쿠버네티스 등)를 맡는 게 CNCF 다. KCNA 는 "CNCF 는 Linux Foundation 산하"만 알면 충분.

Governing Board 랑 TOC, 계속 헷갈려요. 쉽게 구분법 없어요?

Board = 돈(business), TOC = 기술(technical). Board 는 예산·마케팅·회원사·KubeCon 을 굴리는 지갑, TOC 는 어떤 프로젝트를 받고 승격시킬지 투표로 정하는 손이다. 시험에서 "프로젝트 승급은 누구?"가 나오면 무조건 TOC. Board 는 프로젝트 승급에 손대지 않는다.

오픈소스면 다 오픈 거버넌스 아닌가요?

아니다. 오픈소스 = 코드 공개, 오픈 거버넌스 = 결정권 분산이다. 코드가 공개돼 있어도 한 회사가 로드맵·머지 권한을 독점하면 오픈 거버넌스가 아니다. CNCF 가 graduated 기준으로 "maintainer 가 여러 회사에 분산"을 요구하는 이유가 바로 이 결정권 분산을 강제하려는 것이다.

CNCF Ambassador 는 뭐 하는 사람이에요?

클라우드 네이티브 커뮤니티를 전파하는 공인 자원봉사자다. 강연·글쓰기·밋업·멘토링으로 사람들을 돕는다. 주의할 점: ambassador 는 홍보·교육 역할일 뿐 프로젝트의 기술 결정권(머지·승급 투표)은 없다. Maintainer(프로젝트 안 기술 권한)와 헷갈리지 말 것.

End User Community 는 왜 따로 있어요?

CNCF 가 벤더(파는 회사)뿐 아니라 실제 운영에 쓰는 사용자의 목소리를 거버넌스에 넣으려는 장치다. 프로젝트를 프로덕션에서 굴리는 조직들이 모여 "이런 게 필요하다/이게 아프다"를 재단·프로젝트에 전달한다. 사용자 관점이 있어야 특정 벤더 이익으로 치우치지 않으니, 이것도 벤더 중립을 지키는 방법이다.

SIG 랑 프로젝트는 뭐가 달라요?

프로젝트는 배포되는 소프트웨어(예: Prometheus)이고, SIG(Special Interest Group)는 특정 영역(네트워킹, 스토리지, 보안 등)을 지속적으로 맡는 작업 그룹이다. 쿠버네티스 내부도 SIG-Network, SIG-Storage 처럼 나뉜다. KCNA 는 "SIG = 특정 영역의 상설 그룹" 정도면 충분.

페르소나가 정확히 뭐예요? 왜 시험에 나와요?

페르소나는 클라우드 네이티브 일을 역할별 관심사로 나눈 것이다: 개발자(내 앱), 운영/SRE/platform(전체 안정), 보안(접근 통제). KCNA 가 이걸 묻는 이유는 실무에서 "이 작업은 누구 책임인가"를 아는 게 중요하기 때문. 예를 들어 RBAC 설정은 보안, 앱 자원 요청(requests)은 개발자 관심사 — 이렇게 매핑할 수 있으면 된다.

requests 는 개발자, limits 는 운영이라니, 같은 필드 아니에요?

맞다, 0006 에서 본 같은 Pod 스펙의 이웃 필드다. 그래서 좋은 협업 예시다: requests("내 앱엔 이만큼 필요해")는 앱을 아는 개발자가 선언하고, limits("전체를 지키려면 이 상한")는 클러스터를 지키는 운영이 건다. 페르소나는 담이 아니라 책임 분담이라, 이렇게 한 지점에서 두 역할이 만나는 게 자연스럽다.

DCO 랑 CLA 는 꼭 알아야 해요?

KCNA 수준에선 이름과 한 줄 역할이면 충분하다. DCO는 커밋에 Signed-off-by 한 줄로 "이 코드 낼 권리가 내게 있다"고 서명하는 가벼운 방식, CLA는 기여물 라이선스 권리를 재단/회사에 넘기는 좀 더 무거운 계약이다. "기여하려면 서명 절차가 있을 수 있다" 정도의 인지면 된다.

"Cloud Native" 정의를 통째로 외워야 하나요?

통암기보다 다섯 조각을 배운 것과 연결하는 게 낫다: 컨테이너(0012 OCI), 마이크로서비스(0014), 불변 인프라(Pod 교체), 선언적 API(제어 루프·YAML), 회복력(자가 치유). 각 단어를 "내가 만진 것 하나"와 이을 수 있으면, 정의 문제는 암기가 아니라 목차 확인이 된다.
