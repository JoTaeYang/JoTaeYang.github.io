Deployment 는 Pod 를 만들지 않는다

진단에서 틀린 바로 그 문제. 이번엔 외우는 게 아니라 눈으로 직접 확인한다.

이번 레슨의 win: 내 로컬 클러스터에서 Deployment → ReplicaSet → Pod 소유 체인을 직접 추적하고, Pod 를 죽여도 되살아나는 이유를 체인으로 설명할 수 있다.

1. 왜 세 층인가

Deployment 를 만들면 쿠버네티스에는 리소스가 세 종류 생긴다:

Deployment"어떤 버전을, 어떻게 교체할지" → 만든다 → ReplicaSet"이 템플릿의 Pod 를 N 개 유지" → 만든다 → Pod ×N실제로 도는 컨테이너

역할이 다르기 때문이다. ReplicaSet 은 단 하나만 안다: "이 템플릿의 Pod 개수를 N 으로 유지하라." Pod 가 죽으면 새로 만든다 — 그게 전부다. Deployment 는 그 위에서 버전 교체를 담당한다: 이미지를 바꾸면 새 ReplicaSet 을 만들고, 새 것을 0→3 으로 늘리며 옛 것을 3→0 으로 줄인다. 이게 롤링 업데이트의 실체다.

아하 레슨 0001 Q2 에서 맞힌 "제어 루프"가 여기서 두 겹으로 돈다: Deployment 컨트롤러는 ReplicaSet 들을 수렴시키고, ReplicaSet 컨트롤러는 Pod 개수를 수렴시킨다. 각 층은 바로 아래 층만 책임진다 — 그래서 Deployment 는 Pod 를 직접 만들지 않는다.

2. 실습 — 체인을 눈으로 추적한다

환경: 로컬 일회용 클러스터 kind. Docker(또는 Podman/OrbStack)가 떠 있어야 한다. 이미 kind 클러스터가 있으면 1번은 건너뛴다.
terminal $ brew install kind && kind create cluster --name mastery $ kubectl create deployment web --image=nginx:1.27 --replicas=3

추적 1 — 세 층이 전부 생겼는지:

terminal $ kubectl get deploy,rs,pod NAME READY UP-TO-DATE AVAILABLE deployment.apps/web 3/3 3 3 NAME DESIRED CURRENT READY replicaset.apps/web-7c5ddbdf54 3 3 3 NAME READY STATUS RESTARTS pod/web-7c5ddbdf54-8kq2x 1/1 Running 0 pod/web-7c5ddbdf54-p9jtm 1/1 Running 0 pod/web-7c5ddbdf54-zv4wn 1/1 Running 0

이름부터 체인이다: Pod 이름 = Deployment이름-ReplicaSet해시-랜덤. 소유 관계는 이름 규칙이 아니라 메타데이터에 실제로 기록되어 있다:

terminal $ kubectl get pod -l app=web -o jsonpath='{.items[0].metadata.ownerReferences[0].kind}{" / "}{.items[0].metadata.ownerReferences[0].name}' ReplicaSet / web-7c5ddbdf54

추적 2 — Pod 를 죽이면 누가 살리나:

terminal $ kubectl delete pod -l app=web --wait=false && kubectl get pod -w ... 기존 Pod Terminating, 새 이름의 Pod 가 즉시 ContainerCreating → Running (Ctrl-C 로 종료)

당신이 아니라 ReplicaSet 컨트롤러가 살렸다. "DESIRED 3, 현재 0" 을 보고 3개를 다시 만든 것.

추적 3 — 이미지를 바꾸면 ReplicaSet 이 교체된다:

terminal $ kubectl set image deployment/web nginx=nginx:1.28 && kubectl get rs -w web-7c5ddbdf54 3 → 2 → 1 → 0 (옛 ReplicaSet 이 줄어들고) web-6f9d8b7c44 1 → 2 → 3 (새 ReplicaSet 이 늘어난다 — 해시가 다르다!)
옛 ReplicaSet 은 0 이 되어도 삭제되지 않고 남는다 — kubectl rollout undo 가 이걸 다시 키우는 방식으로 롤백하기 때문. 확인: kubectl rollout undo deployment/web && kubectl get rs

끝났으면 정리: kind delete cluster --name mastery (또는 다음 레슨을 위해 남겨둔다).

3. 회상 연습

Q1. 롤링 업데이트 중 쿠버네티스가 실제로 하는 일은?

추적 3 에서 본 그대로. ReplicaSet 은 불변의 템플릿을 지키는 존재라 고치지 않고 교체한다. 공식 문서: Updating a Deployment

Q2. 추적 2 에서 지운 Pod 를 되살린 주체는?

각 층은 바로 아래 층만 책임진다. Deployment 는 ReplicaSet 개수/크기만 보고, Pod 개수 유지는 ReplicaSet 의 일이다. 공식 문서: ReplicaSet

Q3. (지난 레슨 복습) 재생성될 때마다 IP 가 바뀌는 Pod 들에 안정적인 내부 접점을 주는 것은?

간격 반복(spaced retrieval): 레슨 0001 Q4. 방금 실습에서 Pod 이름/IP 가 계속 바뀌는 걸 봤다 — 그래서 Service 의 label selector 가 필요하다. 다음 레슨에서 이 Deployment 에 직접 Service 를 붙인다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

Deployment 를 지우면 Pod 도 같이 지워지나요?

지워진다. 소유 체인 덕분이다 — Deployment 를 지우면 그것이 소유한 ReplicaSet 이, 그리고 ReplicaSet 이 소유한 Pod 들이 연쇄 삭제(cascading delete)된다. 이 연쇄를 실제로 잇는 게 각 리소스의 ownerReferences 다. 참고로 --cascade=orphan 을 주면 자식을 남길 수도 있다.

ownerReferences 는 누가 채워 넣나요?

내가 아니라 컨트롤러가 채운다. ReplicaSet 컨트롤러가 Pod 를 만들 때 "내가 이 Pod 의 주인"이라고 Pod 의 ownerReferences 에 자기를 적고, Deployment 컨트롤러도 ReplicaSet 을 만들 때 똑같이 한다. 추적 1 의 jsonpath 로 본 ReplicaSet / web-... 이 바로 그 기록이다.

ReplicaSet 을 직접 만들면 안 되나요?

만들 수는 있지만, 그러면 롤링 업데이트를 잃는다. 이미지를 바꾸면 Deployment 는 새 ReplicaSet 을 만들어 무중단 교체해 주는데, ReplicaSet 혼자서는 그 위 층이 없어 스스로 버전을 못 바꾼다. 그래서 실무에선 거의 항상 Deployment 를 쓰고 ReplicaSet 은 직접 손대지 않는다.

롤링 업데이트 중에 요청이 실패하지 않나요?

기본적으로 안 하도록 설계돼 있다. Deployment 는 옛 Pod 를 한 번에 다 죽이지 않고, 새 Pod 를 먼저 Running 상태로 올린 뒤에야 옛 것을 줄인다(maxSurge / maxUnavailable 로 조절). 다만 새 Pod 가 "떴다"와 "진짜 요청 받을 준비됐다"는 다르다 — 이 간극을 메우는 게 다음에 배울 readiness probe 다.

옛 ReplicaSet 이 0 인데 왜 안 지워지고 남아 있어요?

kubectl rollout undo 의 롤백 재료로 일부러 남겨 둔다. 롤백은 옛 ReplicaSet 을 다시 0→N 으로 키우는 방식이라, 지워 버리면 되돌아갈 곳이 없어진다. 몇 개까지 보관할지는 Deployment 의 revisionHistoryLimit(기본 10)이 정한다.

kind 가 정확히 뭐예요? 진짜 클러스터인가요?

Kubernetes IN Docker — 쿠버네티스 노드를 통째로 Docker 컨테이너 하나로 돌리는 일회용 로컬 클러스터다. API 는 진짜 쿠버네티스와 똑같아서 학습·CI 에 딱 맞지만, 노드가 컨테이너라 프로덕션용은 아니다. 끝나면 kind delete cluster 로 통째로 버리면 된다.

Pod 이름 뒤 7c5ddbdf54 같은 해시는 뭐예요?

ReplicaSet 이 지키는 Pod 템플릿의 지문이다(pod-template-hash). 템플릿이 같으면 같은 해시, 이미지 하나만 바뀌어도 해시가 달라진다 — 그래서 추적 3 에서 새 ReplicaSet 이 다른 해시로 나타난 것이다. 이 해시 덕에 두 ReplicaSet 이 서로의 Pod 를 헷갈리지 않는다.

Pod 를 지웠는데 이름이 다른 새 Pod 가 떠요. 같은 Pod 아닌가요?

다른 Pod 다. ReplicaSet 은 죽은 Pod 를 "되살리는" 게 아니라 개수를 맞추려고 새로 만든다 — 그래서 이름 뒤 랜덤 부분이 바뀐다. Pod 는 원래 사라졌다 새로 생기는 걸 전제로 하는 일회용 존재라, 이름·IP 가 바뀌어도 정상이다. 안정적인 접점이 필요해서 Service 가 있는 것(레슨 0001 복습).

kubectl scale 로 개수를 바꾸면 Deployment 를 거쳐야 하나요?

Deployment 를 대상으로 하라. kubectl scale deployment/web --replicas=5 하면 Deployment 가 자기 ReplicaSet 의 목표 개수를 바꾸고, ReplicaSet 이 Pod 를 채운다. ReplicaSet 을 직접 스케일하면 Deployment 컨트롤러가 "어? 목표랑 다르네" 하고 도로 되돌린다 — 각 층은 바로 아래 층만 자기 것으로 관리하기 때문이다.

롤아웃이 도중에 멈춘 것 같은데 어떻게 확인해요?

kubectl rollout status deployment/web 가 진행 상황을 실시간으로 보여 주고, 다 끝나면 알려 준다. 멈춰 있다면 대개 새 Pod 가 Running 이 못 돼서다(이미지 오타, ImagePullBackOff 등) — kubectl get pod 의 STATUS 열부터 본다. 이 진단 흐름은 CKA 실기의 단골이다.
