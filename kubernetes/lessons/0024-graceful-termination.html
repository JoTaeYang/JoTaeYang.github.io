Kubernetes Mastery · Lesson 0024 Fundamentals 심화

우아한 종료 — Pod 는 어떻게 곱게 죽나

kubectl delete pod 를 치는 순간부터 컨테이너가 사라지기까지, Terminating 동안 병렬로 벌어지는 두 가지 일 — 트래픽 차단과 SIGTERM — 을 정확한 시퀀스로 새긴다. "죽이는 게 아니라 마무리할 시간을 준다."

이번 레슨의 win: Pod 삭제가 즉시 kill 이 아니라 유예(grace period)를 둔 협상임을 손으로 확인한다. 삭제 순간 (a) 0003 의 EndpointSlice 에서 빠져 새 트래픽이 끊기고, (b) preStop 훅 → SIGTERM → terminationGracePeriodSeconds 대기 → SIGKILL 로 이어지는 순서를 time kubectl delete 로 직접 시간을 재 눈에 새긴다. 이게 0014 의 12-factor "disposability(빠른 종료)"와 0002 롤링 업데이트가 조용히 기대고 있던 바로 그 메커니즘이다.

1. 삭제는 즉사가 아니다 — Terminating 이라는 유예

kubectl delete pod X 를 쳐도 Pod 는 그 자리에서 사라지지 않는다. API 서버는 Pod 에 metadata.deletionTimestamp 를 찍을 뿐이고, 이 순간부터 Pod 는 Terminating 상태로 들어간다. 0023 에서 본 "deletionTimestamp 가 찍힌 Pod = 종료 진행 중"이 바로 여기서 시작된다.

아하 — deletionTimestamp 는 "죽어라"가 아니라 "정리하고 나가라" timestamp 가 찍히면 그때부터 유예 시계(grace period)가 돌기 시작한다. 기본 30초. 그 시간 안에 컨테이너가 스스로 마무리하고 나가면 곱게 끝, 못 나가면 시간이 다 됐을 때 강제로 kill 된다. 즉 삭제는 "이제 곧 자를 테니 하던 일 마무리해"라는 예고이지, 즉각적인 처형이 아니다.

2. Terminating 동안 병렬로 벌어지는 두 가지

deletionTimestamp 가 찍히면 동시에 두 갈래가 진행된다. 이 둘이 병렬이라는 게 뒤(5절 경합)에서 문제의 씨앗이 된다.

(a) 네트워크 갈래(b) 프로세스 갈래
무슨 일 Service 의 EndpointSlice 에서 이 Pod 가 제거됨 컨테이너를 순서대로 종료 시작
효과 새 트래픽이 이 Pod 로 더는 안 감 preStop → SIGTERM → 대기 → SIGKILL
연결 0003 EndpointSlice 컨트롤러의 제어 루프가 "not-ready"로 빼고, 0013 kube-proxy 가 규칙에서 지운다 이 레슨 3절의 훅·시그널 시퀀스
0003 과 이어짐: Service 가 Pod 를 label selector 로 추적하고 그 결과가 EndpointSlice 에 실시간 기록된다고 했다. Pod 가 Terminating 이 되면 EndpointSlice 컨트롤러가 이 Pod 를 ready:false(terminating)로 표시 → 곧 목록에서 빠진다. 그래야 새 연결이 이미 종료 중인 Pod 로 라우팅되지 않는다. 이건 0004 의 readiness probe 가 실패했을 때 endpoint 에서 빠지던 것과 같은 창구다.

3. 프로세스 갈래의 정확한 시퀀스

컨테이너를 종료하는 (b) 갈래는 정해진 순서로 흐른다. 순서를 헷갈리면 대부분의 "왜 요청이 유실되지?" 오해가 나온다.

순서단계설명
1preStop 훅 실행SIGTERM 보내기 전. 블로킹 — 훅이 끝나야 다음으로 간다. (있을 때만)
2컨테이너 PID 1 에 SIGTERM"이제 종료해" 신호. 앱이 이걸 받아 in-flight 요청을 마무리해야 한다.
3grace period 대기terminationGracePeriodSeconds(기본 30s). preStop 에 쓴 시간도 이 예산에서 차감된다.
4아직 살아있으면 SIGKILL커널이 강제 종료(9번, 무시 불가). 여기 걸리면 in-flight 요청은 그냥 끊긴다.
핵심 세 가지 ① preStop 은 SIGTERM "전"에, 블로킹으로 돈다. ② grace period 는 preStop + SIGTERM 후 정리시간을 합쳐서 재는 총예산이다 — preStop 에 25초를 쓰면 SIGTERM 뒤엔 5초밖에 안 남는다. ③ 앱이 SIGTERM 을 무시하면 grace 끝에 무조건 SIGKILL 이라, 곱게 죽을 기회를 스스로 버리는 것이다.
postStart 와 헷갈리지 말 것: postStart 훅은 컨테이너가 시작된 직후 실행되는 짝이지만, 컨테이너 entrypoint 와 실행 순서가 보장되지 않는다(둘이 거의 동시에 뜬다). preStop 은 종료 쪽, postStart 는 시작 쪽 — 대칭이지만 보장 강도가 다르다.

4. 앱이 SIGTERM 을 처리해야 하는 이유 — disposability 의 실제 구현

SIGTERM 은 "정리하고 나가라"는 정중한 요청이다. 잘 만든 앱은 이걸 받으면: ① 새 요청 접수 중단, ② in-flight 요청 마무리, ③ DB/커넥션 정리 후 스스로 종료한다. 그러면 grace period 가 다 차기 전에 곱게 끝나고 SIGKILL 은 발동조차 안 한다.

0014 와 연결 — 이게 12-factor 의 그 원칙이다 0014 12-factor 의 "프로세스는 disposable — 빠르게 뜨고 우아하게 죽어야 한다"가 추상론이 아니라 바로 이 SIGTERM 처리로 구현된다. "언제 죽어도 되는 stateless 프로세스"라는 0002 의 명제도, Pod 가 이렇게 안전하게 종료될 수 있어야 성립한다. 우아한 종료 = disposability 의 하드웨어.
강제 삭제는 최후의 수단: kubectl delete pod X --grace-period=0 --force 는 grace period 를 0으로 만들어 preStop·SIGTERM 을 건너뛰고 즉시 API 오브젝트를 지운다. in-flight 요청은 그냥 끊기고, StatefulSet 같은 경우엔 데이터 정합성까지 깨질 수 있다(다른 노드에서 같은 Pod 가 뜨는 split-brain). 노드가 완전히 죽어 응답이 없을 때 같은 예외 상황에서만. 평소엔 절대 습관화하지 말 것.

5. 경합 문제와 preStop sleep 패턴

2절에서 (a)네트워크와 (b)프로세스가 병렬이라고 했다. 문제는 이 둘의 순서가 완벽히 보장되지 않는다는 것이다. SIGTERM 이 도착해 앱이 종료를 시작했는데, 0013 의 kube-proxy 규칙(iptables/ipvs) 갱신이 아직 안 끝나 새 요청이 이 종료 중인 Pod 로 몇 개 더 들어올 수 있다. 그 요청들은 커넥션이 거부되어 유실된다.

해법 — preStop 에 짧은 sleep preStop 에 sleep 5~10 을 넣는 게 흔한 관용구다. SIGTERM 을 일부러 몇 초 늦춰서, 그 사이 0003 EndpointSlice 제거가 전파되고 0013 kube-proxy 규칙이 갱신될 시간을 벌어준다. 즉 "앱은 아직 멀쩡히 요청을 받는데, 라우팅은 이미 나를 뺐다" 상태를 만들어 요청 유실을 없앤다. sleep 이 끝나야 SIGTERM 이 가므로, 앱은 그동안 정상 응답을 계속한다.
0002 롤링 업데이트가 여기에 기댄다: 0002 의 rolling update 는 "새 Pod 를 Ready 로 올리고 옛 Pod 를 종료"의 반복이었다. 그 "옛 Pod 종료"가 바로 이 우아한 종료 시퀀스다. preStop sleep + SIGTERM 처리가 제대로 안 되어 있으면, 배포할 때마다 매번 요청이 몇 개씩 유실된다 — "무중단 배포"가 실제로는 무중단이 아니게 된다. 우아한 종료는 롤링 업데이트의 숨은 전제다.

6. Hands-on — preStop 이 붙잡는 걸 시간으로 확인 (kind mastery)

kind 클러스터 mastery 이어서 사용. 0006·0020 처럼 lab namespace 에서 격리한다. 없으면: kind create cluster --name mastery.
terminal $ kubectl create namespace lab 2>/dev/null; kubectl config set-context --current --namespace=lab

preStop(sleep 10) + grace 30초 인 Pod 를 만든다. heredoc 으로 바로 apply:

terminal $ kubectl apply -f - <<'EOF' apiVersion: v1 kind: Pod metadata: name: graceful spec: terminationGracePeriodSeconds: 30 # SIGTERM 후 SIGKILL 까지 총예산 containers: - name: app image: nginx:1.27 lifecycle: preStop: exec: command: ["/bin/sh", "-c", "sleep 10"] # SIGTERM 전에 10초 붙잡음(블로킹) EOF pod/graceful created $ kubectl wait --for=condition=Ready pod/graceful --timeout=60s pod/graceful condition met

다른 터미널에서 상태를 지켜본다 — Terminating 이 뜨는 걸 실시간으로 본다:

terminal 2 (watch) $ kubectl get pod graceful -w NAME READY STATUS RESTARTS AGE graceful 1/1 Running 0 20s graceful 1/1 Terminating 0 25s # delete 순간 여기로 — 아직 살아있다 graceful 0/1 Terminating 0 35s # preStop sleep 10 이 끝나고 종료

원래 터미널에서 time 으로 삭제 시간을 잰다 — preStop 이 붙잡아 ~10초 이상 걸리는 게 핵심 관찰:

terminal $ time kubectl delete pod graceful pod "graceful" deleted kubectl delete pod graceful ... 10.4s total # preStop sleep 10 만큼 붙잡힘
무엇을 봤나 ① delete 는 즉시 안 끝나고 preStop sleep 만큼(~10초) 붙잡혔다 = preStop 이 SIGTERM 전에, 블로킹으로 돈다는 증거. ② 그동안 Pod 는 Terminating 이지만 여전히 살아 요청을 받을 수 있다 — 5절 sleep 패턴이 요청 유실을 막는 원리 그대로. ③ nginx 는 SIGTERM 을 잘 처리하므로 grace 30초를 다 쓰지 않고 sleep 직후 곱게 끝났다(SIGKILL 미발동).

대비 실험 — 강제 삭제는 즉시다. 다시 만들고 --grace-period=0 --force 로 지워 시간을 비교해 본다:

terminal $ kubectl apply -f - <<'EOF' apiVersion: v1 kind: Pod metadata: { name: graceful } spec: terminationGracePeriodSeconds: 30 containers: - { name: app, image: nginx:1.27, lifecycle: { preStop: { exec: { command: ["/bin/sh","-c","sleep 10"] } } } } EOF $ kubectl wait --for=condition=Ready pod/graceful --timeout=60s $ time kubectl delete pod graceful --grace-period=0 --force Warning: Immediate deletion does not wait for confirmation that the running resource has been terminated. ... pod "graceful" force deleted kubectl delete ... ... 0.3s total # preStop·SIGTERM 다 건너뛰고 즉시 — 위험한 이유

정리: kubectl delete ns lab && kubectl config set-context --current --namespace=default.

KCNA 범위 표시: KCNA 는 "Pod 삭제 시 grace period 를 두고 우아하게 종료된다 / SIGTERM 후 SIGKILL"이라는 개념까지가 시험 범위다. preStop/postStart 훅의 정확한 문법, sleep 경합 패턴, grace period 튜닝 같은 상세는 CKAD 심화 몫 — 여기선 손으로 한 번 만져 직관만 확실히 새기고 넘어가면 충분하다.

7. 회상 연습

Q1. kubectl delete pod X 를 쳤을 때 컨테이너의 프로세스 갈래에서 벌어지는 순서로 옳은 것은?

preStop 은 SIGTERM 전에 블로킹으로 돈다. 그다음 PID 1 에 SIGTERM, terminationGracePeriodSeconds 만큼 대기, 그때도 살아있으면 SIGKILL(강제). SIGKILL 은 마지막 수단이지 시작이 아니다.

Q2. (레슨 0003·0013 복습) preStop 에 sleep 5 를 넣는 흔한 이유로 가장 정확한 것은?

네트워크(EndpointSlice 제거·kube-proxy 갱신)와 프로세스(SIGTERM)가 병렬이라 순서가 완벽히 보장 안 된다. 짧은 sleep 으로 SIGTERM 을 늦추면 그 사이 라우팅이 나를 빼는 게 전파되어 요청 유실(0003·0013)을 막는다.

Q3. (레슨 0014 복습) 앱이 SIGTERM 을 무시하도록 짜여 있으면 어떻게 되나?

간격 반복: SIGTERM 은 무시 가능하지만 SIGKILL 은 불가능하다. 앱이 SIGTERM 을 안 받으면 grace 끝에 무조건 SIGKILL 이라 곱게 죽을 기회를 버린다. 0014 의 "우아하게 죽어야 한다(disposability)"가 바로 이 SIGTERM 처리로 구현됨을 뒤집어 보여주는 경우다.
판정: Q1·Q2·Q3 를 맞히면 삭제 시퀀스(preStop→SIGTERM→grace→SIGKILL), 경합과 sleep 패턴, SIGTERM 처리의 당위가 ✅ — "Pod 는 왜 즉시 안 죽나", "무중단 배포(0002)는 뭘 믿고 도나"에 정확히 답할 수 있다. 삭제가 처형이 아니라 협상임을 이제 안다.

초보자 질문 (막히면 여기부터)

종료는 "즉시"라는 직관과 어긋나 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

Pod 가 왜 delete 하자마자 안 사라지고 Terminating 에 머물러요?

삭제는 즉사가 아니라 유예를 둔 종료이기 때문이다. API 서버는 deletionTimestamp 를 찍어 "종료 시작"만 표시하고(0023), kubelet 이 preStop→SIGTERM→grace 대기를 거친다. 이 과정 내내 Pod 는 Terminating 으로 보인다. 컨테이너가 다 정리되고 나가야 오브젝트가 진짜 삭제된다.

terminationGracePeriodSeconds 기본값은 얼마예요? 어떻게 바꿔요?

기본 30초다. Pod spec 의 spec.terminationGracePeriodSeconds 로 바꾼다. 종료에 오래 걸리는 앱(큰 배치 마무리, 커넥션 드레인)은 늘리고, 빨리 죽어도 되는 stateless 앱은 그대로 둔다. 삭제할 때 kubectl delete pod X --grace-period=60 로 그 한 번만 덮어쓸 수도 있다.

preStop 훅이랑 SIGTERM, 결국 둘 다 "종료해"인데 왜 둘 다 있어요?

역할이 다르다. SIGTERM 은 앱 프로세스에게 보내는 신호라, 앱이 시그널 핸들러를 짜 넣어야 반응한다. preStop 은 앱을 안 고치고 종료 직전 임의 명령(sleep, drain 스크립트, 종료 API 호출)을 끼워 넣는 훅이다. 앱 소스를 못 건드릴 때, 또는 5절처럼 SIGTERM 을 일부러 늦출 때 preStop 이 필요하다.

preStop 에 쓴 시간도 grace period 에서 까이나요?

그렇다. terminationGracePeriodSeconds 는 preStop + SIGTERM 후 정리를 합친 총예산이다. grace 30초에 preStop sleep 25초를 쓰면 SIGTERM 뒤엔 5초밖에 안 남는다. preStop 을 길게 잡을 거면 grace 도 같이 늘려야 앱이 SIGTERM 후 마무리할 시간이 확보된다.

왜 하필 nginx 로 실습했어요? 우리 앱도 곱게 죽나요?

nginx 는 SIGTERM 을 잘 처리(연결 마무리 후 종료)하도록 만들어져 있어 관찰이 깔끔하다. 여러분 앱은 직접 SIGTERM 핸들러를 넣어야 곱게 죽는다 — 안 넣으면 grace 끝에 SIGKILL 로 끊긴다(Q3). 실습의 nginx 는 "잘 만든 앱은 이렇게 종료된다"의 본보기다.

SIGTERM 이 컨테이너의 "PID 1"에 간다는 게 무슨 뜻이에요?

컨테이너 안에서 entrypoint 로 뜬 맨 처음 프로세스가 PID 1 이고, SIGTERM 은 여기로 간다. 문제는 쉘 스크립트로 앱을 띄우면 쉘이 PID 1 이 되어 SIGTERM 을 자식 앱에 전달 안 하는 경우가 흔하다는 것. 이때 앱은 신호를 못 받아 grace 끝에 SIGKILL 된다. exec 로 앱을 띄우거나 init(tini) 을 쓰면 앱이 PID 1 이 되어 신호를 제대로 받는다.

postStart 훅은 언제 써요? preStop 이랑 짝인가요?

대칭이지만 다르다. postStart 는 컨테이너 시작 직후 실행되는 훅으로, entrypoint 와 실행 순서가 보장되지 않는다(거의 동시에 뜬다). 그래서 "반드시 앱 뜨기 전에 뭘 해야 한다"는 postStart 로 보장할 수 없다 — 그건 0004 의 startupProbe/initContainer 몫이다. preStop(종료 쪽)은 블로킹이라 순서가 확실한 반면, postStart(시작 쪽)는 그렇지 않다.

--grace-period=0 --force 는 언제 써도 되나요?

노드가 완전히 죽어 kubelet 이 응답을 안 해 Pod 가 Terminating 에 무한정 걸린 예외 상황 정도다. 이건 preStop·SIGTERM 을 다 건너뛰고 API 오브젝트만 지우므로 in-flight 요청 유실, StatefulSet 의 경우 같은 Pod 가 두 노드에서 도는 split-brain 위험이 있다(warn 참고). 평소 배포·재시작엔 절대 습관화하지 말 것.

preStop sleep 없이도 요청 유실을 막을 순 없나요?

앱이 SIGTERM 을 받고도 바로 끊지 말고 잠깐 더 요청을 받다가 종료하도록 짜면 비슷한 효과가 난다(앱 레벨 드레인). 하지만 앱을 못 고치거나 여러 언어·프레임워크가 섞였을 때, preStop sleep 은 앱과 무관하게 일괄로 문제를 덮는 가장 단순한 방패다. 근본은 EndpointSlice 제거(0003)·kube-proxy 갱신(0013)의 전파 지연이라, 그 지연을 sleep 으로 흡수하는 것이다.

Job/CronJob 이나 데몬 같은 앱도 이 시퀀스가 똑같이 적용되나요?

Pod 로 도는 모든 워크로드에 같은 종료 시퀀스가 적용된다 — preStop→SIGTERM→grace→SIGKILL 은 컨트롤러 종류와 무관한 Pod 레벨 메커니즘이다. 다만 무엇을 "마무리"해야 하는지가 다를 뿐(웹은 in-flight 요청, 배치는 진행 중 작업 저장). Service 뒤에 없는 Job 은 2절 (a)네트워크 갈래가 사실상 무의미하고 (b)프로세스 갈래만 의미 있다.
