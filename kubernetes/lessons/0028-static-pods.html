Kubernetes Mastery · Lesson 0028 Architecture 심화

Static Pod: 컨트롤 플레인이 스스로를 부팅하는 법

0007 이 남겨둔 닭-달걀 질문 — "apiserver 를 스케줄하려면 apiserver 가 필요한데, 첫 apiserver 는 누가 띄우나?" — 를 static Pod 로 닫는다.

이번 레슨의 win: static Pod 가 API 서버·스케줄러를 거치지 않고 kubelet 이 직접 띄우는 Pod 임을 이해하고, kubeadm 클러스터의 컨트롤 플레인(apiserver·scheduler·controller-manager·etcd) 자체가 static Pod 로 부팅한다는 것을 내 kind 노드 안에서 두 눈으로 확인한다. 0007 FAQ 에서 한 줄로 흘렸던 "부트스트랩의 닭-달걀을 static Pod 가 푼다"가 여기서 완성된다.

0. 한눈에 — 닭이 먼저다

30초 요약 (이것만 외워도 됨)

보통 Pod = kubectl apply → API server → etcd → scheduler 가 노드 고름 → kubelet 이 실행. 즉 컨트롤 플레인을 통째로 거쳐야 Pod 하나가 뜬다.

Static Pod = kubelet 이 노드의 매니페스트 디렉터리(/etc/kubernetes/manifests)를 혼자 감시하다가, 거기 놓인 Pod YAML 을 API server 없이 바로 띄운다. 진실의 원천은 etcd 가 아니라 그 디렉터리의 파일이다.

그래서 닭-달걀이 풀린다: apiserver 자신을 static Pod 로 만들면, kubelet 이 부팅하면서 파일만 보고 apiserver 를 띄운다 — apiserver 를 띄우는 데 apiserver 가 필요 없다.

비유 — 관리인이 붙여둔 공고문

보통 Pod 는 시청(API server)에 서류를 접수해야 건물이 선다. 그런데 시청 건물 자체가 아직 없다면? 관리인(kubelet)이 자기 게시판(/etc/kubernetes/manifests)에 붙은 공고문을 보고, 시청을 거치지 않고 직접 그 건물을 짓는다. 시청 건물도 이렇게 지어진다 — 그래서 첫 시청이 선다.

나중에 시청이 서면, 관리인이 자기가 지은 건물들을 시청 장부에 "읽기 전용 사본(mirror Pod)"으로 등록해 둔다. kubectl get pod 으로 보이긴 하지만, 장부의 사본을 지워도 게시판 원본이 그대로면 관리인이 다시 짓는다.

1. 핵심 — kubelet 이 파일을 보고 직접 띄운다

0007 에서 배운 정상 경로는 API server 라는 단일 관문을 반드시 지났다. static Pod 는 그 관문을 우회하는 유일한 길이다. kubelet 은 시작할 때 두 개의 입력을 본다:

  • API server — "나(이 노드)에게 배정된 Pod 를 달라"고 watch 한다 (0012 의 kubelet 역할).
  • 매니페스트 디렉터리 — --pod-manifest-path(기본 /etc/kubernetes/manifests)에 있는 YAML 파일들. API server 와 무관하게 로컬 파일만 보고 그대로 실행한다.

2. Mirror Pod — 보이지만 제어할 수 없는 그림자

static Pod 는 API server 없이도 도는데, 그럼 kubectl get pods 에는 왜 보일까? kubelet 이 API server 가 살아난 뒤 자기가 띄운 static Pod 를 읽기 전용 미러(mirror Pod)로 API server 에 등록해 주기 때문이다. 관리 편의를 위한 사본일 뿐, 원본이 아니다.

미러 Pod 의 규칙 두 개

① 이름에 노드명이 붙는다. kube-apiserver 가 아니라 kube-apiserver-mastery-control-plane 처럼 -노드명 접미사가 달린다 — "이건 그 노드의 static Pod 미러"라는 표식이다.

② API 로 제어가 안 된다. 미러를 kubectl delete 하면 잠깐 목록에서 사라지지만, kubelet 이 파일이 그대로임을 보고 곧바로 다시 띄우고 미러도 다시 만든다. 진실의 원천은 언제나 노드의 파일이지 etcd 가 아니다.

미러를 손으로 지우지 말 것. kubectl delete pod kube-apiserver-... 는 무해해 보여도, 컨트롤 플레인 컴포넌트를 잠깐 재시작시킨다 — 학습 클러스터라 곧 복구되지만 습관이 되면 위험하다. static Pod 를 진짜로 바꾸려면 노드 안 YAML 파일을 수정하는 게 정석이다. 이것이 CKA 의 실무 포인트: static Pod 매니페스트를 편집 = 그 컴포넌트를 재시작한다는 뜻이다.

3. DaemonSet 과 헷갈리지 않기

"노드마다 하나씩 뜨는 무언가"라는 점에서 static Pod 는 DaemonSet(0009) 과 닮아 보인다. 하지만 관리 주체와 API 의존성이 정반대다.

관리 주체API 가 관리 (DaemonSet 컨트롤러, 0007 의 controller-manager)kubelet 이 관리 (파일 기반)
API server 필요?필요 — 컨트롤 플레인이 이미 떠 있어야 함불필요 — 컨트롤 플레인 전에도 뜸
"노드마다 하나"를 누가 보장?컨트롤러가 노드를 watch 하며 Pod 를 뿌림각 노드에 파일을 깔아둔 만큼만
대표 용례kube-proxy, CNI 에이전트, 로그/모니터링컨트롤 플레인 그 자체 (kubeadm)
한 줄 구분 DaemonSet = 컨트롤 플레인이 있어야 노드마다 뿌린다. Static Pod = 컨트롤 플레인이 없어도 kubelet 이 띄운다. 그래서 "컨트롤 플레인보다 먼저, 또는 컨트롤 플레인 자신"을 띄워야 할 때만 static Pod 다. 그 외 노드-단위 데몬은 전부 DaemonSet 이 맞다. kube-proxy 가 static Pod 가 아니라 DaemonSet 인 이유 — kube-proxy 는 apiserver 가 떠 있어야 의미가 있으니까.

4. 실습 — 내 kind 노드에서 증거를 잡는다

kind 클러스터 mastery. 컨트롤 플레인 노드는 도커 컨테이너 mastery-control-plane 로 떠 있다 — 0013 에서 iptables 를 들여다볼 때 들어갔던 그 노드다. 여기선 kubectl 이 아니라 docker exec 로 노드 안 파일 시스템을 직접 본다.

추적 1 — 매니페스트 디렉터리에 컨트롤 플레인이 파일로 놓여 있다.

terminal $ docker exec mastery-control-plane ls /etc/kubernetes/manifests etcd.yaml kube-apiserver.yaml kube-controller-manager.yaml kube-scheduler.yaml ← 컨트롤 플레인 4형제가 그냥 "파일"이다

이 네 개가 0007 에서 배운 컨트롤 플레인 컴포넌트 전부다. Deployment 도 아니고 etcd 안 오브젝트도 아닌, 그냥 노드 디스크 위의 YAML — kubelet 이 이 파일들만 보고 클러스터를 부팅했다.

추적 2 — 같은 것들이 kube-system 에 "미러 Pod"로 보인다. 이름에 노드명이 붙었다.

terminal $ kubectl get pods -n kube-system | grep mastery-control-plane etcd-mastery-control-plane 1/1 Running kube-apiserver-mastery-control-plane 1/1 Running kube-controller-manager-mastery-control-plane 1/1 Running kube-scheduler-mastery-control-plane 1/1 Running

접미사 -mastery-control-plane 이 미러 Pod 의 지문이다. 보통 Pod(예: coredns) 는 coredns-xxxxx-yyyyy 처럼 랜덤 해시가 붙지만, static Pod 미러는 노드명이 그대로 붙는다.

추적 3 — 이건 미러라는 걸 오브젝트가 스스로 말해준다.

terminal $ kubectl get pod kube-scheduler-mastery-control-plane -n kube-system -o jsonpath='{.metadata.ownerReferences[0].kind}' Node ← 소유자가 ReplicaSet 이 아니라 "Node" 다. 즉 kubelet 이 진실의 원천

보통 Pod 의 소유자는 ReplicaSet/DaemonSet 같은 컨트롤러다(0002·0009). 미러 Pod 의 소유자는 Node — "이 Pod 의 진실은 API 가 아니라 저 노드에 있다"는 선언이다.

한 패킷의 일생 대신, 한 부팅의 일생. 지금까지의 조각을 부트스트랩 순서로 묶으면:

컨트롤 플레인 부팅 순서 노드 전원 ON │ ① kubelet 시작 아직 API server 없음. 로컬 파일만 봄 │ └ watch 대상: /etc/kubernetes/manifests ▼ ② 매니페스트 발견 etcd.yaml, kube-apiserver.yaml … 를 읽음 │ ③ static Pod 로 직접 실행 API server 없이 컨테이너를 띄움 (0012 의 CRI 로) │ └ etcd 뜸 → apiserver 뜸 → controller-manager·scheduler 뜸 ▼ ④ API server 살아남 이제 정상 경로(0007)가 열림 │ ⑤ kubelet 이 미러 등록 자기가 띄운 static Pod 를 API 에 읽기 전용 사본으로 ▼ └ 이름에 -노드명 붙여서 kubectl get pods -n kube-system 에 4형제가 보인다
(선택) 미러 삭제 실험 — 개념만, 직접은 권장 안 함. 만약 kubectl delete pod kube-scheduler-mastery-control-plane -n kube-system 를 치면, 미러는 잠깐 사라지지만 kubelet 이 /etc/kubernetes/manifests/kube-scheduler.yaml 이 그대로임을 보고 즉시 컨테이너를 재시작하고 미러도 다시 만든다. "API 로는 못 지운다 = 파일이 원본"의 산 증거지만, 학습 클러스터라도 스케줄러를 잠깐 죽이는 셈이니 굳이 하지 말고 이 문단으로 대신하자.

5. 회상 연습

Q1. kubeadm/kind 클러스터에서 "apiserver 를 스케줄하려면 apiserver 가 필요한데 첫 apiserver 는 어떻게 뜨나?"의 답은?

닭-달걀은 static Pod 가 푼다. kubelet 은 API server 없이도 로컬 매니페스트 파일만 보고 Pod 를 띄울 수 있다 — 그래서 apiserver 자신을 static Pod 로 만들면 apiserver 를 띄우는 데 apiserver 가 필요 없다. scheduler 도 etcd 도 apiserver 를 거쳐야 하니 부트스트랩을 맡을 수 없다.

Q2. kubectl delete pod kube-apiserver-mastery-control-plane -n kube-system 를 실행하면?

미러 Pod 는 읽기 전용 사본일 뿐 진실의 원천이 아니다. 원본은 노드의 /etc/kubernetes/manifests/kube-apiserver.yaml 파일이고, kubelet 이 그 파일을 감시하므로 미러를 지워도 파일이 남아 있으면 재생성한다. 진짜로 바꾸려면 파일을 수정해야 한다(그러면 컴포넌트가 재시작된다).

Q3. (레슨 0009 복습) kube-proxy 는 노드마다 하나씩 뜨는데 DaemonSet 이다. 왜 static Pod 가 아닐까?

간격 반복: static Pod 는 "컨트롤 플레인 전/없이 떠야 하는 것" 전용이다. kube-proxy 는 Service(EndpointSlice)를 apiserver 로 watch 해야 규칙을 심으므로(0013), 애초에 apiserver 가 있어야 한다. 컨트롤 플레인이 있는 뒤에 노드마다 뿌리는 일은 DaemonSet 컨트롤러(controller-manager)가 맡는 게 정석이다.
판정: Q1·Q2·Q3 을 맞히면 0007 이 남긴 부트스트랩의 닭-달걀이 ✅ — 컨트롤 플레인 아키텍처의 마지막 조각이 닫혔다. static Pod 는 CKA 주제지만, 이걸로 "컴포넌트가 어떻게 뜨는가"까지 설명할 수 있게 됐다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

매니페스트 디렉터리 경로는 항상 /etc/kubernetes/manifests 인가요?

기본값이 그렇고 kubeadm/kind 는 이 경로를 쓴다. 정확히는 kubelet 설정의 staticPodPath(옛 플래그 --pod-manifest-path)가 정한다. kubelet 이 이 디렉터리를 주기적으로 스캔하다 파일이 생기면 Pod 를 띄우고, 파일을 지우면 Pod 를 지운다. 즉 디렉터리에 파일을 두는 것 = Pod 를 만드는 것이다.

static Pod 를 지우려면 어떻게 해요?

노드 안에서 그 YAML 파일을 삭제(또는 디렉터리 밖으로 이동)하면 된다. kubelet 이 파일이 없어진 걸 감지해 해당 Pod 를 종료한다. kubectl delete 로는 안 된다 — 미러만 잠깐 지워질 뿐 파일이 원본이라 재생성된다(Q2). 그래서 컨트롤 플레인 컴포넌트를 진짜로 다루는 정석은 언제나 "노드의 파일"이다.

static Pod 매니페스트를 수정하면 어떻게 되나요?

kubelet 이 파일 변경을 감지해 그 Pod 를 재시작한다 — 새 스펙으로. 그래서 CKA 에서 "apiserver 에 플래그 하나 추가하라"는 문제는 kube-apiserver.yaml 을 편집하는 일이고, 저장하는 순간 apiserver 가 잠깐 내려갔다 새 설정으로 올라온다. 매니페스트 편집 = 컴포넌트 재시작이라는 실무 감각을 여기서 잡아둬야 한다.

미러 Pod 랑 진짜 static Pod 는 뭐가 다른가요?

같은 하나를 두 곳에서 본 것이다. 진짜(원본)는 노드에서 kubelet 이 돌리는 컨테이너 — 진실의 원천은 매니페스트 파일. 미러는 그걸 API server 에 등록한 읽기 전용 사본으로, kubectl get pods 로 보라고 있는 것뿐이다. 미러를 지워도 원본(파일)이 남으면 다시 생긴다. "보이는 것"과 "실제 도는 것"이 분리돼 있다는 게 핵심.

왜 소유자(ownerReference)가 Node 인가요?

보통 Pod 는 ReplicaSet·DaemonSet 같은 컨트롤러가 소유한다(그 컨트롤러가 개수를 맞춘다). 미러 Pod 는 컨트롤러가 아니라 그 노드의 kubelet 이 관리하므로, 소유자를 Node 로 표시한다. "이 Pod 의 생사는 컨트롤 플레인이 아니라 저 노드가 쥐고 있다"는 뜻 — 추적 3 에서 jsonpath 로 직접 확인한 그 값이다.

워커 노드에도 static Pod 를 놓을 수 있나요?

가능하다 — 어느 노드든 kubelet 이 돌고 매니페스트 디렉터리가 있으면 그 노드에 static Pod 가 뜬다. 다만 kubeadm 은 컨트롤 플레인 노드에만 컨트롤 플레인 4형제 매니페스트를 깔아둔다. 워커에 뭔가를 노드-고정으로 띄우고 싶을 때 쓸 수 있지만, 컨트롤 플레인이 이미 있다면 보통은 DaemonSet 이 더 관리하기 좋다(3절 표).

etcd 도 static Pod 인데, 데이터는 컨테이너가 죽으면 사라지나요?

아니다. etcd.yaml 을 열어 보면 hostPath 볼륨(예: /var/lib/etcd)이 노드 디스크에 마운트돼 있다(0011 의 볼륨 개념). 컨테이너가 재시작돼도 데이터는 노드 파일 시스템에 남는다. static Pod 는 실행 방식일 뿐, 상태 저장은 볼륨이 따로 책임진다 — 둘은 별개다.

kind 는 노드가 도커 컨테이너인데, static Pod 컨테이너는 그 안의 컨테이너인가요?

맞다 — "컨테이너 안의 컨테이너". kind 노드(mastery-control-plane)는 도커 컨테이너이고, 그 안에서 kubelet 이 또 다른 컨테이너 런타임(containerd)으로 apiserver 등을 돌린다(0012 의 CRI). 그래서 docker exec mastery-control-plane ... 로 노드에 들어가 ls /etc/kubernetes/manifests 를 보는 것이다(0013 과 같은 진입 방식).

static Pod 는 스케줄러를 안 거친다는데, requests/limits(0006)는 어떻게 되나요?

스케줄러가 노드를 고르는 단계는 없지만(이미 그 노드 고정), 매니페스트에 resources.requests 를 쓰면 kubelet 이 그대로 적용한다. 다만 scheduler 가 개입 안 하므로 "requests 로 다른 Pod 를 밀어낸다" 같은 스케줄링 계산에는 참여하지 않는다 — 컨트롤 플레인 컴포넌트는 어차피 그 노드에 반드시 있어야 하니 자리 경쟁 대상이 아니다.

이 내용이 KCNA 에 직접 나오나요?

static Pod 의 상세(미러 Pod, ownerReference 등)까지는 CKA 영역이다. KCNA 는 "컨트롤 플레인 컴포넌트가 무엇인지"(0007) 수준이면 충분하다. 하지만 이 그림이 있으면 "그 컴포넌트들은 어떻게 뜨나"라는 자연스러운 다음 질문에 답할 수 있어, 0007 아키텍처가 암기가 아니라 이야기로 남는다 — 그리고 CKA 로 갈 때 그대로 재활용된다.
