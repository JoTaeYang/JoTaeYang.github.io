인터페이스 3형제: CRI · CNI · CSI

쿠버네티스는 실행·네트워크·스토리지를 직접 하지 않는다. 대신 표준 소켓 세 개를 뚫어두고, 거기에 꽂히는 플러그인이 실제 일을 한다. 지금까지 손으로 만진 것들의 밑단에 이름을 붙인다.

이번 레슨의 win: 실습으로 이미 겪은 세 가지 — 컨테이너가 실행되고(0007), Pod 가 IP를 받고(0003), PV 가 스토리지를 얻은(0011) — 그 밑에 각각 CRI · CNI · CSI 라는 표준 인터페이스가 있었음을 내 클러스터에서 직접 확인한다.

1. 프레임 — 쿠버네티스는 소켓의 집합이다

쿠버네티스는 "무엇을 할지"(desired state)를 정하고, "어떻게 할지"의 실제 구현은 교체 가능한 플러그인에 위임한다. 그 위임의 규격이 인터페이스다. 이름은 전부 C_I 꼴이다.

인터페이스무엇을 꽂나담당내가 이미 본 곳
CRI (Container Runtime Interface) containerd, CRI-O 컨테이너 실행 kubelet 이 nodeName 을 보고 Pod 를 띄운 것 (0007)
CNI (Container Network Interface) Calico, Cilium, kindnet Pod 네트워크(IP 부여) 모든 Pod 가 IP 를 받고 서로 통신한 것 (0003)
CSI (Container Storage Interface) ebs.csi, 각종 드라이버 스토리지 프로비저닝 StorageClass 가 PV 를 자동 생성한 것 (0011)
아하 왜 직접 안 하고 소켓으로 뺐을까? 선택의 자유 때문이다. AWS 는 EBS(CSI), 사내는 Ceph(CSI), 네트워크는 보안 정책이 세면 Cilium(CNI) — 환경마다 최적이 다르다. 쿠버네티스가 하나를 강제하는 대신 "이 규격만 맞추면 뭐든 꽂아라"라고 열어둔 덕에, 같은 YAML 이 어디서나 돈다. 0008 의 "규칙과 실행 주체 분리", 0011 의 "요청과 공급 분리"와 같은 철학의 확장이다.

2. 실습 — 내 클러스터에서 셋을 찾아낸다

새로 만들 것 없이 지금 돌고 있는 kind 클러스터를 관찰만 한다. 어휘를 실물에 붙이는 게 목적이다.

CRI — 이 노드의 런타임은?

terminal $ kubectl get nodes -o wide NAME STATUS ... CONTAINER-RUNTIME mastery-control-plane Ready ... containerd://1.7.x ← Docker 가 아니라 containerd

kubelet 은 컨테이너를 직접 못 띄운다 — CRI 라는 gRPC 규격으로 containerd 에게 "이 이미지로 컨테이너 켜"라고 시킨다. 예전엔 Docker 를 썼지만, Docker 는 CRI 를 직접 말하지 못해 dockershim 어댑터가 필요했고, 그 군더더기를 걷어내며 1.24 에서 제거됐다 (KCNA 단골 문제).

CNI — Pod 에게 IP 를 준 자는?

terminal $ kubectl get pods -n kube-system | grep -E 'kindnet|kube-proxy' kindnet-abc12 1/1 Running ← kind 의 CNI 플러그인 kube-proxy-xyz34 1/1 Running ← Service 규칙(0003)을 노드에 심는 자

0003 에서 Pod 마다 IP 가 있고 web 이름으로 통신됐던 건, CNI 플러그인(kindnet)이 Pod 가 뜰 때마다 IP 를 배정하고 노드 간 경로를 깔아줬기 때문이다. 실무에선 여기 Calico/Cilium 이 들어가 NetworkPolicy(진단 Q8, Level 2 예정) 같은 정책을 집행한다.

CSI — PV 를 만든 자는?

terminal $ kubectl get pods -n local-path-storage local-path-provisioner-... 1/1 Running ← 0011 에서 PV 를 찍어낸 프로비저너
정확히 하자: kind 의 local-path-provisioner 는 "스토리지 플러그인"의 자리를 채우지만 엄밀한 CSI 드라이버는 아니다(간이 로컬 프로비저너다). 진짜 CSI 는 클라우드의 ebs.csi.aws.com, pd.csi.storage.gke.io 같은 드라이버들이다. 0011 에서 StorageClass 가 채운 그 슬롯이 CSI 의 자리라는 것만 기억하면 된다 — kind 는 그 슬롯에 간단한 대체품을 꽂았을 뿐.

3. 부록 어휘 — 오케스트레이션 상식 (개념만)

KCNA 비중상 개념 수준이면 충분하다. 실기(CKA/CKAD)에서 실습으로 다시 만난다 — 지금은 이름과 한 줄만.

용어한 줄
taint / toleration노드에 "출입 금지" 딱지(taint)를 붙이고, 그 딱지를 견디는(toleration) Pod 만 배치 허용. 전용 노드 만들 때.
node affinity"GPU 노드에 붙여줘"처럼 Pod 가 노드를 선호/요구하는 규칙. requests(0006)가 양이라면, affinity 는 위치.
HPA (Horizontal Pod Autoscaler)CPU/메모리 사용률을 보고 replicas 를 자동으로 늘렸다 줄인다. Pod 개수를 조절(수평).
VPA / Cluster AutoscalerVPA=Pod 의 requests/limits 를 조절(수직), CA=노드 자체를 늘렸다 줄임. 셋을 헷갈리지 않게 "무엇을 늘리나"로 구분.

4. 회상 연습

Q1. kubelet 이 컨테이너를 실제로 띄울 때 대화하는 표준 인터페이스는?

CRI(Container Runtime Interface). kubelet 은 직접 실행하지 않고 CRI 로 containerd/CRI-O 에 위임한다. 이미지=실행=CRI, 네트워크=CNI, 스토리지=CSI 로 역할이 깔끔히 갈린다.

Q2. "Docker 가 쿠버네티스에서 제거됐다"는 말의 정확한 의미는?

빠진 건 "kubelet↔Docker" 사이의 군더더기 어댑터(dockershim)뿐이다. Docker 로 빌드한 이미지는 OCI 표준이라 containerd 가 그대로 실행한다 — 빌드 도구로서의 Docker 와 런타임으로서의 Docker 를 구분하는 게 이 문제의 핵심이다.

Q3. (레슨 0011 복습) 클라우드에서 StorageClass 가 PV 를 자동 생성할 때 실제로 그 슬롯에 꽂히는 것은?

간격 반복: 0011 의 "공장(StorageClass)"이 부리는 실제 일꾼이 CSI 드라이버다. kind 는 그 자리에 간이 local-path 를 꽂았지만, 클라우드에선 ebs.csi/pd.csi 같은 CSI 드라이버가 진짜 블록 볼륨을 프로비저닝한다.
판정: Q1·Q2 를 맞히면 Container Orchestration 도메인의 핵심 어휘 ✅. 남은 건 비중 낮은 도메인들(Cloud Native Architecture 16% / Observability 8% / Delivery 8%)의 어휘 일괄 — 다음 레슨에서 CNCF 생태계·12-factor·관측성 3기둥·GitOps·Helm 을 몰아서 훑고(과투자 금지), 그다음이 KCNA 전 도메인 모의 진단 → 응시 일정 확정이다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

인터페이스 이름이 왜 다 C_I 예요? 외우기 헷갈려요.

가운데 글자만 바꾸면 된다: Runtime(실행), Network(네트워크), Storage(스토리지). "컨테이너의 실행·연결·저장" 세 가지를 각각 표준화한 것이다. 앞의 C(Container)와 뒤의 I(Interface)는 공통.

그럼 Docker 는 이제 아예 쓸모없어진 건가요?

아니다. Docker 는 여전히 최고의 이미지 빌드/개발 도구다. 빠진 건 "쿠버네티스 노드가 컨테이너를 실행하는 런타임"으로서의 역할뿐이다. docker build 로 만든 이미지는 OCI 표준이라 containerd 가 그대로 돌린다 — 빌드와 런타임은 다른 이야기다.

containerd 랑 CRI-O 는 뭐가 달라요? 뭘 골라야 하죠?

둘 다 CRI 를 만족하는 경량 런타임이라, 쿠버네티스 입장에선 바꿔 꽂아도 된다. containerd 는 Docker 에서 갈라져 나와 널리 쓰이고(kind·EKS 등 기본), CRI-O 는 쿠버네티스 전용으로 설계됐다(OpenShift 기본). 초보 단계에선 "둘 다 CRI 구현체" 정도면 충분하다.

CNI 플러그인은 왜 이렇게 종류가 많아요? (Calico, Cilium, kindnet…)

"Pod 에 IP 주고 연결"이라는 최소 임무는 같지만, 그 위에 얹는 기능이 다르다. kindnet 은 최소 기능(로컬 실습용), Calico/Cilium 은 NetworkPolicy 집행·암호화·관측성 등을 더한다. 보안 요구가 세면 Cilium, 단순하면 가벼운 걸 고른다.

local-path 가 CSI 가 아니면, kind 엔 CSI 가 없는 건가요?

기본 kind 엔 진짜 CSI 드라이버가 없고, 간이 프로비저너(local-path)가 그 역할을 대신한다. "StorageClass 가 채우는 슬롯 = CSI 의 자리"라는 구조만 같을 뿐이다. 실제 CSI 드라이버(ebs.csi 등)를 kind 에 설치할 수도 있지만, 학습엔 불필요하다.

kube-proxy 도 CNI 인가요? 같이 네트워크 하는 것 같은데요.

아니다, 역할이 다르다. CNI(kindnet)는 Pod 에 IP 를 주고 Pod 끼리 연결하고, kube-proxy는 Service(0003)의 가상 IP 를 실제 Pod 로 라우팅하는 규칙을 노드에 심는다. 하나는 "Pod 네트워크", 하나는 "Service 네트워크" 담당이다.

taint 랑 node affinity, 방향이 헷갈려요.

주체가 반대다. taint 는 노드가 "아무나 오지 마"라고 미는 것(Pod 이 toleration 으로 견뎌야 허용), affinity 는 Pod 이 "저 노드로 가고 싶어"라고 당기는 것이다. 미는 노드 / 당기는 Pod 로 외우면 안 헷갈린다.

HPA 는 0006 의 requests/limits 랑 무슨 관계예요?

HPA 는 보통 "CPU 사용률이 requests 대비 몇 %"를 기준으로 판단한다 — 그래서 requests(0006)를 잘 잡아야 HPA 가 제대로 동작한다. 사용률이 목표를 넘으면 replicas 를 늘리고, 내려가면 줄인다. requests 가 스케줄링의 기준이자 HPA 의 기준이기도 한 셈이다.

HPA, VPA, Cluster Autoscaler — 셋을 한 번에 정리해 주세요.

"무엇을 늘리나"로 가른다. HPA=Pod 개수(수평), VPA=Pod 하나의 requests/limits(수직), Cluster Autoscaler=노드 수. 트래픽이 늘면 HPA 가 Pod 를 늘리고, 그러다 노드가 모자라면 CA 가 노드를 늘리는 식으로 함께 works.

이 인터페이스들은 KCNA 시험에 어떻게 나와요?

대개 "kubelet 이 컨테이너 실행에 쓰는 인터페이스는?"(CRI), "Docker 제거의 의미는?"(dockershim), "Pod IP 를 배정하는 것은?"(CNI) 같은 객관식이다. 깊은 설정이 아니라 역할 매칭이 핵심이라, 오늘 표의 "무엇을 꽂나 / 담당"만 확실하면 된다.
