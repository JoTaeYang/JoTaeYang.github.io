설정은 이미지 밖으로: ConfigMap 과 Secret

같은 이미지를 dev 와 prod 에서 다르게 돌리는 방법 — 그리고 env 주입과 volume 주입의 결정적 차이.

이번 레슨의 win: ConfigMap 을 env 와 volume 두 방식으로 Pod 에 주입하고, 설정을 바꿨을 때 한쪽만 살아서 갱신되는 것을 직접 관찰한다. Secret 이 "암호화가 아니라 인코딩"이라는 것도 눈으로 확인한다.

1. 왜 분리하는가

DB 주소를 이미지 안에 구우면, dev/prod 마다 이미지를 따로 빌드해야 하고 비밀번호가 이미지 레이어에 영원히 남는다. 그래서 쿠버네티스는 설정을 별도 리소스로 뗐다:

담는 것저장 형태예시
ConfigMap민감하지 않은 설정평문로그 레벨, 기능 플래그, 설정 파일
Secret민감한 값base64 인코딩 (암호화 아님!)DB 비밀번호, API 키, TLS 인증서

Pod 에 주입하는 통로는 둘 다 같다: 환경 변수로 넣거나, volume 으로 마운트해 파일로 보이게 하거나.

2. 실습 — 두 통로의 결정적 차이

kind 클러스터 mastery 이어서 사용. 이번 YAML 은 레슨 0004 보다 한 단계 복잡하다 — 손으로 직접 타이핑해 보는 것을 권한다(복붙 금지). CKAD 근육은 그렇게 생긴다.

추적 1 — ConfigMap 하나를 두 통로로 주입:

terminal $ kubectl create configmap app-config --from-literal=LOG_LEVEL=info --from-literal=GREETING=hello
config-demo.yaml apiVersion: v1 kind: Pod metadata: name: config-demo spec: containers: - name: app image: busybox:1.36 command: ["sleep", "3600"] env: - name: LOG_LEVEL # 통로 1: 환경 변수 valueFrom: configMapKeyRef: name: app-config key: LOG_LEVEL volumeMounts: - name: config-vol # 통로 2: 파일 마운트 mountPath: /etc/config volumes: - name: config-vol configMap: name: app-config
terminal $ kubectl apply -f config-demo.yaml $ kubectl exec config-demo -- sh -c 'echo env: $LOG_LEVEL && echo file: $(cat /etc/config/LOG_LEVEL)' env: info file: info ← 두 통로 모두 info

추적 2 — 설정을 바꾸면 무슨 일이? 여기가 오늘의 핵심이다:

terminal $ kubectl create configmap app-config --from-literal=LOG_LEVEL=debug --from-literal=GREETING=hello --dry-run=client -o yaml | kubectl apply -f - $ sleep 60 && kubectl exec config-demo -- sh -c 'echo env: $LOG_LEVEL && echo file: $(cat /etc/config/LOG_LEVEL)' env: info ← 환경 변수는 그대로! (프로세스 시작 시 고정) file: debug ← 마운트된 파일은 갱신됐다 (kubelet 이 주기적으로 동기화)
아하 환경 변수는 프로세스가 시작되는 순간 박제된다 — 리눅스의 본질이지 쿠버네티스 잘못이 아니다. 바꾸려면 Pod 를 재시작해야 하고, Deployment 라면 kubectl rollout restart 가 그 일을 한다 (레슨 0002 에서 본 ReplicaSet 교체가 그대로 일어난다). volume 마운트는 kubelet 이 갱신해 주지만 앱이 파일을 다시 읽어야 의미가 있다.

추적 3 — Secret 은 암호화가 아니다:

terminal $ kubectl create secret generic db-pass --from-literal=password='S3cret!' $ kubectl get secret db-pass -o jsonpath='{.data.password}' UzNjcmV0IQ== $ kubectl get secret db-pass -o jsonpath='{.data.password}' | base64 -d S3cret! ← 읽기 권한만 있으면 누구나 원문을 꺼낸다
base64 는 잠금이 아니라 포장이다. 진짜 보호는 RBAC 로 읽기 권한을 좁히고, etcd 저장 시 암호화(encryption at rest)를 켜는 것 — 정확히 CKS(Level 5)에서 다룰 주제다. 지금은 "Secret ≠ 암호화"만 몸에 새기면 된다.

정리: kubectl delete pod config-demo && kubectl delete cm app-config && kubectl delete secret db-pass

3. 회상 연습

Q1. ConfigMap 값을 바꿨는데 env 로 주입받은 앱에 반영이 안 된다. 올바른 이해는?

추적 2 에서 본 그대로. 실무에서는 kubectl rollout restart deployment/... 로 새 Pod 를 굴려 반영한다. volume 마운트라면 파일은 갱신되지만 앱의 재읽기 여부는 앱 책임.

Q2. Secret 의 base64 에 대한 올바른 설명은?

추적 3 에서 base64 -d 한 줄로 원문을 꺼냈다. 보호의 실체는 RBAC + encryption at rest — CKS 에서 직접 설정하게 된다.

Q3. (레슨 0002 복습) rollout restart 로 env 를 반영할 때 내부에서 일어나는 일은?

간격 반복: 소유 체인. rollout restart 는 Pod 템플릿에 재시작 타임스탬프 annotation 을 찍어 "템플릿이 바뀌었다"로 만들고, 레슨 0002 의 ReplicaSet 교체 메커니즘이 그대로 굴러간다. 새 프로세스 = 새 env.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

ConfigMap 이랑 Secret, 언제 뭘 써요?

기준은 딱 하나: 이 값이 새면 곤란한가? 아니오(로그 레벨, 기능 플래그, 설정 파일) → ConfigMap. 예(비밀번호, API 키, TLS 인증서) → Secret. 둘의 사용법(env·volume 주입)은 완전히 똑같고, 다른 건 저장 형태(평문 vs base64)와 취급 습관뿐이다.

env 로 넣은 값은 왜 안 바뀌나요?

환경 변수는 프로세스가 시작되는 순간 박제되기 때문이다 — 리눅스의 본질이지 쿠버네티스 버그가 아니다. ConfigMap 을 고쳐도 이미 떠 있는 프로세스의 env 는 그대로다. 반영하려면 새 프로세스가 필요하고, 그게 Pod 재시작(또는 rollout restart)이다.

그럼 volume 으로 마운트하면 자동으로 갱신되나요?

파일 내용은 갱신된다 — kubelet 이 주기적으로(보통 1분 내) 동기화한다. 하지만 앱이 그 파일을 다시 읽어야 의미가 있다. 파일이 바뀌어도 시작할 때 한 번 읽고 마는 앱이면 결국 재시작이 필요하다. "파일은 새것, 프로세스 기억은 옛것"을 구분하라.

설정 하나 바꾸자고 Pod 를 통째로 재시작해요? 낭비 아닌가요?

Deployment 라면 낭비가 아니다 — kubectl rollout restart 가 새 Pod 를 하나씩 띄우고 준비되면 옛 것을 내리는 롤링 교체를 한다(레슨 0002 의 ReplicaSet 교체). 다운타임 없이 새 env 가 반영된다. 무중단이 이 방식의 핵심이다.

base64 는 암호화 아니에요?

아니다. base64 는 가역 인코딩일 뿐이라 base64 -d 한 줄이면 누구나 원문을 꺼낸다(추적 3 에서 직접 봤다). 암호화는 키가 있어야 풀리지만, base64 는 키가 없다 — 그냥 포장지다. "Secret ≠ 암호화"를 몸에 새기자.

그럼 Secret 은 안 쓰느니만 못한 거 아닌가요?

그렇지 않다. Secret 은 별도 리소스라서 RBAC 로 읽기 권한을 따로 좁힐 수 있고, 로그·describe 출력에 값이 안 찍히며, etcd 암호화(encryption at rest)를 켜면 저장 단계도 보호된다. base64 자체가 보호가 아닐 뿐, Secret 이라는 그릇에는 진짜 보호 장치를 걸 자리가 있다.

Secret 을 진짜 안전하게 하려면 뭘 해야 하나요?

세 겹이다: ① RBAC 로 get secrets 권한을 최소한의 주체에게만, ② etcd encryption at rest 로 디스크 저장을 암호화, ③ 외부 금고(Vault, 클라우드 Secrets Manager) 연동. ①은 레슨 0008 RBAC 에서, ②③의 본격 설정은 CKS(Level 5) 에서 손으로 다룬다. 지금은 "base64 는 잠금이 아니다"만 확실히.

같은 값을 env 랑 volume 둘 다로 넣으면 충돌하나요?

충돌하지 않는다 — 서로 다른 통로라 독립적으로 각자 값을 전달한다(추적 1 에서 한 ConfigMap 을 두 방식으로 넣었다). 다만 갱신 동작이 다르다는 걸 잊지 마라: 설정을 바꾸면 volume 쪽만 파일이 새로워지고 env 쪽은 옛날 값 그대로다. 그 차이를 직접 본 게 오늘의 핵심이다.

키를 하나하나 configMapKeyRef 로 쓰는 게 귀찮아요.

envFrom 을 쓰면 ConfigMap(또는 Secret)의 모든 키를 한 번에 환경 변수로 넣는다. 편하지만 어떤 키가 들어갔는지 YAML 만 봐선 안 보이고, 키 이름이 그대로 env 이름이 되니 대문자·언더스코어 규칙에 안 맞으면 조용히 누락될 수 있다. 소수의 값은 keyRef 로 명시하는 게 더 안전하다.

ConfigMap 을 자주 바꾸는데 매번 재시작이 걱정돼요. 고정할 방법은?

반대 방향이지만 immutable: true 라는 옵션이 있다 — 한 번 만들면 수정 불가로 잠그는 것이다. 실수로 값이 바뀌는 사고를 막고, kubelet 이 변경 감시를 안 해도 되니 대규모 클러스터에선 성능 이점도 있다. 값을 바꾸려면 새 이름의 ConfigMap 을 만들어 Deployment 가 그걸 가리키게 롤아웃한다.
