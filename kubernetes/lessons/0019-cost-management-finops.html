Kubernetes Mastery · Lesson 0019 심화 · Observability

비용 관리와 FinOps: requests 가 곧 청구서다

비용도 하나의 신호다 — 측정 못 하면 최적화 못 한다. 0006 에서 배운 requests 가 어떻게 그대로 매달 나가는 돈이 되는지, 눈으로 본다.

이번 레슨의 win: "왜 비용이 관측성 도메인이지?"의 답을 잡는다 — 비용은 메트릭·로그·트레이스(0014)의 사촌인 네 번째 신호다. 그리고 0006 의 requests(예약) 가 어떻게 유휴 노드 → 더 많은 노드 → 더 큰 청구서로 이어지는지를 kubectl describe node 로 직접 보고, OpenCost·FinOps·showback/chargeback 이라는 KCNA 어휘에 손맛을 붙인다.

0. 왜 관측성 도메인인가 — 비용은 신호다

딱 이 한 줄 측정할 수 없으면 최적화할 수 없다. 메트릭이 "CPU 얼마나 쓰나"를 보여주듯, 비용은 "그 리소스에 얼마 내고 있나"를 보여주는 신호다. 0014 의 관측성 3기둥(메트릭·로그·트레이스)이 "시스템이 어떻게 도나"를 본다면, 비용 가시성은 "그게 얼마짜리인가"를 본다. 그래서 KCNA 는 비용을 관측성(Observability) 도메인에 넣는다 — 놓치기 쉬운 항목이다.

클라우드에서 쿠버네티스는 편하지만, 편한 만큼 돈이 어디로 새는지 안 보인다. Pod 하나가 노드 어디에 떠서 얼마를 먹는지 청구서엔 안 찍힌다 — 청구서엔 "노드 10대" 라고만 나온다. 그 10대가 왜 10대인지, 절반이 놀고 있진 않은지를 보이게 만드는 것이 이 레슨의 주제다.

1. THE 핵심 연결 — requests 는 예약이고, 예약은 돈이다

0006 에서 못 박은 문장을 다시 꺼낸다: requests 는 예약이다. 스케줄러는 노드의 실제 사용량이 아니라 requests 의 합계로 자리를 잡는다(호텔 예약 장부). 이 한 문장이 비용의 뿌리다. requests 를 실사용보다 높게 잡으면 무슨 일이 벌어지는지 따라가 보자:

단계무슨 일결과
① 과대 requestsPod 가 실제로 200m 쓰는데 requests: 1 로 예약노드 장부에 800m 이 "예약됐지만 유휴"로 잠김
② bin-packing 악화장부가 빨리 차니 노드에 Pod 를 촘촘히 못 채움노드가 반만 차도 "예약 만석" → 새 Pod 는 Pending(0006)
③ 오토스케일 발동Pending 을 풀려고 Cluster Autoscaler 가 노드 추가(0012)실사용은 그대로인데 노드 수만 증가
④ 청구서노드 = 클라우드 인스턴스 = 시간당 요금안 쓰는 예약에 매달 돈을 낸다
아하 — requests 는 비용 레버다 0006 의 requests 가 그대로 비용 다이얼이다. 노드 실사용이 30% 인데 청구서가 꽉 찬 이유는 거의 항상 "requests 과대 예약 → bin-packing 이 나빠 노드가 더 필요" 이 경로다. 0006 Q1("실사용 10% 인데 Pending")에서 봤던 그 예약 장부가, 규모가 커지면 낭비되는 노드 대수로 번역된다. CPU 를 아무리 안 써도 예약이 잡혀 있으면 그 자리엔 돈이 나간다.
그림으로: 노드 용량 100. Pod A 가 실제 20 을 쓰는데 requests 80 으로 예약하면, 남은 20 에는 작은 Pod 하나밖에 못 들어간다. 나머지 60 은 "비어 있지만 예약 만석"이라 새 Pod 는 Pending → 노드 한 대가 더 뜬다. 물리적으론 40 만 쓰는데 노드 2대 값을 낸다.

2. Right-sizing — 예약을 실사용에 맞추는 기술

해법은 단순하다: requests 를 실제 사용량에 가깝게 맞춘다(right-sizing). 하지만 양쪽 다 위험하다 — 이건 0006 의 두 심판이 그대로 비용 언어로 번역된 것이다:

방향증상0006 과의 연결
과대 예약 (requests 너무 큼)노드 낭비 · 유휴 예약 · 더 많은 노드 = 돈Pending 유발(예약 장부가 참)
과소 예약 (requests 너무 작음)메모리 부족 시 OOMKilled(Exit 137) · 성능 저하0006 의 137, incompressible memory
균형점실사용 + 여유(headroom) 만큼만 예약right-sizing 의 목표
누가 균형점을 찾아주나 — VPA 손으로 맞추기 어려우니 VPA(Vertical Pod Autoscaler, 0012) 가 실사용을 관찰해 "이 워크로드는 requests 를 250m/300Mi 로 낮추라"고 추천한다. VPA 의 recommendation 모드는 곧 right-sizing 추천기다 — 과대 예약을 자동으로 찾아준다. HPA 가 Pod 수를 늘려 부하를 받는다면(0012), VPA 는 Pod 크기(requests)를 실사용에 맞춰 낭비를 줄인다.

4. 도구 — OpenCost 와 Kubecost

비용을 "보이게" 하려면 노드 청구서를 Pod·namespace 단위로 쪼개는(allocation) 도구가 필요하다. 쿠버네티스 자체는 비용을 모른다 — Pod 가 어느 노드의 얼마짜리 자원을 얼마나 예약/사용했는지 계산해 클라우드 가격표와 곱해줘야 한다. 그게 이 두 도구다:

도구정체포지션
OpenCostCNCF 프로젝트. 벤더 중립 비용 배분 표준(오픈소스)Kubecost 가 기증한 오픈 코어. "쿠버네티스 비용의 Prometheus"
KubecostOpenCost 위에 UI·리포트·최적화 추천을 얹은 제품(상용 티어 있음)OpenCost 를 감싼 완제품
OpenCost 가 하는 일 한 줄 Pod 의 requests(예약)와 실사용을 노드의 클라우드 가격표와 곱해서 "namespace=payments 가 이번 달 $340" 처럼 라벨·namespace 단위로 비용을 배분한다. 0014 에서 Prometheus 가 벤더 중립 메트릭 표준이었듯, OpenCost 는 벤더 중립 비용 표준이다 — CNCF 우산 아래 있어 특정 클라우드에 묶이지 않는다.

0012 의 오토스케일러들은 성능 도구인 동시에 비용 도구다. 부하가 빠지면 자원을 반납하게 만드는 게 핵심:

6. FinOps 3원칙 — 비용을 문화로

FinOps = Finance + DevOps. 클라우드 비용을 엔지니어·재무·경영이 함께 다루는 문화·실천이다 (FinOps Foundation 은 CNCF 처럼 Linux Foundation 산하). 세 원칙만 기억하면 된다:

원칙뜻이 레슨에서
가시성(Visibility)먼저 보이게 만든다 — 못 보면 못 고침OpenCost 로 namespace 별 비용 배분
최적화(Optimization)보이면 줄인다 — right-sizing, 오토스케일, spot2·5절
책임(Accountability)팀이 자기 비용에 책임진다showback / chargeback
순서가 핵심 가시성 → 최적화 → 책임은 순서가 곧 논리다. 보여야(가시성) 줄일 수 있고(최적화), 누구 것인지 붙여야(책임/showback) 팀이 스스로 줄인다. 이건 0014 의 관측성 루프 "메트릭으로 감지 → 로그로 원인 → 트레이스로 병목"과 같은 골격 — 측정이 먼저다.

7. 관찰 (선택, 2분) — 예약됐지만 유휴를 눈으로

도구 설치 없이, 0006 의 예약 장부를 클러스터에서 직접 본다. describe node 의 Allocated resources 가 바로 "예약 % (requests 합)"이다:

terminal $ kubectl describe node | grep -A6 'Allocated resources' Allocated resources: Resource Requests Limits cpu 1850m (92%) 3200m (160%) memory 2400Mi (78%) 4Gi (133%)

여기서 92% 는 예약률이다 — 실사용이 아니라 requests 의 합. 이제 실사용과 비교한다:

terminal $ kubectl top nodes NAME CPU(cores) CPU% MEMORY MEMORY% mastery-node-0 210m 10% 900Mi 29% # 예약 92% vs 실사용 10% !
이 두 줄이 이 레슨 전부다 예약(describe node)은 92%인데 실사용(kubectl top)은 10% — 바로 이 간극이 돈이다. 노드는 "예약 만석"이라 새 Pod 를 못 받아(Pending, 0006) 오토스케일러가 새 노드를 붙이지만, 물리적으론 90% 가 놀고 있다. 이 82%p 의 간극 × 노드 대수 × 인스턴스 시간요금 = 낭비되는 청구서다. right-sizing 은 이 간극을 좁히는 일이다.
metrics-server 가 없으면 kubectl top 이 "Metrics API not available" 을 낸다 — 0014 에서 본 그 메시지다. kind 라면 kubectl apply -f https://.../metrics-server 후 --kubelet-insecure-tls 패치가 필요할 수 있다. 없어도 개념은 describe node 의 Allocated 만으로 충분히 보인다.

8. 회상 연습

Q1. 노드의 CPU 실사용률이 10% 인데 클러스터가 계속 새 노드를 띄워 비용이 오른다. 가장 유력한 근본 원인은?

0006 복습: 스케줄러는 실사용이 아니라 requests 합으로 판단한다. 과대 requests → bin-packing 악화 → Pending → Cluster Autoscaler 가 노드 추가(0012). 실사용은 10% 여도 예약이 꽉 차 노드 값이 계속 나간다. requests right-sizing 이 근본 해법.

Q2. 팀별로 "이번 달 얼마 썼는지 내역서를 보내되, 실제 예산 차감은 하지 않는" 방식을 뭐라 하나?

showback = 보여주기(청구 없음), chargeback = 진짜 차감(내부 청구). 둘 다 FinOps 3원칙 중 "책임(accountability)"을 구현하는 방식이다. 헷갈리면 "show=보여줌, charge=청구함"으로 어원을 잡으면 된다.

Q3. 쿠버네티스 비용을 namespace·label 단위로 배분해주는, CNCF 의 벤더 중립 오픈소스 표준은?

OpenCost = 비용의 벤더 중립 표준(Kubecost 가 CNCF 에 기증한 오픈 코어). Prometheus 가 메트릭 표준이듯 OpenCost 는 비용 배분 표준이다. VPA(0012)는 right-sizing 도구로 최적화를 돕지만 비용 배분 자체는 아니다.
판정: Q1·Q2·Q3 를 맞히면 KCNA 관측성 도메인의 숨은 항목(비용/FinOps)이 ✅ — "requests 가 비용을 좌우한다 · OpenCost · FinOps 3원칙 · showback/chargeback" 이 시험 깊이의 전부다. 그 이상(실제 OpenCost 배포, spot 노드 운영, 예산 알림)은 심화·실무 몫이니 과투자 금지.

초보자 질문 (막히면 여기부터)

비용은 개념 여러 개가 얽혀 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

비용이 왜 "관측성" 도메인이에요? 재무 아닌가요?

KCNA 는 비용을 Observability(관측성)에 넣는다. 이유는 "측정 없이는 최적화 없다"는 관측성의 핵심 논리가 비용에도 그대로 적용되기 때문이다. 비용은 메트릭·로그·트레이스(0014)와 나란한 또 하나의 신호 — 시스템이 얼마짜리인지를 보여준다. 그래서 재무 주제이면서 동시에 "보이게 만드는" 관측성 주제다.

requests 를 낮추면 무조건 이득 아닌가요? 왜 균형을 찾아요?

너무 낮추면 0006 의 OOMKilled(Exit 137)가 난다. requests 는 메모리 여유를 예약하는 것이기도 해서, 실사용보다 지나치게 낮게 잡으면 부하가 몰릴 때 자원이 부족해 죽거나(메모리) 느려진다(CPU throttling). 그래서 "실사용 + 약간의 여유(headroom)"가 균형점이고, VPA 추천(0012)이 그 지점을 잡아준다. 과대 예약=낭비, 과소 예약=장애.

OpenCost 랑 Kubecost, 뭘 알아야 해요?

KCNA 는 OpenCost = CNCF 벤더 중립 비용 배분 표준(오픈소스)만 알면 된다. Kubecost 는 그 위에 UI·리포트·추천을 얹은 상용 제품이고, OpenCost 는 Kubecost 가 CNCF 에 기증한 오픈 코어다. 관계만 잡으면 충분 — "OpenCost 는 표준, Kubecost 는 그걸 감싼 완제품".

showback 이랑 chargeback, 계속 헷갈려요.

어원으로 잡는다. showback = "보여주기"(내역서만 발송, 돈은 안 움직임). chargeback = "청구하기"(팀 예산에서 실제 차감). 둘 다 "각 팀이 자기 비용에 책임진다(FinOps 의 accountability)"를 구현하지만, showback 은 인식 유도, chargeback 은 실제 내부 청구다. 조직이 성숙하면 showback → chargeback 으로 넘어가는 경우가 많다.

spot / preemptible 노드가 싸면 다 그걸 쓰면 되나요?

안 된다 — spot 노드는 클라우드가 언제든 회수(수 분 예고 후)할 수 있다. 그래서 죽어도 되는 워크로드(stateless, 재시작 가능한 배치·워커, 0014 의 12-factor "disposable")에만 적합하다. 데이터베이스처럼 상태를 가진(0011 PV) 워크로드나 항상 떠 있어야 하는 서비스엔 위험하다. "60~90% 싼 대신 언제든 뺏길 수 있음"이 핵심 트레이드오프.

scale-to-zero 는 HPA 로 되나요?

기본 HPA(0012)는 보통 minReplicas: 1 이라 0 까지 안 내려간다 — 요청을 받을 Pod 가 최소 1개는 떠 있어야 하기 때문이다. 진짜 0개까지 내리려면 0017 의 Knative(또는 KEDA) 같은 serverless 계층이 필요하다. 요청이 오면 그때 Pod 를 띄우고(cold start), 없으면 0개 → 안 쓰는 동안 공짜. 트래픽이 뜸한 워크로드의 비용 절감 핵심 기법이다.

kubectl describe node 의 Allocated 가 100% 넘게 나와요(예: limits 160%).

정상이다. requests 합은 100% 를 넘을 수 없지만(예약이라서), limits 합은 넘을 수 있다 — 모든 Pod 가 동시에 상한까지 쓰진 않는다는 전제로 오버커밋(overcommit)을 허용하기 때문이다(0006 의 두 심판: requests=예약, limits=상한). 비용을 볼 땐 Requests %(예약률)를 봐야 한다. 그게 노드에 얼마나 자리가 잠겼는지를 나타낸다.

egress 요금이 뭐예요? 클러스터 안 통신도 돈 드나요?

보통 같은 AZ(가용영역) 안의 통신은 공짜지만, 다른 AZ·리전으로 넘어가거나 인터넷으로 나가는(egress) 데이터엔 GB 당 요금이 붙는다. 마이크로서비스가 무심코 리전을 넘나들거나 대용량을 외부로 내보내면 청구서의 숨은 큰 항목이 된다. KCNA 수준에선 "egress = 나가는 방향엔 돈이 든다" 정도만 알면 충분하고, 세부 최적화는 실무 몫이다.

PV 를 지웠는데 비용이 계속 나와요.

PVC/PV 의 reclaim policy가 Retain 이면, PVC 를 지워도 뒤의 실제 클라우드 디스크(볼륨)는 남아 계속 과금된다(0011 PV/PVC). Delete 정책이면 함께 삭제된다. "지웠는데 왜 돈이?"의 단골 원인이 이 orphaned volume 이다 — 비용 도구(OpenCost)가 이런 유휴 스토리지를 찾아주기도 한다.

FinOps 를 공부하려면 뭐부터 봐야 해요?

KCNA 는 3원칙(가시성·최적화·책임)과 "requests 가 비용을 좌우한다", "OpenCost", "showback/chargeback" 정도면 충분하다 — 그 이상은 자격증 범위 밖이다. 더 깊게 가려면 FinOps Foundation 문서(원전 링크)와 OpenCost 문서를 보되, 실습은 실제 클라우드 클러스터가 있을 때 하는 게 낫다. 지금은 0006 requests 와의 연결만 확실히 잡으면 시험엔 충분하다.
