Kubernetes Mastery · Lesson 0025 Fundamentals 심화

Init 컨테이너 & 네이티브 사이드카 — 앱 앞뒤로 붙는 컨테이너

0016 메시가 Pod 에 몰래 넣던 그 사이드카의 정체와 메커니즘이 여기 있다. 앱보다 먼저 도는 것(init), 앱과 나란히 도는 것(sidecar) — 순서를 쿠버네티스가 어떻게 보장하는가.

이번 레슨의 win: 0002 에서 "한 Pod 에 여러 컨테이너"를 배웠고 0016 에서 메시가 사이드카를 자동 주입하는 걸 봤다. 이제 그 아래 메커니즘 — init 컨테이너(앱 전에 순차 실행·완료 대기)와 네이티브 사이드카(1.29 stable, 앱 전에 뜨고 뒤에 종료) — 를 손으로 만진다. 덤으로 0009 의 "사이드카 때문에 Job 이 Complete 못 하는" 고전 문제가 왜 네이티브 사이드카로 풀리는지 확실히 갈라 둔다.

1. Init 컨테이너 — 앱보다 "먼저, 순서대로, 끝까지"

Init 컨테이너는 Pod 의 앱 컨테이너들이 시작되기 전에 실행되는 컨테이너다. 세 가지 성질이 전부다:

성질뜻왜 유용한가
먼저모든 앱 컨테이너보다 앞서 실행앱이 뜨기 전에 판을 깔 수 있다
순서대로(sequential)init 이 여럿이면 하나씩 차례로 (병렬 아님)1번이 끝나야 2번 — 단계적 준비
끝까지(완료 대기)init 이 성공 종료(exit 0) 해야 다음으로 넘어감준비가 끝나기 전엔 앱을 안 띄운다
아하 — 앱은 판이 깔린 뒤에만 뜬다 init 컨테이너들이 전부 순차로 성공해야 비로소 앱 컨테이너들이 시작된다. 그전까지 Pod 는 Init:0/1 같은 상태에 멈춰 있다. 그래서 "DB 가 준비될 때까지 대기", "설정 파일 다운로드", "DB 마이그레이션", "볼륨 파일 권한 세팅" 같은 선행 작업을 앱 코드에 넣지 않고 init 으로 분리한다 — 앱 이미지는 앱만 하면 된다.

대표 용도 두 갈래:

용도예
의존성 대기DB/다른 Service 가 응답할 때까지 until nslookup db; do sleep 2; done
셋업설정/시크릿 다운로드, 스키마 마이그레이션, emptyDir 볼륨에 파일 권한 부여
init 이 실패하면? 앱은 아예 안 뜬다. init 컨테이너의 재시작은 Pod 의 restartPolicy(0023) 를 따른다 — Always/OnFailure 면 init 이 성공할 때까지 계속 재시도(Pod 는 Init:CrashLoopBackOff 로 보임), Never 면 Pod 가 Init:Error 로 실패한다. 어느 쪽이든 앱 컨테이너는 시작조차 안 된다 — 그래서 "의존성 대기"를 init 에 넣으면 의존성이 준비될 때까지 앱이 절대 안 뜨는 안전한 게이트가 된다.

2. 실행 순서 — init 은 순차, 앱은 병렬

한 Pod 안에서 시간 축은 이렇게 흐른다:

타임라인 한 줄 init-1 → init-2 → … → (전부 exit 0) → 앱 컨테이너들 병렬 시작. init 은 하나씩 순서대로 돌고 각자 성공 종료를 기다린다. 반면 앱 컨테이너(containers:)들은 서로 순서 보장이 없고 준비되는 대로 동시에 뜬다. 그래서 앱들 사이에 "이게 먼저 떠야 한다"가 필요하면 일반 앱 컨테이너로는 보장이 안 된다 — 이 빈틈이 다음 절의 네이티브 사이드카를 부른다.
0002 와 이어짐: 0002 에서 "한 Pod = 여러 컨테이너가 네트워크·볼륨 공유"를 배웠다. init 컨테이너도 같은 Pod 의 볼륨을 공유하므로, init 이 emptyDir 에 받아둔 설정 파일을 앱 컨테이너가 그대로 읽는 식의 바통 넘기기가 된다.

3. 네이티브 사이드카 — restartPolicy: Always 인 init 컨테이너

쿠버네티스 1.28 베타 · 1.29 stable 부터, initContainers 항목에 restartPolicy: Always 를 주면 그 컨테이너는 네이티브 사이드카가 된다. 이름은 init 리스트에 있지만 동작은 다르다:

아하 — init 자리에 있지만 안 끝난다 일반 init 은 "성공 종료해야 다음"이지만, restartPolicy: Always 인 init 은 계속 돌게 두고 Ready 가 되면 다음으로 넘어간다. 그래서 이 컨테이너는 (1) 앱 컨테이너보다 먼저 뜨고, (2) 앱과 나란히 계속 돌고, (3) 앱이 끝난 이후에 종료된다. "앱 전에 up, 앱 후에 down" 이 보장된다.

이게 왜 큰일이냐면, 옛날 방식 사이드카(그냥 containers: 에 컨테이너 하나 더 넣기)의 3대 고질병을 한 번에 고치기 때문이다:

옛 사이드카의 문제왜 아팠나네이티브 사이드카의 해결
시작 순서앱 컨테이너들은 병렬 시작 — 프록시가 앱보다 늦게 뜨면 앱 초기 요청이 샌다init 자리라 앱보다 먼저 뜨고 Ready 후 앱 시작
종료 순서종료 때 프록시가 앱보다 먼저 죽으면 앱의 마지막 요청이 끊긴다사이드카는 앱 이후에 종료
Job 완료 차단사이드카가 안 죽어 Job 의 main 이 끝나도 Pod 가 Complete 안 됨main 이 끝나면 사이드카도 자동 종료 → Job 완료

4. 일반 사이드카 vs 네이티브 사이드카 (0016 메시와 대비)

0016 에서 메시가 주입하던 Envoy 프록시가 바로 사이드카다. 그때는 대개 일반 멀티컨테이너 방식이었다 — 같은 Pod 의 containers: 에 프록시를 하나 더 넣는. 두 방식을 나란히 놓으면 차이가 선명하다:

일반 사이드카 (containers:)네이티브 사이드카 (initContainers + restartPolicy: Always)
어디에 선언spec.containersspec.initContainers + restartPolicy: Always
시작 순서앱과 병렬 — 보장 없음앱보다 먼저 (init 단계)
종료 순서보장 없음앱 이후 종료
Job 에서안 죽어 Job Complete 차단main 끝나면 종료 → Job 완료
버전항상 됨1.29+ (stable)
0016 과 이어짐: 0016 의 메시 사이드카는 "앱은 손 안 대고 옆에 프록시를 붙여 트래픽을 대신 처리"하는 패턴이었다. 그 패턴은 그대로고, 네이티브 사이드카는 그 패턴을 순서까지 안전하게 만든 최신 메커니즘이다. 요즘 메시들이 네이티브 사이드카로 옮겨가는 이유가 이 표다.

6. Job 에서 왜 네이티브 사이드카가 결정적인가 (0009)

0009 에서 Job 은 "main 컨테이너가 성공 종료하면 완료(Complete)"라고 배웠다. 여기에 일반 사이드카를 붙이면 사고가 난다:

고전 함정 Job 에 로깅/프록시 일반 사이드카를 붙이면, main 작업이 exit 0 로 끝나도 사이드카는 계속 돌기 때문에 Pod 안에 살아있는 컨테이너가 남아 Job 이 영원히 Complete 하지 못한다. 사이드카는 "끝"이라는 개념이 없으니까. 네이티브 사이드카는 이걸 정의로 해결한다 — main(앱) 컨테이너가 끝나면 사이드카도 종료된다. 그래서 Job 이 정상 완료된다.

7. Hands-on (a) — init 컨테이너로 서비스 대기 관찰 (kind mastery)

kind 클러스터 mastery 이어서 사용. 0020 처럼 lab namespace 에서 격리해 실습한다. init 이 성공할 때까지 앱이 안 뜨는 걸 눈으로 보는 게 목표다.
terminal $ kubectl create namespace lab 2>/dev/null; kubectl config set-context --current --namespace=lab

init 이 15초 기다리는 Pod. init-demo.yaml — 앱은 nginx, init 은 그저 sleep 으로 "준비 대기"를 흉내 낸다:

init-demo.yaml apiVersion: v1 kind: Pod metadata: name: init-demo spec: initContainers: - name: wait-a-bit image: busybox:1.36 command: ['sh', '-c', 'echo waiting...; sleep 15; echo ready'] # 앱 전에 15초 순차 실행 containers: - name: app image: nginx:1.27 # init 이 exit 0 해야 비로소 시작
terminal $ kubectl apply -f init-demo.yaml pod/init-demo created $ kubectl get pod init-demo -w # -w = watch, 상태 전이를 실시간으로 NAME READY STATUS RESTARTS AGE init-demo 0/1 Init:0/1 0 2s # init 실행 중 — 앱은 아직 안 뜸 init-demo 0/1 Init:0/1 0 14s init-demo 0/1 PodInitializing 0 16s # init 끝, 앱 이미지 준비 중 init-demo 1/1 Running 0 18s # init 성공 → 앱 시작 (Ctrl+C 로 watch 종료)

init 의 로그는 -c 로 컨테이너를 지정해서 본다 (앱 로그와 섞이지 않게):

terminal $ kubectl logs init-demo -c wait-a-bit # -c <init-name> 으로 init 컨테이너 로그 waiting... ready
직접 확인해 볼 것 init 의 command 를 ['sh','-c','exit 1'] 로 바꿔 다시 apply 해 보라. Pod 는 Init:Error → (기본 restartPolicy Always 라) Init:CrashLoopBackOff 로 맴돌고 앱은 영영 안 뜬다 — 0023 의 재시작 정책이 init 에도 그대로 적용된다는 걸 몸으로 확인하는 순간이다.

8. Hands-on (b) — 네이티브 사이드카 (kind mastery)

사이드카가 앱과 나란히 도는 Pod. sidecar-demo.yaml — initContainers 에 있지만 restartPolicy: Always 라 끝나지 않고 앱과 공존한다:

sidecar-demo.yaml apiVersion: v1 kind: Pod metadata: name: sidecar-demo spec: initContainers: - name: logger # init 자리에 있지만… image: busybox:1.36 restartPolicy: Always # ← 이 한 줄이 네이티브 사이드카로 만든다 command: ['sh', '-c', 'while true; do echo "[sidecar] tick"; sleep 5; done'] containers: - name: app image: nginx:1.27 # 사이드카가 Ready 된 뒤 시작, 사이드카는 계속 돎
terminal $ kubectl apply -f sidecar-demo.yaml pod/sidecar-demo created $ kubectl get pod sidecar-demo NAME READY STATUS RESTARTS AGE sidecar-demo 2/2 Running 0 8s # 2/2 — 사이드카와 앱이 나란히 Running $ kubectl logs sidecar-demo -c logger --tail=2 [sidecar] tick [sidecar] tick
일반 사이드카였다면? 위 logger 를 initContainers 대신 containers: 에 넣고 restartPolicy 줄을 지우면 동작은 비슷해 보인다(2/2 Running). 차이는 순서 보장과 Job 에서의 종료에서 드러난다 — kind create cluster 가 1.29+ 이미지를 쓰므로 네이티브 사이드카 필드가 그대로 먹는다. 구버전 kind 라면 restartPolicy: Always 가 무시될 수 있으니 kubectl version 으로 서버 1.29+ 를 확인하라.

정리: kubectl delete ns lab && kubectl config set-context --current --namespace=default.

시험 범위 표시: KCNA 는 "init 컨테이너 개념"(앱 전에 순차·완료 대기)까지가 범위다. 네이티브 사이드카·사이드카 패턴·Job 완료 상호작용은 CKAD 심화 — 여기선 "왜 이렇게 설계됐나"의 감을 잡아 두면 CKAD 로 매끄럽게 이어진다.

9. 회상 연습

Q1. init 컨테이너가 2개인 Pod 를 만들었다. 실행 방식으로 옳은 것은?

init 컨테이너는 하나씩 순차로 돌며 각자 exit 0 를 기다린다. 전부 성공한 뒤에야 앱 컨테이너(containers:)들이 병렬로 시작된다. 그전까지 Pod 는 Init:0/2 등에 멈춘다.

Q2. (레슨 0009 복습) Job 의 main 컨테이너에 로깅 일반 사이드카(그냥 containers: 에 추가)를 붙였다. main 이 exit 0 로 끝났다. Job 은?

일반 사이드카는 "끝"이 없어 계속 돈다. Job 은 Pod 안에 실행 중인 컨테이너가 남아 있어 Complete 하지 못하고 매달린다. 이를 정의로 푸는 것이 네이티브 사이드카 — main 이 끝나면 사이드카도 종료되어 Job 이 완료된다.

Q3. (레슨 0016 대비) 네이티브 사이드카를 만드는 방법과 그 보장으로 옳은 것은?

네이티브 사이드카(1.29 stable)는 initContainers 안의 컨테이너에 restartPolicy: Always 를 준 것이다. init 자리라 앱보다 먼저 뜨고, Always 라 끝나지 않고 앱과 공존하다, 앱 종료 후 내려간다. 일반 containers: 사이드카는 이 순서 보장이 없다.
판정: Q1·Q2·Q3 를 맞히면 init 컨테이너의 순차·완료 대기 규칙, Job 에서의 사이드카 완료 차단 문제, 그리고 네이티브 사이드카의 "앞에 up, 뒤에 down" 보장이 ✅ — 0016 메시 사이드카의 정체가 메커니즘 수준에서 열렸다. 다음(0026+)은 이 컨테이너들이 공유 볼륨으로 바통을 넘기는 실전 조립으로 이어진다.

초보자 질문 (막히면 여기부터)

init 과 사이드카는 "언제 뜨고 언제 죽나"의 타이밍이 핵심이라 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

init 컨테이너랑 그냥 앱 컨테이너, 뭐가 달라요?

둘 다 같은 Pod 안에 있지만 시점이 다르다. init 은 앱 컨테이너 전에, 하나씩 순서대로, 성공 종료할 때까지 돈다. 앱 컨테이너(containers:)는 init 이 모두 끝난 뒤 병렬로 시작되어 계속 산다. init 은 "판 깔기(준비 작업)", 앱은 "본 작업"이라고 보면 된다.

왜 준비 작업을 앱 코드 안에서 하지 않고 init 으로 빼나요?

관심사 분리 때문이다. "DB 준비 대기·설정 다운로드" 같은 인프라성 선행 작업을 앱 이미지에 넣으면 앱이 무거워지고, 다른 앱에서 재사용도 안 된다. init 으로 빼면 앱 이미지는 앱만 하면 되고, 대기/셋업 로직은 작은 busybox 같은 이미지로 재사용·교체가 쉽다. 또 init 은 앱보다 강한 권한(예: 볼륨 권한 변경)을 잠깐만 가질 수 있어 보안에도 유리하다.

Init:0/1 에서 Pod 가 안 넘어가요. 뭘 봐야 하나요?

init 컨테이너가 아직 성공 종료(exit 0)를 못 한 것이다. kubectl logs <pod> -c <init-name> 로 init 로그를, kubectl describe pod <pod> 로 이벤트를 본다. "DB 대기" 같은 init 이라면 진짜로 그 의존성이 안 떠서 정상적으로 기다리는 중일 수도 있다. init 이 계속 실패하면 0023 의 restartPolicy 에 따라 재시도(CrashLoopBackOff)한다.

init 이 실패하면 앱은 조금이라도 뜨나요?

아니다. 전혀 안 뜬다. init 이 성공해야만 앱 컨테이너가 시작되므로, init 이 실패하면 Pod 는 Init:Error 또는 Init:CrashLoopBackOff 에 머물고 앱은 시작조차 되지 않는다. 이게 오히려 장점 — "의존성이 준비 안 됐는데 앱이 반쯤 떠서 에러를 뿌리는" 상황을 원천 차단하는 게이트가 된다.

사이드카가 정확히 뭐예요? 0016 에서 본 그거랑 같아요?

같다. 사이드카는 앱 컨테이너 옆에 붙어 앱을 대신하거나 보조하는 컨테이너다(로깅 수집·프록시·설정 리로더 등). 0016 의 메시가 자동 주입하던 Envoy 프록시가 대표적인 사이드카다. 이 레슨은 그 사이드카를 "어떻게 안전하게(순서 보장) 붙이나"라는 메커니즘을 다룬다.

네이티브 사이드카는 왜 하필 initContainers 에 넣어요? 이상하지 않나요?

영리한 재활용이다. 사이드카에 필요한 건 "앱보다 먼저 떠야 한다"는 성질인데, 그건 이미 init 컨테이너가 가진 성질이다. 그래서 init 자리에 넣되 restartPolicy: Always 를 붙여 "먼저 뜨지만 끝나지 않고 계속 도는" 컨테이너로 만든 것이다. 새 필드를 만들지 않고 기존 init 순서 보장을 그대로 얻는 설계다.

그냥 containers: 에 컨테이너 하나 더 넣는 옛 방식은 이제 쓰면 안 되나요?

여전히 동작하고, 순서가 안 중요한 단순한 보조 컨테이너면 충분하다. 다만 시작/종료 순서가 중요하거나(프록시가 앱보다 먼저 떠야 함) Job 에서 사이드카를 쓸 때는 옛 방식이 문제를 일으키므로 네이티브 사이드카가 정답이다. "순서·Job 이 걸리면 네이티브, 아니면 아무거나"로 기억하면 된다.

Job 이 Complete 안 되고 계속 Running 이에요. 사이드카 때문인가요?

그럴 확률이 높다. 0009 에서 Job 은 "main 이 성공 종료하면 완료"인데, containers: 에 넣은 일반 사이드카는 안 죽으니 살아있는 컨테이너가 남아 Job 이 영영 Complete 못 한다. 사이드카를 initContainers + restartPolicy: Always(네이티브)로 바꾸면 main 종료 시 사이드카도 내려가 Job 이 완료된다.

init 컨테이너와 네이티브 사이드카가 앱과 볼륨을 공유하나요?

그렇다. 0002 에서 배운 대로 한 Pod 의 컨테이너들은 볼륨(예: emptyDir)과 네트워크를 공유한다. 그래서 init 이 emptyDir 에 설정 파일을 받아두면 앱이 그대로 읽고(바통 넘기기), 로깅 사이드카는 앱이 볼륨에 쓴 로그 파일을 읽어 내보낸다. 컨테이너를 나누되 데이터는 공유하는 게 이 패턴들의 핵심이다.

내 kind 클러스터에서 네이티브 사이드카가 안 먹으면요?

서버 버전이 1.29 미만이면 initContainers 안의 restartPolicy: Always 가 무시되거나 거부될 수 있다. kubectl version 으로 Server Version 을 확인하고, 낮으면 최신 kind 로 kind create cluster 를 다시 만들면 된다(요즘 kind 는 1.29+ 이미지를 기본으로 쓴다). init 컨테이너 자체(Hands-on a)는 아주 오래전부터 되므로 버전과 무관하게 실습 가능하다.
