Kubernetes Mastery · Lesson 0020 Fundamentals 심화

QoS 클래스 & eviction — 누가 먼저 쫓겨나나

0006 의 requests/limits 가 Pod 에 등급(QoS)을 매기고, 노드가 쪼들리면 그 등급 순서로 쫓아낸다. "예산 안 낸 Pod 가 먼저 나간다."

이번 레슨의 win: 0006 에서 넣은 requests/limits 만으로 쿠버네티스가 Pod 에 자동으로 붙이는 세 등급(Guaranteed·Burstable·BestEffort)의 판정 규칙을 손으로 확인하고, 노드 메모리 압박 때 어느 순서로 쫓겨나는지를 직관으로 새긴다. 덤으로 자주 뒤섞이는 OOMKill(개별 컨테이너) vs eviction(노드 전체)을 확실히 갈라 둔다.

1. 세 등급은 requests/limits 에서 저절로 나온다

0006 에서 우리는 requests(예약)와 limits(단속)를 넣었다. 그런데 그걸 어떻게 넣었는지에 따라 쿠버네티스가 Pod 에 QoS 클래스라는 등급을 자동으로 붙인다. 내가 따로 지정하는 필드가 아니라 requests/limits 조합의 결과로 계산되는 값이라는 게 핵심이다.

QoS 클래스판정 조건 (Pod 안 모든 컨테이너 기준)한 줄 직관
Guaranteed 모든 컨테이너가 cpu·memory 둘 다 requests 와 limits 를 설정했고, 각각 requests == limits (값이 정확히 같음) "예산을 딱 못박아 완납" — 가장 안전
Burstable Guaranteed 는 아니지만, 최소 한 컨테이너가 cpu 또는 memory 의 request 나 limit 을 하나라도 설정 "일부라도 예약함" — 중간
BestEffort 어떤 컨테이너도 requests/limits 를 하나도 설정 안 함 "예산 0" — 가장 먼저 희생
아하 — 판정은 위에서 아래로 순서대로 걸러진다: 먼저 모두 requests==limits (cpu·memory 다)면 Guaranteed. 아니면서 뭐라도 하나 설정했으면 Burstable. 정말 아무것도 없으면 BestEffort. 그래서 "limits 만 넣어도" 쿠버네티스가 requests 를 같은 값으로 채우니 (0006 에서 본 그 자동 채움) Guaranteed 가 될 수 있다. 반대로 memory 만 requests==limits 이고 cpu 는 비었으면 Guaranteed 가 아니라 Burstable — "둘 다, 모든 컨테이너"라는 and 조건을 놓치기 쉽다.
0006 과 이어짐: requests 는 스케줄러가 배치 시점에 보는 예약, limits 는 kubelet/커널이 실행 내내 보는 상한이었다. QoS 클래스는 이 두 숫자를 어떻게 조합했느냐를 요약한 라벨일 뿐 — 새 개념이 아니라 0006 의 파생물이다.

2. 노드가 쪼들리면 — eviction 순서

노드의 메모리가 압박(memory pressure)받으면 kubelet 은 노드를 지키려고 Pod 를 쫓아낸다(evict). 이때 순서가 바로 QoS 클래스다. 예산 장부의 직관 그대로 — 예산을 안 낸 Pod 가 먼저 나간다.

쫓겨나는 순서대상왜
1순위 (먼저)BestEffort예약이 0 — 잃을 게 없는 셈, 가장 먼저 희생
2순위Burstable 중 자기 memory request 를 초과해 쓰는 것부터"예약한 것보다 더 쓰고 있는" Pod 가 먼저
3순위 (마지막)Guaranteed예약=상한으로 딱 맞게 산다 — 노드가 정말 못 버틸 때만
직관 한 줄 "예산을 안 낸 Pod 가 먼저 쫓겨난다." BestEffort(예산 0) → 예약보다 초과 사용 중인 Burstable → Guaranteed(완납) 순. 스케줄러가 0006 의 requests 로 배치를 결정했듯, kubelet 은 같은 requests 를 기준으로 퇴거를 결정한다. 같은 장부, 다른 심판.
0012 와의 차이: 0012 의 taint/affinity 는 "이 Pod 를 어느 노드에 놓을까"라는 배치 규칙이었다. eviction 은 이미 놓인 Pod 를 노드가 내보내는 것 — 반대 방향이다. 쫓겨난 Pod 는 (컨트롤러가 관리한다면) 다시 스케줄되어 taint/affinity 규칙을 또 통과해야 한다. 그리고 0002 에서 배운 대로 Pod 는 ephemeral — evict 는 그 무상함이 노드 레벨에서 발동하는 한 가지 경우다.

3. OOMKill 과 eviction 은 다른 사건이다

둘 다 "Pod 가 메모리 때문에 죽거나 사라진다"라서 뭉뚱그리기 쉽지만, 층이 다르다. 0006 의 Exit Code 137 은 OOMKill 쪽이었다.

OOMKilleviction
누가 리눅스 커널 (OOM killer) 노드의 kubelet
무엇 때문에 그 컨테이너 하나가 자기 memory limit 초과 노드 전체가 메모리 부족(pressure)
범위 개별 컨테이너 (개별) 노드 위 Pod 들 (노드 레벨)
신호/상태 Exit Code 137, Reason OOMKilled Pod status Evicted (describe/events)
순서 기준 없음 — 넘긴 그 컨테이너가 죽음 QoS 클래스 (BestEffort 먼저)
가장 흔한 혼동: "메모리 부족이면 QoS 낮은 게 먼저 죽는다"는 eviction에만 맞는 말이다. OOMKilled(137) 은 QoS 와 무관하게 자기 limit 을 넘긴 바로 그 컨테이너가 죽는다 — Guaranteed 여도 자기 상한을 넘기면 OOMKill 된다. "limit 초과 = 개별 처형(OOMKill), 노드 부족 = QoS 순 퇴거(eviction)"로 갈라서 기억하라.

4. Hands-on — 세 등급을 눈으로 확인 (kind mastery)

kind 클러스터 mastery 이어서 사용. 0006 처럼 lab namespace 에서 격리해 실습한다. kubectl 버전마다 run 의 리소스 플래그가 다를 수 있으니, 안전하게 매니페스트 YAML 을 apply 하는 방식으로 간다.
terminal $ kubectl create namespace lab 2>/dev/null; kubectl config set-context --current --namespace=lab

세 Pod 를 한 매니페스트로. qos.yaml — 각 Pod 의 requests/limits 만 다르게 했다:

qos.yaml apiVersion: v1 kind: Pod metadata: name: g-pod # Guaranteed 노리기 spec: containers: - name: app image: nginx:1.27 resources: requests: cpu: "100m" memory: "64Mi" limits: cpu: "100m" # cpu·memory 둘 다 requests==limits memory: "64Mi" --- apiVersion: v1 kind: Pod metadata: name: b-pod # Burstable 노리기 spec: containers: - name: app image: nginx:1.27 resources: requests: memory: "64Mi" # request 만, limit 은 비움 → Guaranteed 조건 미달 --- apiVersion: v1 kind: Pod metadata: name: e-pod # BestEffort 노리기 spec: containers: - name: app image: nginx:1.27 # resources 아예 없음
terminal $ kubectl apply -f qos.yaml pod/g-pod created pod/b-pod created pod/e-pod created $ kubectl get pod g-pod -o jsonpath='{.status.qosClass}{"\n"}' Guaranteed $ kubectl get pod b-pod -o jsonpath='{.status.qosClass}{"\n"}' Burstable $ kubectl get pod e-pod -o jsonpath='{.status.qosClass}{"\n"}' BestEffort

한 번에 셋 다 보기 — 판정이 requests/limits 에서 나온 게 눈에 보인다:

terminal $ kubectl get pods -o custom-columns='NAME:.metadata.name,QOS:.status.qosClass' NAME QOS b-pod Burstable e-pod BestEffort g-pod Guaranteed
직접 확인해 볼 것 b-pod 에서 memory 를 requests==limits 로 맞추되 cpu 를 안 넣으면 여전히 Guaranteed 가 아니다 — "cpu·memory 둘 다, 모든 컨테이너"라는 조건을 몸으로 확인하는 순간이다. 고쳐서 다시 apply 하고 jsonpath 로 등급이 바뀌는지 보라. (Pod 는 immutable 필드가 있어 resources 변경엔 delete 후 재생성이 편하다 — 0002 의 ephemeral 성질 그대로.)

정리: kubectl delete ns lab && kubectl config set-context --current --namespace=default.

심화 한 줄: eviction 순서를 QoS 위에서 더 세밀하게 조절하고 싶으면 PriorityClass(Pod 우선순위)와 preemption(높은 우선순위 Pod 가 낮은 걸 밀어내고 자리 차지)이 있다. KCNA 범위 밖 — 여기선 "그런 게 있다"까지만. 시험엔 3 클래스·판정 규칙·eviction 우선순위 세 가지면 충분하다.

5. 회상 연습

Q1. 한 Pod 의 유일한 컨테이너가 memory: requests==limits==128Mi 인데 cpu 는 아무것도 설정 안 함이다. QoS 클래스는?

Guaranteed 는 모든 컨테이너가 cpu·memory 둘 다 requests==limits 여야 한다. cpu 가 비면 그 and 조건이 깨져 Burstable 로 떨어진다. 뭐라도 하나 설정했으니 BestEffort 도 아니다.

Q2. 노드가 memory pressure 에 빠졌다. kubelet 이 가장 먼저 쫓아내는(evict) Pod 는?

순서: BestEffort → (request 초과 사용 중인) Burstable → Guaranteed. "예산 안 낸 Pod 가 먼저 나간다." Guaranteed 는 마지막, 노드가 정말 못 버틸 때만이다.

Q3. (레슨 0006 복습) Guaranteed Pod 의 컨테이너가 자기 memory limit 을 넘겼다. 무슨 일이 일어나나?

간격 반복: OOMKill 은 커널이, limit 넘긴 그 컨테이너 하나를 죽이는 것(0006 의 137)이라 QoS 등급과 무관하다. QoS 순서는 노드 전체 memory pressure 때의 eviction에만 적용된다. 둘은 다른 층이다.
판정: Q1·Q2·Q3 를 맞히면 QoS 3 클래스의 판정 규칙, eviction 우선순위, 그리고 OOMKill vs eviction 의 층 구분이 ✅ — 0006 자원 관리의 마지막 조각이 채워졌다. 이걸로 "Pending·OOMKilled·Evicted" 세 실패 모드를 각각 다른 심판에 정확히 귀속시킬 수 있다.

초보자 질문 (막히면 여기부터)

QoS 는 "내가 넣는 값"이 아니라 "계산되어 나오는 라벨"이라 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

QoS 클래스를 내가 직접 지정할 수 있나요?

없다. qosClass 는 읽기 전용 — 내가 넣은 requests/limits 조합을 보고 쿠버네티스가 계산해서 .status.qosClass 에 채운다. 등급을 바꾸고 싶으면 requests/limits 를 바꾸는 수밖에 없다. 그래서 이 레슨은 "QoS 필드를 배우는" 게 아니라 "0006 의 두 숫자가 등급으로 어떻게 번역되나"를 배우는 것이다.

Guaranteed 가 되려면 왜 cpu 도 requests==limits 여야 하나요?

Guaranteed 의 정의가 "모든 컨테이너의 cpu·memory 둘 다 requests==limits"이기 때문이다. memory 만 딱 맞고 cpu 를 비우면 그 컨테이너는 cpu 를 "얼마든 burst" 할 수 있어 Burstable 성격이 남는다. 그래서 쿠버네티스는 이 경우를 Guaranteed 로 인정하지 않는다. 실습의 b-pod 로 직접 확인할 수 있다.

limits 만 넣고 requests 를 안 넣으면 QoS 가 뭐가 되나요?

0006 에서 봤듯 limits 만 넣으면 쿠버네티스가 requests 를 같은 값으로 자동 채운다. 그래서 cpu·memory 둘 다 limits 를 넣었다면 requests==limits 가 되어 Guaranteed 가 된다. 한쪽만 넣었다면 나머지는 여전히 비어 Burstable 이다. "자동 채움"을 기억하면 예측이 쉽다.

OOMKill 이랑 eviction, 결국 둘 다 죽는 거 아닌가요?

결과는 비슷해 보여도 원인·주체·범위가 다르다. OOMKill 은 커널이 limit 넘긴 컨테이너 하나를 죽이는 것(개별, Exit 137). eviction 은 kubelet이 노드 전체 메모리 부족 때 QoS 순서로 Pod 를 내보내는 것(노드 레벨, status=Evicted). "limit 초과냐, 노드 부족이냐"로 원인을 가르면 안 헷갈린다.

evict 된 Pod 는 어떻게 되나요? 사라지고 끝인가요?

Pod 자체는 Evicted 상태로 남고 컨테이너는 종료된다. 그 Pod 를 Deployment/ReplicaSet 이 관리한다면(0002) 컨트롤러가 대체 Pod 를 새로 만들어 다른 노드에 다시 스케줄한다 — 이때 0012 의 taint/affinity·requests 를 또 통과해야 한다. 반면 단독 Pod(관리자 없음)면 그냥 Evicted 로 남는다. Pod 는 ephemeral 하다는 0002 의 명제 그대로다.

Burstable 중에서도 "request 초과 사용"이 먼저 쫓겨난다는 게 무슨 뜻이에요?

kubelet 은 Burstable Pod 들 사이에선 "예약한 memory request 보다 얼마나 더 쓰고 있나"를 본다. request 이하로 얌전히 쓰는 Pod 보다, request 를 넘겨 초과 사용 중인 Pod 를 먼저 내보낸다 — "예약한 만큼만 쓴 Pod 는 덜 미안하다"는 직관이다. KCNA 는 "BestEffort → Burstable → Guaranteed" 큰 순서면 충분하고, 이 세부는 감으로만 잡아도 된다.

kubectl get pod ... -o jsonpath='{.status.qosClass}' 가 빈 값이면요?

Pod 가 아직 스케줄/생성 초기라 status 가 덜 채워졌을 수 있다. 잠깐 뒤 다시 조회하거나 kubectl describe pod <name> 의 QoS Class: 줄로 확인하면 된다. describe 는 0004·0006 에서처럼 언제나 든든한 진단 창구다.

실무에선 뭘 목표로 해야 하나요? 다 Guaranteed 로?

중요한(핵심 경로) 워크로드는 Guaranteed 로 두어 eviction 에서 마지막까지 살아남게 하는 게 안전하다. 다만 requests==limits 는 0006 의 예약 장부를 그만큼 크게 잡는 것이라 스케줄 여유를 먹는다. 배치성·비핵심 작업은 Burstable/BestEffort 로 두어 자원을 탄력적으로 쓴다. "중요할수록 Guaranteed, 유연할수록 Burstable" 정도의 감이면 KCNA 엔 충분.

PriorityClass 는 QoS 랑 뭐가 다른가요?

둘 다 "누가 살아남나"를 다루지만 축이 다르다. QoS 는 requests/limits 에서 자동 계산되어 주로 eviction 순서에 쓰인다. PriorityClass 는 내가 명시하는 우선순위로, 스케줄 자리가 없을 때 낮은 Pod 를 밀어내는 preemption에 쓰인다. KCNA 는 QoS 3 클래스와 eviction 순서까지가 범위 — PriorityClass 는 "존재만" 알면 된다(CKA 심화).

eviction 은 memory 부족일 때만 일어나나요?

이 레슨의 초점은 memory pressure(가장 자주, 그리고 QoS 순서가 뚜렷하게 적용되는 경우)다. 실제로 kubelet 은 디스크(ephemeral-storage) 부족 같은 다른 압박에도 Pod 를 evict 할 수 있지만, KCNA 는 "노드 자원 부족 시 QoS 낮은 순으로"라는 핵심 골격이면 충분하다. 세부 임계값·신호 종류는 CKA 몫이다.
