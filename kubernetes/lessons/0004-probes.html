Kubernetes Mastery · Lesson 0004 Level 1→3 · 실습

readiness 는 과녁에서 빼고, liveness 는 재시작한다

진단에서 틀린 그 문제. 두 probe 의 차이를 EndpointSlice 와 RESTARTS 숫자로 직접 확인한다.

이번 레슨의 win: readiness 실패와 liveness 실패의 결과가 완전히 다른 곳에 나타나는 것을 관찰하고, 어떤 상황에 어떤 probe 를 써야 하는지 판단할 수 있다. 그리고 처음으로 YAML 매니페스트를 직접 적용한다.

1. 두 probe, 두 명의 감독관

kubelet 은 컨테이너마다 두 가지 질문을 반복해서 던질 수 있다:

질문실패하면결과가 보이는 곳
readiness "지금 트래픽 받을 준비 됐어?" Service 과녁에서 제외 (재시작 없음) EndpointSlice, READY 0/1
liveness "아직 살아는 있어?" 컨테이너 재시작 RESTARTS 카운트, Events
아하 레슨 0003 에서 "EndpointSlice 에는 Ready 상태 Pod 만 실린다"고 했다 — 그 Ready 를 결정하는 게 바로 readiness probe 다. 두 레슨이 여기서 만난다: readiness 는 Service 층의 문제(과녁 목록), liveness 는 컨테이너 층의 문제(생사). 층이 다르니 결과도 다르다.

2. 실습 — 같은 실패, 다른 운명

kind 클러스터(mastery)와 레슨 0003 의 web Service 를 그대로 쓴다. 이번엔 kubectl create 가 아니라 YAML 파일을 직접 적용한다 — CKAD 로 가는 첫 걸음.

추적 1 — readiness: 준비 안 된 Pod 는 과녁에 안 실린다. 아래를 ready-demo.yaml 로 저장하고 적용하라:

ready-demo.yaml apiVersion: v1 kind: Pod metadata: name: ready-demo labels: app: web # ← web Service 의 selector 에 걸린다! spec: containers: - name: nginx image: nginx:1.27 readinessProbe: exec: command: ["cat", "/tmp/ready"] # 파일이 있어야 Ready periodSeconds: 2
terminal $ kubectl apply -f ready-demo.yaml && kubectl get pod ready-demo -w NAME READY STATUS RESTARTS ready-demo 0/1 Running 0 ← Running 인데 Ready 가 아니다! (Ctrl-C) $ kubectl get endpointslices -l kubernetes.io/service-name=web ENDPOINTS: 기존 web Pod 3개의 IP 뿐 — ready-demo 는 라벨이 맞아도 목록에 없다

이제 준비됐다고 알려주자:

terminal $ kubectl exec ready-demo -- touch /tmp/ready $ kubectl get pod ready-demo && kubectl get endpointslices -l kubernetes.io/service-name=web ready-demo 1/1 Running 0 ← Ready! ENDPOINTS 에 ready-demo 의 IP 가 추가되었다 (이제 과녁 4개)

거꾸로 kubectl exec ready-demo -- rm /tmp/ready 하면 몇 초 안에 다시 과녁에서 빠진다. RESTARTS 는 끝까지 0 — readiness 는 절대 재시작시키지 않는다.

추적 2 — liveness: 죽었다고 판정되면 재시작된다. live-demo.yaml:

live-demo.yaml apiVersion: v1 kind: Pod metadata: name: live-demo spec: containers: - name: nginx image: nginx:1.27 livenessProbe: exec: command: ["cat", "/tmp/alive"] initialDelaySeconds: 5 periodSeconds: 2
terminal $ kubectl apply -f live-demo.yaml && kubectl get pod live-demo -w live-demo 1/1 Running 0 live-demo 1/1 Running 1 ← /tmp/alive 가 없으니 곧 재시작! live-demo 1/1 Running 2 ← 그리고 또... (Ctrl-C) $ kubectl describe pod live-demo | grep -A3 Events | tail -5 Warning Unhealthy Liveness probe failed: cat: /tmp/alive: No such file or directory Normal Killing Container nginx failed liveness probe, will be restarted

kubectl exec live-demo -- touch /tmp/alive 로 파일을 만들어 주면 재시작이 멈춘다 (재시작 직후 타이밍이면 한 번 더 걸릴 수 있다 — 그것도 관찰거리다).

실무 함정: liveness 에 DB 연결 같은 외부 의존성을 넣으면, DB 장애 때 멀쩡한 앱 컨테이너들이 전부 재시작 폭풍에 빠진다. liveness 는 "프로세스가 회복 불능인가"만, 외부 준비 상태는 readiness 로. 공식 문서의 주의사항 참고.

정리: kubectl delete pod ready-demo live-demo

3. 회상 연습

Q1. 배포 직후 앱이 워밍업(캐시 로딩)에 30초 걸린다. 그동안 트래픽만 안 받으면 된다. 어떤 probe 인가?

워밍업은 정상 과정이지 죽은 게 아니다. liveness 로 걸면 워밍업 중 재시작 → 다시 워밍업 → 무한 루프. 진단 Q5 의 함정이 정확히 이것이었다.

Q2. 추적 1 에서 ready-demo 가 EndpointSlice 에 없었던 이유는?

EndpointSlice 는 selector 일치 그리고 Ready, 두 조건이다. 그리고 소유와 무관하게 라벨만 맞으면 어떤 Pod 든 과녁이 된다는 것도 확인했다 — Service 는 소유 체인 밖(레슨 0003).

Q3. (레슨 0002–0003 종합 복습) 배포한 앱에 연결이 안 된다. kubectl describe svc web 의 Endpoints 가 비어 있다. 가능한 원인 둘은?

빈 Endpoints 의 2대 원인 = "과녁에 안 걸림(라벨)" 또는 "걸렸지만 Ready 아님(probe)". 레슨 0003 warning + 이번 레슨이 합쳐진, 실기 시험 그대로의 디버깅 체크리스트다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

readiness 랑 liveness, 둘 다 꼭 넣어야 하나요?

아니다 — 필요한 것만 넣으면 된다. 워밍업/외부 의존성이 있어 "준비 상태"가 흔들리는 앱이면 readiness, 데드락처럼 프로세스가 스스로 회복 못 하는 경우가 있으면 liveness. 둘의 목적이 아예 다르니(과녁 제외 vs 재시작) "둘 다 없어도 되는가"부터 판단하는 게 맞다.

probe 를 아예 안 넣으면 어떻게 되나요?

기본값은 "항상 통과"다 — 컨테이너 프로세스가 살아만 있으면 Ready 로 치고, kubelet 은 재시작을 안 시킨다. 그래서 워밍업 중인 앱한테도 트래픽이 바로 꽂히고(readiness 없음), 데드락에 걸려 멈춘 앱도 영영 안 고쳐진다(liveness 없음). "괜찮아 보이는데 왜 오류가?"의 흔한 뿌리다.

추적 1 에서 STATUS 는 Running 인데 READY 가 0/1 이에요. 모순 아닌가요?

아니다 — 다른 층의 얘기다. STATUS: Running 은 "컨테이너 프로세스가 떠 있다", READY 0/1 은 "readiness probe 를 아직 통과 못 해 트래픽 받을 준비가 안 됐다"다. 프로세스는 살아 있지만 준비는 안 된 상태 — 그래서 EndpointSlice 과녁에서 빠져 있는 것이다.

readiness 가 실패하는 동안 Pod 는 그냥 방치되나요?

재시작은 안 되지만 방치는 아니다 — kubelet 이 periodSeconds 간격으로 계속 다시 물어본다. 준비되면(추적 1 에서 touch /tmp/ready) 몇 초 안에 자동으로 1/1 이 되고 과녁에 다시 실린다. readiness 는 "지금 상태"를 계속 갱신하는 스위치이지 일회성 판정이 아니다.

RESTARTS 숫자가 계속 오르는데 정상인가요?

정상이 아니다 — liveness probe 가 계속 실패해 컨테이너를 반복 재시작한다는 신호다(추적 2 가 정확히 이 그림). kubectl describe pod 의 Events 에서 Unhealthy/Killing 을 보면 원인이 나온다. 참고로 readiness 실패로는 RESTARTS 가 절대 안 오른다 — 숫자가 오르면 범인은 항상 liveness(또는 크래시) 쪽이다.

왜 liveness 에 DB 체크를 넣으면 안 되나요?

liveness 실패 = 재시작인데, DB 가 죽으면 멀쩡한 앱 컨테이너 전부가 동시에 재시작 폭풍에 빠지기 때문이다(실습 함정 박스). 재시작해도 DB 는 여전히 죽어 있으니 아무것도 안 고쳐지고 상황만 악화된다. 외부 의존성은 readiness(과녁에서만 빠짐), liveness 는 "이 프로세스 자체가 회복 불능인가"만.

그럼 DB 연결 확인은 어느 probe 에 넣어요?

readiness 다. DB 가 끊기면 그 Pod 만 조용히 과녁에서 빠지고(트래픽 안 받음), DB 가 돌아오면 자동으로 다시 실린다 — 재시작 없이. 앞 질문과 짝으로 외우면 좋다: 외부 의존성 = readiness, 프로세스 생사 = liveness.

exec probe 말고 다른 방식도 있나요?

있다 — httpGet(엔드포인트에 GET, 2xx/3xx 면 성공), tcpSocket(포트가 열려 있으면 성공), 그리고 오늘 쓴 exec(명령의 종료코드 0 이면 성공). 웹 앱은 보통 httpGet /healthz 를 쓴다. 실습에서 파일 존재 여부로 실패를 껐다 켰다 하려고 exec cat 을 골랐을 뿐이다.

startupProbe 는 언제 쓰나요?

부팅이 느린 앱(레거시 JVM 등)에서, 뜨는 동안 liveness 가 성급히 죽이는 걸 막을 때 쓴다. startupProbe 가 통과하기 전까지는 liveness/readiness 를 아예 잠가 둔다 — "일단 다 뜰 때까지 기다려" 담당이다. initialDelaySeconds 를 크게 잡는 것보다 깔끔하다. 세부는 오늘 안 다뤘고 CKAD 재료다.

이번엔 왜 kubectl create 대신 YAML 파일을 쓰나요?

probe 처럼 중첩이 깊은 설정은 명령어 플래그로 표현하기 힘들고, 무엇보다 실무·시험에서 리소스는 대부분 매니페스트(YAML)로 관리하기 때문이다. 파일로 두면 버전 관리·재적용·리뷰가 된다. CKAD 는 사실상 YAML 을 빠르게 쓰고 apply 하는 시험이라, 오늘이 그 첫 걸음이다.

periodSeconds, initialDelaySeconds 는 무슨 차이예요?

initialDelaySeconds 는 "컨테이너 시작 후 첫 검사까지 기다리는 시간"(추적 2 에서 5초), periodSeconds 는 "그 다음부터 몇 초 간격으로 반복하나"(2초)다. 여기에 failureThreshold(몇 번 연속 실패해야 실패로 판정)까지 조합해 민감도를 조절한다. 너무 촘촘하면 예민하고, 너무 느슨하면 장애 감지가 늦는다.
