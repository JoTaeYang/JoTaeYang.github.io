Kubernetes Mastery · Lesson 0027 Fundamentals 심화

API 그룹 · 버전 · deprecation

지금까지 쓴 모든 매니페스트는 apiVersion: 으로 시작했다. 그 한 줄이 정확히 무엇을 가리키는지 — 어느 API 그룹의 몇 번째 버전인지 — 를 읽을 줄 알면, 쿠버네티스 API가 하나의 덩어리가 아니라 그룹들의 집합임이 보인다. 이번엔 kubectl api-resources·api-versions·explain 으로 내 클러스터의 API 지도를 직접 걸어 내려간다.

이번 레슨의 win: apiVersion: apps/v1 을 보면 "apps 그룹의 stable v1 버전"이라고 즉시 해석하고, v1alpha1·v1beta1·v1 의 성숙도 차이를 안다. 0021의 CRD가 새 그룹을 추가하는 것, 0010의 Ingress가 그룹을 옮긴 deprecation 사례, 0008의 RBAC 그룹이 하나로 꿰인다. KCNA 리터러시의 마지막 조각 — 이제 매니페스트 첫 줄을 눈감고도 읽는다.

0. 한눈에 — 세 줄

딱 이 세 줄

모든 오브젝트 = apiVersion + kind. apiVersion 은 group/version 형식. Core 그룹은 이름이 빈 문자열이라서 그냥 v1(Pod·Service·ConfigMap). 이름 있는 그룹은 apps/v1·batch/v1처럼 앞에 그룹명이 붙는다.

왜 그룹인가 = API를 모듈화하고 독립적으로 버저닝하기 위해. 그래서 CRD(0021)가 새 그룹을 통째로 추가할 수 있다 — 쿠버네티스 API는 하나의 거대한 덩어리가 아니라 그룹의 집합.

버전은 성숙 단계를 뜻한다: v1alpha1(실험, 기본 off) → v1beta1(검증 중, 바뀔 수 있음) → v1(stable, 하위호환 보장). stable API를 제거할 땐 사전 공지(deprecation 정책)를 거친다 — 0010 Ingress의 그룹 이동이 그 사례.

1. 모든 오브젝트는 apiVersion + kind 로 시작한다

지금까지 손으로 쓴 매니페스트를 떠올려 보라. Pod든 Deployment든 Ingress든, 맨 위 두 줄은 언제나 같은 골격이었다:

manifest · 공통 골격 apiVersion: apps/v1 # <group>/<version> — "어느 그룹의 몇 버전 API로 이걸 해석하라" kind: Deployment # 그 그룹 안의 리소스 종류(명사) metadata: name: web spec: ...

apiVersion 은 API server에게 "이 오브젝트를 어느 그룹·버전의 스키마로 읽어라"를 말하는 좌표다. kind 는 그 그룹 안의 명사. 이 둘이 있어야 API server가 저장·검증·라우팅을 할 수 있다.

아하 — Core 그룹은 왜 이름이 없나 Pod·Service·ConfigMap·Namespace 의 apiVersion 은 그냥 v1 — 그룹명이 안 보인다. 이건 이들이 Core(레거시) 그룹에 속하고, 그 그룹의 이름이 빈 문자열("")이기 때문이다. 쿠버네티스 초창기에 그룹 개념이 생기기 전부터 있던 원조 리소스들이라, 하위호환을 위해 "그룹명 없이 v1" 형태를 유지한다. 그래서 apiVersion: v1 을 보면 "이름 없는 Core 그룹의 v1"이라고 읽으면 된다.

이름 있는 그룹은 <group>/<version> 형태로 그룹명이 앞에 붙는다. 지금까지 이 트랙에서 쓴 kind 들이 어느 그룹에 속하는지 정리하면 — 매니페스트 첫 줄이 갑자기 읽힌다:

apiVersion그룹대표 kind이 트랙 연결
v1Core(빈 문자열 "")Pod · Service · ConfigMap · Secret · Namespace · Node · PersistentVolumeClaim0002 · 0005 · 0011
apps/v1appsDeployment · StatefulSet · DaemonSet · ReplicaSet0002 · 0009
batch/v1batchJob · CronJob0009
networking.k8s.io/v1networking.k8s.ioIngress · NetworkPolicy0010
rbac.authorization.k8s.io/v1rbac.authorization.k8s.ioRole · RoleBinding · ClusterRole · ClusterRoleBinding0008
apiextensions.k8s.io/v1apiextensions.k8s.ioCustomResourceDefinition0021
왜 Deployment는 apps/v1 인데 Pod는 v1 일까? Pod는 원조 Core 리소스라 그룹명이 없고, Deployment·StatefulSet 등은 나중에 apps 그룹으로 묶여 추가됐다. 그래서 같은 워크로드라도 apiVersion이 다르다 — 헷갈리면 kubectl explain <kind> 나 kubectl api-resources(5절)로 언제든 확인할 수 있다.

2. 왜 그룹으로 나누나 — 모듈화·독립 버저닝·확장

쿠버네티스 API를 하나의 거대한 v1 로 만들 수도 있었을 것이다. 굳이 그룹으로 쪼갠 이유는 셋이다:

이유뜻결과
모듈화기능별로 API를 묶어 관리네트워킹은 networking.k8s.io, 권한은 rbac.authorization.k8s.io — 관심사가 분리됨
독립 버저닝그룹마다 따로 버전을 올린다batch 가 v1인데 어떤 그룹은 아직 v1beta1일 수 있음 — 한쪽 변경이 다른 쪽을 안 흔든다
확장(extensibility)새 그룹을 추가할 수 있음CRD(0021)가 stable.example.com 같은 새 그룹을 통째로 API에 붙인다
아하 — 0021과 직접 연결 0021 실습에서 group: stable.example.com 으로 CRD를 apply 하자 kubectl get crontabs 가 동작했다. 그게 바로 새 API 그룹 하나가 런타임에 추가된 것이다. 즉 API 그룹 구조가 있기 때문에 쿠버네티스는 코어를 안 건드리고도 무한히 확장된다. cert-manager 는 cert-manager.io 그룹을, Prometheus Operator 는 monitoring.coreos.com 그룹을 각자 추가한다 — 전부 "그룹의 집합"이라는 이 구조 위에 얹힌다. 그래서 "쿠버네티스 API = 하나의 덩어리가 아니라 그룹들의 집합"이라는 문장이 리터러시의 핵심이다.

3. 버전 = 성숙 단계 (alpha · beta · stable)

버전 문자열의 모양이 그 API의 성숙도와 안정성 보장을 그대로 말해 준다. v1alpha1·v2beta3·v1 을 보면 아래 표로 즉시 해석할 수 있다:

단계모양기본 활성화하위호환 보장뜻
alphav1alpha1기본 off (feature gate로 켜야)없음 — 예고 없이 바뀌거나 사라질 수 있음실험적. 프로덕션 금지
betav1beta1과거엔 기본 on (지금은 정책상 off 경향)제한적 — 필드가 바뀔 수 있음, 마이그레이션 예고는 함검증 중. 신중히 사용
stablev1기본 on보장 — 오래 유지되고 제거는 사전 공지프로덕션 안전
아하 — CRD의 served/storage와 같은 뿌리다 0021 CRD 실습에서 versions: - name: v1, served: true, storage: true 를 썼다. 바로 그 버전 성숙 개념이 빌트인 API에도 그대로 적용된다. CRD가 v1beta1 → v1 로 성숙하는 것과, 빌트인 Ingress가 v1beta1 → v1 로 성숙한 것은 완전히 같은 메커니즘이다. API server가 여러 버전을 동시에 served 하면서 하나를 storage 로 고정하고 버전 간 변환을 해주는 것도 빌트인·CRD 공통. 버전 성숙 규칙은 하나 — 빌트인이냐 커스텀이냐만 다르다.
alpha 버전은 프로덕션에 쓰지 마라. v1alpha1 은 기본으로 꺼져 있고(feature gate 필요), 클러스터 업그레이드 한 번에 필드가 바뀌거나 리소스 자체가 사라질 수 있다 — 하위호환 보장이 전혀 없기 때문이다. "기능이 있다"와 "안전하게 의존해도 된다"는 다른 이야기. 버전 문자열이 그 경계를 그대로 알려 준다.

4. deprecation 정책 — stable은 오래 보장, 제거는 예고

API가 영원하진 않다. 더 나은 설계로 옮겨가면 옛 버전은 deprecated(폐기 예정) 표시가 붙고, 정해진 기간 뒤 제거된다. 핵심은 쿠버네티스가 사전 공지 정책을 지킨다는 것 — stable(v1) API는 최소 여러 릴리스에 걸쳐 유지된 뒤에야 제거된다.

아하 — 0010 Ingress가 바로 그 사례다 Ingress 는 역사적으로 extensions/v1beta1 → networking.k8s.io/v1beta1 → networking.k8s.io/v1 로 그룹과 버전을 옮겼다. 그래서 오래된 튜토리얼의 apiVersion: extensions/v1beta1 을 그대로 apply 하면 요즘 클러스터(v1.22+)에서는 거부되거나 경고가 뜬다. 0010에서 우리가 networking.k8s.io/v1 을 쓴 게 바로 "제거된 옛 그룹 대신 현재 stable 그룹을 쓴 것". 이게 deprecation 정책이 실제로 작동한 흔적이다.
kubectl apply 가 가끔 경고를 내는 이유: 클러스터가 아직 deprecated 버전을 served 하고 있지만 "이 apiVersion 은 곧 제거된다, 새 버전으로 옮겨라"라고 미리 알려주는 것이다. 예: Warning: extensions/v1beta1 Ingress is deprecated in v1.14+, unavailable in v1.22+; use networking.k8s.io/v1 Ingress. 경고가 뜨면 무시하지 말고 apiVersion 을 최신 stable로 갱신하는 게 정석 — 다음 업그레이드에서 진짜로 깨지기 전에.

5. 실습 — 내 클러스터의 API 지도를 걸어 내려간다 (kind 클러스터)

말로만 하지 말고 손으로 확인한다. mastery/lab 의 kind 클러스터에서 세 명령으로 API 그룹·버전·스키마를 직접 읽는다.

5-1. kubectl api-resources — kind 가 어느 그룹·버전에, namespaced 인지 읽기:

terminal · kind 클러스터 $ kubectl api-resources | head -8 # NAME / SHORTNAMES / APIVERSION / NAMESPACED / KIND 컬럼 NAME SHORTNAMES APIVERSION NAMESPACED KIND configmaps cm v1 true ConfigMap namespaces ns v1 false Namespace nodes no v1 false Node pods po v1 true Pod services svc v1 true Service deployments deploy apps/v1 true Deployment jobs batch/v1 true Job $ kubectl api-resources --api-group=apps # 한 그룹만 필터 NAME SHORTNAMES APIVERSION NAMESPACED KIND daemonsets ds apps/v1 true DaemonSet deployments deploy apps/v1 true Deployment replicasets rs apps/v1 true ReplicaSet statefulsets sts apps/v1 true StatefulSet
컬럼 읽는 법: APIVERSION 이 바로 매니페스트에 쓸 apiVersion 값이다 — Pod 은 v1, Deployment 는 apps/v1. NAMESPACED 는 그 리소스가 네임스페이스에 속하는지 (Pod=true, Node=false). 1절 표를 통째로 외울 필요 없이 이 명령 하나로 언제든 확인하면 된다.

5-2. kubectl api-versions — 이 클러스터에 설치된 그룹/버전 전체 목록:

terminal $ kubectl api-versions # 각 줄이 하나의 group/version admissionregistration.k8s.io/v1 apiextensions.k8s.io/v1 apps/v1 batch/v1 networking.k8s.io/v1 rbac.authorization.k8s.io/v1 v1 ... $ kubectl api-versions | grep -c / # 이름 있는 그룹의 버전 수 (v1은 슬래시 없어 제외) 25
아하 — 여기에 CRD 그룹이 나타난다 0021 실습에서 stable.example.com CRD를 apply 했다면, 이 목록에 stable.example.com/v1 한 줄이 추가로 등장했을 것이다. kubectl api-versions 는 곧 "이 클러스터의 API 표면 지도" — 빌트인 그룹과 CRD가 추가한 그룹이 구분 없이 나란히 뜬다. 그래서 이 목록만 봐도 "이 클러스터에 어떤 Operator/확장이 깔려 있나"를 읽을 수 있다.

5-3. kubectl explain — 스키마를 걸어 내려간다 (0021 CRD 스키마와 같은 뿌리):

terminal $ kubectl explain deployment # 최상위: GROUP·VERSION·KIND 를 먼저 알려줌 GROUP: apps KIND: Deployment VERSION: v1 DESCRIPTION: Deployment enables declarative updates for Pods and ReplicaSets. FIELDS: apiVersion <string> kind <string> metadata <ObjectMeta> spec <DeploymentSpec> status <DeploymentStatus> $ kubectl explain deployment.spec.template.spec # 점(.)으로 스키마를 깊이 파고듦 KIND: Deployment VERSION: apps/v1 FIELD: spec <PodSpec> DESCRIPTION: PodSpec is a description of a pod. FIELDS: containers <[]Container> -required- volumes <[]Volume> nodeName <string> ...
아하 — explain은 openAPI 스키마를 읽어주는 것 kubectl explain 이 보여주는 필드 구조는 API server가 각 그룹·버전에 대해 발행하는 OpenAPI 스키마에서 온다. 0021의 CRD가 openAPIV3Schema 로 커스텀 리소스의 필드·타입을 정의한 것과 정확히 같은 뿌리다 — 빌트인 리소스는 쿠버네티스가 그 스키마를 내장하고, CRD는 여러분이 직접 써 넣을 뿐. 그래서 CRD를 등록하면 kubectl explain crontab.spec 도 바로 동작한다. explain = "이 kind는 어떤 필드를 갖나"를 스키마에서 걸어 내려가는 도구.
버전을 명시해 explain 하기: 같은 kind가 여러 버전을 가질 때 kubectl explain ingress --api-version=networking.k8s.io/v1 처럼 버전을 못박아 스키마를 볼 수 있다. deprecated 버전과 stable 버전의 필드 차이를 비교할 때 유용하다.

Aggregation layer·conversion webhook(여러 API 버전 간 자동 변환, 외부 API server 붙이기)은 CKA 심화 — KCNA/이 트랙에선 "그런 확장 지점이 있다"는 이름만 알면 충분하다. 오늘의 핵심은 매니페스트 첫 줄을 읽고, 세 명령으로 API 지도를 걷는 것.

6. 회상 연습

Q1. apiVersion: v1 (그룹명 없이 v1) 로 표현되는 리소스는 어느 그룹에 속하는가?

그룹명 없이 v1 인 것들(Pod·Service·ConfigMap·Namespace)은 초창기부터 있던 Core 그룹 소속이고, 그 그룹의 이름은 빈 문자열이다. Deployment는 나중에 apps 그룹으로 묶여 apps/v1 이 됐다. kubectl api-resources 의 APIVERSION 컬럼으로 언제든 확인 가능.

Q2. 매니페스트에서 apiVersion: something.k8s.io/v1alpha1 을 봤다. 이 버전에 대해 옳은 설명은?

버전 문자열이 성숙도를 말한다: v1alpha1=실험(기본 off, 예고 없이 변경/제거 가능), v1beta1=검증 중(바뀔 수 있음), v1=stable(하위호환 보장, 제거는 사전 공지). alpha는 프로덕션에 의존하면 안 된다. 이건 0021 CRD의 served/storage 버전 성숙과 같은 개념이다.

Q3. (레슨 0010·0021 복습) 오래된 튜토리얼의 apiVersion: extensions/v1beta1 Ingress를 요즘 클러스터에 apply했더니 거부됐다. 왜이고, 무엇이 이를 가능케 하는 구조인가?

Ingress는 extensions/v1beta1 → networking.k8s.io/v1beta1 → networking.k8s.io/v1 로 그룹·버전을 옮겼고(0010), 옛 버전은 사전 공지 후 제거됐다(deprecation 정책). 이런 독립적 이동이 가능한 건 API가 그룹의 집합으로 모듈화돼 있고 각 그룹이 따로 버저닝되기 때문 — CRD(0021)가 새 그룹을 추가하는 것과 같은 구조 위에 있다. 해결은 apiVersion을 최신 stable로 갱신하는 것.

초보자 질문 (막히면 여기부터)

이 주제는 문자열이 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

apiVersion 하고 kind, 왜 둘 다 필요해요?

kind 만으로는 어느 그룹·버전의 스키마로 해석할지 모른다. 예를 들어 Ingress 는 옛 extensions/v1beta1 과 지금 networking.k8s.io/v1 이 필드 구조가 다르다. apiVersion 이 "이 오브젝트를 정확히 어느 버전의 규칙으로 읽어라"를 못박아 주기 때문에 둘 다 필요하다. 좌표(apiVersion) + 명사(kind) 조합이라고 보면 된다.

Core 그룹 이름이 "빈 문자열"이라는 게 무슨 뜻이에요?

말 그대로 그룹명이 없다. 그래서 apiVersion 에 <group>/<version> 대신 버전만(v1) 쓴다. RBAC의 Role 에서 Core 리소스에 권한을 줄 때 apiGroups: [""](빈 문자열)로 쓰는 것도 이 때문이다(0008). Pod·Service 같은 원조 리소스가 여기 속하고, 하위호환을 위해 이 형태를 유지한다.

같은 kind가 apiVersion만 다를 수 있나요?

있다. Ingress 가 대표적 — 시기에 따라 extensions/v1beta1, networking.k8s.io/v1beta1, networking.k8s.io/v1 세 apiVersion으로 존재했다. API server가 버전 간 변환을 해줘서, 클러스터가 여러 버전을 동시에 제공(served)하다가 옛 버전을 제거하는 식으로 옮겨간다. 그래서 "어떤 apiVersion을 써야 하나"는 클러스터 버전에 달렸고, kubectl explain <kind> 로 현재 권장 버전을 확인한다.

alpha / beta / stable, 실무에서 어떻게 판단해요?

stable(v1): 안심하고 프로덕션에 쓴다. beta(v1beta1): 기능이 거의 완성됐지만 필드가 바뀔 수 있으니 신중히, 업그레이드 노트를 챙긴다. alpha(v1alpha1): 기본 off이고 예고 없이 사라질 수 있으니 테스트/실험에만. "기능이 존재한다"와 "안전하게 의존해도 된다"는 다르다 — 버전 문자열이 그 경계다.

왜 Deployment는 apps/v1 인데 Pod는 v1 이에요? 둘 다 워크로드잖아요?

Pod는 쿠버네티스 초창기 Core 리소스라 그룹명이 없다(v1). Deployment·StatefulSet·DaemonSet·ReplicaSet은 나중에 apps 그룹으로 묶여 추가됐다. 같은 "워크로드"라도 API에 들어온 시기와 그룹이 달라 apiVersion이 다르다. 외우기보다 kubectl api-resources 의 APIVERSION 컬럼으로 확인하는 습관이 낫다.

kubectl apply 할 때 뜨는 "deprecated" 경고, 무시해도 되나요?

당장은 동작하지만 무시하면 안 된다. 경고는 "이 apiVersion은 곧 제거되니 새 버전으로 옮겨라"는 사전 공지다. 방치하면 다음 클러스터 업그레이드에서 진짜로 apply가 실패한다. 경고에 대개 대체 버전이 적혀 있으니(예: "use networking.k8s.io/v1"), 그걸로 apiVersion 을 갱신하고 다시 apply하면 된다.

kubectl api-resources 와 kubectl api-versions 는 뭐가 달라요?

api-resources 는 리소스(kind) 목록을 준다 — 각 kind가 어느 apiVersion에, namespaced인지, shortname은 뭔지. api-versions 는 그룹/버전 목록을 준다 — 이 클러스터에 어떤 group/version이 설치돼 있나. 전자는 "무엇을 만들 수 있나(명사)", 후자는 "어떤 API 표면이 있나(좌표)"라고 보면 된다. 둘 다 5절 실습에서 직접 쳐봤다.

kubectl explain 은 어디서 그 정보를 가져와요?

API server가 발행하는 OpenAPI 스키마에서 가져온다. 각 그룹·버전마다 "이 kind는 어떤 필드를 갖고 타입이 뭔지"가 스키마로 정의돼 있고, explain 은 그걸 사람이 읽게 풀어준다. 0021의 CRD가 openAPIV3Schema 로 커스텀 리소스 스키마를 써 넣은 것과 같은 뿌리 — 그래서 CRD를 등록하면 kubectl explain 이 커스텀 리소스에도 바로 동작한다.

CRD(0021)가 "새 그룹을 추가한다"는 게 이 레슨과 어떻게 이어져요?

직접 이어진다. CRD의 spec.group 에 stable.example.com 을 적으면, 그 순간 kubectl api-versions 목록에 stable.example.com/v1 이 새 그룹으로 등장한다. 즉 API가 그룹의 집합으로 설계돼 있어서, 코어를 안 건드리고 그룹을 무한히 붙일 수 있다. cert-manager는 cert-manager.io, Prometheus Operator는 monitoring.coreos.com 그룹을 각자 추가한다 — 이 레슨의 "그룹 구조"가 0021의 확장성이 가능한 이유다.

aggregation layer, conversion webhook은 꼭 알아야 하나요?

KCNA/이 트랙 범위에선 이름만 알면 된다. conversion webhook = 같은 kind의 여러 버전(v1beta1↔v1) 사이를 자동 변환해 주는 확장 지점. aggregation layer = 쿠버네티스 API server 뒤에 외부 API server를 붙여 새 API를 제공하는 방법(metrics-server가 이걸 쓴다). 둘 다 API를 확장하는 심화 메커니즘으로, 자세한 건 CKA 몫이다. "그런 확장 지점이 있다"는 지도만 챙기면 충분.
