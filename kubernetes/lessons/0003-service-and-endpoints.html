Kubernetes Mastery · Lesson 0003 Level 1 · 실습

Service 는 Pod 를 어떻게 찾아내나

Pod 는 계속 죽고 태어난다. 그런데 주소는 왜 안 바뀌는가 — label selector 와 EndpointSlice 의 이야기.

이번 레슨의 win: Service 가 label 로 Pod 집합을 추적하는 과정을 EndpointSlice 에서 직접 관찰하고, "Pod 가 바뀌어도 주소가 안정적인 이유"를 체인으로 설명할 수 있다.

1. 문제: 움직이는 과녁

레슨 0002 실습에서 Pod 를 지울 때마다 새 이름, 새 IP 로 태어나는 걸 봤다. 클라이언트가 Pod IP 를 직접 알고 있다면 매번 끊긴다. 그래서 쿠버네티스는 중간에 한 층을 더 둔다:

클라이언트 Pod"web 에 연결해줘" → Service web고정 ClusterIP + DNS 이름 → selector: app=web → EndpointSlice지금 살아있는 Pod IP 목록

Service 는 Pod 를 소유하지 않는다 — 레슨 0002 의 소유 체인과 완전히 별개다. 그저 label selector 로 "지금 app=web 라벨을 단 Ready 상태 Pod 전부"를 계속 조회할 뿐이다. 그 결과 목록이 EndpointSlice 라는 리소스에 실시간으로 기록된다.

아하 여기서도 제어 루프다. EndpointSlice 컨트롤러가 "selector 에 걸리는 Pod 목록"과 "EndpointSlice 내용"의 차이를 끊임없이 좁힌다. Deployment 가 Pod 를 갈아치워도 루프가 목록을 갱신하니, 클라이언트가 보는 주소(ClusterIP/DNS)는 흔들리지 않는다.

2. 실습 — 과녁 추적을 눈으로 본다

레슨 0002 의 kind 클러스터(mastery)와 web Deployment 를 그대로 쓴다. 지웠다면: kind create cluster --name mastery && kubectl create deployment web --image=nginx:1.27 --replicas=3

추적 1 — Service 를 붙이고 세 조각 확인:

terminal $ kubectl expose deployment web --port=80 $ kubectl get svc web -o wide && kubectl get endpointslices -l kubernetes.io/service-name=web NAME TYPE CLUSTER-IP PORT(S) SELECTOR web ClusterIP 10.96.143.7 80/TCP app=web NAME ADDRESSTYPE PORTS ENDPOINTS web-x8k2p IPv4 80 10.244.0.5,10.244.0.6,10.244.0.7

ENDPOINTS 의 세 IP 를 Pod IP 와 대조해 보라: kubectl get pod -l app=web -o wide — 정확히 일치한다.

추적 2 — 다른 Pod 에서 이름으로 접속:

terminal $ kubectl run client --rm -it --image=busybox:1.36 --restart=Never -- wget -qO- http://web <title>Welcome to nginx!</title> ...

Pod IP 를 하나도 몰랐는데 연결됐다. 클러스터 DNS 가 web → web.default.svc.cluster.local → ClusterIP 로 풀어준 것.

추적 3 — Pod 를 갈아치워도 목록이 따라온다:

terminal $ kubectl delete pod -l app=web --wait=false && kubectl get endpointslices -l kubernetes.io/service-name=web -w ... 옛 IP 들이 빠지고, ReplicaSet 이 만든 새 Pod 의 IP 들로 갱신된다 (Ctrl-C 로 종료)

ClusterIP(kubectl get svc web)는 그대로다. 주소는 고정, 과녁 목록만 갱신 — 이게 Service 의 전부다.

selector 를 오타 내면(예: app=Web) Service 는 만들어지지만 EndpointSlice 가 빈다 — 연결이 안 될 때 "Service 는 있는데 endpoints 가 비어 있나?"는 실기 시험 단골 디버깅 절차다. kubectl describe svc web 의 Endpoints 줄을 확인하는 습관을 들이자.

3. 회상 연습

Q1. Service 와 Pod 를 잇는 메커니즘은?

Service 는 소유 체인 밖에 있다. 오직 라벨로만 느슨하게 연결되며, 그래서 Deployment 가 아닌 어떤 Pod 든(라벨만 맞으면) 과녁이 될 수 있다. 공식 문서: Service

Q2. 추적 3 에서 Pod 삭제 후, 클라이언트 연결이 유지되는 이유는?

역할 분담: Pod 를 되살리는 건 ReplicaSet(레슨 0002), 과녁 목록을 갱신하는 건 EndpointSlice 컨트롤러, 고정 주소를 제공하는 건 Service. 세 루프가 각자 돈다.

Q3. (레슨 0002 복습) kubectl rollout undo 가 롤백하는 방식은?

간격 반복: 레슨 0002 의 warning 박스. 옛 ReplicaSet 이 0 으로 남아 있는 이유가 바로 이것이다. 템플릿은 불변 — 고치지 않고 교체/재사용한다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

ClusterIP 는 왜 안 바뀌나요? Pod 는 계속 바뀌는데요.

ClusterIP 는 Service 를 만들 때 한 번 할당되고, Service 를 지우기 전까지 고정이다. Pod 는 죽고 태어나며 IP 가 바뀌지만, 그건 EndpointSlice 안의 과녁 목록일 뿐 클라이언트가 보는 주소가 아니다. "주소는 고정, 과녁 목록만 갱신" — 이 한 줄이 Service 의 전부다.

selector 에 오타를 내면 어떻게 되나요?

Service 는 정상적으로 만들어진다 — 다만 selector 에 걸리는 Pod 가 없어서 EndpointSlice 가 빈다. 연결하면 아무 곳에도 안 닿아 timeout 이 난다. kubectl describe svc web 의 Endpoints 줄이 비어 있으면 selector 를 의심하라 — 실기 시험 단골 디버깅이다.

http://web 이름은 누가 풀어주나요?

클러스터 안에 상주하는 CoreDNS 다. Service 를 만들면 자동으로 web.default.svc.cluster.local DNS 레코드가 생기고, Pod 안에서 web 만 쳐도 same-namespace 규칙으로 그 이름까지 확장돼 ClusterIP 로 풀린다. IP 를 하나도 몰라도 이름만으로 닿는 이유다.

EndpointSlice 랑 Endpoints 는 뭐가 달라요?

둘 다 "지금 살아있는 Pod IP 목록"을 담지만, 옛 Endpoints 는 한 리소스에 전부 몰아넣어 Pod 가 수천 개면 무거웠다. EndpointSlice 는 이를 여러 조각(기본 100개씩)으로 나눠 확장성을 얻은 신형이다. 요즘 컨트롤러는 EndpointSlice 를 쓰고, Endpoints 는 호환용으로만 남아 있다.

Service 도 Pod 처럼 죽나요?

아니다. Service 는 노드 위에서 도는 프로세스가 아니라 API server 에 등록된 가상의 라우팅 규칙이다. 실제 트래픽은 각 노드의 kube-proxy(또는 그 후속 데이터플레인)가 iptables/IPVS 규칙으로 ClusterIP 를 Pod IP 로 넘겨준다. 그래서 특정 Pod 가 죽어도 Service 자체는 멀쩡하다.

kubectl expose 없이 YAML 로도 만들 수 있나요?

물론이다. expose 는 그냥 Service YAML 을 대신 써주는 편의 명령이다. 직접 쓰려면 kind: Service 에 spec.selector: {app: web} 와 spec.ports 를 적으면 된다. 실전 매니페스트는 대부분 YAML 이니, kubectl expose ... --dry-run=client -o yaml 로 뽑아 보며 익혀 두면 좋다.

selector 없는 Service 도 있다던데요?

있다. selector 를 비우면 EndpointSlice 를 손으로 채워야 한다 — 클러스터 밖 데이터베이스처럼 Pod 가 아닌 외부 IP 로 트래픽을 보내고 싶을 때 쓰는 고급 패턴이다. 지금은 "보통은 selector 로 자동, 예외적으로 수동도 가능"만 알아 두면 충분하다.

포트를 80 으로 열었는데 Pod 는 8080 이면요?

port(Service 가 받는 포트)와 targetPort(Pod 안 컨테이너 포트)를 따로 지정한다: --port=80 --target-port=8080. 안 적으면 targetPort 는 port 와 같다고 가정한다 — "연결은 되는데 응답이 없다"의 흔한 원인이 이 둘의 불일치다.

Service 를 다른 namespace 에서도 이름으로 부를 수 있나요?

같은 namespace 면 web, 다른 namespace 면 web.<namespace> 형태로 불러야 한다. 짧은 이름 web 은 "내 namespace 안"으로만 풀리기 때문이다. 전체 이름 web.default.svc.cluster.local 은 어디서든 통한다.

Ready 가 아닌 Pod 도 과녁에 들어가나요?

아니다. readinessProbe 가 통과해 Ready 상태인 Pod 만 EndpointSlice 에 실린다. 그래서 아직 부팅 중인 Pod 로는 트래픽이 안 간다 — 이게 무중단 배포의 핵심 장치다. "Pod 는 Running 인데 왜 트래픽이 안 와요?"의 답이 대개 readiness 미통과다.

replicas 를 0 으로 줄이면 Service 는 어떻게 되나요?

Service 와 ClusterIP 는 그대로 남지만 EndpointSlice 가 빈다(과녁이 하나도 없으니). 연결하면 닿을 곳이 없어 실패한다. 다시 replicas 를 올리면 새 Pod IP 로 목록이 자동으로 채워진다 — Service 는 손댈 필요가 없다.
