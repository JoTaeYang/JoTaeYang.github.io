Kubernetes Mastery · Lesson 0026 Fundamentals 심화

ownerReferences · GC · finalizer

0002 에서 "소유 체인"이라고 부른 것의 실제 정체 — metadata 안의 필드 두 개다. 하나(ownerReferences)는 삭제를 연쇄시키고, 다른 하나(finalizers)는 삭제를 붙잡는다.

이번 레슨의 win: Deployment 를 지우면 왜 Pod 까지 사라지는지, 반대로 namespace·PVC 가 왜 Terminating 에서 안 없어지는지를 추측이 아니라 metadata 필드로 설명한다. ownerReferences(연쇄 삭제의 링크)와 finalizers(삭제 차단 목록)를 kind 클러스터에서 직접 만들고, 만지고, 고아로 만들고, Terminating 에 일부러 멈춰 세운다.

1. ownerReferences — "소유 체인"은 이 필드였다

0002 에서 우리는 Deployment → ReplicaSet → Pod 를 "소유 체인"이라 불렀다. 그 체인은 이름 규칙(deploy-hash-random)이 아니라 각 종속 객체의 metadata.ownerReferences 라는 실제 필드에 기록돼 있다. ReplicaSet 이 Pod 를 만들 때 Pod 의 ownerReferences 에 "내가 주인"이라고 자기를 적고, Deployment 컨트롤러도 ReplicaSet 을 만들 때 똑같이 한다.

Deployment주인 (owner) ← ownerRef ← ReplicaSet주인이자 종속 ← ownerRef ← Pod종속 (dependent)

화살표가 아래에서 위로 향하는 데 주목하자. 종속 객체가 주인을 가리킨다 — Pod 가 "내 주인은 이 ReplicaSet" 이라고 자기 안에 적어 둔다. 주인은 자기 종속이 누군지 목록으로 들고 있지 않다. 그래서 삭제 시점에 "누가 이 객체를 주인으로 지목했나"를 역으로 훑어야 한다.

아하 — controllerRef ownerReferences 항목 중 controller: true 인 것이 딱 하나 있는데, 이게 실제로 관리하는 주인(controllerRef)이다. 0002 에서 "각 층은 바로 아래 층만 책임진다"고 했던 그 관리 관계가 이 한 비트로 표현된다. 나머지 controller 없는 ownerRef 는 "참조는 하지만 관리는 안 하는" 약한 소유(예: 여러 주인이 공유하는 종속). GC 는 모든 ownerRef 가 사라져야 종속을 지운다.

2. 가비지 컬렉션 — "Deployment 지우면 Pod 도" 의 진짜 이유

Deployment 를 지우면 가비지 컬렉터(GC 컨트롤러, kube-controller-manager 안)가 깨어난다. GC 는 클러스터 전체를 훑으며 "주인이 없어진 종속"을 찾아 지운다. Deployment 가 사라지면 그것을 ownerReferences 로 지목하던 ReplicaSet 이 고아가 되어 삭제되고, 그 ReplicaSet 을 지목하던 Pod 들이 다시 고아가 되어 삭제된다 — 이게 연쇄 삭제(cascading delete)다. 0002 의 "Deployment 지우면 Pod 도 사라진다"는 마법이 아니라 이 필드를 따라간 결과일 뿐이다.

그런데 "연쇄"에도 순서가 있다. kubectl delete 의 --cascade 옵션이 세 가지 정책을 고른다:

정책동작플래그
Background (기본) 주인을 먼저 즉시 삭제하고, GC 가 백그라운드에서 종속을 정리. kubectl delete 가 바로 리턴한다. --cascade=background
Foreground 주인에 foregroundDeletion finalizer 를 걸고 종속을 먼저 다 지운 뒤 주인을 지운다. 주인은 그동안 Terminating. --cascade=foreground
Orphan 주인만 지우고 종속의 ownerReferences 를 지워 고아로 남긴다. Pod 가 살아남는다. --cascade=orphan
왜 기본이 Background 인가: 사용자가 delete 하고 오래 안 기다려도 되게 하려는 것. Foreground 는 "종속이 완전히 사라진 걸 확인한 뒤에야 주인이 없어졌다고 보고 싶은" 자동화(예: 다른 리소스가 이 삭제 완료에 의존)에서 쓴다. Orphan 은 "관리자만 갈아끼우고 실행 중인 Pod 는 건드리지 말자"는 무중단 시나리오에 쓴다 — 2번째 실습에서 직접 본다.

3. 실습 A — ownerReferences 를 눈으로 보고, 고아로 만든다

환경: 0002·0006 에서 쓰던 kind 클러스터 mastery. 없으면 kind create cluster --name mastery. 실습 격리를 위해 전용 namespace 를 쓴다.
terminal $ kubectl create namespace lab $ kubectl -n lab create deployment web --image=nginx:1.27 --replicas=2 $ kubectl -n lab get deploy,rs,pod NAME READY UP-TO-DATE AVAILABLE deployment.apps/web 2/2 2 2 NAME DESIRED CURRENT READY replicaset.apps/web-6f8d9c5b7 2 2 2 NAME READY STATUS RESTARTS pod/web-6f8d9c5b7-abcde 1/1 Running 0 pod/web-6f8d9c5b7-fghij 1/1 Running 0

추적 1 — Pod 의 ownerRef 가 진짜로 있는지:

terminal $ POD=$(kubectl -n lab get pod -l app=web -o jsonpath='{.items[0].metadata.name}') $ kubectl -n lab get pod $POD -o jsonpath='{.metadata.ownerReferences}' | jq [ { "apiVersion": "apps/v1", "kind": "ReplicaSet", "name": "web-6f8d9c5b7", "uid": "...", "controller": true, "blockOwnerDeletion": true } ]

controller: true — 이 ReplicaSet 이 이 Pod 의 controllerRef다(1번 절의 그것). jq 가 없으면 뒤의 | jq 만 빼면 된다. 같은 방식으로 ReplicaSet 의 ownerRef 를 보면 Deployment 가 나온다:

terminal $ RS=$(kubectl -n lab get rs -l app=web -o jsonpath='{.items[0].metadata.name}') $ kubectl -n lab get rs $RS -o jsonpath='{.metadata.ownerReferences[0].kind}{" / "}{.metadata.ownerReferences[0].name}' Deployment / web

추적 2 — Orphan 으로 지우면 Pod 가 살아남는다:

terminal $ kubectl -n lab delete deployment web --cascade=orphan deployment.apps "web" deleted $ kubectl -n lab get deploy,rs,pod No resources found ... (deploy 없음) NAME DESIRED CURRENT READY replicaset.apps/web-6f8d9c5b7 2 2 2 # RS 가 살아있다! NAME READY STATUS RESTARTS pod/web-6f8d9c5b7-abcde 1/1 Running 0 # Pod 도 그대로 pod/web-6f8d9c5b7-fghij 1/1 Running 0

Deployment 만 사라지고 ReplicaSet·Pod 는 고아로 살아남았다. RS 의 ownerReferences 를 다시 보면 이제 비어 있다 — GC 가 주인 지목을 지웠기 때문. 비교로, 기본(Background) 삭제였다면 RS·Pod 가 전부 연쇄 삭제됐을 것이다. 남은 RS 는 이제 아무도 관리하지 않으니 직접 지우자(이건 종속이 없어 정책 무관):

terminal $ kubectl -n lab get rs $RS -o jsonpath='{.metadata.ownerReferences}'; echo " <- 비었으면 고아 확정" $ kubectl -n lab delete rs $RS # 기본 background 로 RS+Pod 연쇄 삭제
--cascade=orphan 은 0002 의 "Pod 를 지워도 되살아난다"와 정반대 상황을 만든다. 주인(관리자)이 사라졌으니 이제 Pod 가 죽어도 아무도 안 채워 준다. 실무에서 Orphan 은 "컨트롤러만 교체하고 실행 중 워크로드는 무중단 유지" 같은 고급 마이그레이션에 한정해 쓴다. 잘못 쓰면 관리 안 되는 유령 리소스가 남는다.

4. finalizer — 삭제를 "붙잡는" 필드

ownerReferences 가 삭제를 퍼뜨린다면, metadata.finalizers 는 삭제를 막는다. finalizer 는 문자열 키의 목록이다. 객체를 지우라고 하면 API 서버는 바로 지우지 않고, 대신 metadata.deletionTimestamp 에 시각을 찍는다. 그리고 이렇게 약속한다: "finalizers 목록이 완전히 빌 때까지 실제로는 안 지운다."

아하 — Terminating 은 "삭제 중"이 아니라 "삭제 대기 중" deletionTimestamp 가 찍혔지만 finalizer 가 남아 있으면 객체는 Terminating 상태로 멈춰 있다. 각 finalizer 는 담당 컨트롤러가 있어서, 자기 몫의 정리 작업(클라우드 로드밸런서 회수, PV 볼륨 분리, namespace 안 리소스 비우기)을 마친 뒤 자기 키를 목록에서 제거한다. 마지막 키가 빠지는 순간 API 서버가 객체를 진짜로 삭제한다. 그래서 Terminating 에서 안 없어지는 건 "누군가 자기 finalizer 를 아직 안 뗐다"는 뜻이다.
흔한 finalizer담당 정리 작업어디서 봤나
service.k8s.io/load-balancer-cleanupService 삭제 전 클라우드 LB·규칙 회수0010 LoadBalancer 타입
kubernetes.io/pv-protection, pvc-protection아직 쓰는 PV/PVC 를 못 지우게 보호0011 stuck PVC
kubernetes (namespace)namespace 안 모든 리소스를 다 지운 뒤에야 namespace 삭제아래 실습 B
foregroundDeletionForeground 연쇄 삭제(2번 절)에서 종속이 다 빠질 때까지 주인 붙잡음2번 표

namespace 가 특히 finalizer 로 돌아간다. kubectl delete namespace lab 을 하면 namespace 에 kubernetes finalizer 가 있어서, 안의 Deployment·Pod·ConfigMap 이 전부 삭제될 때까지 namespace 는 Terminating 에 머문다. 안의 무언가가 자기 finalizer 로 버티면 namespace 도 덩달아 영원히 Terminating 이 된다 — 이게 "namespace 가 안 지워져요"의 흔한 원인이다.

5. 실습 B — finalizer 로 삭제를 멈춰 세우고, 손으로 풀어준다

커스텀 finalizer 를 단 ConfigMap 을 만들어, delete 가 Terminating 에 멈추는 것을 관찰하고, patch 로 finalizer 를 떼어 삭제를 완료시킨다. (담당 컨트롤러가 없는 finalizer 라 자동으로는 절대 안 빠진다 — 우리가 컨트롤러 역할을 대신한다.)

terminal $ kubectl -n lab apply -f - <<'EOF' apiVersion: v1 kind: ConfigMap metadata: name: protected finalizers: - example.com/protect data: note: "try to delete me" EOF configmap/protected created

추적 1 — 삭제 명령이 리턴하지 않고 매달린다:

terminal $ kubectl -n lab delete configmap protected --wait=false configmap "protected" deleted # 이렇게 떠도 실제로는 안 지워졌다! $ kubectl -n lab get configmap protected -o jsonpath='{.metadata.deletionTimestamp}{"\n"}{.metadata.finalizers}{"\n"}' 2026-07-07T09:15:22Z # deletionTimestamp 찍힘 = 삭제 요청됨 ["example.com/protect"] # 그런데 finalizer 가 남아 붙잡고 있음

--wait=false 를 안 주면 kubectl 이 finalizer 가 빠지길 기다리며 영원히 멈춘다(Ctrl-C 로 탈출 가능). 객체는 여전히 조회된다 — deletionTimestamp 는 찍혔지만 finalizer 가 있어 실제 삭제가 보류된 Terminating 상태다. 이게 stuck namespace/PVC 와 완전히 같은 메커니즘이다.

추적 2 — finalizer 를 떼면 그 즉시 사라진다:

terminal $ kubectl -n lab patch configmap protected --type=merge -p '{"metadata":{"finalizers":[]}}' configmap/protected patched $ kubectl -n lab get configmap protected Error from server (NotFound): configmaps "protected" not found # 목록이 비자마자 삭제 완료

마지막 finalizer 를 patch 로 비우는 순간, 이미 찍혀 있던 deletionTimestamp 때문에 API 서버가 객체를 즉시 지웠다. 실무에서 이 patch 는 최후의 수단이다 — 원래는 담당 컨트롤러가 정리 작업을 마치고 스스로 떼어야 한다.

finalizer 를 손으로 강제 제거하면 그 finalizer 가 하려던 정리 작업을 건너뛴다. namespace 의 kubernetes finalizer 를 강제로 비우면 안에 리소스가 남은 채 namespace 만 사라져 유령 리소스가 될 수 있다(0011 의 stuck PVC 를 이 방식으로 억지로 풀면 볼륨이 안 회수될 수 있다). "왜 안 빠지나"를 먼저 진단하고(대개 담당 컨트롤러가 죽었거나 외부 의존이 막힘), 강제 제거는 정말 마지막에.

실습 정리: kubectl delete namespace lab — 이제 안의 리소스가 없으니 namespace 의 kubernetes finalizer 도 곧바로 빠지고 깔끔히 사라진다.

6. 회상 연습

Q1. Deployment 를 kubectl delete deploy web(플래그 없음)로 지웠을 때 실제로 일어나는 일은?

플래그가 없으면 Background(기본)다: 주인 먼저 삭제, 종속은 GC 가 뒤에서 정리. 1번 보기는 --cascade=orphan, 3번은 --cascade=foreground 의 동작이다. 공식 문서: Cascading deletion

Q2. namespace 가 Terminating 에서 몇 분째 안 없어진다. 가장 정확한 설명은?

namespace 삭제는 안의 모든 리소스가 지워질 때까지 kubernetes finalizer 로 붙잡는다. 안의 무언가(예: 5번 실습의 ConfigMap 같은)가 자기 finalizer 로 버티면 namespace 도 함께 멈춘다. 진단: 그 namespace 안에서 Terminating 인 리소스와 그 metadata.finalizers 를 찾는다. 공식 문서: Finalizers

Q3. (0002 복습) Pod 의 ownerReferences 안에서 controller: true 로 표시된 항목이 뜻하는 것은?

간격 반복: 0002 에서 ReplicaSet 컨트롤러가 Pod 개수를 수렴시켰다. 그 관리 주체가 controller: true 인 ownerRef 로 metadata 에 박혀 있다. 여러 ownerRef 중 controller 는 최대 하나. 공식 문서: Owners and Dependents
판정: Q1·Q2·Q3 를 맞히면 0002 의 "소유 체인"이 이제 추상 개념이 아니라 만질 수 있는 metadata 필드가 됐다는 뜻이다. GC 정책 3종과 finalizer 진단은 KCNA 범위를 넘어 CKA 실기의 단골(stuck namespace/PVC 풀기)이니, 위 실습은 손이 기억할 때까지.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다.

ownerReferences 는 내가 채우나요, 컨트롤러가 채우나요?

컨트롤러가 채운다. ReplicaSet 컨트롤러가 Pod 를 만들 때 Pod 의 ownerReferences 에 자기를 controller: true 로 적고, Deployment 컨트롤러도 ReplicaSet 에 똑같이 한다. 내가 YAML 에 직접 쓰는 필드가 아니다(쓸 수는 있지만 보통 안 쓴다). 3번 실습에서 jsonpath 로 본 그 값이 컨트롤러가 남긴 기록이다.

Background 랑 Foreground, 눈으로 보면 뭐가 달라요?

Background: delete 명령이 즉시 리턴하고 주인이 바로 없어진다. 종속은 잠시 뒤 사라진다(GC 가 뒤처리). Foreground: 주인이 Terminating 으로 남아 있고, 종속이 다 지워진 다음에야 주인이 없어진다. 그래서 "이 객체가 사라졌다 = 종속도 확실히 다 사라졌다"를 보장받고 싶을 때 Foreground 를 쓴다.

--cascade=orphan 으로 남긴 Pod 는 이제 누가 관리해요?

아무도 관리하지 않는다. 주인이 사라지고 ownerReferences 가 비워졌으니, 그 Pod 가 죽어도 되살릴 컨트롤러가 없다(0002 의 "죽으면 되살아난다"가 깨진다). 그래서 Orphan 은 임시·마이그레이션 용도이고, 끝나면 새 컨트롤러에 다시 입양시키거나 직접 정리해야 한다. 방치하면 유령 리소스가 된다.

finalizer 랑 ownerReferences 는 둘 다 metadata 인데 무슨 차이예요?

방향이 반대다. ownerReferences 는 삭제를 퍼뜨린다(주인 지우면 종속도 연쇄). finalizers 는 삭제를 막는다(목록이 빌 때까지 실제 삭제 보류). 하나는 "링크를 따라 지워라", 다른 하나는 "정리 끝날 때까지 기다려라". 한 객체가 둘 다 가질 수 있다.

Terminating 은 "지금 지워지는 중"이라는 뜻 아니에요?

정확히는 "삭제가 요청됐지만(=deletionTimestamp 찍힘) 아직 finalizer 때문에 못 지우는 중"이다. finalizer 가 없거나 다 빠졌으면 Terminating 은 눈 깜짝할 새 지나간다. 오래 머문다면 = 누군가의 finalizer 가 안 빠지고 있다는 신호. 그래서 stuck 진단의 첫걸음은 -o jsonpath='{.metadata.finalizers}' 로 남은 키를 보는 것.

PVC 가 Terminating 에서 안 없어져요(0011). 왜죠?

PVC 에는 보통 kubernetes.io/pvc-protection finalizer 가 있어서, 그 PVC 를 아직 쓰는 Pod 가 있으면 삭제를 막는다. 쓰는 Pod 를 먼저 지우면 담당 컨트롤러가 finalizer 를 떼고 PVC 가 사라진다. 즉 "안 지워진다"가 아니라 "아직 쓰고 있으니 못 지운다"는 보호 장치다 — 억지로 finalizer 를 떼기 전에 쓰는 Pod 부터 찾아라.

Service(LoadBalancer)를 지웠는데 클라우드 LB 가 안 없어지면?

0010 의 LoadBalancer Service 에는 service.k8s.io/load-balancer-cleanup finalizer 가 붙는다. cloud-controller-manager 가 실제 클라우드 LB 를 회수한 뒤에야 이 finalizer 를 떼어 Service 삭제를 완료한다. 이걸 손으로 강제 제거하면 클라우드 LB 가 과금된 채 방치될 수 있으니, LB 정리가 끝날 때까지 기다리는 게 맞다.

내 kind 클러스터에는 왜 load-balancer-cleanup finalizer 가 안 보여요?

kind 에는 클라우드 LB 가 없어서 LoadBalancer Service 가 실제 외부 IP 를 못 받고(0010 에서 pending), 그 finalizer 도 안 붙는다. 그래서 이 레슨의 finalizer 실습은 커스텀 finalizer 를 단 ConfigMap으로 대신했다 — 담당 컨트롤러가 없으니 우리가 patch 로 직접 떼 주며 메커니즘만 그대로 체험한다.

GC 컨트롤러는 어디서 도나요? 별도 프로세스인가요?

별도가 아니라 kube-controller-manager 안의 한 컨트롤러다(garbage collector). 클러스터의 모든 객체를 감시하며 "주인이 사라진 종속"을 찾아 지운다. 그래서 0002 의 ReplicaSet 컨트롤러, Deployment 컨트롤러와 같은 프로세스 안에서 나란히 도는, 제어 루프의 또 다른 한 가지다.

실습에서 delete 가 "deleted" 라고 떴는데 왜 객체가 남아 있어요?

kubectl delete 의 "deleted" 메시지는 "삭제를 요청했다(deletionTimestamp 를 찍었다)"는 뜻이지 "물리적으로 사라졌다"가 아니다. finalizer 가 남아 있으면 요청만 접수되고 실제 삭제는 보류된다. 그래서 5번 실습에서 "deleted" 가 떠도 get 하면 Terminating 인 객체가 여전히 조회됐다. 진짜 사라졌는지는 항상 get 으로 확인하라.
