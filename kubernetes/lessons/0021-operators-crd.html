Kubernetes Mastery · Lesson 0021 심화 · API 확장

Operator 패턴과 CRD

쿠버네티스에게 새 리소스 종류(kind)를 가르치고, 그것을 지켜보며 원하는 상태로 수렴시키는 제어 루프(0007)를 내 앱을 위해 쓴다. 이번엔 개념만이 아니라 — 작은 CRD 하나를 실제로 kubectl apply 해서 API가 확장되는 걸 눈으로 본다.

이번 레슨의 win: "CRD가 API를 확장한다, Operator = CRD + 커스텀 컨트롤러 = 운영 지식을 코드로 인코딩한 것"을 말로 설명할 수 있게 된다. 그리고 0007의 제어 루프, 0008의 RBAC, 0010의 "규칙엔 컨트롤러가 필요", 0014의 Helm(day-1) vs 살아있는 자가치유(day-2) — 이미 배운 네 조각이 하나로 꿰인다.

0. 한눈에 — 세 단어

딱 이 세 줄

CRD = API 서버에게 새 명사(kind)를 가르치는 것. kind: CronTab 을 등록하면 그 순간부터 kubectl get crontabs 가 동작하고, etcd 저장·RBAC·label selector 가 기본 리소스처럼 붙는다.

커스텀 컨트롤러 = 그 새 명사를 watch 하며 원하는 상태로 수렴시키는 제어 루프. 바로 0007의 reconciliation 을 "내 앱을 위해" 쓴 것 — kube-controller-manager 의 내장 컨트롤러들과 같은 골격이다.

Operator = CRD + 컨트롤러 + 특정 앱의 "운영 지식"(백업·페일오버·업그레이드·스케일)을 코드로 인코딩한 것. 사람 운영자(operator)가 하던 day-2 작업을 자동화한다.

1. CRD — API 서버에게 새 명사를 가르친다

지금까지 쓴 Pod, Deployment, Service 는 전부 쿠버네티스가 미리 아는 kind 였다. CustomResourceDefinition(CRD) 은 여기에 내가 정의한 새 kind — 예: CronTab, PostgresCluster, Certificate — 를 런타임에 추가한다. 등록하는 순간 API 서버는 그 kind 에 대한 REST 엔드포인트를 열어 준다.

아하 — "내 도메인 언어로 API를 확장" CRD를 apply 하면 그 kind 는 1급 시민이 된다: kubectl get postgresclusters 가 동작하고, 매니페스트는 etcd에 저장되며(0007의 그 etcd), 0008의 RBAC이 그대로 적용되고(Role 의 resources 에 새 리소스명을 넣으면 됨), label selector·kubectl describe·-o yaml 이 전부 기본 리소스와 똑같이 붙는다. 쿠버네티스 API 표면을 통째로 물려받는 것.

2. 커스텀 컨트롤러 — 0007의 제어 루프를 내 앱에

CRD만으로는 모양만 생긴다. 그 리소스를 지켜보며 실제 세계를 그 선언에 맞춰 바꾸는 주체가 컨트롤러다. 그 동작은 0007에서 배운 그 제어 루프와 한 글자도 다르지 않다:

아하 — 같은 골격, 다른 대상 kube-controller-manager 안의 Deployment 컨트롤러는 "replicas: 3 인데 지금 2개 → 하나 더" 를 무한 반복한다. 커스텀 컨트롤러는 똑같이 "PostgresCluster 가 replicas: 3, version: 15 라는데 지금 상태와 다르네 → 맞춘다" 를 반복한다. watch → 현재 상태와 원하는 상태 비교 → 좁힌다(reconcile) — 이 루프가 전부다. 그래서 커스텀 컨트롤러는 "kube-controller-manager 의 내장 컨트롤러를 내 도메인용으로 하나 더 쓴 것"이라고 보면 정확하다.
컨트롤러는 클러스터 안에서 보통 Deployment로 도는 하나의 Pod다. 그 Pod가 API server의 watch 스트림을 구독하고, 커스텀 리소스가 생기거나 바뀌면 reconcile 함수를 실행한다. 특별한 마법이 아니라 — 여러분이 지금까지 본 -n kube-system 의 컨트롤러들과 똑같은 방식으로 도는, 그냥 또 하나의 컨트롤러다.

3. Operator = CRD + 컨트롤러 + 운영 지식(day-2)

Operator 는 이 둘(CRD + 컨트롤러)에 더해, 특정 앱을 운영하는 사람의 노하우를 코드로 담은 것이다. DB를 예로 들면 사람 운영자가 손으로 하던 일 — 백업 스케줄, 리더 장애 시 페일오버, 무중단 버전 업그레이드, 안전한 스케일 — 을 reconcile 루프 안에 인코딩한다. 그래서 이름이 Operator(운영자)다.

한 줄 대비 Helm은 사진 한 장(설치 순간의 상태)을 남기고 떠난다. Operator는 살아서 지켜본다. Helm의 render는 0014에서 본 대로 설치 시점의 펼침이고, Operator의 reconcile은 0007/0014 GitOps처럼 끊임없이 현재를 원함에 맞추는 살아있는 루프다. 둘은 배타적이지 않다 — Helm 차트로 Operator를 설치하는 조합도 흔하다.

실제로 널리 쓰이는 Operator들: Prometheus Operator(모니터링 스택을 kind: Prometheus·ServiceMonitor 로 선언), cert-manager(kind: Certificate 를 선언하면 TLS 인증서를 발급·자동 갱신), 각종 DB Operator(Postgres·MySQL·Kafka…). 전부 "그 앱의 운영 지식"을 CRD + 컨트롤러로 감싼 것이다.

4. "규칙만 있고 컨트롤러 없으면 무력" — 익숙한 그 패턴

아하 — 이미 두 번 본 구조다 CRD만 등록하고 컨트롤러가 없으면, 커스텀 리소스는 etcd에 저장될 뿐 아무 일도 안 일어난다. 이건 새 규칙이 아니라 이 트랙을 관통하는 패턴이다:
  • 0010 Ingress: Ingress 규칙만 만들고 Ingress 컨트롤러가 없으면 라우팅이 안 된다 — 규칙엔 그걸 실현할 컨트롤러가 필요.
  • 0008 Role: Role만 만들고 RoleBinding이 없으면 권한이 무력 — 규칙만으론 아무에게도 안 붙는다.
  • 여기 CRD: kind만 등록하고 컨트롤러가 없으면 리소스는 그냥 저장된 데이터일 뿐.
세 경우 모두 "선언(규칙)과 그것을 집행하는 주체(컨트롤러)는 별개"라는 같은 골격이다.

5. 실습 — CRD 하나를 실제로 apply (kind 클러스터)

말로만 하지 말고 손으로 확인한다. mastery/lab 의 kind 클러스터에서 아주 작은 CRD를 등록하고, 그 새 kind가 진짜로 API에 나타나는지 본다. 컨트롤러는 만들지 않는다 — 오히려 그게 4절의 "무력" 패턴을 눈으로 각인시킨다.

5-1. CRD를 apply — apiextensions.k8s.io/v1 스키마로 CronTab kind 정의:

terminal · kind 클러스터 $ kubectl apply -f - <<'EOF' apiVersion: apiextensions.k8s.io/v1 kind: CustomResourceDefinition metadata: name: crontabs.stable.example.com # 반드시 <plural>.<group> 형식 spec: group: stable.example.com scope: Namespaced names: plural: crontabs singular: crontab kind: CronTab shortNames: [ct] versions: - name: v1 served: true storage: true schema: openAPIV3Schema: type: object properties: spec: type: object properties: cronSpec: type: string image: type: string replicas: type: integer EOF customresourcedefinition.apiextensions.k8s.io/crontabs.stable.example.com created
이름 규칙 주의: metadata.name 은 반드시 <plural>.<group> — 여기선 crontabs.stable.example.com. served: true = 이 버전을 API로 제공, storage: true = etcd에 이 버전으로 저장. 버전이 하나면 둘 다 true. openAPIV3Schema 는 커스텀 리소스의 구조 검증 스키마 — 여기 없는 필드를 쓰면 API server가 거부한다(0007의 admission/validation 흐름).

5-2. 새 kind가 API에 나타났는지 확인:

terminal $ kubectl get crd crontabs.stable.example.com NAME CREATED AT crontabs.stable.example.com 2026-07-07T09:00:00Z $ kubectl api-resources | grep crontab # 이제 기본 리소스 목록에 등장 crontabs ct stable.example.com/v1 true CronTab

5-3. 이제 그 kind의 인스턴스(커스텀 리소스)를 apply:

terminal $ kubectl apply -f - <<'EOF' apiVersion: stable.example.com/v1 kind: CronTab metadata: name: my-first-crontab spec: cronSpec: "*/5 * * * *" image: busybox:1.36 replicas: 2 EOF crontab.stable.example.com/my-first-crontab created $ kubectl get crontab # singular/plural/shortName 다 동작 NAME AGE my-first-crontab 10s $ kubectl describe ct my-first-crontab # shortName 'ct'도, describe도 기본 리소스처럼 Name: my-first-crontab API Version: stable.example.com/v1 Kind: CronTab Spec: Cron Spec: */5 * * * * Image: busybox:1.36 Replicas: 2 ...
여기서 멈춰서 보라 — 아무 일도 안 일어난다. replicas: 2 라고 선언했지만 Pod는 하나도 안 뜬다. kubectl get pods 를 쳐도 비어 있다. 왜? 이 kind를 watch하는 컨트롤러가 없기 때문이다. 커스텀 리소스는 그저 etcd에 저장된 구조화된 데이터일 뿐 — 누군가 그것을 읽고 실제 세계를 바꿔줘야 비로소 동작한다. 이것이 4절의 "규칙만 있고 컨트롤러 없으면 무력"(0010 Ingress, 0008 Role)과 정확히 같은 상황이다. Operator를 설치했다면 바로 이 지점에서 그 컨트롤러가 Pod·CronJob 등을 만들어 냈을 것이다.

5-4. 정리(cleanup): CRD를 지우면 그 kind의 모든 인스턴스도 함께 사라진다.

terminal $ kubectl delete crd crontabs.stable.example.com customresourcedefinition.apiextensions.k8s.io "crontabs.stable.example.com" deleted $ kubectl get crontab error: the server doesn't have a resource type "crontab" # kind 자체가 API에서 사라짐 → 되돌리기 완료
실제 컨트롤러를 쓰려면? 컨트롤러 코드는 보통 Kubebuilder 나 Operator SDK 로 골격을 생성해 Go(또는 Ansible/Helm 기반)로 reconcile 로직을 채운다. 그건 이 트랙의 범위를 넘는 심화 — KCNA 수준에선 "Operator는 이런 도구로 만든다"는 이름만 알면 충분하다. 오늘의 핵심은 API 확장 자체를 손으로 본 것.

6. 회상 연습

Q1. CRD를 apply해서 kind: CronTab 을 등록했다. 하지만 이 kind를 watch하는 컨트롤러는 배포하지 않았다. kind: CronTab 인스턴스를 apply하면?

CRD는 API 표면(저장·조회·RBAC)만 확장한다. 실제 동작은 그 kind를 watch하는 컨트롤러의 몫 — 없으면 리소스는 그저 저장된 데이터다. 0010 Ingress(규칙만 있고 컨트롤러 필요)·0008 Role(Binding 없으면 무력)과 같은 "선언과 집행은 별개" 패턴.

Q2. Helm과 Operator의 가장 결정적인 차이는?

Helm은 0014에서 본 대로 설치·업그레이드 순간에 render해 apply하는 패키지 매니저 — 이후 스스로 고치지 못한다. Operator는 reconcile 루프(0007)로 계속 drift를 교정한다. "사진 한 장 vs 살아있는 운영 엔지니어".

Q3. (레슨 0007 복습) 커스텀 컨트롤러의 reconcile 루프가 kube-controller-manager 안의 Deployment 컨트롤러와 "같은 골격"이라는 말의 뜻은?

0007의 제어 루프가 전부다: watch → desired vs actual 비교 → 좁힘. Deployment 컨트롤러는 replicas를, 커스텀 컨트롤러는 내 도메인 리소스를 대상으로 할 뿐 골격은 동일하다. 그래서 Operator는 "내 앱을 위한 컨트롤러를 하나 더 쓴 것". etcd 직접 접근이 아니라 API server의 watch로 상태를 본다(0007 Q2).

초보자 질문 (막히면 여기부터)

이 주제는 조각이 많아 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

CRD랑 커스텀 리소스(CR)는 같은 말인가요?

다르다. CRD(CustomResourceDefinition)는 "새 kind를 정의하는 틀"이고, CR(Custom Resource)은 그 틀로 찍어낸 인스턴스다. 실습으로 치면 kind: CustomResourceDefinition(5-1) 이 CRD, kind: CronTab(5-3) 이 CR. 클래스와 객체의 관계라고 보면 된다.

Operator랑 컨트롤러는 같은 건가요?

Operator는 컨트롤러를 포함하는 더 큰 개념이다. 컨트롤러 = watch하며 reconcile하는 루프(0007). Operator = 그 컨트롤러 + CRD + 특정 앱의 운영 지식(백업·페일오버·업그레이드)을 하나로 묶은 것. 그래서 "모든 Operator는 컨트롤러를 갖지만, 모든 컨트롤러가 Operator는 아니다" — 내장 Deployment 컨트롤러는 특정 앱 운영 지식이 없으니 Operator라 부르지 않는다.

왜 그냥 Deployment로 앱을 깔지 않고 Operator를 써요?

단순한 stateless 앱이면 Deployment로 충분하다. Operator가 빛나는 건 운영이 복잡한 상태 저장 앱 — 예: Postgres 클러스터에서 리더가 죽으면 팔로워를 승격(페일오버)하고, 버전 업그레이드를 순서대로 무중단으로, 정기 백업을 뜬다. 이 "사람이 새벽에 깨서 하던 일"을 코드로 자동화한 게 Operator다. day-2 운영이 힘든 앱일수록 값어치가 크다.

day-1, day-2가 무슨 뜻이에요?

업계 용어다. day-1 = 설치·초기 배포(한 번의 이벤트). day-2 = 그 이후의 지속적 운영 — 모니터링, 백업, 업그레이드, 장애 복구, 스케일. Helm은 day-1을 잘하고(설치), Operator는 day-2를 자동화한다(계속 돌봄). "설치는 쉽고 운영이 어렵다"는 통찰을 담은 구분이다.

CRD를 등록하면 RBAC은 자동으로 되나요?

API 표면은 자동으로 생기지만, 권한은 명시해야 한다. 0008의 Role/ClusterRole 의 resources 에 새 리소스명(예: crontabs)과 apiGroups(예: stable.example.com)를 넣어야 그 리소스를 만질 수 있다. 즉 커스텀 리소스도 기본 리소스와 똑같이 RBAC이 걸린다 — 그래서 Operator를 설치하면 대개 자기 컨트롤러용 ServiceAccount와 Role이 함께 딸려 온다.

컨트롤러 없이 CRD만 있는 게 쓸모가 있나요?

있다. 컨트롤러 없이도 CRD는 "구조화된 설정을 API에 저장·조회"하는 용도로 유용하다 — 예: 다른 도구가 읽어가는 설정 저장소, kubectl로 관리하는 앱별 메타데이터. 다만 "선언대로 실제 세계가 바뀌는" 자동화를 원하면 컨트롤러가 필요하다(5-3 warn). 실습에서 본 게 정확히 "컨트롤러 없는 CRD = 저장만 되는 상태"다.

served와 storage가 뭐가 달라요?

CRD가 여러 버전(v1, v2beta1…)을 가질 때 쓴다. served: true = 그 버전을 API로 제공(클라이언트가 그 버전으로 읽고 쓸 수 있음). storage: true = etcd에 실제로 저장되는 버전(정확히 하나만 true여야 함). API server가 버전 간 변환을 해주므로, 여러 버전을 served로 열어두고 storage는 하나로 고정하는 식이다. 버전이 하나뿐이면 둘 다 true(실습 5-1).

openAPIV3Schema는 꼭 써야 하나요?

apiextensions.k8s.io/v1(현재 정식 버전)에서는 필수다. 이 스키마가 커스텀 리소스의 필드 구조와 타입을 검증한다 — 예: replicas 를 integer 로 선언하면 문자열을 넣었을 때 API server가 거부한다(0007의 validation 흐름). 스키마 없이 아무 필드나 받던 옛 v1beta1 은 폐기됐다. 즉 "이 kind는 이런 모양이어야 한다"를 강제하는 안전장치다.

Prometheus Operator, cert-manager 같은 건 어떻게 설치해요?

대개 Helm 차트나 매니페스트 한 번으로 설치한다 — 그러면 CRD들과 컨트롤러 Deployment가 함께 깔린다. 여기서 재미있는 점: Helm(0014)으로 Operator를 설치하는 조합이다. Helm이 day-1(설치)을 하고, 설치된 Operator가 day-2(운영)를 맡는 — 둘이 배타적이지 않고 층을 이룬다. 설치 후엔 kind: Certificate, kind: Prometheus 같은 CR을 apply하며 쓴다.

Operator를 직접 만들려면 뭘 배워야 하나요? (KCNA 범위 밖)

KCNA엔 안 나오지만 궁금하다면: Kubebuilder 나 Operator SDK 로 프로젝트 골격을 생성하고, reconcile 함수(0007 루프의 몸통)를 주로 Go로 채운다. "이 CR을 보면 실제 세계를 이렇게 맞춰라"를 코드로 쓰는 것. Helm/Ansible 기반으로 코드 없이 만드는 길도 있다. 핵심 개념(watch → reconcile)은 이 레슨에서 이미 잡았으니, 만들기는 그 위에 얹는 손기술일 뿐이다.
