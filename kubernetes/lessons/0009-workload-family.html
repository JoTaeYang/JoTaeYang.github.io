워크로드 4형제: 언제 Deployment 가 아닌가

Deployment 는 "똑같고, 서로 바꿔도 되고, 영원히 도는 복제본"을 가정한다. 그 가정이 깨지는 네 지점에 각각 다른 워크로드가 산다.

이번 레슨의 win: Job 이 Completed 로 끝나는 것(재시작이 아니라)과, StatefulSet 의 Pod 가 web-0, web-1 처럼 이름을 갖고 순서대로 태어나 지워도 같은 이름으로 되살아나는 것을 관찰한다. 0002 의 "랜덤 해시 Pod"와 정반대다.

1. 프레임 — Deployment 의 세 가정, 그리고 그걸 깨는 형제들

Deployment 를 고를 때 우리는 은연중에 세 가지를 가정한다. 다른 워크로드는 이 중 하나씩을 부순다.

워크로드깨는 가정한 줄 정의실제 쓰임
Deployment — (기준) 똑같은 stateless 복제본 N 개를 영원히 유지 웹서버, API 등 대부분의 stateless 앱
StatefulSet "서로 바꿔도 된다" 안정된 이름·순서·스토리지를 가진 복제본 (web-0, web-1…) DB, Kafka 등 각 인스턴스가 고유 신원을 갖는 것
DaemonSet "N 개를 아무 데나" 노드마다 정확히 하나 (replicas 개념이 없음) 로그 수집기, 모니터링 에이전트, CNI
Job "영원히 돈다" 끝날 때까지 실행 후 Completed — 성공하면 재시작 안 함 배치 작업, 마이그레이션, 백업
CronJob "영원히 돈다" 일정에 맞춰 Job 을 찍어내는 틀 (cron 문법) 야간 백업, 주기 리포트
아하 "어떤 워크로드를 쓰지?"는 "Deployment 의 어떤 가정이 여기서 안 맞나?" 한 질문으로 풀린다. 바꿔 써도 되면 Deployment, 신원이 필요하면 StatefulSet, 노드마다 하나면 DaemonSet, 언젠가 끝나야 하면 Job(주기적이면 CronJob). 다섯을 외우는 게 아니라 가정 세 개를 기억한다.

2. 실습 — 가장 다른 둘을 직접 본다

kind 클러스터 mastery 이어서, 실습 namespace lab 에서 (없으면 kubectl create namespace lab && kubectl config set-context --current --namespace=lab).

추적 1 — Job 은 끝난다. Deployment 의 Pod 는 프로세스가 끝나면 "죽었다"며 재시작되지만(0004), Job 의 Pod 는 끝나는 게 목표다. pi.yaml:

pi.yaml apiVersion: batch/v1 # Deployment 와 다른 API 그룹: batch kind: Job metadata: name: pi spec: backoffLimit: 4 # 실패 시 최대 4번까지 재시도 template: spec: restartPolicy: Never # Job 은 Always 를 못 쓴다 containers: - name: pi image: perl:5.34 command: ["perl", "-Mbignum=bpi", "-wle", "print bpi(200)"]
terminal $ kubectl apply -f pi.yaml && kubectl get pod -l job-name=pi -w NAME READY STATUS RESTARTS pi-abc12 0/1 ContainerCreating 0 pi-abc12 1/1 Running 0 pi-abc12 0/1 Completed 0 ← 끝났고, 재시작하지 않는다 (RESTARTS 0) $ kubectl get job pi NAME STATUS COMPLETIONS DURATION pi Complete 1/1 7s $ kubectl logs -l job-name=pi | head -c 40 3.14159265358979323846264338327950288...

Completed 는 0006 에서 본 CrashLoopBackOff 의 정반대다 — 같은 "종료"라도 Deployment 에겐 장애, Job 에겐 성공이다. 그래서 Job 은 restartPolicy: Always 를 아예 금지한다(영원히 돌 거면 Job 이 아니니까).

추적 2 — StatefulSet 은 이름을 갖는다. 0002 에서 Deployment Pod 들은 web-7c5ddbdf54-x9k2p 같은 랜덤 해시로 태어났다. StatefulSet 은 정반대다. web-sts.yaml:

web-sts.yaml apiVersion: v1 kind: Service # headless service (clusterIP: None) — 0003 의 변형 metadata: name: web spec: clusterIP: None selector: { app: web } ports: [{ port: 80 }] --- apiVersion: apps/v1 kind: StatefulSet metadata: name: web spec: serviceName: web replicas: 3 selector: { matchLabels: { app: web } } template: metadata: { labels: { app: web } } spec: containers: - name: nginx image: nginx:1.27
terminal $ kubectl apply -f web-sts.yaml && kubectl get pod -l app=web -w web-0 0/1 ContainerCreating web-0 1/1 Running ← 0 이 Ready 가 되어야 web-1 0/1 Pending web-1 1/1 Running ← 그 다음 1 이 시작된다 (순서 보장) web-2 1/1 Running $ kubectl delete pod web-1 $ kubectl get pod -l app=web web-0 1/1 Running web-1 1/1 Running ← 랜덤 이름이 아니라 똑같은 web-1 로 되살아났다 web-2 1/1 Running
0002 와 비교하라. Deployment 에서 Pod 를 지우면 다른 랜덤 이름의 Pod 가 대신 떴다 (누가 죽든 상관없으니까). StatefulSet 에서는 web-1 이 죽으면 정확히 web-1 이 돌아온다 — DB 클러스터에서 "1번 노드"의 정체성과 그에 붙은 디스크(PVC)가 유지돼야 하기 때문이다. 스토리지 연결(PV/PVC)은 곧 이어질 스토리지 레슨(0011)에서 다룬다.

DaemonSet 과 CronJob 은 개념으로. kind 는 기본 단일 노드라 DaemonSet 이 Pod 1 개만 띄워 극적이지 않다 — "노드마다 하나"라는 정의만 잡고 넘어간다. CronJob 은 위 Job 에 schedule: "*/1 * * * *" 를 감싼 것뿐이라, 1 분마다 pi 같은 Job 이 새로 찍히는 틀이다. 과투자 금지: 이 둘은 KCNA 에선 어휘 수준, CKAD 에서 실습으로 다시 만난다.

정리: kubectl delete ns lab — Job, StatefulSet, Service 까지 한 번에.

3. 회상 연습

Q1. 매일 새벽 3 시에 DB 를 백업하는 스크립트를 돌려야 한다. 알맞은 워크로드는?

"언젠가 끝나야 하고(=Job) + 주기적(=schedule)" → CronJob. Deployment 로 하면 스크립트가 끝나는 순간 "죽었다"며 재시작되어 백업이 폭주한다 — Deployment 의 "영원히 돈다" 가정이 안 맞는 자리다.

Q2. StatefulSet 의 web-1 Pod 를 지웠다. 무슨 일이 일어나나?

StatefulSet 의 핵심은 안정된 신원. 이름(그리고 붙은 PVC)이 인덱스에 고정돼 있어, 죽어도 같은 이름으로 돌아온다. 0002 의 Deployment(교체 가능한 랜덤 Pod)와 정확히 반대되는 성질이다.

Q3. (레슨 0006 복습) Job 의 Pod 가 성공적으로 끝나 Completed 다. 이건 어떤 상태와 헷갈리면 안 되나?

간격 반복: Completed(Job 의 목표)와 CrashLoopBackOff(Deployment 의 장애)는 같은 "프로세스 종료"에서 갈린다. 워크로드 종류가 "종료"의 의미를 바꾼다 — restartPolicy 가 그 차이의 스위치다.
판정: Q1·Q2 를 맞히면 워크로드 4형제 ✅ — 다음은 외부 노출 3단 (NodePort → LoadBalancer → Ingress). 0003 의 ClusterIP 위에 "클러스터 밖에서 접속"을 얹는다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

워크로드가 5 개나 되는데 다 외워야 하나요?

외우는 건 Deployment 의 가정 3개(교체 가능·N개·영원히)뿐이다. 나머지 넷은 그 가정을 하나씩 깬 특수 케이스라 "이 가정이 안 맞네 → 저거"로 유도된다. 암기 5개가 아니라 분기 1개다.

Job 이 성공했는데 왜 Pod 가 안 사라지고 Completed 로 남아 있어요?

로그를 나중에 볼 수 있게 일부러 남긴다. 정리하려면 kubectl delete job pi(Pod 도 함께 사라짐), 또는 Job spec 에 ttlSecondsAfterFinished 를 넣으면 완료 후 자동 삭제된다. CKAD 단골 옵션이다.

Job 에서 restartPolicy: Always 를 쓰면 왜 안 되나요?

Always 는 "끝나도 계속 다시 켜라"는 뜻이라, 언젠가 끝나야 하는 Job 의 목적과 정면 충돌한다. 그래서 API 가 아예 거부한다 — Job 은 Never 또는 OnFailure만 허용한다. 실패 재시도는 restartPolicy 가 아니라 backoffLimit 이 담당한다.

backoffLimit 이랑 restartPolicy: OnFailure 는 뭐가 달라요?

둘 다 "실패하면 재시도"지만 층이 다르다. OnFailure 는 같은 Pod 안에서 컨테이너를 재시작하고, backoffLimit 은 Job 이 새 Pod 를 몇 번까지 만들지의 상한이다. backoffLimit 을 넘기면 Job 이 Failed 로 확정된다.

StatefulSet 의 Pod 가 순서대로 뜨는 게 왜 중요해요?

DB 클러스터 같은 데선 0 번(주로 primary)이 먼저 준비돼야 1 번, 2 번이 거기 붙어 복제를 시작할 수 있다. 아무 순서로나 뜨면 부팅이 꼬인다. 그래서 web-0 이 Ready 가 된 뒤에야 web-1 이 시작된다. 삭제는 역순(2→1→0)이다.

headless service(clusterIP: None)는 뭐예요? 0003 의 Service 랑 달라요?

0003 의 보통 Service 는 하나의 가상 IP(ClusterIP) 뒤로 Pod 들을 숨겼다. headless 는 그 대표 IP 를 없애고(None), 대신 각 Pod 에 web-0.web, web-1.web 같은 고유 DNS 이름을 준다. "아무나 하나"가 아니라 "특정 인스턴스"를 직접 부르기 위한 것 — StatefulSet 의 신원과 짝을 이룬다.

DaemonSet 은 replicas 를 몇으로 두나요?

둘 수 없다 — replicas 필드 자체가 없다. "노드 수 = Pod 수"가 규칙이라, 노드를 새로 추가하면 거기에 자동으로 하나 뜨고, 노드를 빼면 함께 사라진다. 그래서 로그 수집기·모니터링 에이전트처럼 "모든 노드에 하나씩" 있어야 하는 것에 쓴다.

StatefulSet 이면 스토리지(디스크)도 자동으로 붙나요?

이번 실습은 신원(이름·순서)만 봤고 디스크는 안 붙였다. StatefulSet 에 volumeClaimTemplates 를 더하면 Pod 마다 전용 PVC가 생기고, web-1 이 죽었다 살아나도 같은 디스크를 다시 문다. 이 PV/PVC 가 곧 이어질 스토리지 레슨(0011) 주제다 — 그 사이에 외부 노출(0010)이 먼저 온다.

CronJob 의 schedule: "*/1 * * * *" 는 어떻게 읽어요?

표준 cron 5칸: 분·시·일·월·요일. */1 * * * * = "매 1분마다". 예: 0 3 * * * = 매일 새벽 3시, 0 0 * * 0 = 매주 일요일 자정. crontab.guru 같은 사이트로 확인하는 습관을 들이면 실수를 줄인다.

그냥 전부 Deployment 로 하면 안 되나요? 굳이 나눠야 해요?

가정이 맞으면 Deployment 가 제일 편하다 — 대부분의 stateless 앱이 그렇다. 문제는 가정이 틀린 자리에 Deployment 를 쓰면 조용히 망가진다는 것: 배치 작업이 무한 재시작되거나, DB Pod 가 매번 새 이름·새 디스크로 태어나 데이터를 잃는다. "맞는 도구"의 값어치는 장애 났을 때 드러난다.
