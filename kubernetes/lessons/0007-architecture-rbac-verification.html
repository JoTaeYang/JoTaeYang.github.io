Kubernetes Mastery · Lesson 0007 검증 · 퀴즈만

검증: 아키텍처와 RBAC, 아직 단단한가

새로 배우는 것 없음. 인접 트랙(../teach/)에서 배운 지식이 KCNA 기준으로 살아 있는지 8문항으로 확인한다.

이번 레슨의 win: 갭 체크의 🔶 두 칸(아키텍처, RBAC)을 ✅ 또는 "보강 필요"로 판정한다. 6개 이상 정답이면 ✅ — 틀린 주제만 다음에 보강한다.
회상 연습의 정석대로: 문서를 다시 보지 말고, 기억만으로 풀 것. 막히는 감각 자체가 유용한 데이터다.

Part A — 컨트롤 플레인과 요청의 여정

Q1. kubectl apply -f deploy.yaml 직후, 매니페스트가 가장 먼저 도착해 저장되는 경로는?

모든 것의 관문은 API server 다. 인증(authn) → 인가(authz/RBAC) → admission → etcd 저장. 다른 어떤 컴포넌트도 이 순서를 건너뛸 수 없다.

Q2. etcd 와 직접 통신하는 컴포넌트는?

etcd 의 유일한 클라이언트는 API server 다. scheduler 도 controller 도 kubelet 도 전부 API server 의 watch/list 를 통해서만 상태를 본다 — 그래서 API server 가 단일 관문이자 병목이다.

Q3. scheduler 가 실제로 하는 일의 전부는?

scheduler 는 "결정"만 한다 — nodeName 이 비어 있는 Pod 를 발견하면 필터/점수 계산으로 노드를 골라 그 필드를 채울 뿐. 실행은 그 nodeName 을 watch 하던 kubelet 의 몫이다.

Q4. 레슨 0002 에서 본 Deployment/ReplicaSet 의 제어 루프들 — 이 컨트롤러들이 실제로 돌고 있는 곳은?

Deployment 컨트롤러, ReplicaSet 컨트롤러, EndpointSlice 컨트롤러(레슨 0003)까지 — 수십 개의 루프가 kube-controller-manager 한 프로세스에 모여 돈다. 트랙 전체를 관통하던 "제어 루프"의 실체.

Q5. scheduler 프로세스가 죽었다. 클러스터에서 일어나는 일은?

각 컴포넌트는 독립적이다. 실행 중인 Pod 는 kubelet 이 지키고, scheduler 부재의 증상은 정확히 "새 Pod 의 영원한 Pending" — 레슨 0006 의 Pending 감별에 한 가지 원인이 추가된 셈이다 (describe Events 에 스케줄 기록 자체가 없다는 점이 다르다).

Part B — 인증과 인가 (RBAC)

Q6. Role 과 ClusterRole 의 차이는?

경계는 namespace 다. 노드/PV 같은 namespace 없는 리소스나 전 namespace 권한은 ClusterRole 이 필요하다. 주체(사람/SA)나 동사(get/delete)와는 무관하다.

Q7. "누가(subject) 무엇을 할 수 있는지"가 실제로 성립되는 순간은?

Role = 권한 목록(명사), RoleBinding = 부여(동사). Role 만 만들어 두면 아무 일도 일어나지 않는다 — 실기에서 "권한이 안 먹혀요"의 단골 원인이 Binding 누락이다.

Q8. ServiceAccount 가 존재하는 이유는?

사람은 kubeconfig 로, Pod 는 ServiceAccount 토큰으로 API server 에 인증한다. 모든 Pod 는 지정 없으면 그 namespace 의 default SA 를 받는다 — 그리고 그 토큰의 권한을 좁히는 것이 CKS 의 단골 주제가 된다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

API server 가 죽으면 클러스터가 통째로 멈추나요?

이미 돌고 있는 Pod 는 계속 돈다 — 각 노드의 kubelet 이 로컬에서 컨테이너를 지키기 때문이다. 하지만 새로운 변경은 전부 막힌다: kubectl 도, scheduler·controller-manager 의 watch 도 전부 API server 를 거치므로 배치·복구·스케일이 멈춘다. "관문 하나가 곧 단일 장애점"이라 실무에선 API server 를 다중화한다.

etcd 는 왜 API server 만 접근하나요? 다른 컴포넌트가 직접 읽으면 안 되나요?

일부러 그렇게 막아 뒀다. etcd 앞에 API server 하나만 세워야 인증·인가(RBAC)·admission·검증이 딱 한 곳에서 강제된다 — 여러 클라이언트가 제각각 etcd 를 만지면 그 관문을 우회할 길이 생긴다. 그래서 etcd 의 유일한 클라이언트는 API server 이고, 나머지는 전부 watch/list 로 상태를 본다(Q2 의 핵심).

scheduler 랑 controller-manager 는 뭐가 다른가요?

scheduler 는 "어디에?"만 답한다 — nodeName 이 빈 Pod 를 보고 노드를 골라 그 필드를 채운다(Q3). controller-manager 는 "얼마나·상태 맞추기" 담당 — Deployment·ReplicaSet 같은 수십 개 조정 루프가 desired ↔ actual 을 계속 좁힌다(Q4). 배치 결정 vs 상태 조정, 서로 다른 프로세스다.

컨트롤 플레인 컴포넌트도 그냥 Pod 인가요?

kubeadm·kind 로 만든 클러스터에선 대체로 그렇다 — API server·scheduler·controller-manager·etcd 가 static Pod 로 뜬다(kubelet 이 /etc/kubernetes/manifests 의 YAML 을 보고 직접 띄우는, API server 없이도 뜨는 특수 Pod). kubectl get pods -n kube-system 으로 눈으로 확인할 수 있다. 부트스트랩의 닭-달걀 문제를 static Pod 가 푼다.

admission controller 가 뭐예요? (Q1 에 나왔는데 이 레슨은 안 다뤘어요)

요청이 인증(authn) → 인가(authz/RBAC) 를 통과한 뒤, etcd 에 저장되기 직전에 끼어드는 마지막 검문소다. 값을 바꾸거나(mutating: 기본값 주입, sidecar 삽입) 통째로 거부한다(validating: 정책 위반 차단). "RBAC 은 통과했는데 왜 거부되죠?"의 흔한 범인 — CKA/CKS 로 이어지는 실이다.

kube-proxy 는 어디서 뭘 하나요? 컨트롤 플레인 컴포넌트인가요?

아니다 — kube-proxy 는 모든 워커 노드에 하나씩 도는 데이터 평면 요소다. Service 의 가상 IP 로 온 트래픽을 실제 Pod 들로 넘기는 iptables/IPVS 규칙을 각 노드에 심는다(레슨 0003 의 EndpointSlice 를 watch 해서). 컨트롤 플레인이 "결정"이라면 kube-proxy 는 노드에서 그걸 "집행"하는 쪽이다.

scheduler 가 죽었다 살아나면 밀렸던 Pending Pod 들은 어떻게 되나요?

알아서 처리된다. scheduler 는 nodeName 이 빈 Pod 를 watch 로 계속 지켜보다가, 다시 뜨는 순간 쌓여 있던 Pending 들을 하나씩 배치한다 — 상태(desired)는 etcd 에 그대로 남아 있으니 잃는 것이 없다. 이 "선언한 상태는 남고, 컨트롤러가 나중에 따라잡는다"가 쿠버네티스 전반을 관통하는 원리다.

Role 만 만들고 RoleBinding 을 안 만들면 어떻게 되나요?

아무 일도 안 일어난다(Q7). Role 은 권한 목록(명사)일 뿐, 존재만으로는 누구에게도 붙지 않는다. RoleBinding 이 주체(사람/SA)와 Role 을 이어야 비로소 권한이 성립한다. 실기에서 "권한 줬는데 안 먹혀요"의 단골 원인이 바로 이 Binding 누락 — 0008 실습에서 손으로 각인한다.

ServiceAccount 랑 그냥 사용자 계정은 뭐가 다른가요?

인증하는 주체가 다르다: 사람은 kubeconfig 의 인증서/토큰으로, Pod 안 프로세스는 ServiceAccount 토큰으로 API server 에 자신을 증명한다(Q8). SA 는 클러스터가 관리하는 신원이고, 모든 Pod 는 지정이 없으면 그 namespace 의 default SA 를 자동으로 받는다. 그 기본 SA 의 권한을 좁히는 게 CKS 단골 주제다.

Role 과 ClusterRole, 시험에서 뭘로 가르나요?

딱 한 질문: "이 권한이 한 namespace 안에서만 필요한가?" 예 → Role, 아니오 → ClusterRole(Q6). 주체가 사람이냐 SA 냐, 동사가 읽기냐 쓰기냐는 전혀 상관없다 — 오답 보기들이 노리는 함정이 정확히 그것이다. node·PV 처럼 애초에 namespace 가 없는 리소스도 ClusterRole 영역이다.

회상 퀴즈에서 오래 망설였는데 정답이면 ✅ 로 쳐도 되나요?

표시는 해 두는 게 좋다. 맞았어도 오래 망설인 문항은 아직 "인출 강도"가 약하다는 신호 — 며칠 뒤 다시 물으면 틀릴 확률이 높은 후보다. 그래서 0008 이 controller-manager 문항을 재시험으로 다시 낸다. 간격 반복은 "맞았나"보다 "얼마나 매끄럽게 나왔나"를 본다.
