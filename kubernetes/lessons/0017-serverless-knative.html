Kubernetes Mastery · Lesson 0017 심화 · Architecture

Serverless 와 Knative: 안 쓰면 0으로

"서버가 없다"가 아니라 "서버를 신경 안 쓴다". 0012 의 HPA 가 못 하는 것 — Pod 를 0 까지 내렸다 첫 요청에 되살리기 — 을 Knative 가 어떻게 하는지 파본다.

이번 레슨의 win: scale-to-zero 라는 단 하나의 차별점을 이해한다 — 요청이 없으면 Pod 가 0 개가 되고(유휴 비용 0), 첫 요청이 오면 그때 켜진다. 0012 의 HPA(보통 항상 1 개 이상)와 정확히 어디서 갈라지는지, 그 대가(cold start)가 뭔지가 여기서 풀린다.

0. 한눈에 — 세 문장

딱 이 세 줄

Serverless = 서버를 신경 안 쓴다. 서버가 사라지는 게 아니라, 용량·스케일링·유휴 관리를 내가 안 하고 플랫폼에 맡긴다. 이벤트/요청이 오면 돌고, 안 오면 0 으로 내려가고, 쓴 만큼만 낸다.

Knative Serving = 쿠버네티스 위에 얹는 serverless 층. 요청량에 따라 Pod 수를 자동 조절하는데, 핵심은 다른 오토스케일러가 못 하는 0 까지 내렸다 첫 요청에 되살리기.

대가는 cold start. 0 에서 깨어날 땐 Pod 를 새로 띄우느라 첫 요청이 느리다. 유휴 비용 0 을 얻는 대신 첫 손님을 기다리게 하는 트레이드오프.

1. Serverless 란 — "서버가 없다"가 아니다

Serverless 는 이름이 오해를 부른다. 서버는 여전히 있다 — 어딘가의 노드에서 컨테이너가 돈다. 바뀌는 건 누가 그 서버를 신경 쓰느냐다. 세 가지 특성으로 정의된다:

특성뜻내가 안 하는 일
서버 관리 위임용량·프로비저닝을 플랫폼이 함"노드 몇 대? replicas 몇 개?" 고민 안 함
이벤트/요청 구동요청·이벤트가 있을 때만 실행"항상 켜둬야 하나?" 안 함
scale-to-zero안 쓰면 0 개로 내려감유휴 시간에 노는 Pod 를 안 굴림
아하 — 0009 의 가정이 여기서 깨진다 0009 의 워크로드 4 형제(Deployment·StatefulSet·DaemonSet·Job)는 대부분 "항상 도는 복제본"을 전제로 했다 — Deployment 는 replicas 만큼 Pod 를 계속 유지한다. Serverless 는 그 전제를 뒤집는다: 요청이 없으면 유지할 복제본도 0 개다. "일이 없으면 아무것도 안 돈다"가 핵심 발상이다.
한 줄 기억법: "서버가 없다"(X) → "서버를 신경 안 쓴다"(O). 청구서에 idle 요금이 안 찍히는 게 체감 차이다.

2. Knative Serving — scale-to-zero 라는 차별점

Knative 는 쿠버네티스 위에 얹어 serverless 워크로드를 굴리는 CNCF 프로젝트다. 두 부분으로 나뉜다 — 이번 레슨의 주인공은 Serving이다:

구성하는 일
Knative ServingHTTP 요청량에 따라 Pod 를 자동 스케일. 핵심 = 0↔N 오토스케일링
Knative Eventing이벤트를 라우팅 (소스→브로커→구독). 요청이 아니라 이벤트로 구동

Serving 의 결정적 차별점은 하나다: scale-to-zero. 요청이 한동안 없으면 Pod 수를 0 개까지 내리고, 첫 요청이 도착하면 그때 Pod 를 기동한다. 이걸 가능하게 하는 숨은 조연이 activator다 — Pod 가 0 개일 때 요청을 대신 받아 붙잡아 두고, Pod 가 뜰 때까지 기다렸다 넘겨준다.

0007 의 제어 루프, 요청량 버전 Knative 오토스케일러(KPA)도 결국 0007 의 제어 루프다 — "원하는 상태(요청량에 맞는 Pod 수)"와 "현재 상태"를 계속 맞춘다. 차이는 신호가 CPU 가 아니라 동시 요청 수(concurrency)이고, 원하는 값이 0 이 될 수 있다는 점뿐이다.

Knative Eventing은 한 문단으로: 요청(HTTP) 대신 이벤트로 구동하는 세계다. 이벤트 소스(예: 메시지 큐, 타이머)가 이벤트를 만들면 브로커(broker)가 받아 구독(trigger) 조건에 맞는 서비스로 전달한다. 0014 에서 말한 느슨한 결합의 구현체 — 보내는 쪽은 받는 쪽을 몰라도 된다.

3. 결정적 대비 — HPA(0012) vs Knative

0012 의 HPA는 CPU/메모리 등 메트릭을 보고 replicas 를 늘리고 줄였다. 하지만 보통 minReplicas 가 1 이상 — 항상 최소 1 개는 켜져 있다. Knative 는 그 바닥을 0 까지 뚫는다:

HPA (0012)Knative Serving
스케일 신호CPU·메모리·커스텀 메트릭동시 요청 수(concurrency)·RPS
최소 Pod 수보통 minReplicas ≥ 1 (항상 켜짐)0 까지 내려감
요청 없을 때1 개는 계속 돌며 대기 (idle 비용 발생)0 개 — 아무것도 안 돎 (idle 비용 0)
첫 요청이미 켜져 있어 즉시 응답Pod 를 새로 띄움 → cold start 지연
대상기존 Deployment 에 얹음Knative Service(Revision) 단위
한 줄 구분 HPA 는 "1 개와 N 개 사이"를 오간다. Knative 는 "0 개와 N 개 사이"를 오간다. 그 0 이 되느냐가 serverless 인지 아닌지를 가르는 선이다. HPA 로도 트래픽엔 잘 대응하지만, "쓰지 않는 시간에 Pod 를 완전히 끄기"는 HPA 단독으론 못 한다.

4. Cold start — scale-to-zero 의 대가

공짜 점심은 없다. Pod 가 0 개일 때 첫 요청이 오면, 쿠버네티스는 그 순간 Pod 를 스케줄→이미지 풀(있으면 캐시)→컨테이너 시작→앱 초기화를 거친다. 이 시간만큼 첫 요청이 느려진다 — 이게 cold start다.

Cold start (0 에서 깨어남)Warm (이미 켜져 있음)
첫 요청 지연큼 (Pod 기동 시간이 얹힘)작음 (바로 처리)
비용유휴 비용 0유휴에도 비용 발생
트레이드오프를 정직하게. scale-to-zero 는 유휴 비용 0 을 얻는 대신 첫 손님을 기다리게 한다. 그래서 "가끔 오는 트래픽·내부 도구·이벤트 처리"엔 이상적이지만, "밀리초가 중요한 항상 켜진 API"엔 minScale=1 로 바닥을 1 로 올리기도 한다 — 그 순간 scale-to-zero 를 포기하고 HPA 에 가까워진다. 선택의 문제지 정답이 하나가 아니다.

5. FaaS vs Knative — 함수 단위 vs 컨테이너 단위

Serverless 를 들으면 흔히 AWS Lambda 같은 FaaS(Function as a Service)를 떠올린다. Knative 와 무엇이 다른지 한 줄로 나눈다:

FaaS (Lambda, OpenFaaS)Knative
배포 단위함수 하나 (핸들러 코드)컨테이너 하나 (OCI 이미지)
실행 모델이벤트당 함수 호출요청/이벤트에 컨테이너가 응답
포장런타임에 코드 조각을 올림0014 의 OCI 이미지 그대로
이식성대체로 벤더 종속적쿠버네티스 있으면 어디서나(CNCF)
아하 FaaS = 함수를 올린다. Knative = 컨테이너를 올린다. Knative 는 이미 쓰던 OCI 이미지(0014)를 그대로 serverless 화한다 — 새 프로그래밍 모델을 배울 필요가 없고, 벤더에 묶이지도 않는다. "쿠버네티스 위의 serverless" 라는 포지션이 여기서 나온다.

6. 왜 Architecture 도메인인가

KCNA 에서 serverless/Knative 는 Cloud Native Architecture 도메인에 속한다. 왜냐하면 이것이 느슨한 결합·탄력성이라는 클라우드 네이티브 원칙의 극단적 실현이기 때문이다.

0014 의 12-factor 와 이어진다

scale-to-zero 가 가능하려면 앱이 stateless·disposable 해야 한다 — 0014 의 12-factor 원칙 그대로다. Pod 를 아무 때나 0 으로 죽였다 되살리려면, 그 Pod 안에 잃으면 안 되는 상태가 없어야 한다(상태는 0011 의 PV 나 외부 저장소에).

0002 에서 "Pod 는 ephemeral(언제든 교체)"라고 배운 그 성질이, 여기선 기능이 된다 — 언제든 죽여도 되니까 안 쓸 때 0 으로 죽일 수 있는 것이다. disposability 가 scale-to-zero 의 전제조건이다.

성숙도 한 줄(0014): Knative 는 CNCF Incubating 단계를 거쳐 성장한 프로젝트다. "graduated 만큼 보편적이진 않지만 실사용처가 탄탄한" 위치 — KCNA 는 이름·역할·scale-to-zero 개념까지면 충분하다.
비용 예고(0019): scale-to-zero 의 진짜 매력은 돈이다 — 유휴 비용 0. 안 쓰는 시간에 청구서가 안 늘어난다. 이 "쓴 만큼만" 모델이 비용 최적화와 어떻게 엮이는지는 0019 비용 레슨에서 다룬다.

7. 개념 트레이스 — 첫 요청이 0 을 깨우는 여정

정직한 고지: Knative 설치·운영은 KCNA 범위를 넘는 CKAD+ 영역이다(CRD 설치, networking layer 구성 등). 그래서 여기선 손으로 깔지 않는다. 대신 0013 스타일로 "요청 0 → 응답"까지 한 요청의 여정을 개념으로 따라간다 — 이 그림이 scale-to-zero 의 전부다.
scale-to-zero 를 깨우는 한 요청의 일생 초기 상태: 요청이 한동안 없었다 │ ① Pod 0 개 Revision 의 replicas = 0. 아무것도 안 돎 (idle 비용 0) │ └ HPA 라면 여기서도 1 개는 살아있었을 자리 ▼ ② 첫 curl 도착 사용자: curl http://hello.default.example.com │ ③ activator 가 붙잡음 Pod 가 0 이라 요청이 activator 로 라우팅됨 │ └ activator: 요청을 버퍼에 붙잡아 두고 응답을 미룸 ▼ ④ 오토스케일러에 신호 "요청 왔다!" → KPA 가 원하는 replicas 를 0 → 1 로 │ └ 0007 의 제어 루프: desired ≠ current 이면 맞춘다 ▼ ⑤ Pod 1 개 기동 스케줄 → 컨테이너 시작 → 앱 준비(readiness) │ └ 이 시간이 cold start 지연 ▼ ⑥ activator 가 넘겨줌 Pod 가 Ready 되면 붙잡아 둔 요청을 그 Pod 로 전달 │ ⑦ 응답 Hello World! ← 첫 요청은 느렸지만 성공 │ ⑧ 이후 요청 warm Pod 가 살아있는 동안은 빠름. 트래픽 멈추면 다시 ① 로 (0 으로 회귀)
activator 가 핵심 트릭이다

scale-to-zero 가 요청을 잃지 않는 비결은 ③ 의 activator다. Pod 가 0 개일 때 요청이 그냥 실패하면 쓸모가 없다. activator 는 첫 요청을 대신 받아 붙잡아 두고, Pod 가 뜨는 동안 기다렸다 넘겨준다 — 그래서 사용자는 "느렸다"고 느낄 뿐 "실패했다"고 느끼지 않는다.

트래픽이 충분히 오르면 activator 는 경로에서 빠지고(warm 상태), 요청이 끊기면 다시 0 으로 내려가며 activator 가 재등장한다. 0↔N 을 오가는 문지기인 셈이다.

KCNA 는 activator/KPA 의 내부 구현까지는 안 묻는다 — "scale-to-zero 가 되고, 그 대가로 cold start 가 있다"만 알면 된다. 위 트레이스는 왜 그런지 이해를 위한 심화(그래서 심화 태그)다.

8. 회상 연습

Q1. "serverless"의 가장 정확한 설명은?

"서버가 없다"가 아니라 "서버를 신경 안 쓴다"이다. 서버는 존재하되 용량·스케일링을 내가 관리하지 않고, 요청/이벤트로 구동되며, scale-to-zero 로 유휴 시 0 이 되고, 쓴 만큼만 과금된다.

Q2. (레슨 0012 복습) Knative Serving 이 HPA 와 갈라지는 결정적 차이는?

HPA 는 "1 개와 N 개 사이"를, Knative 는 "0 개와 N 개 사이"를 오간다. 그 0 이 되느냐가 serverless 의 핵심선이다. (노드를 늘리는 건 Cluster Autoscaler — 0012.)

Q3. scale-to-zero 를 켠 Knative 서비스에서 오랜만에 온 첫 요청이 느린 이유는?

cold start: Pod 0 개 상태에서 activator 가 요청을 붙잡아 두고 오토스케일러가 0→1 로 올려 Pod 를 새로 띄운다. 그 스케줄·시작·초기화 시간만큼 첫 요청이 느려진다 — 유휴 비용 0 의 대가다. 이후 warm 상태에선 빠르다.
판정: Q1·Q2·Q3 를 맞히면 Architecture 도메인의 serverless 조각이 개념 수준으로 ✅ — "serverless 특성 · Knative 역할 · scale-to-zero · event-driven · cold start" 라는 KCNA 체크리스트가 채워진다. 다음은 0018 로 이어지고, scale-to-zero 의 비용 이야기는 0019 에서 매듭짓는다.

초보자 질문 (막히면 여기부터)

이름은 낯설어도 개념은 단순하다 — "안 쓰면 0". 궁금한 것만 펼쳐 보면 된다.

그래서 서버가 진짜로 없는 거예요, 있는 거예요?

있다. 어딘가의 노드에서 컨테이너가 돈다. 바뀌는 건 내가 그 서버를 신경 쓰느냐다 — "노드 몇 대 살까, replicas 몇으로 둘까, 유휴에 뭘 끌까"를 내가 안 하고 플랫폼에 맡긴다. "serverless"는 "server-less(서버 없음)"가 아니라 "server 관리에서 나를 free 하게"라고 읽는 게 맞다.

Knative 랑 그냥 Deployment(0002) 는 뭐가 달라요?

Deployment 는 replicas 만큼 Pod 를 계속 유지한다 — 요청이 없어도 그 수만큼 돈다(0009 의 "항상 도는 복제본"). Knative Service 는 요청량에 따라 자동 스케일하고, 특히 요청이 없으면 0 개까지 내린다. "항상 켜둠"과 "필요할 때만 켬"의 차이다.

HPA 로도 오토스케일 되잖아요? 왜 Knative 가 따로 필요해요?

HPA(0012)는 트래픽 대응은 잘 하지만 보통 minReplicas 가 1 이상이라 "완전히 끄기"를 못 한다. 항상 최소 1 개는 idle 비용을 먹는다. Knative 는 그 바닥을 0 으로 뚫는다 — 안 쓰는 시간엔 아무것도 안 돌아 유휴 비용이 0 이 된다. 그 대가가 cold start 다.

cold start 는 얼마나 느린 거예요? 몇 초씩 걸려요?

앱과 이미지에 따라 다르다 — 이미지가 노드에 캐시돼 있고 앱 초기화가 가벼우면 수백 ms, 무거운 앱·큰 이미지면 몇 초까지 간다. 핵심은 절대값이 아니라 트레이드오프다: "가끔 오는 트래픽에 유휴 비용 0"이 좋으면 감수하고, "매 요청이 빨라야" 하면 minScale=1 로 바닥을 올린다.

minScale=1 로 바닥을 올리면 그거 그냥 Deployment 아니에요?

거의 그렇다 — scale-to-zero 를 포기하면 "항상 1 개 이상"이라 HPA+Deployment 에 가까워진다. 그래도 Knative 는 요청 기반 스케일·리비전 관리·트래픽 분할 같은 기능은 남는다. 요점: scale-to-zero 는 켜고 끌 수 있는 선택지이지 강제가 아니다.

FaaS(Lambda)랑 Knative, 하나만 알아야 하면 뭘?

KCNA 는 구분만 물으면 충분하다: FaaS = 함수 단위(코드 핸들러를 올림, 대체로 벤더 종속), Knative = 컨테이너 단위(OCI 이미지를 그대로 serverless 화, CNCF 라 이식성 좋음). "둘 다 serverless 인데 배포 단위가 함수냐 컨테이너냐"가 한 줄 정답.

scale-to-zero 하려면 앱이 stateless 여야 한다는 게 무슨 말이에요?

Pod 를 아무 때나 0 으로 죽였다 되살리는 게 scale-to-zero 다. 만약 Pod 안(메모리·로컬 디스크)에 잃으면 안 되는 상태가 있으면, 0 으로 죽이는 순간 그게 날아간다. 그래서 상태는 밖(0011 의 PV, DB, 캐시)에 두고 Pod 는 언제 죽어도 되게(disposable, 0014 의 12-factor) 만들어야 한다. stateless 가 scale-to-zero 의 전제다.

Knative Eventing 은 Serving 이랑 뭐가 달라요?

Serving = HTTP 요청에 응답하고 요청량으로 스케일(0↔N). Eventing = 이벤트로 구동 — 소스(큐·타이머 등)가 이벤트를 내면 브로커가 받아 구독 조건에 맞는 서비스로 라우팅한다. Serving 이 "요청 오면 돈다"면 Eventing 은 "이벤트 오면 돈다". 느슨한 결합(0014)의 구현체다.

activator 가 정확히 뭐예요? 왜 필요해요?

Pod 가 0 개일 때 첫 요청을 대신 받아 붙잡아 두는 Knative 컴포넌트다. 없으면 Pod 0 개 상태의 요청은 그냥 실패한다. activator 가 요청을 버퍼에 잡아두고, 오토스케일러가 Pod 를 0→1 로 띄우는 동안 기다렸다, Pod 가 준비되면 넘겨준다. 덕분에 첫 요청은 "느릴 뿐 실패는 안 함". KCNA 깊이는 넘지만 scale-to-zero 를 이해하는 열쇠라 짚어둔다.

이거 실습으로 깔아봐야 하나요? KCNA 시험에 설치가 나와요?

아니다. Knative 설치·운영은 CKAD+ 영역(CRD·networking layer 구성)이라 KCNA 에선 안 나온다. KCNA 는 개념만 묻는다 — serverless 특성, Knative 의 역할, scale-to-zero, event-driven, cold start. 그래서 이 레슨도 손으로 안 깔고 "요청 0 → 응답" 여정을 개념 트레이스로만 따라갔다.
