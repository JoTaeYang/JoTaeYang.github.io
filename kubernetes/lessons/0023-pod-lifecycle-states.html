Kubernetes Mastery · Lesson 0023 Fundamentals 심화

Pod 생명주기 상태 머신 — phase · 컨테이너 상태 · restartPolicy

0004·0006·0009 에 흩어져 있던 Pending·OOMKilled·CrashLoopBackOff·Succeeded 를 하나의 상태 머신으로 꿴다. "Pod 는 지금 어느 칸에 있고, 왜 거기 있나?"

이번 레슨의 win: 지금까지 각 레슨에서 따로 마주친 실패·성공 상태들을 두 층의 상태 머신으로 정리한다 — 바깥층 Pod phase 5종(Pending·Running·Succeeded·Failed·Unknown)과 안층 컨테이너 상태 3종(Waiting·Running·Terminated). 그 위에 restartPolicy(0009 Job 에서 본 그것)와 probe(0004)가 얹혀 상태 전이를 만든다는 걸 손으로 확인한다. 이걸로 kubectl describe pod 의 State / Last State / Exit Code 세 줄을 정확히 읽게 된다.

0. 한눈에 — 두 층의 상태 머신

딱 이 두 줄

바깥층 = Pod phase(요약, 5종): 이 Pod 가 전체적으로 어느 단계인가. kubectl get pod 의 STATUS 뒤에 숨은 큰 그림.

안층 = 컨테이너 상태(디테일, 3종): 그 안의 각 컨테이너가 실제로 뭘 하고 있나. kubectl describe pod 이 보여주는 진짜 정보. CrashLoopBackOff·ImagePullBackOff·OOMKilled 는 전부 이 안층에서 벌어지는 일이다 — phase 가 아니라.

왜 두 층인가: Pod 는 컨테이너를 여러 개 담을 수 있다(0002). 그래서 "Pod 전체"의 상태(phase)와 "그 안 각 컨테이너"의 상태를 분리한다. 컨테이너 하나가 재시작 중이어도 Pod phase 는 여전히 Running일 수 있다 — 이 어긋남이 초보자를 가장 많이 헷갈리게 한다. 이번 레슨의 핵심이 바로 이 어긋남이다.

1. 바깥층 — Pod phase 5종

.status.phase 는 Pod 의 생애 요약이다. 딱 5개뿐이고, 그중 둘(Succeeded·Failed)은 터미널(끝난 상태, 더 안 바뀜)이다:

phase뜻어디서 봤나
Pending API 서버가 수락은 했지만 아직 컨테이너가 다 못 떴다 — 스케줄링 대기 또는 이미지 pull 대기 0006 requests 가 노드 여유를 초과 → Pending. 0012 taint 로 배치 못 함도 여기
Running 노드에 바인딩됐고 최소 한 컨테이너가 실행 중(또는 시작·재시작 중) 정상 상태. 하지만 안에서 컨테이너가 재시작 중일 수도 있다(2절)
Succeeded (터미널) 모든 컨테이너가 정상 종료(exit 0)했고 재시작 안 함. 끝 0009 Job 이 완료되면 그 Pod 가 여기로
Failed (터미널) 모든 컨테이너가 종료됐고 최소 하나가 비정상 종료(exit ≠ 0)했으며 재시작 안 함. 끝 0009 Job 이 backoffLimit 을 다 쓰고 실패하면 여기
Unknown Pod 상태를 알 수 없음 — 보통 노드와 통신이 끊김(kubelet 무응답) 노드 장애 시. 드물다
아하 — Succeeded/Failed 는 "재시작이 없을 때"만 나온다 Pod 가 Succeeded·Failed 라는 터미널에 도달하려면 컨테이너가 종료된 채 다시 살아나지 않아야 한다. 그런데 재시작 여부를 결정하는 게 바로 restartPolicy(3절)다. 그래서 phase 5종은 restartPolicy 없이는 반쪽짜리 — restartPolicy: Always인 Pod 는 컨테이너가 죽어도 계속 되살아나 절대 Failed 에 안 닿고 Running 을 맴돈다. 이게 0009 Job 이 왜 restartPolicy: Always를 못 쓰는지의 이유다(3절에서 연결).

2. 안층 — 컨테이너 상태 3종 (디테일이 여기 있다)

Pod 안 각 컨테이너는 항상 셋 중 하나다. kubectl describe pod 의 State: 줄이 이걸 보여준다. 실무에서 트러블슈팅하는 정보는 거의 다 여기다:

컨테이너 상태뜻 · 대표 reason
Waiting 아직 실행 못 함. reason 이 왜인지 말해준다:
· ContainerCreating — 정상, 이미지 pull·마운트 중
· ErrImagePull / ImagePullBackOff — 이미지 못 가져옴(오타·권한, 0006)
· CrashLoopBackOff — 계속 죽어서 재시작 백오프 중(4절)
Running 실제로 프로세스가 돌고 있음. Started 시각이 찍힌다
Terminated 실행 끝남. reason + exit code 가 왜·어떻게 끝났는지 말해준다:
· Completed (exit 0) — 정상 종료
· OOMKilled (exit 137) — 메모리 limit 초과, 커널이 죽임(0006)
· Error (exit ≠ 0) — 앱이 비정상 종료
Last State — 왜 지금 이 꼴인지의 단서 describe 는 State(지금)뿐 아니라 Last State(직전 종료 기록)도 보여준다. 컨테이너가 지금 Waiting: CrashLoopBackOff이면 "지금은 재시작 대기 중"이라는 뜻이고, 왜 죽는지는 Last State 의 Terminated · Exit Code: 137 같은 줄에 있다. "State 로 지금을, Last State 로 왜를 읽는다" — 이 두 줄 조합이 Pod 디버깅의 핵심 습관이다.
phase 와 컨테이너 상태의 매핑: 컨테이너가 Waiting: ContainerCreating이면 보통 Pod phase 는 Pending. 컨테이너가 Running이거나 Waiting: CrashLoopBackOff(재시작을 전제로 살아있음)면 Pod phase 는 Running. 그래서 kubectl get pod 에 STATUS: CrashLoopBackOff로 뜨는 건 편의상 컨테이너 reason 을 끌어올려 보여주는 것일 뿐, 진짜 phase 는 여전히 Running이다.

3. restartPolicy 3종 — phase 를 좌우하는 스위치

Pod 스펙의 restartPolicy 는 "컨테이너가 종료되면 되살릴까?"를 정한다. 3종이고, 0009 에서 Job 이 왜 특정 값을 쓰는지가 여기서 풀린다:

restartPolicy동작어디서
Always (기본) exit code 0이든 1이든 무조건 재시작. 그래서 Pod 는 계속 Running — 터미널에 안 닿음 Deployment 가 관리하는 Pod. "서비스는 계속 떠 있어야" 하니까
OnFailure exit ≠ 0일 때만 재시작. exit 0이면 그대로 Succeeded 0009 Job 에 흔함 — "성공할 때까지 재시도, 성공하면 끝"
Never 재시작 안 함. exit 0 → Succeeded, exit ≠ 0 → Failed (한 번에 터미널) 일회성 작업, 결과를 그대로 보고 싶을 때
아하 — exit 1 하나가 policy 에 따라 세 갈래로 같은 "exit 1으로 죽는 컨테이너"라도: Never → 재시작 없이 Pod phase Failed (한 방에 터미널, 실습 a). OnFailure/Always → 재시작 → 또 죽음 → 재시작… → CrashLoopBackOff(4절)로 빠지며 Pod 는 Running 유지(실습 b). Deployment 가 Always 를 기본으로 쓰기 때문에 우리가 흔히 보는 크래시 증상이 대부분 CrashLoopBackOff 인 것이다. 그리고 Job 이 Always 를 금지하는 이유도 이제 명확 — Always 면 exit 0 후에도 재시작해 버려 "작업 완료"라는 개념 자체가 성립 안 한다(0009).

4. CrashLoopBackOff 의 정체 — 상태 머신 위의 백오프

0006 에서 이름만 스쳤던 CrashLoopBackOff 를 이제 정확히 놓는다. 이건 에러가 아니라 kubelet 의 재시작 전략이다:

일어나는 일
Crash컨테이너가 시작하자마자(또는 곧) exit ≠ 0으로 죽는다 → Terminated: Error
LooprestartPolicy(Always/OnFailure)가 재시작 → 또 죽음 → 또 재시작…
BackOffkubelet 이 재시작 간격을 지수적으로 늘린다: 10s → 20s → 40s … 최대 5분(300s)에서 상한. 그 대기 동안 컨테이너는 Waiting: CrashLoopBackOff
왜 backoff 를 두나: 즉시 무한 재시작하면 계속 죽는 컨테이너가 노드 CPU 를 태우고 로그를 폭주시킨다. 간격을 벌려 "숨 쉴 틈"을 준다. 그래서 CrashLoopBackOff 라는 글자를 봤다는 건 "이미 여러 번 죽어서 kubelet 이 재시작을 미루는 중"이라는 뜻 — 원인은 Last State 의 exit code·kubectl logs 에 있지, 이 글자 자체엔 없다.
혼동 주의: CrashLoopBackOff 는 컨테이너 상태(Waiting)지 Pod phase 가 아니다. 그리고 ImagePullBackOff 와 헷갈리지 마라 — 전자는 컨테이너가 뜬 뒤 죽어서 재시작 백오프(원인은 앱·설정), 후자는 애초에 이미지를 못 가져와서 pull 백오프(원인은 이미지 이름·레지스트리 권한, 0006). "돌다 죽나 vs 뜨지도 못하나"로 가른다.

5. probe 는 이 상태 머신 위에 얹힌다 (0004 연결)

0004 의 liveness/readiness probe 를 이 상태 머신 위에 올려 보면 각 probe 가 어느 전이를 건드리는지가 선명해진다:

probe실패하면 상태 머신에서 무슨 일
liveness kubelet 이 컨테이너를 죽이고 재시작한다 → Running → Terminated → Waiting → Running. 계속 실패하면 CrashLoopBackOff 로 빠진다 — probe 가 만든 크래시 루프
readiness 재시작 안 함. 컨테이너는 Running 그대로지만 Pod 가 Ready 아님으로 표시돼 Service endpoint 에서 빠진다(0004·0013) — 트래픽만 끊고 목숨은 안 건드림
한 줄 구분 liveness = 목숨(재시작), readiness = 트래픽(endpoint 포함/제외). liveness 실패는 Terminated→Waiting 전이를 일으켜 상태 머신을 돌리고, readiness 실패는 상태 머신을 안 건드리고 Running 유지한 채 0013 의 Service 뒤에서 빠질 뿐이다. probe 를 "새 개념"이 아니라 "상태 전이를 트리거하는 버튼"으로 보면 0004 가 이 레슨과 한 몸이 된다.

6. Hands-on — 상태 전이를 눈으로 (kind mastery)

kind 클러스터 mastery 이어서 사용. 0006·0020 처럼 lab namespace 에서 격리해 실습한다. 일부러 exit 1로 죽는 컨테이너를 만들어 restartPolicy 만 바꿔가며 phase 가 어떻게 갈리는지 본다.
terminal $ kubectl create namespace lab 2>/dev/null; kubectl config set-context --current --namespace=lab

(a) restartPolicy: Never — 한 번 죽고 Failed 로 직행. heredoc 으로 바로 apply:

terminal $ kubectl apply -f - <<'EOF' apiVersion: v1 kind: Pod metadata: name: die-never spec: restartPolicy: Never containers: - name: app image: busybox:1.36 command: ["sh", "-c", "echo booting; sleep 2; exit 1"] EOF pod/die-never created $ sleep 8; kubectl get pod die-never NAME READY STATUS RESTARTS AGE die-never 0/1 Failed 0 8s $ kubectl get pod die-never -o jsonpath='{.status.phase}{"\n"}' Failed

RESTARTS 0, phase Failed — Never 는 재시작 없이 한 번에 터미널에 닿았다(3절 표 그대로).

(b) restartPolicy: Always — 같은 컨테이너가 CrashLoopBackOff 로. -w로 전이를 실시간 관찰:

terminal $ kubectl apply -f - <<'EOF' apiVersion: v1 kind: Pod metadata: name: die-always spec: restartPolicy: Always containers: - name: app image: busybox:1.36 command: ["sh", "-c", "echo booting; sleep 2; exit 1"] EOF pod/die-always created $ kubectl get pod die-always -w NAME READY STATUS RESTARTS AGE die-always 0/1 Running 0 3s die-always 0/1 Error 0 5s # exit 1 → Terminated: Error die-always 0/1 Running 1 6s # Always 가 재시작 (RESTARTS 1) die-always 0/1 Error 1 11s die-always 0/1 CrashLoopBackOff 1 24s # 반복되자 백오프 진입 die-always 0/1 Running 2 38s # backoff 만료 후 또 재시작…

Ctrl-C 로 watch 종료. 같은 컨테이너인데 Always 라서 재시작을 반복하며 CrashLoopBackOff 로 빠졌다. phase 는?

terminal $ kubectl get pod die-always -o jsonpath='{.status.phase}{"\n"}' Running

phase 는 여전히 Running이다(2절의 어긋남!) — get pod 의 STATUS 칸에 뜬 CrashLoopBackOff 는 컨테이너 reason 을 끌어올려 보여준 것일 뿐.

(c) describe 로 State / Last State / Exit Code 읽기 — "지금"과 "왜"를 분리해 읽는다:

terminal $ kubectl describe pod die-always | grep -A 12 'State:' State: Waiting Reason: CrashLoopBackOff # 지금: 재시작 대기 중 Last State: Terminated Reason: Error Exit Code: 1 # 왜: 직전에 exit 1로 죽음 Started: Mon, 07 Jul 2026 09:12:03 +0900 Finished: Mon, 07 Jul 2026 09:12:05 +0900 Ready: False Restart Count: 3
읽는 법 State: Waiting / CrashLoopBackOff = 지금은 백오프 대기(4절). Last State: Terminated / Exit Code 1 = 왜 여기 왔는지 = 앱이 exit 1로 죽는다. 다음 수사는 kubectl logs die-always --previous(직전 컨테이너 로그)로 왜 exit 1인지를 본다. 만약 여기 Exit Code: 137·Reason: OOMKilled였다면? 그건 0006·0020 의 메모리 limit 초과 크래시 루프다 — 같은 자리를 읽어 원인만 다르게 진단한다.

정리: kubectl delete ns lab && kubectl config set-context --current --namespace=default.

예고(0024): 컨테이너가 어떻게 종료되는가 — SIGTERM → grace period → SIGKILL, 그리고 PID 1 이 신호를 안 받으면 왜 kubectl delete 가 느려지는지 — 는 다음 레슨 0024(종료 & graceful shutdown)에서 이 상태 머신의 Running → Terminated 전이를 확대해 다룬다. 이번 레슨은 "어느 칸에 있나"까지.

7. 회상 연습

Q1. 컨테이너가 exit 1로 죽는 Pod 를 restartPolicy: Never로 만들었다. 잠시 뒤 Pod 의 .status.phase 는?

Never 는 재시작을 안 한다. 컨테이너가 exit≠0으로 끝나면 Pod 는 곧장 터미널 phase Failed. CrashLoopBackOff 는 재시작하는(Always/OnFailure) policy 에서만 나온다 — Never 엔 루프 자체가 없다.

Q2. kubectl get pod 에 STATUS: CrashLoopBackOff로 뜨는 Pod 의 실제 .status.phase 는 무엇이며, 그 이유는?

두 층을 가르는 문제. CrashLoopBackOff 는 컨테이너 상태 Waiting 의 reason이지 phase 가 아니다. restartPolicy 가 재시작을 계속 시도하므로 터미널(Failed)에 안 닿고 phase 는 Running — get 의 STATUS 칸이 편의상 컨테이너 reason 을 끌어올려 보여줄 뿐이다.

Q3. (레슨 0004·0006 복습) describe pod 의 Last State 가 Terminated · Reason: OOMKilled · Exit Code: 137, 현재 State 는 Waiting · CrashLoopBackOff다. 무슨 일이 일어나는 중인가?

간격 반복: Exit Code 137 · OOMKilled 는 0006 의 memory limit 초과(커널이 죽임)다. 그게 반복되니 0006 의 CrashLoopBackOff 로 빠진 것. ImagePullBackOff 는 뜨지도 못한 경우라 Terminated 기록이 없고, readiness 실패는 재시작을 안 시켜 CrashLoopBackOff 가 안 나온다. "Last State 로 왜, State 로 지금"을 읽는 습관 그대로다.
판정: Q1·Q2·Q3 를 맞히면 phase 5종 · 컨테이너 상태 3종 · restartPolicy 3종이 하나의 상태 머신으로 ✅ — 0004·0006·0009 에 흩어졌던 Pending·OOMKilled·CrashLoopBackOff·Succeeded 를 이제 "어느 층, 어느 칸, 왜"로 정확히 귀속시킬 수 있다. 다음 0024 에서 Running → Terminated 전이(종료 신호)를 확대한다.

초보자 질문 (막히면 여기부터)

phase 와 컨테이너 상태가 두 층이라는 게 핵심이자 함정이다. 궁금한 것만 펼쳐 보면 된다.

Pod phase 랑 컨테이너 상태, 결국 같은 말 아닌가요?

다르다. phase(5종)는 Pod 전체의 큰 요약이고, 컨테이너 상태(3종)는 그 안 각 컨테이너의 실제 상태다. Pod 하나에 컨테이너가 여럿일 수 있어(0002) 두 층으로 나눈다. 트러블슈팅 정보(왜 죽나, exit code)는 거의 다 컨테이너 상태 층에 있다 — get 은 phase 를 요약해 보여주고, describe 는 컨테이너 상태를 낱낱이 보여준다.

kubectl get pod 의 STATUS 에 뜨는 값이 phase 가 아니라고요?

항상 phase 인 건 아니다. Running·Pending·Succeeded·Failed 는 phase 그대로지만, CrashLoopBackOff·ImagePullBackOff·ContainerCreating 같은 건 컨테이너 상태의 reason을 kubectl 이 편의상 끌어올려 STATUS 칸에 보여주는 것이다. 진짜 phase 를 보려면 kubectl get pod <name> -o jsonpath='{.status.phase}' 로 직접 뽑는다 — 실습 (b) 에서 CrashLoopBackOff 인데 phase 가 Running 인 걸 확인했다.

restartPolicy 를 나중에 바꿀 수 있나요?

Pod 의 restartPolicy 는 immutable(생성 후 변경 불가) 필드다. 바꾸려면 Pod 를 delete 하고 새 값으로 다시 만들어야 한다 — Pod 는 ephemeral(0002)이라 재생성이 자연스럽다. 실무에선 보통 Deployment(Always 고정)나 Job(OnFailure/Never)이 이 값을 대신 관리하므로 직접 만질 일은 드물다.

왜 Deployment 는 restartPolicy 를 Always 로만 쓰나요?

Deployment 는 "이 서비스를 계속 떠 있게" 하는 게 목적이라, 컨테이너가 무슨 이유로 죽든 되살려야 한다 → Always. 실제로 Deployment 의 Pod 템플릿엔 Always 외의 값을 넣을 수 없다(검증에서 거부). 반대로 0009 의 Job 은 "끝나야 하는 작업"이라 Always 를 금지하고 OnFailure/Never 만 허용한다 — Always 면 exit 0 후에도 재시작해 "완료"가 불가능해지기 때문.

CrashLoopBackOff 가 떴어요. 뭘 먼저 봐야 하나요?

글자 자체엔 원인이 없다 — "여러 번 죽어서 재시작을 미루는 중"이라는 뜻일 뿐이다. 순서: (1) kubectl describe pod <name> 의 Last State 의 Reason·Exit Code 로 어떻게 죽었는지 본다(예: 137=OOMKilled, 0006). (2) kubectl logs <name> --previous 로 직전 컨테이너가 남긴 에러 로그를 본다(--previous 가 핵심 — 지금 컨테이너는 아직 로그가 없을 수 있다). 이 두 개면 대부분 원인이 잡힌다.

backoff 간격은 정확히 얼마씩 늘어나나요?

kubelet 은 재시작 대기를 지수적으로 늘린다: 대략 10s → 20s → 40s → 80s … 식으로 배가되다 최대 5분(300s)에서 상한에 걸린다. 컨테이너가 일정 시간(기본 10분) 안정적으로 잘 돌면 이 카운터가 리셋된다. KCNA/CKAD 는 "지수 백오프, 최대 5분, 재시작 전략이지 에러가 아님"까지면 충분하고 정확한 숫자를 외울 필요는 없다.

Exit Code 137 은 왜 137이에요? 다른 숫자도 있나요?

리눅스에서 신호로 죽으면 exit code = 128 + 신호번호다. SIGKILL(9)로 죽으면 128+9=137 — OOM killer 가 SIGKILL 을 쓰기 때문에 OOMKilled 가 137로 나온다(0006·0020). SIGTERM(15)로 죽으면 128+15=143. 앱이 스스로 exit 1 하면 그냥 1(Reason: Error). 이 규칙을 알면 exit code 만 보고 "신호로 죽었나 스스로 죽었나"를 가른다.

liveness probe 랑 CrashLoopBackOff 는 무슨 관계예요?

0004 의 liveness probe 가 실패하면 kubelet 이 그 컨테이너를 죽이고 재시작한다. probe 가 계속 실패하면 이 "죽이고→재시작"이 반복돼 결국 CrashLoopBackOff 로 빠진다 — 즉 앱이 스스로 안 죽어도 probe 가 크래시 루프를 만들 수 있다. 반대로 readiness probe 실패는 재시작을 안 시키니 CrashLoopBackOff 를 안 만든다(그냥 endpoint 에서 빠짐). "CrashLoopBackOff 인데 앱은 멀쩡한 것 같다"면 liveness 설정(경로·타임아웃)을 의심하라.

readiness 실패면 Pod 상태가 어떻게 보이나요?

phase 는 Running 그대로, 컨테이너 State 도 Running 이다 — 목숨은 멀쩡하다. 다만 kubectl get pod 의 READY 칸이 0/1 로 뜨고, 그 Pod 는 0013 의 Service endpoint 목록에서 빠진다(트래픽이 안 감). "STATUS 는 Running 인데 READY 가 0/1" = 살아 있지만 트래픽은 안 받는 상태 = readiness 신호. liveness(재시작) 와 정확히 반대다.

Unknown phase 는 언제 보나요? 위험한가요?

보기 드물다. Pod 가 뜬 노드의 kubelet 이 API 서버와 통신이 끊겼을 때(노드 다운·네트워크 단절) 컨트롤 플레인이 상태를 못 받아 Unknown 으로 표시한다. 즉 "Pod 가 나빠서"가 아니라 "노드 소식을 못 들어서"다. 컨트롤러가 관리하는 Pod 라면 일정 시간 뒤 다른 노드에 대체 Pod 를 만든다(0002·0007 의 self-healing). KCNA 는 "Unknown = 노드 통신 두절"만 알면 충분.
