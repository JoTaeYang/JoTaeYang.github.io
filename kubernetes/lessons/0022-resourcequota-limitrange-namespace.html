Kubernetes Mastery · Lesson 0022 Fundamentals 심화

namespace 예산: ResourceQuota 와 LimitRange

0006 이 Pod 하나의 예산이었다면, 이번엔 그 위층 — namespace 전체의 예산 장부다. 그리고 "namespace = 격리"라는 흔한 오해를 손으로 바로잡는다.

이번 레슨의 win: 새 namespace 에 ResourceQuota 와 LimitRange 를 걸고, (a) requests 를 안 적은 Pod 에 LimitRange 가 기본값을 자동으로 채워주는 것과 (b) quota 를 넘긴 Pod 가 생성 자체에서 거부되는 것을 kind 클러스터에서 직접 관찰한다. "namespace 는 네트워크 격리가 아니다"라는 오해도 여기서 못을 박는다.

1. 층을 다시 세운다 — Pod 예산 위의 namespace 예산

0006 에서 배운 requests(예약)와 limits(상한)는 Pod 하나의 예산이었다. 그런데 한 팀이 Pod 를 100 개 만들면? 각 Pod 는 얌전해도 팀 전체가 클러스터를 삼킨다. 그래서 그 위층이 필요하다 — namespace 단위의 총량 장부.

층대상누가 거는가레슨
requests/limits컨테이너 하나Pod 작성자(개발자)0006
LimitRangenamespace 안 개별 Pod/컨테이너의 기본값·경계플랫폼/관리자이 레슨
ResourceQuotanamespace 전체의 합계 상한플랫폼/관리자이 레슨
아하 — 예산은 두 방향으로 잰다 ResourceQuota = 위에서 내려다본 총합("이 팀 전부 합쳐 CPU 10 코어, memory 20Gi, Pod 20 개까지"). LimitRange = 아래에서 하나씩 본 경계("Pod 하나당 최대 2 코어, 안 적으면 기본 requests 를 채워줌"). 같은 namespace 를 총량과 낱개 두 각도로 재는 것 — 층이 다르니 헷갈리지 말 것.

2. namespace 복습 — 그리고 가장 큰 오해

0006 에서 실습을 lab namespace 에 격리했고, 0008 의 RBAC 도 namespace 스코프였다. namespace 는 클러스터를 논리적으로 나누는 가상 파티션이다 — 이름 충돌을 막고, 권한(RBAC)과 예산(Quota)을 팀별로 나누는 경계선. 이걸 소프트 멀티테넌시라 부른다.

흔한 오해 — namespace 는 네트워크/보안 격리가 아니다. 0013 에서 본 "평평한(flat) 네트워크"를 떠올려라: 기본 상태에서 team-a namespace 의 Pod 는 team-b namespace 의 Pod 에 그냥 접속된다. namespace 는 이름과 권한과 예산을 나눌 뿐, 패킷을 막지 않는다. 진짜 네트워크 격리는 NetworkPolicy(0013 에서 씨를 뿌렸고, KCSA·Level 2 에서 본격화)가 필요하다. "namespace 를 나눴으니 안전하다"는 시험·실무 양쪽의 단골 함정이다.

3. ResourceQuota — namespace 총량 상한

ResourceQuota 는 한 namespace 안 모든 Pod 를 합친 값에 천장을 씌운다. 세 종류를 제한한다:

제한 대상필드 예뜻
requests 합계requests.cpu, requests.memorynamespace 안 모든 컨테이너 requests 의 총합 상한
limits 합계limits.cpu, limits.memory모든 컨테이너 limits 의 총합 상한
오브젝트 개수pods, services, configmaps…만들 수 있는 리소스 개수 상한
함정 — Quota 가 requests 를 재면, Pod 는 requests 를 반드시 적어야 한다. namespace 에 requests.cpu quota 가 걸리면, 쿠버네티스는 새 Pod 의 requests 합을 장부에 더해야 하는데 requests 를 안 적은 Pod 는 얼마를 더할지 몰라 아예 생성을 거부한다 (must specify requests.cpu). 그래서 Quota 는 보통 LimitRange 와 짝으로 쓴다 — LimitRange 가 기본 requests 를 채워주면 개발자가 매번 안 적어도 통과한다. 이 짝을 다음 절과 실습에서 본다.

4. LimitRange — 개별 Pod 의 기본값과 경계

LimitRange 는 그 namespace 안 낱개 컨테이너에 두 가지를 한다:

필드역할
defaultRequestrequests 를 안 적은 컨테이너에 자동으로 채울 requests 기본값
defaultlimits 를 안 적은 컨테이너에 자동으로 채울 limits 기본값
min / max허용하는 requests/limits 의 하한·상한 — 벗어나면 생성 거부
아하 — 이 둘이 짝인 이유가 여기서 맞물린다 Quota 는 "requests 를 안 적으면 거부"하고, LimitRange 는 "requests 를 안 적으면 채워준다". 그래서 LimitRange 가 Quota 보다 먼저 개발자의 Pod 에 기본값을 주입하고 → 채워진 값으로 Quota 장부에 더한다. 개발자는 image 만 적은 얌전한 Pod 를 던져도, 관리자가 정한 기본 예산이 자동으로 붙어 장부에 잡히는 것. 0020 의 QoS 와도 연결된다 — LimitRange 가 requests 와 limits 를 같게 채우면 그 Pod 는 Guaranteed, requests 만 채우고 limits 가 비면 Burstable 이 된다. 기본값 설계가 QoS 클래스를 좌우한다.

5. 실습 — 짝을 kind 에서 직접 굴린다

kind 클러스터 mastery 이어서 사용. 이번엔 team-a namespace 를 하나 만들어 예산을 걸고, 두 가지를 관찰한다: (a) 기본값 주입, (b) quota 초과 거부. 정리는 마지막에 namespace 하나 지우는 걸로 끝난다(0006 의 격리 습관).

1) namespace + ResourceQuota + LimitRange 를 한 번에 apply (heredoc):

terminal $ kubectl create namespace team-a $ kubectl apply -n team-a -f - <<'EOF' apiVersion: v1 kind: ResourceQuota metadata: name: team-a-quota spec: hard: requests.cpu: "1" # 팀 전체 requests 합 최대 1 코어 requests.memory: 1Gi limits.cpu: "2" limits.memory: 2Gi pods: "3" # Pod 최대 3 개 --- apiVersion: v1 kind: LimitRange metadata: name: team-a-limits spec: limits: - type: Container defaultRequest: # requests 를 안 적으면 이걸 채움 cpu: 200m memory: 128Mi default: # limits 를 안 적으면 이걸 채움 cpu: 500m memory: 256Mi min: cpu: 100m memory: 64Mi max: # 이보다 크면 생성 거부 cpu: "1" memory: 512Mi EOF resourcequota/team-a-quota created limitrange/team-a-limits created

2) 관찰 (a) — requests 를 안 적은 Pod 에 기본값이 채워진다:

terminal $ kubectl run bare -n team-a --image=nginx:1.27 # resources 를 하나도 안 적었다 pod/bare created $ kubectl get pod bare -n team-a -o jsonpath='{.spec.containers[0].resources}'; echo {"limits":{"cpu":"500m","memory":"256Mi"},"requests":{"cpu":"200m","memory":"128Mi"}}

개발자는 --image 만 줬는데, LimitRange 가 defaultRequest/default 를 조용히 채웠다. requests 와 limits 가 다르므로 이 Pod 의 QoS 는 Burstable(0020) — kubectl get pod bare -n team-a -o jsonpath='{.status.qosClass}' 로 확인할 수 있다.

3) 관찰 (b) — quota 를 넘기면 생성이 거부된다. 장부부터 본다:

terminal $ kubectl describe resourcequota team-a-quota -n team-a Name: team-a-quota Resource Used Hard -------- ---- ---- limits.cpu 500m 2 limits.memory 256Mi 2Gi pods 1 3 requests.cpu 200m 1 requests.memory 128Mi 1Gi $ # requests.memory 400Mi 짜리 Pod 5 개를 시도 → memory 총합(1Gi)·pods(3) 두 벽에 부딪힌다 $ kubectl apply -n team-a -f - <<'EOF' apiVersion: v1 kind: Pod metadata: name: big spec: containers: - name: app image: nginx:1.27 resources: requests: memory: 900Mi # 이미 128Mi 쓴 장부에 900Mi → 1Gi 초과 EOF Error from server (Forbidden): error when creating "STDIN": pods "big" is forbidden: exceeded quota: team-a-quota, requested: requests.memory=900Mi, used: requests.memory=128Mi, limited: requests.memory=1Gi

Pod 가 Pending 이 되는 게 아니라 생성 자체가 거부됐다(0006 의 Pending 과 다른 층 — 스케줄러까지 가지도 못하고 API 서버가 admission 단계에서 막는다). 에러 메시지가 친절히 used / limited 를 보여준다.

Quota 인데 LimitRange 를 지웠다면? LimitRange 없이 requests.cpu quota 만 걸린 namespace 에서 kubectl run 으로 맨몸 Pod 를 만들면 Error ... must specify requests.cpu 로 거부된다. 바로 4 절의 함정 — 짝이 깨지면 개발자가 매번 requests 를 손으로 적어야 한다.

정리: kubectl delete namespace team-a — namespace 를 지우면 Quota·LimitRange·Pod 가 통째로 사라진다.

6. 왜 중요한가 — 비용 가드레일과 멀티테넌시

ResourceQuota 는 곧 팀별 예산 가드레일이다. 0019 의 비용 관리 관점에서 보면, quota 는 "이 팀은 클러스터의 이만큼만 쓴다"를 강제로 못박는 장치 — 한 팀의 폭주가 다른 팀의 Pod 를 굶기지 못하게 막는 소프트 멀티테넌시의 핵심이다. LimitRange 는 그 안에서 개발자가 예산을 안 적어도 합리적 기본값이 붙게 해 QoS(0020) 와 장부(Quota) 양쪽을 동시에 만족시킨다.

KCNA 깊이 — 딱 세 줄 namespace = 소프트 멀티테넌시(이름·권한·예산의 논리적 파티션, 네트워크 격리 아님). ResourceQuota = namespace 총량(합계 requests/limits·개수 상한). LimitRange = 개별 기본값·경계(default·min·max). 그 이상(스코프 셀렉터, priority class quota 등)은 심화 몫.

7. 회상 연습

Q1. requests.cpu quota 가 걸린 namespace 에 requests 를 안 적은 Pod 를 만들었더니 거부됐다. 이 거부를 없애는 가장 알맞은 방법은?

Quota 가 requests 를 재면 requests 없는 Pod 는 "얼마를 장부에 더할지 몰라" 거부된다. LimitRange 의 defaultRequest 가 기본값을 채워주면 개발자가 매번 안 적어도 통과한다 — 둘이 짝으로 쓰이는 바로 그 이유다.

Q2. team-a 와 team-b 두 namespace 로 앱을 나눴다. 기본 상태에서 team-b 의 Pod 는 team-a 의 Pod 에 네트워크로 접근할 수 있나?

0013 의 평평한 네트워크: namespace 는 이름·권한·예산을 나눌 뿐 패킷을 막지 않는다. 진짜 네트워크 격리는 NetworkPolicy(KCSA·Level 2)의 몫이다. "namespace = 격리"는 대표적 함정.

Q3. ResourceQuota 와 LimitRange 의 관계로 옳은 것은?

Quota = 위에서 내려다본 총합(팀 전체 CPU·memory·개수), LimitRange = 아래에서 하나씩 본 경계(Pod 당 default·min·max). 같은 namespace 를 총량과 낱개 두 각도로 재는, 서로 다른 층의 도구다.
판정: Q1·Q2·Q3 를 맞히면 namespace 예산 층(총량 Quota + 낱개 LimitRange)과 "namespace ≠ 네트워크 격리" 오해까지 ✅ — 0006 위에 한 층을 올렸다. 다음 단계는 그 못 박은 오해를 실제로 푸는 곳, 즉 NetworkPolicy(Level 2 · KCSA)다.

초보자 질문 (막히면 여기부터)

예산 두 도구가 층이 달라 헷갈리기 쉽다. 궁금한 것만 펼쳐 보면 된다.

ResourceQuota 랑 LimitRange, 한 문장으로 구분하면?

ResourceQuota = namespace 전체의 합계 천장("팀 다 합쳐 CPU 10 코어까지"), LimitRange = Pod 하나당 기본값·경계("안 적으면 200m 채우고, 2 코어 넘으면 거부"). 총량 대 낱개 — 층이 다르다. 보통 짝으로 같이 건다.

namespace 만 나누면 팀끼리 격리되는 거 아닌가요?

절반만 맞다. namespace 는 이름·권한(RBAC)·예산(Quota)을 나눈다 — 이걸 소프트 멀티테넌시라 한다. 하지만 네트워크는 안 나뉜다: 기본 상태에서 다른 namespace 의 Pod 끼리 그냥 통신된다(0013 평평한 네트워크). 패킷을 막으려면 NetworkPolicy 가 별도로 필요하다.

왜 Quota 만 걸면 Pod 가 거부되나요?

Quota 가 requests.cpu 를 재기 시작하면, 새 Pod 마다 requests 를 장부에 더해야 한다. 그런데 requests 를 안 적은 Pod 는 더할 값이 없어서 쿠버네티스가 "requests.cpu 를 명시하라"며 거부한다. LimitRange 의 defaultRequest 를 같이 걸면 기본값이 채워져 이 문제가 사라진다.

default 랑 defaultRequest 는 뭐가 달라요?

defaultRequest 는 requests 를 안 적었을 때 채우는 값(예약), default 는 limits 를 안 적었을 때 채우는 값(상한)이다. 0006 의 requests/limits 두 얼굴이 그대로 이어진다 — LimitRange 는 그 둘의 기본값을 각각 정한다.

LimitRange 의 min/max 는 requests 를 재나요 limits 를 재나요?

type: Container LimitRange 에서 min/max 는 requests 와 limits 둘 다의 허용 범위다. 예컨대 max.cpu: "1" 이면 requests 든 limits 든 1 코어를 넘기려 하면 생성이 거부된다. 너무 크거나 작은 값으로 클러스터를 어지럽히지 못하게 하는 가드레일이다.

quota 초과면 Pod 가 Pending 되나요?

아니다. 생성 자체가 거부된다(Error from server (Forbidden): exceeded quota). 이건 API 서버의 admission 단계에서 막히는 것 — 스케줄러까지 가지도 못한다. 0006 의 Pending(스케줄러가 노드를 못 찾음)과는 완전히 다른 층의 실패다.

LimitRange 기본값이 QoS 클래스에 영향을 주나요?

준다. LimitRange 가 defaultRequest 와 default 를 같은 값으로 채우면 그 Pod 는 Guaranteed, requests 만 채우고 limits 가 비면 Burstable(0020)이 된다. 즉 관리자의 기본값 설계가 개발자 모르게 QoS 등급을 정하는 셈 — 축출 우선순위까지 좌우한다.

ResourceQuota 로 Pod 개수도 제한할 수 있나요?

있다. pods: "20" 처럼 오브젝트 개수도 quota 대상이다. CPU·memory 뿐 아니라 services, configmaps, persistentvolumeclaims(0011 의 PVC) 개수까지 제한할 수 있다. 한 팀이 리소스를 무한정 찍어내는 걸 막는 용도다.

RBAC(0008)랑 ResourceQuota 는 같은 건가요?

다르다. 둘 다 namespace 스코프라 헷갈리지만, RBAC 는 누가 무엇을 할 수 있나(권한), ResourceQuota 는 얼마나 쓸 수 있나(예산)를 정한다. 팀 namespace 를 줄 때 보통 셋을 같이 건다 — RBAC(권한 경계) + Quota(예산 경계) + (원하면) NetworkPolicy(네트워크 경계).

이미 quota 를 초과 중인 namespace 에 quota 를 더 낮추면 어떻게 되나요?

기존 Pod 는 죽지 않는다 — quota 는 새 리소스 생성 시점에만 검사한다. 다만 그 상태에선 Used 가 Hard 를 넘은 채로 남아, 새 Pod 는 계속 거부된다. kubectl describe resourcequota 로 Used/Hard 를 보며 조정한다.

Primary Source 공식 문서: Resource Quotas — namespace 총량 제한(requests/limits 합계·오브젝트 개수)의 원전. Limit Ranges 로 default/min/max 를, Namespaces 로 "논리적 파티션·소프트 멀티테넌시"의 정의를 확인할 수 있다.
