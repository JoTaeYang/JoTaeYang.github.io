Kubernetes Mastery · Lesson 0006 Level 1 · 실습

requests 는 예약, limits 는 단속

Pod 가 Pending 에 멈추는 가장 흔한 이유, 그리고 OOMKilled 의 정체 — 자원 예산의 두 얼굴.

이번 레슨의 win: 스케줄 불가능한 Pod 를 일부러 만들어 Pending 의 사유를 Events 에서 읽어내고, memory limit 을 넘긴 컨테이너가 OOMKilled 되는 것을 관찰한다. 진단 때 "기억으로" 맞힌 Q7 이 증거로 바뀐다.

1. 두 숫자, 두 심판

의미심판어기면
requests "최소 이만큼은 확보해 달라" — 예약 스케줄러 (배치 시점, 딱 한 번) 맞는 노드가 없으면 Pending
limits "이 이상은 못 쓴다" — 상한 kubelet/커널 (실행 내내) 메모리 초과 → OOMKilled / CPU 초과 → throttling
아하 스케줄러는 노드의 실제 사용량이 아니라 requests 의 합계로 계산한다. 노드가 한가해도 예약이 꽉 찼으면 Pending 이고, 예약이 널널하면 실제로 바빠도 배치된다. 호텔 예약 장부와 같다 — 방이 비어 보여도 장부가 차 있으면 체크인 불가. 그리고 memory 와 CPU 는 초과 시 운명이 다르다: memory 는 뺏을 수 없어 죽이고(OOMKill), CPU 는 조절할 수 있어 느려지게만 한다(throttle).

2. 실습 — 두 심판을 각각 화나게 해 본다

kind 클러스터 mastery 이어서 사용. 이번엔 실습 전용 namespace 를 만들어 정리를 쉽게 한다 — 앞으로 모든 실습은 이렇게 격리하는 습관을 들인다.
terminal $ kubectl create namespace lab && kubectl config set-context --current --namespace=lab

추적 1 — 스케줄러를 화나게: 불가능한 예약. greedy.yaml:

greedy.yaml apiVersion: v1 kind: Pod metadata: name: greedy spec: containers: - name: app image: nginx:1.27 resources: requests: memory: "999Gi" # 이런 노드는 없다 cpu: "100"
terminal $ kubectl apply -f greedy.yaml && kubectl get pod greedy NAME READY STATUS RESTARTS greedy 0/1 Pending 0 $ kubectl describe pod greedy | tail -3 Warning FailedScheduling 0/1 nodes are available: 1 Insufficient cpu, 1 Insufficient memory. preemption: 0/1 nodes are available: 1 No preemption victims found.

진단 Q7 의 정답이 눈앞에 있다: Pending = 컨테이너가 시작도 안 됨 = 로그 없음. 사유는 언제나 describe 의 Events 에 스케줄러가 남긴다. requests 를 현실적으로 (예: memory 64Mi, cpu 100m) 고쳐 apply 하면 즉시 Running 이 되는 것도 확인하라.

추적 2 — kubelet 을 화나게: 상한 초과. hungry.yaml:

hungry.yaml apiVersion: v1 kind: Pod metadata: name: hungry spec: containers: - name: stress image: polinux/stress command: ["stress"] args: ["--vm", "1", "--vm-bytes", "250M", "--vm-hang", "1"] # 250Mi 를 쓰려는 앱 resources: requests: memory: "64Mi" limits: memory: "128Mi" # 상한은 128Mi
terminal $ kubectl apply -f hungry.yaml && kubectl get pod hungry -w hungry 0/1 OOMKilled 0 ← 128Mi 를 넘는 순간 커널이 죽였다 hungry 0/1 CrashLoopBackOff 1 ← 재시작해도 또 넘으니 반복... (Ctrl-C) $ kubectl describe pod hungry | grep -B1 -A2 'Last State' Last State: Terminated Reason: OOMKilled Exit Code: 137

실무에서 Exit Code: 137 을 보면 조건반사로 "memory limit"을 떠올리게 된다 (128 + 9, SIGKILL). limits.memory 를 300Mi 로 올려 apply 하면 안정적으로 Running 이 된다.

Pending vs OOMKilled vs CrashLoopBackOff — 셋은 층이 다르다: Pending 은 배치 실패(스케줄러, 노드에 못 감), OOMKilled 는 실행 중 처형(kubelet/커널), CrashLoopBackOff 는 "뭐가 됐든 반복해서 죽으니 점점 천천히 재시도 중"이라는 상태다. 원인이 아니라 증상이니, 원인은 항상 describe 와 Last State 에서 찾는다.

정리: kubectl delete ns lab && kubectl config set-context --current --namespace=default — namespace 를 지우면 안의 모든 것이 함께 사라진다. 이게 격리의 힘이다.

3. 회상 연습

Q1. 노드들의 CPU 실사용률이 10% 인데 새 Pod 가 Pending 이다. 가장 유력한 원인은?

스케줄러는 실사용량이 아니라 예약 장부(requests 합)로 판단한다. requests 를 과하게 잡은 Pod 들이 장부를 채우면, 한가한 클러스터에서도 Pending 이 난다 — 실무 단골.

Q2. memory 초과는 OOMKill, CPU 초과는 throttling. 이 비대칭의 이유는?

compressible(CPU) vs incompressible(memory) 자원. CPU 는 스케줄링 시간을 줄이면 앱이 느려질 뿐이지만, 쓰고 있는 메모리를 회수하면 프로세스가 성립 안 되니 죽이는 수밖에 없다.

Q3. (레슨 0004 복습) Pod 상태별 첫 번째 확인 위치 — 옳은 짝은?

간격 반복: 시작 전 문제(Pending)는 스케줄러의 기록(Events)에, 시작 후 문제(CrashLoop/OOM)는 프로세스의 기록(logs, Last State)에 있다. 층으로 기억하면 안 잊는다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

requests 랑 limits, 꼭 둘 다 넣어야 하나요?

필수는 아니다. 둘 다 생략하면 예약 0, 상한 없음(BestEffort) — 편하지만 노드가 쪼들리면 가장 먼저 쫓겨난다. 실무 기본기는 둘 다, 최소한 requests 만이라도 명시하는 것이다. "예약(requests) 없이 스케줄된 Pod 는 장부에 안 잡힌다"를 떠올리면 왜 위험한지 보인다.

cpu: 100m 의 m 은 뭐예요?

millicore, 즉 CPU 코어의 1/1000이다. 1000m = 1 코어, 100m = 코어의 10%. memory 의 Mi(=1024²바이트)와 헷갈리기 쉬운데, cpu 의 m은 소문자 밀리, memory 의 M/Mi는 용량 단위로 아예 다른 뜻이다.

노드가 한가한데 왜 Pending 인가요?

스케줄러는 실제 사용량이 아니라 requests 의 합계로 판단하기 때문이다. 이미 배치된 Pod 들의 requests 가 노드 용량을 채웠으면, 그 Pod 들이 실제로는 놀고 있어도 새 예약은 거절된다. 호텔 예약 장부와 같다 — 방이 비어 보여도 장부가 차 있으면 체크인 불가.

Pending 이면 로그를 어디서 봐야 하나요?

로그는 없다 — Pending 은 컨테이너가 시작조차 안 한 상태라 찍을 게 없다. 사유는 언제나 kubectl describe pod 의 Events 에 스케줄러가 남긴다(FailedScheduling). 간격 반복: 시작 전 문제는 Events, 시작 후 문제는 logs·Last State — 층으로 기억한다.

Exit Code 137 은 무슨 뜻인가요?

128 + 9, 즉 SIGKILL(9번 시그널)로 강제 종료됐다는 신호다. 컨테이너 세계에서 137 은 거의 항상 memory limit 초과 → OOMKilled를 의미한다. 실무에서 137 을 보면 조건반사로 "memory limit" 을 떠올리게 된다. 확정은 describe 의 Last State → Reason 이 OOMKilled 인지로 짓는다.

limits 만 넣고 requests 를 안 넣으면요?

쿠버네티스가 requests 를 limits 와 같은 값으로 자동 설정한다. 즉 limits 만 적어도 그 값만큼 예약이 잡히니, 의도치 않게 큰 예약이 되어 Pending 을 만들 수 있다. 반대로 requests 만 있고 limits 가 없으면 상한이 없다 — 두 숫자는 서로 다른 심판(스케줄러 vs kubelet)임을 기억하면 헷갈리지 않는다.

memory 는 죽이는데 CPU 는 왜 안 죽이나요?

자원의 성질이 다르다. CPU 는 압축 가능(compressible) — 시간을 덜 주면 앱이 느려질 뿐(throttling)이라 살려둘 수 있다. memory 는 압축 불가능(incompressible) — 이미 쓰고 있는 메모리를 뺏으면 프로세스가 성립 안 되니 죽이는 수밖에 없다(OOMKill). 이 비대칭이 회상 연습 Q2 의 핵심이다.

OOMKilled 랑 CrashLoopBackOff 는 뭐가 다른가요?

OOMKilled 는 왜 죽었나(원인, 이번 한 번), CrashLoopBackOff 는 계속 죽어서 재시도를 점점 늦추는 중(상태, 반복)이다. 즉 CrashLoopBackOff 는 증상이지 원인이 아니다. 진짜 원인은 describe 의 Last State 에서 찾는다 — OOMKilled 일 수도, 다른 에러일 수도 있다.

QoS 클래스가 뭐예요?

requests/limits 를 어떻게 넣었는지에 따라 쿠버네티스가 Pod 에 매기는 우선순위 등급이다. Guaranteed(둘이 같음) > Burstable(requests<limits) > BestEffort(둘 다 없음). 노드 메모리가 부족하면 BestEffort 부터 쫓겨난다. 오늘 원전 문서에서 바로 이어지는 개념이고, KCNA/CKA 단골이다.

실습마다 namespace lab 을 왜 따로 만드나요?

정리가 한 줄로 끝나기 때문이다 — kubectl delete ns lab 하나로 안의 Pod·설정이 통째로 사라진다. default 에 실습 잔해를 쌓으면 나중에 뭐가 내 것인지 헷갈린다. 이 격리 습관은 앞으로 모든 레슨에서 이어진다(0008 도 같은 lab 을 쓴다).
