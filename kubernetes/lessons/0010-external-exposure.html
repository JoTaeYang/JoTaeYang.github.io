외부 노출 3단: ClusterIP 밖으로 나가는 세 개의 문

0003 의 ClusterIP 는 클러스터 안에서만 통했다. 밖에서 접속하려면 세 단계로 문을 넓힌다 — NodePort, LoadBalancer, Ingress.

이번 레슨의 win: 같은 Deployment 에 Service type 을 ClusterIP → NodePort → LoadBalancer 로 바꿔가며 kubectl get svc 한 줄이 어떻게 달라지는지 본다. NodePort 의 3xxxx 포트가 열리고, LoadBalancer 의 EXTERNAL-IP 가 <pending> 에 멈추는 "그 멈춤"이 바로 오늘의 핵심 교훈이다.

1. 프레임 — 같은 Pod, 점점 넓어지는 문

네 type 은 계단이다. 위 칸은 아래 칸을 포함한다 — NodePort 는 ClusterIP 를 그대로 갖고 그 위에 포트를 열고, LoadBalancer 는 NodePort 를 갖고 그 위에 외부 LB 를 얹는다.

type도달 범위누가 만들어 주나계층
ClusterIP (기본) 클러스터 안에서만 쿠버네티스 (0003) L4 (TCP/UDP)
NodePort 모든 노드의 노드IP:3xxxx 쿠버네티스 (kube-proxy 가 포트 오픈) L4
LoadBalancer 하나의 외부 IP 클라우드 (AWS/GCP…의 LB 프로비저닝) L4
Ingress host/path 로 여러 Service를 한 입구에 Ingress 컨트롤러 (nginx 등, 별도 설치) L7 (HTTP)
아하 Service type 3 개는 전부 L4(포트 단위)다 — "이 포트로 온 걸 이 Pod 들에게". Ingress 만 L7(HTTP 단위)이라 URL 을 읽고 /api 는 이 Service, shop.example.com 은 저 Service 로 나눈다. 그래서 Ingress 는 Service 를 대체하는 게 아니라 여러 ClusterIP Service 앞에 서는 문지기다. LoadBalancer 를 Service 마다 하나씩 사면 비싸니까, Ingress 하나로 묶는다.

2. 실습 — type 을 바꾸며 get svc 를 읽는다

kind 클러스터 mastery, namespace lab 이어서. 없으면 kubectl create namespace lab && kubectl config set-context --current --namespace=lab.
terminal $ kubectl create deployment web --image=nginx:1.27 --replicas=2 $ kubectl expose deployment web --port=80 # 기본 type=ClusterIP $ kubectl get svc web NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) web ClusterIP 10.96.145.12 <none> 80/TCP ← 밖에서 갈 문이 없다

추적 1 — NodePort 로 승격. 기존 Service 의 type 만 바꾼다.

terminal $ kubectl patch svc web -p '{"spec":{"type":"NodePort"}}' $ kubectl get svc web NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) web NodePort 10.96.145.12 <none> 80:31842/TCP ← 80 은 그대로, 31842 가 새로

80:31842 를 읽어라: 안쪽 ClusterIP 의 80 은 그대로 살아 있고(포함!), 그 위에 모든 노드의 31842 포트가 새로 열렸다. 범위는 항상 30000–32767 이다. kind 는 노드가 도커 컨테이너라 노드IP:31842 를 호스트에서 바로 치기 번거로우니, 도달 자체는 port-forward 로 확인한다:

terminal $ kubectl port-forward svc/web 8080:80 $ curl -s localhost:8080 | grep -o '<title>.*</title>' <title>Welcome to nginx!</title>
실제 클라우드/온프렘 노드라면 curl http://<노드IP>:31842 가 바로 됐을 것이다. kind 에서 진짜 NodePort 를 호스트로 뚫으려면 클러스터 생성 시 extraPortMappings 가 필요하다 — 지금은 개념 확인이 목적이라 port-forward 로 충분하다.

추적 2 — LoadBalancer 로 승격, 그리고 멈춤을 관찰.

terminal $ kubectl patch svc web -p '{"spec":{"type":"LoadBalancer"}}' $ kubectl get svc web NAME TYPE CLUSTER-IP EXTERNAL-IP PORT(S) web LoadBalancer 10.96.145.12 <pending> 80:31842/TCP ← 영원히 pending
이 <pending> 이 오늘의 진짜 교훈이다. LoadBalancer 는 쿠버네티스가 아니라 클라우드가 실제 로드밸런서를 만들어 IP 를 채워 넣어야 완성된다. 순수 kind 에는 그 클라우드가 없으니 영원히 <pending> 이다 — 고장이 아니라 "나는 외부 인프라에 의존한다"는 선언이다. EKS/GKE 였다면 몇 초 뒤 진짜 IP 가 찍힌다. (kind 에서도 보고 싶으면 cloud-provider-kind 나 MetalLB 를 설치하면 된다 — 선택 사항.)

정리: kubectl delete ns lab. (port-forward 는 Ctrl-C 로 먼저 종료.)

3. Ingress — 문 하나로 여러 앱 (L7)

Service 셋은 "포트 하나 = 앱 하나"였다. 앱이 열 개면 LoadBalancer 열 개(=비용 열 배)가 된다. Ingress 는 HTTP 를 읽어 하나의 입구에서 URL 로 갈래를 튼다:

web-ingress.yaml (개념 예시) apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: shop spec: rules: - host: shop.example.com http: paths: - path: /api # shop.example.com/api → api-svc pathType: Prefix backend: { service: { name: api-svc, port: { number: 80 } } } - path: / # 나머지 전부 → web-svc pathType: Prefix backend: { service: { name: web-svc, port: { number: 80 } } }
Ingress 는 규칙(YAML)일 뿐, 그 규칙을 실행할 Ingress 컨트롤러(ingress-nginx 등)가 클러스터에 설치돼 있어야 실제로 동작한다 — Role 만 있고 실행 주체가 없으면 무력했던 RBAC(0008)과 같은 구조다. kind 에서 손으로 돌려보고 싶으면 아래 Source 의 "Ingress in kind" 가이드를 따르면 되지만, KCNA 는 개념(L7 라우팅, 컨트롤러 필요)까지면 충분하다 — 실습은 CKAD 에서.

4. 회상 연습

Q1. Service type 을 ClusterIP 에서 NodePort 로 바꿨다. 안쪽 ClusterIP 는 어떻게 되나?

type 은 계단이라 위 칸이 아래 칸을 포함한다. get svc 의 80:31842 가 증거 — 80(ClusterIP)은 그대로고 31842(노드 포트)가 추가됐다. 클러스터 내부 접근은 여전히 ClusterIP 로 된다.

Q2. 순수 kind 클러스터에서 LoadBalancer Service 의 EXTERNAL-IP 가 계속 <pending> 이다. 원인은?

LoadBalancer 는 쿠버네티스 자체 기능이 아니라 "클라우드야, LB 하나 만들고 IP 알려줘"라는 요청이다. 그 클라우드가 없으면 요청은 영원히 대기한다 — 고장이 아니라 환경의 한계다. EKS/GKE 나 MetalLB 가 그 자리를 채운다.

Q3. (레슨 0003 복습) 한 도메인 아래 /api 와 / 를 서로 다른 Service 로 보내려 한다. 무엇을 쓰나?

path/host 로 가르는 건 URL(HTTP)을 읽어야 하는 L7 일 = Ingress 의 몫이다. Service(L4)는 포트만 보므로 /api 와 / 를 구분할 수 없다. Ingress 뒤에는 여전히 0003 의 평범한 ClusterIP Service 들이 선다.
판정: Q1·Q2 를 맞히면 외부 노출 3단 ✅ — 다음은 스토리지(PV / PVC / StorageClass). 0009 에서 예고한 StatefulSet 의 volumeClaimTemplates 가 여기서 이어진다. 이걸로 KCNA Fundamentals(비중 46%)의 ⬜ 칸이 거의 다 채워진다.

초보자 질문 (막히면 여기부터)

실습하다 떠오르는 "어? 이건 왜 이러지?"를 미리 모았다. 궁금한 것만 펼쳐 보면 된다 — 안 궁금하면 접힌 채 넘어가도 된다.

NodePort 포트는 왜 하필 30000–32767 이에요? 그냥 80 쓰면 안 되나요?

낮은 포트(80, 443 등)는 노드의 시스템 서비스와 충돌할 수 있어, 겹치지 않는 높은 대역을 따로 떼어 뒀다. 특정 번호를 원하면 spec.ports[].nodePort: 30080 처럼 그 범위 안에서 지정할 수 있고, 안 정하면 자동 배정된다.

NodePort 면 그 포트가 딱 그 Pod 가 있는 노드에서만 열리나요?

아니다 — 모든 노드에서 열린다. 3번 노드로 요청이 와도 kube-proxy 가 실제 Pod 가 있는 노드로 내부 전달한다. 그래서 아무 노드 IP 로나 접속해도 된다. "어느 노드에 Pod 가 떴는지" 신경 안 써도 되는 게 장점이다.

port-forward 랑 NodePort 는 같은 건가요?

다르다. port-forward 는 내 kubectl 이 API server 를 거쳐 만드는 임시 개인 터널이다 — 명령을 끄면 사라지고, 나만 쓴다. NodePort 는 클러스터에 상시로 열린 진짜 문이다. 오늘은 kind 사정상 도달 확인만 port-forward 로 편하게 한 것뿐이다.

LoadBalancer 가 <pending> 이면 실습이 실패한 건가요?

아니다, 그게 정상이고 오늘의 교훈이다. LoadBalancer 는 "클라우드야 LB 만들어줘"라는 요청이라, 클라우드가 없는 kind 에선 응답할 주체가 없어 대기 상태로 남는다. EKS/GKE 라면 IP 가 찍힌다. 굳이 kind 에서 보려면 cloud-provider-kind 나 MetalLB 를 설치하면 된다.

Ingress 를 만들었는데 아무 일도 안 일어나요.

Ingress 리소스는 규칙일 뿐, 그걸 실행할 Ingress 컨트롤러(ingress-nginx 등)를 클러스터에 설치해야 동작한다. Role 만 있고 RoleBinding 이 없으면 무력했던 0008 과 똑같은 구조다 — 규칙과 실행 주체는 별개다.

그럼 Ingress 자체는 어떻게 밖에서 접속돼요?

Ingress 컨트롤러 자신이 보통 LoadBalancer(또는 NodePort) Service 하나로 노출된다. 즉 "외부 IP 하나 → Ingress 컨트롤러 → path/host 로 갈라 여러 ClusterIP Service". LB 를 앱마다 사는 대신 컨트롤러 앞에 하나만 두는 게 비용 절감의 핵심이다.

path 라우팅이 되면 Ingress 하나로 다 되는데 왜 Service 가 여전히 필요해요?

Ingress 는 "어느 Service 로 보낼지"만 정하고, 실제로 살아있는 Pod 목록을 추적해 부하를 나누는 건 여전히 Service+EndpointSlice(0003)의 몫이다. Ingress 뒤에는 항상 평범한 ClusterIP Service 들이 선다 — 대체가 아니라 층이 하나 더 얹힌 것이다.

LoadBalancer 랑 Ingress, 둘 다 외부 노출인데 뭘 써야 해요?

TCP/포트 단위면 LoadBalancer(L4), HTTP/URL 단위 라우팅이 필요하면 Ingress(L7). 웹앱 여러 개를 도메인·경로로 나눠 한 입구에 모으는 대부분의 경우가 Ingress 다. gRPC 아닌 임의 TCP(예: DB 포트 노출)는 LoadBalancer 가 맞다.

ClusterIP 없이 바로 외부로 열 수는 없나요?

없다 — NodePort·LoadBalancer 는 내부에 ClusterIP 를 반드시 갖고 그 위에 층을 얹는 구조다(get svc 의 80:31842 가 그 증거). 안쪽 ClusterIP 가 Pod 들에게 부하를 나누는 실체이고, 바깥 문들은 그리로 가는 통로일 뿐이다.

kubectl expose 랑 kubectl patch, YAML 로도 되죠?

물론이다. 오늘은 흐름을 빨리 보려 명령형(imperative)을 썼지만, 실무·GitOps 에선 Service YAML 에 type: NodePort 를 적어 apply 한다. CKAD 실기에선 expose 같은 명령형이 시간 절약이라, 둘 다 손에 익혀 두면 좋다.
